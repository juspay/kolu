PR plan: srid/agency — calibrate /lowy on extraction shape

Two calibrations, both surfaced by 20-cycle /ralph loop on juspay/kolu#998: (1) single in-tree consumer is not disqualifying; (2) the extracted package must be a coherent library, not a grab-bag of helpers bundled by topic. Bug 2 demonstrated and fixed in 4af1c647 (@kolu/solid-xterm@0.2.0) — case study below.

PR title + summary (copy into the srid/agency PR description)

Title: lowy: single-consumer is not disqualifying; add §6.5 package coherence (+ hickey cross-ref)

Summary:

The /lowy default reasoning treats "1 in-tree consumer" as a fail-on-reuse signal — contradicting both the skill's own §5 hedge (the reuse signal is about interface stability, not import count) and the lived evidence of @kolu/surface, a published framework with a single in-tree consumer. A 20-cycle /ralph loop on kolu#998 exposed this in cycle 2 ("fails Lowy's reuse test" → don't extract @kolu/solid-xterm) and reversed it in cycle 6 only after the prompt was explicitly primed with the Surface precedent. Bug 1 calibrates §6 + the fact-check list to admit single-consumer extractions when the interface is stable under the encapsulated axis.
Bug 2: once the extraction is admitted, the skill never asks whether the resulting package is a coherent library. @kolu/solid-xterm@0.1 shipped three loose public exports — createXtermWebgl, attachXtermStyleSync, createScrollLock — that the consumer had to integrate by hand alongside new XTerm(...) and eight addon constructors. The package's name promised "SolidJS adapter for xterm"; the surface delivered "three xterm-adjacent helpers." New §6.5 ("Package coherence") names this failure mode, the Surface-shape test, the consumer-wires-it-together smell, and the corrective action. A parallel paragraph in hickey/SKILL.md Layer 3 names the package-surface flavour of Layer 2 fragmentation.
Worked example: @kolu/solid-xterm@0.2.0 (kolu commit 4af1c647) reshaped the package behind one createSolidXterm primitive, demoting the three former exports to ./internal/. The consumer (Terminal.tsx) lost 95 LOC; just smoke still passes. That commit is the demonstration of what a §6.5-calibrated reviewer should produce.

Diagnosis A — single-consumer false-negative

Bug 1 /lowy's default reasoning treats "1 in-tree consumer" as a fail-on-reuse signal, contradicting both (a) the skill's own §5 hedge that the reuse signal is about interface stability not import count and (b) the in-the-wild precedent set by @kolu/surface — a published framework with one in-tree consumer.

The contradiction lives in the same file. lowy/SKILL.md:102 warns "If a module can only be used in one context, it may be encapsulating functionality rather than volatility" as a soft-hedged signal. lowy/SKILL.md:124 hardens it in the fact-check phrase-shape list: "single-use is the reuse signal firing. Investigate." A reviewer running the fact-check pass without §5 in hand treats single-use as disqualifying.

Diagnosis B — package-coherence false-positive

Bug 2 Once /lowy admits the extraction, it does not check whether the resulting package is a coherent library. Three exports that share a topic (e.g. all xterm-adjacent) but no shared concept get bundled into one package — partial wiring instead of a socket.

Concrete failure (since fixed in kolu#998 commit 4af1c647): @kolu/solid-xterm@0.1. The package shipped three loose helpers — createXtermWebgl, attachXtermStyleSync, createScrollLock — that the consumer (Terminal.tsx) had to import separately and wire together by hand with new XTerm(...), addon constructors, term.open, the WebGL load policy, and scroll-lock attach. The package's name suggested "SolidJS adapter for xterm.js"; the surface delivered "three xterm-adjacent SolidJS helpers." Those are different things.

"Electricity is electricity and is not partial wiring." The exports list of a real solid-xterm resembles Surface's exports list — one entry point per coherent concept, not one per internal aspect.

Demonstration — @kolu/solid-xterm@0.2.0

The Kolu commit 4af1c647 ships the re-shape. One primitive (createSolidXterm) replaces the three loose helpers; the helpers move to ./internal/ and are no longer re-exported.

How /lowy §5 already says this — just not loudly enough

Atomic business verbs, not implementation operations. Lowy's key interface design principle: interfaces should expose indivisible business-level operations (credit, debit, transfer) rather than CRUD or implementation details. […] An interface that exposes OpenPort(), ClosePort(), AdjustBeam() alongside ReadCode() is mixing communication volatility with reading volatility — two axes jammed behind one interface. — lowy/SKILL.md:96

Three loose exports in one package is the inverse: two axes plus a third, separated behind a name (@kolu/solid-xterm) that suggests one. The reviewer reads each export's interface and finds each one stable under its own axis — and the package-level question of "do these three belong in one library?" never gets asked because §5 is framed per-interface, not per-package.

The published precedent: `@kolu/surface` (and its peers

The interface-stability check admits them — correctly.

The shape that disqualifies is *"the interface mirrors the

2. New §6.5 "Package coherence" — the electricity-is-not-partial-wiring calibration

Insert as a new H3 between §6 (Reuse Signal, ends line ~104) and §7 (The Almost-Expendable Test, line ~106). The cycle-3 extraction shipped this bug because no layer of /lowy asked the question; kolu#998 commit 4af1c647 is the post-feedback fix that demonstrates what the calibrated reviewer should produce.

### 6.5 Package Coherence

When the extraction crosses a *package* boundary (not just a module
within the same package), the reviewer's job is not done after
naming one volatility axis. The package as a whole must read as a
**coherent library** — one concept, one socket. If the package
ships three exports for three internal aspects of what should be
one primitive, you have shipped *partial wiring*, not a receptacle.

Run this check whenever the extraction adds a new published-shape
package (`@org/foo`):

1. **Read the package's exports list as if you were a new consumer.**
   Does it suggest one coherent thing or a topic-bundle?
   - `@kolu/surface` exports `defineSurface` → one entry, one
     concept (typed reactive layer). Coherent.
   - `@kolu/solid-xterm@0.1` (kolu#998 cycle 3–5) exported
     `createXtermWebgl`, `attachXtermStyleSync`, `createScrollLock`
     → three entries, three internal aspects of "xterm lifecycle"
     leaked through three exports. Not a coherent SolidJS adapter
     for xterm; a topic-bundle of three xterm-adjacent helpers.
     The fix shipped in `@kolu/solid-xterm@0.2.0` (commit
     `4af1c647`) is one `createSolidXterm({ container, theme,
     fontSize, addons, webgl, scrollLock, ... })` primitive that
     hides WebGL / style / scroll as internal submodules.

2. **Apply §5's atomic-verb rule at the package level.** §5 already
   warns that an interface exposing `OpenPort` / `ClosePort` /
   `AdjustBeam` alongside `ReadCode` mixes axes. A package
   exporting `createX_webgl` / `attachX_style` / `createX_scroll`
   does the same thing one altitude up: the package's surface is
   three operations on three axes, not one atomic abstraction.

3. **The Surface test.** If the package's exports list does not
   resemble Surface's shape — *one entry point per coherent
   concept, with internal submodules hidden* — the package is
   shaped around the implementation, not around a stable contract.
   Even if each individual export passes §5 in isolation, the
   *package* fails the test.

4. **The "consumer wires it together" smell.** If the only
   in-tree consumer imports several of the package's exports and
   then composes them by hand — the way `Terminal.tsx` had to wire
   `createXtermWebgl` + `attachXtermStyleSync` + `createScrollLock`
   + a bare `XTerm` constructor + 8 addon imports in v0.1 — the
   missing primitive is the composition. The package is shipping
   submodules and asking the consumer to be the integrator. Wrap
   them.

**Action when this fires.** Re-extract behind a single primitive
that owns the integrated lifecycle; demote the current exports to
internal submodules of that primitive. The Lowy verdict is not
"don't extract" — it's "extract one socket, not three wires."

3. Two phrase-shapes for the fact-check list (lines 120–130)

Append to the bulleted phrase-shape list under "Flag these phrase shapes" in the §"Fact-Check Your Own Evaluation" block:

- *"Fails Lowy's reuse test"* (when based on import count alone) —
  reuse-count is a symptom, not a diagnosis. The diagnosis is §5's
  interface-stability check. An interface can have one importer
  today and a perfectly stable contract; ten importers and still be
  shaped around its implementation. Cite the axis, not the count.

- *"Each export passes §5 in isolation"* (without checking the
  package surface) — §5 fires per interface; §6.5 fires per package.
  Three coherent helpers in one package can collectively fail the
  package-coherence check if their union suggests one thing the
  package doesn't actually deliver. Read the exports list as a
  consumer would and ask "what library is this?" — if the answer
  is a topic-bundle ("xterm-adjacent helpers") rather than a
  primitive ("SolidJS adapter for xterm"), §6.5 applies.

Proposed change to .apm/skills/hickey/SKILL.md

One hunk. Insert as a new bullet (call it step 4) at the end of Layer 3's numbered list (after step 3 at hickey/SKILL.md:67, before the "Two abstractions serving one user-level concern" paragraph at line ~71). The Hickey-side of bug 2: three exports in one package, where the consumer reconstitutes the unity at integration time, is the package-surface flavour of Layer 2 Fragmentation.

**Package surface as a fragmentation site.** When evaluating a
new published-shape package (`@org/foo`), read its exports list
the way Layer 2 reads a per-entity structure — *does the consumer
have to reconstitute one concept by wiring multiple exports
together?* If yes, the package has fragmented one primitive into
N exports and shipped the integration cost to the consumer. The
fix is the same as any Layer-2 fix: collapse to one primitive at
the natural layer (one entry point, internal submodules hidden).
See `/lowy` §6.5 for the volatility-side argument.

Calibration tension

Bug 1 says "extract under single-consumer is fine"; bug 2 says "but check the shape of the extracted package." Together they're not in tension — they compose into a single rule:

Extract when (a) a stable volatility axis exists behind a stable interface, AND (b) the resulting package reads as one coherent library, not a topic-bundle. The first calibration unblocks extractions the default heuristic was killing; the second prevents the unblocked extractions from shipping as partial wiring.

Why the original ralph loop missed this

  1. The loop's metric was "graduate framework packages." Once a package was extracted, the cycle moved on; no cycle audited the package's surface coherence separately from the per-export volatility check. /lowy on each export passed because each export does encapsulate an axis. The package-as-library question wasn't a cycle.
  2. /lowy didn't ask it either. The skill's per-interface focus (§5 atomic verbs, §6 reuse signal) is correct at the function/module level. The package-as-library altitude was a blind spot. This PR closes it.
  3. User-as-reviewer caught what the loop's automated reviewers didn't. The fix (commit 4af1c647) is the post-feedback re-shape, which is itself the demonstration of what a §6.5-calibrated /lowy pass should have produced on its own.

Remote terminals over SSH — design plan

Starting point: master @ 4b55c58a (after R1 #944 + R2 #945 merged — kolu-io with refcounted dir-watcher exists; kolu-pty with PTY primitives exists). Goal: a new terminal in Kolu can run on any host in ~/.ssh/config, with branch chip / agent badge / Code-tab parity to local terminals. Zed-inspired architecture: the PTY runs locally with ssh inside; the remote machine hosts a node daemon that runs Kolu's existing integration packages unchanged and streams their events back.

TL;DR. The remote machine runs a node daemon (kolu-remote-server) that imports kolu-git, kolu-claude-code, kolu-codex, kolu-opencode unmodified and exposes their subscription APIs over an NDJSON+Zod RPC channel. The controller speaks to that daemon for metadata; the user's terminal is a local PTY whose program is ssh -t <host> '...'. There is no executor abstraction. There is no async cascade through the integration packages. The integration packages are untouched. The complexity lives in two new packages (kolu-host, kolu-remote) and a small dispatch change in kolu-server's metadata orchestrator.

What we're building

  • "New terminal" palette can target a host from ~/.ssh/config. The PTY runs locally; ssh runs inside it and drops the user into a shell on the remote.
  • Branch chip, worktree create/remove, Code-tab file read, agent badges (Claude Code / Codex / OpenCode) all reflect the remote machine's state.
  • Terminal metadata persists hostId; session restore re-spawns on the original host. Sub-terminals inherit the parent's host.
  • Cross-arch deployment via Nix: nix copy the controller's exact kolu-remote-server closure or .drv to the remote.

Architecture

Two SSH channels per remote terminal, multiplexed over one TCP connection via OpenSSH's ControlMaster=auto + ControlPersist=10m:

┌── controller (laptop) ────────────────────┐ ┌── remote host ─────────────────────────────┐ │ │ │ │ │ ┌─ LOCAL PTY (kolu-pty) ──────┐ │ │ │ │ │ program: ssh │ ── ch A ───→│ bash --rcfile /tmp/kolu-rc-${id} │ │ │ args: -t host '...' │←─────────────│ (OSC 7/2/633;E hooks → bytes back via ssh)│ │ │ xterm.js renders here │ │ │ │ │ └─────────────────────────────┘ │ │ │ │ │ │ ┌─ kolu-remote-server (node daemon) ──┐ │ │ ┌─ kolu-remote RPC client ────┐ │ │ │ │ │ │ │ Host.subscribe(...) │ ── ch B ───→│ │ imports kolu-git UNCHANGED │ │ │ │ Host.invoke(...) │←─────────────│ │ imports kolu-claude-code UNCHANGED │ │ │ │ NDJSON envelope frames │ │ │ │ imports kolu-codex UNCHANGED │ │ │ │ request / response / event │ │ │ │ imports kolu-opencode UNCHANGED │ │ │ │ / endStream / ping │ │ │ │ │ │ │ └─────────────────────────────┘ │ │ │ on subscribe("git", { cwd }): │ │ │ │ │ │ subscribeGitInfo(cwd, emit, log) │ │ │ meta/{git,agent}.ts dispatches: │ │ │ │ │ │ - local terminal: in-process subscribe │ │ │ on invoke("worktree.create", …): │ │ │ - remote terminal: host.subscribe(...) │ │ │ worktreeCreate(…) → response │ │ │ │ │ └─────────────────────────────────────┘ │ └───────────────────────────────────────────┘ └────────────────────────────────────────────┘

The integration packages don't know there's a remote. They run in-process on the controller for local terminals; they run in-process on the remote (inside kolu-remote-server) for remote terminals. Same source, same behaviour, same callbacks. The dispatch — "is this a local or remote terminal?" — happens once in meta/git.ts and meta/agent.ts, not in the providers themselves.

The Host abstraction

Generic interface in kolu-io (domain-free). Concrete discriminated unions in kolu-host. This split keeps kolu-io a library-shaped leaf — it has zero knowledge of git, agents, or any other domain. Adding a new provider = adding a variant in kolu-host's unions, never editing kolu-io.

// packages/integrations/io/src/host.ts — generic, domain-free import type { PtyHandle, SpawnPtyOpts } from "kolu-pty"; export interface HostLogger { debug, info, warn, error: ... } export interface Subscription { stop(): void } /** Host — generic transport-identity abstraction. * Type parameters define the closed set of subscriptions and invokes * a concrete Host supports. kolu-host declares the concrete bindings * (KoluHost = Host<SubscriptionRequest, SubscriptionEventMap, InvokeRequest, * InvokeResultMap>); consumers work in terms of KoluHost and get full * inference on host.subscribe({ kind: "git", cwd }) etc. */ export interface Host< SubReq extends { kind: string }, SubEventMap extends Record<SubReq["kind"], unknown>, InvReq extends { kind: string }, InvResultMap extends Record<InvReq["kind"], unknown>, > { readonly id: string; readonly label: string; readonly kind: "local" | "remote-ssh"; spawnPty(log: HostLogger, opts: SpawnPtyOpts): Promise<PtyHandle>; subscribe<R extends SubReq>( req: R, onEvent: (event: SubEventMap[R["kind"]]) => void, log: HostLogger, ): Subscription; invoke<R extends InvReq>(req: R): Promise<InvResultMap[R["kind"]]>; shutdown(): Promise<void>; } // packages/integrations/host/src/requests.ts — concrete unions live here export type SubscriptionRequest = | { kind: "git"; cwd: string } | { kind: "agent"; provider: "claude-code" | "codex" | "opencode"; terminalId: string; state: AgentTerminalState } | { kind: "foreground"; terminalId: string; sshPid: number } // LocalHost ignores; RemoteHost /proc-walks via this export type SubscriptionEventMap = { git: GitInfo | null; agent: ClaudeCodeInfo | CodexInfo | OpenCodeInfo | null; foreground: { pid: number; name: string; pgid: number }; }; export type InvokeRequest = | { kind: "git.worktreeCreate"; repo: string; name: string } | { kind: "git.worktreeRemove"; path: string } | { kind: "fs.readFile"; path: string; maxBytes?: number } | { kind: "fs.listDir"; path: string }; export type InvokeResultMap = { ... }; export type KoluHost = Host< SubscriptionRequest, SubscriptionEventMap, InvokeRequest, InvokeResultMap >;

Two impls in kolu-host:

  • LocalHost (R3): spawnPty calls kolu-pty's spawnPty directly. subscribe uses a switch (req.kind) with assertNever exhaustiveness (compile error if a variant is added without a dispatch arm), calling the relevant provider's in-process subscribe function. invoke same pattern for one-shots.
  • RemoteHost (F1): spawnPty calls kolu-pty's spawnPty with program: "ssh" and args: [...sshOpts, host, "-t", remoteCmd]. subscribe sends a streaming RPC to kolu-remote-server; events flow back as wire envelopes. invoke sends request/response.

The providers never see Host. They keep their existing signatures (subscribeGitInfo(cwd, onChange, log), etc.). LocalHost calls them directly; kolu-remote-server calls them on the remote.

Design review The Host shape went through a hickey + lowy review pass that caught a structural smell in the first draft: discriminated unions placed in kolu-io, making it depend on every domain package. The fix here — generic Host in kolu-io + concrete unions in kolu-host — keeps kolu-io domain-free. Other findings folded in: collapsed three agent.* variants into one with provider discriminator (saves three-site → two-site update rule); added sshPid to the foreground variant (without it RemoteHost can't resolve the remote shell PID — was a latent F1 runtime defect); activation/dedup registry moves to LocalHost (volatility-correct — separates dedup policy from reconcile orchestration).

Wire protocol

NDJSON + Zod with a Zed-shaped envelope discriminator. Every line on the SSH stdio channel is a JSON object matching one variant:

// packages/integrations/remote/src/protocol.ts const FrameSchema = z.discriminatedUnion("kind", [ z.object({ kind: z.literal("request"), id: z.number(), method: z.string(), params: z.unknown() }), z.object({ kind: z.literal("response"), respondingTo: z.number(), result: z.unknown() }), z.object({ kind: z.literal("error"), respondingTo: z.number(), error: ErrorShape }), z.object({ kind: z.literal("event"), respondingTo: z.number(), payload: z.unknown() }), z.object({ kind: z.literal("endStream"),respondingTo: z.number() }), z.object({ kind: z.literal("ping") }), ]); // Type aliases to disambiguate the three concerns the union braids: // - one-shot RPC: request, response, error // - streaming: request, event…, endStream // - keepalive: ping // No wire change — consumers narrow via these aliases for clarity. export type OneShotFrame = Extract<Frame, { kind: "request" | "response" | "error" }>; export type StreamFrame = Extract<Frame, { kind: "request" | "event" | "endStream" }>; export type TransportFrame = Extract<Frame, { kind: "ping" }>;

RPC patterns:

  • One-shot request (worktreeCreate, readFile): client sends { kind: "request", id, method: "invoke.git.worktreeCreate", params }; server responds with one { kind: "response", respondingTo: id, result } or one { kind: "error", respondingTo: id, error }.
  • Subscription (git info, agent state, foreground): client sends { kind: "request", id, method: "subscribe.git", params }; server streams zero-or-more { kind: "event", respondingTo: id, payload }; client closes by sending { kind: "endStream", respondingTo: id } upstream (or server emits endStream on its own termination).
  • Keepalive: either side may send { kind: "ping" } at any time; receiver echoes. Used to keep ControlPersist alive across idle periods.

Why NDJSON+Zod over protobuf

OptionProsConsPick
Protobuf (Zed's choice)Schema-versioned, binary-efficient, mature tooling, the EndStream pattern is cleanly typed via the oneofCodegen step in the build; TypeScript ecosystem doesn't lean here; binary debuggingNo
NDJSON + Zod with envelope discriminatorTS-native; Zod schemas double as runtime validators; debuggable via cat; no codegen; the discriminator gives us protobuf-style oneof typingHand-roll subscription/endStream semanticsYes
MessagePack + Zod50-70% the wire size of JSON; TS support is fineWire size rarely the bottleneck for our message volume; binary debuggingNo
JSON-RPC 2.0Standard, has notifications (one-way)"Notifications" aren't subscriptions; still have to layer subscription lifecycle on topNo

The envelope discriminator pattern is structurally Zed's Envelope.payload oneof — just human-readable on the wire and validated by Zod instead of generated by prost. See the zed skill for the full comparison.

How each feature works under this architecture

For every Kolu feature touching the remote, the table below shows what happens when the user spawns a remote terminal and the feature lights up.

Why this partition

The R-vs-F split has the same payoff here as it did for R1+R2:

  • Each refactor improves master regardless of whether the feature ships. R3's Host indirection is a clean abstraction for "where does this PTY run?" even if every host is local. R4's via-Host metadata dispatch is a small move toward a more uniform orchestrator, useful even in the local-only world.
  • The feature PR (F1) is smaller because R3+R4 already exist. F1 just adds the SSH transport + remote daemon + a second Host impl that satisfies the interface R3 set up. No "rewire the entire orchestrator while also introducing remote" mega-PR.
  • Reviewer cost is partitioned correctly. R3+R4 are "did anything break?" reviews; F1 is the "is the new functionality correct?" review.

What's NOT in the phasing

No "thread executor through kolu-git" phase. No "thread executor through agent providers" phase. No async cascade in the integration packages. Those phases existed in earlier drafts of this plan when the architecture was "abstract the IO substrate"; under the run-providers-on-remote model they are unnecessary.

Why we abandoned the executor approach Earlier drafts had F1-F3 threading an Executor parameter through every IO-bearing function in kolu-git and the agent providers, with the remote daemon serving as a thin RPC shim for that IO. Two implementing PRs (#948, #949) showed the real cost: ~2,500 lines of churn across the integration packages, plus async-cascade race conditions that forced state-machine rewrites in the watcher code. The "just thread executor" framing turned out to require structural rewrites in code that should be untouched. The run-providers-on-remote model — Zed's approach — moves the substrate seam outside the integration packages: the daemon runs them, the controller talks to the daemon. Integration code is unchanged. See .apm/skills/zed/SKILL.md for the architectural reasoning.

The kolu-remote-server daemon

A node process. Single binary, deployed via Nix to the remote host. Its responsibilities, in full:

  1. Accept NDJSON envelope frames on stdin, write them on stdout. No socket, no port, no auth beyond SSH's own. The daemon runs as the user's UID on the remote; everything it can do, the user can do.
  2. Handle invoke requests by calling the corresponding integration function: git.worktreeCreate → worktreeCreate(repo, name, log) from kolu-git; fs.readFile → node's fs.readFile; etc.
  3. Handle subscribe requests by calling the corresponding integration subscribe function and wiring its callback to envelope-emit: { kind: "git", cwd } → subscribeGitInfo(cwd, (info) => emit({ kind: "event", respondingTo: subId, payload: info }), log).
  4. Dedup subscriptions across multiple subscribers on the same resource (same cwd → one underlying watcher, fan out events to N RPC streams).
  5. Track foreground process for remote terminals: maintains a map of { controllerSshPid → remoteShellPid } built by walking /proc when a foreground subscription is established. Polls tcgetpgrp on the resolved shell PID; emits foregroundChange events on flip.
  6. Receive OSC rcfile content from the controller (one-shot invoke("fs.writeTemp", { content }) at terminal-spawn time), write to /tmp/kolu-rc-${id}, return the path. The controller's ssh -t host '... exec bash --rcfile /tmp/kolu-rc-${id}' then sources it.
  7. Idle ping: every 30 seconds, send { kind: "ping" } to keep ControlMaster's TCP idle timeout from firing.
  8. On stdin EOF or stdout broken-pipe, exit. No daemon-mode persistence across SSH drops — a new connection spawns a fresh daemon. (Subscriptions re-establish on reconnect; see P4.)

Estimated daemon size: ~400 lines of node code, plus the deps it imports (kolu-git, kolu-claude-code, kolu-codex, kolu-opencode, kolu-io, kolu-pty's PtyHandle types — though the daemon doesn't spawn PTYs). Closure size is dominated by node-modules of the integration packages.

Cross-arch deployment via Nix

The remote may be x86_64 Linux while the controller is aarch64 macOS (or vice versa). Zed solves this by downloading prebuilt binaries per-arch from zed.dev; Kolu solves it via Nix:

  1. Same-arch path (controller and remote on same OS+arch): nix copy --to ssh-ng://<alias> <closure-path> ships the realised closure. Subsequent connects find it in the remote's Nix store; no rebuild.
  2. Cross-arch path: nix copy --derivation --to ssh-ng://<alias> <closure-path>.drv ships only the derivation (the build recipe). The remote then realises (nix-store --realise <drv> or equivalent) — building kolu-remote-server from source there. Substituters (e.g. cache.nixos.asia/oss) cache subsequent builds.

No env-override escape hatch. No prebuilt-binary distribution channel. Nix is the deployment contract.

Trade-offs honestly

  • Integration packages are untouched. Zero diff in kolu-git, kolu-claude-code, kolu-codex, kolu-opencode, anyagent. The whole point.
  • No executor abstraction. No async cascade, no race condition rewrites in watchers, no concept multiplication between Executor and Host.
  • One code path runs on both sides. Same provider source executes locally for local terminals and remotely (inside kolu-remote-server) for remote terminals. No risk of behavioral drift.
  • Smaller total diff. ~+2,000 lines net across all phases vs the prior plan's ~+6,500. The ~+4,500 saved is the work we don't do on the integration packages.
  • SSH drops kill the PTY. The local PTY's ssh process dies; user spawns a new terminal to reconnect. Scrollback is gone. Accepted as the cost of dropping the helper-side PTY engine + ring buffer + replay (~1,200 lines we don't write).
  • Remote needs node + the integration packages' native deps (@parcel/watcher, node:sqlite). Same constraint as PR #929 had. Nix deployment handles it.
  • Foreground-PID tracking on the remote needs /proc traversal. Linux: straightforward. macOS-as-remote: needs proc_listpids via native binding. Defer until someone wants macOS-as-remote.
  • Subscription protocol is more complex than plain request/response. NDJSON envelope handles it; well-trodden pattern.
  • Per-host dedup logic lives in kolu-remote-server, not kolu-server. The activation registry idea moves to the daemon. Cleaner because there's only one "executor" per daemon instance.

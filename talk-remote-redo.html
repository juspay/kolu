Current master HEAD: 4b55c58a (post-R1 #944, post-R2 #945). Goal: a new terminal in Kolu can run its PTY on any host in your ~/.ssh/config, with branch chip / agent badge / Code-tab parity to local terminals; SSH drops survive via a daemon on the remote side. Plan is captured end-to-end — no deferred follow-ups. Reviewed through Hickey + Lowy; numbers verified against current master. PR #929 (still in draft) is the prior implementation cited for structural observations — those citations are point-in-time and may drift if PR #929 is revised.

What we're building (the user-visible feature)

  • "New terminal" palette can target a remote host from ~/.ssh/config. The PTY runs there; bytes flow over SSH.
  • Branch chip, worktree create/remove, Code-tab file read all reflect the remote machine's state.
  • Agent badges (Claude Code, Codex, OpenCode) detect and live-update against the agent process running remotely.
  • Helper survives SSH drops: PTYs and their scrollback live in a daemon; reconnect replays missed output.
  • Terminal metadata persists hostId; session restore re-spawns on the original host. Sub-terminals inherit the parent's host.

The activation registry (new mechanism in F6b)

When five terminals on the same SSH host all run codex, we want one WAL watcher per host, not five. The orchestrator keeps a Map<string, Activation> keyed by (provider.kind, executorId) where executorId uses WeakMap<Executor, string> identity. The "local" singleton gets id "local"; each remote Host instance gets "exec#N". First terminal installs; subsequent terminals join the fan-out set.

~120 of F6b's ~150 lines are the registry itself.

Cross-host gotchas

  • $HOME per executor. Paths like ~/.claude/sessions/ are host-relative. Resolve via executor.exec("printenv", ["HOME"]) and memoize per-executor.
  • Tilde expansion. executor.readFile("~/.foo") does NOT expand ~. Resolve home first.
  • SQLite WAL across wire. Helper opens node:sqlite with readOnly: true so concurrent OpenCode/Codex writes don't block.
  • Watch latency. Local refcounted parcel-watcher: <50 ms. SSH-relayed fs.watch: 200-500 ms.
  • Test-time env overrides. KOLU_CLAUDE_SESSIONS_DIR only affects controller's local path; remote tests use mock-executor.

Local path — symmetric with remote, not asymmetric

After R1 + F4, LocalHost.watch consumes integrations/io/refcounted-dir-watcher directly — the same generic primitive kolu-git's existing watcher stack uses. Routing local terminals through LocalHost doesn't pessimise relative to the existing master code path; they share the same underlying refcounted machinery.

The "executor?: Executor where undefined = optimised local" pattern from earlier drafts is gone. meta/git.ts always passes the terminal's Host as the executor; local terminals get LocalHost (which uses the refcounted primitive), remote terminals get RemoteHost (which uses executor.watch over RPC). One code path; the routing decision is host identity, not a special-case.

PR #929 note PR #929 carries an explicit executor = undefined shortcut at meta/git.ts:37-40 to preserve the parcel-watcher fast path for local. That asymmetry exists in PR #929 because PR #929 doesn't extract the refcounted primitive — it stays in kolu-git, only the bare fs.watch ends up in localExecutor. R1 fixes this at the root.

Why not AsyncLocalStorage?

Tempting alternative: stash the executor in AsyncLocalStorage at the orchestrator, read it implicitly inside kolu-git / agents. Functions wouldn't need an executor parameter. Rejected — five concrete failure modes:

// REJECTED: AsyncLocalStorage executor injection import { AsyncLocalStorage } from "node:async_hooks"; const executorContext = new AsyncLocalStorage<Executor>(); // Orchestrator: executorContext.run(host, () => startProviders(entry)); // kolu-git (unchanged signature): export async function detectDefaultBranch(repoPath: string): Promise<string> { const executor = executorContext.getStore() ?? localExecutor; return executor.exec("git", ["symbolic-ref", …], { cwd: repoPath }); }

1. Tests silently pass against the wrong executor

test("worktree create on remote", async () => { const mockExec = vi.fn().mockResolvedValue({ stdout: "main", exitCode: 0 }); const mockExecutor: Executor = { exec: mockExec, /* … */ }; // FORGOT executorContext.run(mockExecutor, …) wrapper const result = await worktreeCreate("/tmp/repo", "feature-a"); expect(mockExec).toHaveBeenCalled(); // ← FAILS silently });

The function ran successfully against localExecutor's default fallback. With explicit params, the test won't compile without supplying an executor.

2. Static analysis becomes useless

With explicit params, grep "executor" packages/integrations/ tells you exactly which functions are host-aware. With ALS, every async function is implicitly host-aware. Refactor decisions become guesswork.

3. New code silently bypasses the executor

// Six months later, a maintainer adds: import fs from "node:fs/promises"; async function readGitConfig(repoPath: string): Promise<string> { return fs.readFile(`${repoPath}/.git/config`, "utf-8"); // ← Direct fs.* call }

With explicit params, the missing executor parameter is a code-review red flag. With ALS, this silently reads the controller's filesystem for remote terminals — bug surfaces only when a user reports "Code tab shows wrong file on remote."

4. Context propagation across boundaries is fragile

ALS propagates through await and Promise chains. It does NOT propagate cleanly across EventEmitter listeners attached outside the context, setImmediate scheduled before context was set, worker threads, or third-party libraries with internal queues.

executorContext.run(remoteHost, () => { startWatcher(session, onChange); // registers an fs.watch listener }); // …time passes, watcher fires: onChange(); // ← getStore() here returns undefined. // Falls back to localExecutor. Wrong host.

5. Concurrent contexts mix unpredictably

await Promise.all([ executorContext.run(host1, () => getStatus("/repo")), executorContext.run(host2, () => getStatus("/repo")), ]);

Works if every IO call is awaited cleanly. Any non-awaited scheduling (setImmediate, queueMicrotask) picks up whichever context the event loop is in. Unreproducible.

The explicit alternative

Every IO-bearing function takes an executor parameter. Noisy. Adding a new IO call requires threading through callers. That's the volatility tax: "which host's filesystem does this read run against?" is a question with a real per-call answer, and the answer should be visible at every site that needs to know.

Trade-offs honestly

  • Each user-visible PR is small. F6a-d are ~20-150 lines touching one meta/*.ts file each. Reviewers can reason about one provider's remote behaviour at a time.
  • Refactor PRs are no-functional-change. R1, R2 (merged), F1+F2+F3 (bundled, #947), F4 are mechanical refactors. Tests pass identically. Reviewers verify "did anything break?"
  • All refactors captured in the plan. R1, R2 land first as standalone PRs; F1-F4 fold in additional refactor work; nothing is deferred to a "future cleanup."
  • R1 + R2 already shipped value to master. Master is unambiguously better even if the feature never lands — the two extracted libraries stand on their own.
  • No partial-symmetry asymmetry. Local and remote routing are structurally identical post-R1.
  • Adding a new Executor op later (e.g. streamFile) edits 4 places (io/ interface, localExecutor, RemoteHost, helper protocol + manager). Acceptable; only matters when a second op forces the move.
  • AsyncLocalStorage rejected. See the dedicated section for the five concrete failure modes.

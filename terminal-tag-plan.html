v3 — intent supplants branch

One slot. Intent if set, branch as fallback.

Closing #930 and #858. Fresh /do-driven PR on a single intent field. The intent annotation now replaces the branch-name display wherever a single annotation line is rendered — branch is the fallback when no intent is set. Terminals without a git repo show nothing until the user types one.

0 The supplant rule new in v3

Every render site that today shows a branch name now shows whichever of these is first non-empty:

  1. Intent — line 1 (if the user typed one)
  2. Branch name (if the terminal lives in a git repo)
  3. Nothing (no intent, no repo → empty slot; user can still add an intent)

What changed:

  • xterm buffer tail in Awaiting card — removed.
  • Intent markdown body replaces it (only when intent has lines past line 1).
  • Annotation slot in each row: intent line-1 if set, else branch, else nothing. No more stacked branch + intent.
  • Eyebrow IntentGlyph chip — removed. The glyph already lives in the annotation slot. One place, not two.
  • Reply input stays — that's the live interaction.
  • Rail mode: emoji painted on the colored swatch (no change from #930) — this is the rail's only annotation surface, so no duplication risk.
  • New: rows for non-git terminals get an empty annotation slot (—) until the user adds intent. Click it to open the editor.

The xterm tail was redundant noise — agent state pills + the reply input already communicate "what's the agent doing." Intent is the part the user actually wrote about the work.
Tail render code in Dock.tsx:520-540 can be deleted. Bonus simplification: peekTick, tailLines, the resize listener, all go.

F Decisions remaining

Smaller calls before /do kicks off

  1. Quick-row "insert" behavior — at cursor, or always at line 1? Recommended: at cursor. Matches the standard mental model for emoji-insert. Inserting at line 1 is surprising when you're editing line 5.
  2. Empty intent on the Awaiting dock card — collapse to header+reply, or show a placeholder hint? Recommended: collapse. The reply input is the only live affordance that matters when no intent is set. Header+reply is a clean, smaller card.
  3. Should the chip still display the line-1 emoji, or just any printable char? Recommended: any printable char. The field is already free-form; the chip should mirror that.
  4. If line 1 is long (say, 50 chars), what does the chip show? Recommended: first grapheme cluster only. The chip is a glyph slot; the pill carries the rest.
  5. Branch name for the fresh PR? Recommended: feat/terminal-intent.

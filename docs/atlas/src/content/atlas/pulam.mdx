## The core model

Three claims, each verified against the shipped code:

- **The unit is a generic awareness value, not kolu's record.** `pulam` exposes `Collection<TerminalId, AwarenessValue>` ‚Äî exactly `cwd ¬∑ git ¬∑ lastAgentCommand ¬∑ lastActivityAt` plus `pr ¬∑ agent ¬∑ foreground`. That schema lives in `@kolu/terminal-workspace` (the R4.1 `@kolu/terminal-awareness`, renamed when it grew fs/git in R6), built from the vendor-neutral `anyagent` / `anyforge` / `kolu-git` shapes with **no kolu content** ‚Äî not even `location` (no sensor can know its own kolu-side `hostId`). Kolu's `TerminalServerMetadata` is `AwarenessValue` **plus** `location` and UI fields: built *on* it, never carved *from* it.
- **kaval stays byte-for-byte minimal.** `pulam` *dials* kaval as a `ptyHostSurface` client for the four taps + `getScreenText`; it adds zero awareness/git/gh logic to kaval. A layer above, never a fork.
- **Ephemerality is the simplifier.** Awareness never has to survive a restart ‚Äî it's re-derivable from live taps + current host fs. So `pulam` sheds all of kaval's durability machinery: no single-instance lock, no spawn/cleanup, no persisted list (it borrows kaval's `terminal.list`), no adoption or reconnect. Every (re)start just re-runs the sensors from now.

## One library, two homes ‚Äî kolu reads, never folds

Local and remote share the *sensor library* and *one schema* ‚Äî and, after [remote-terminals](remote-terminals.html) **R8**, one **shape of consumption** too: the sensors fill the `terminalWorkspaceSurface.awareness` collection, and kolu **reads** it. The only thing that differs is where that collection is *backed*:

- **Local ‚Üí in-process.** kolu-server runs the sensors and they fill the in-process awareness collection; kolu reads it through `awarenessFor(id)` ‚Äî no transport, no framing, and crucially **no fold** (the sensors no longer write kolu's `terminalMetadata`). This reverses the earlier *local writes `terminalMetadata` directly* stance: at #1406 an in-process surface bought nothing (kolu folded into its record either way), so it read as ceremony; R8 makes it earn its keep ‚Äî giving observation its own collection is exactly what deletes the fold and the persisted/live fence, and unifies the read seam with remote.
- **Remote ‚Üí serve (pulam), then mirror (kolu).** `pulam` is Nix-provisioned over ssh (kaval's staleKey recipe), runs the identical sensors, and **serves** the slice over `stdioLink`. kolu mirrors it (R7's total dual) and reads the same `awarenessFor(id)` seam ‚Äî the backing is a mirror instead of an in-process collection.

So the old asymmetry ‚Äî *local mutates kolu's record; remote serves ‚Üí mirrors ‚Üí folds* ‚Äî collapses to **one shape, two backings**: both fill the observation surface, kolu reads it, and local-vs-remote is a backing swap behind the seam. The one new artifact is the host-side hooks impl that reifies each `mutate` closure into a serialized slice frame ‚Äî a wire carries values, not closures. (Until R8 lands, kolu's *local* path still routes awareness into `terminalMetadata`; R8 is what moves it onto the surface kolu reads.)

> **`mirrorRemoteSurface` has graduated** (<PrLink pr={1497} />). It is now the one public mirror in `@kolu/surface` ‚Äî the spec-driven dual of `implementSurface` that drives a served surface's **streaming** primitives (cells/collections/streams) into caller-supplied sinks ‚Äî mirroring *procedures* to a **total** dual is [remote-terminals](remote-terminals.html) R7; `mirrorRemoteCollection` is demoted to its private per-key engine ("you don't sell half a house"). The trigger was pulam growing a real **`activity` stream** (the green dot) ‚Äî the second stream-bearing consumer the graduation waited for. It does not by itself dissolve kolu's server-side fold ‚Äî remote-terminals **R8** does, by composing the surface into kolu's own; until then the fold consumes `mirrorRemoteSurface`. As a `@kolu/surface` API change it rode `surface.md`'s drishti merge-gate.

## The standalone tool ‚Äî `pulam` + `pulam-tui`

Like kaval, the daemon is a deliverable in its own right, shipped and proven *before* kolu touches it. Where `kaval-tui` shows what's *running* in each PTY, `pulam-tui` shows what each terminal *is in* ‚Äî a "what is every agent doing, across every repo" dashboard with **zero kolu-server**, dialable over ssh against a prod box.

## R4.5 ‚Äî the fleet board

The user's framing made literal ‚Äî *"what is every agent doing, across every repo, across every machine"* ‚Äî as a **dashboard you leave open on a second monitor**: glance over from the game and a colour you can read across the room tells you whether any agent is **blocked on you**, and where. It stays a TUI, runs zero kolu-server, and shipped as four steps:

- **R4.5.1 ‚Äî Bun runtime** <PrLink pr={1470} />. Retire `list`/`watch` + the `columnify` text; render a SolidJS-canonical OpenTUI list of one endpoint's terminals (the already-shipped `--socket` or single `--host`, reused verbatim ‚Äî no new connection capability), with a **one-second clock** beside it as the liveness proof. ## Why this stays a TUI, and what is electricity vs. a leaf

The fleet board is the awareness analog of **drishti**. Per [electricity](electricity.html), `@kolu/surface` proved it was real electricity by graduating to a second consumer (drishti's fleet process-monitor). pulam's fleet is the exact same shape one layer up ‚Äî a dashboard that mirrors N remote pulam daemons into one host-keyed view, proving pulam's **own** surface is itself a receptacle a different app plugs into. The clincher: the fan-out + aggregation needed **no contract change** to exist (R4.5.3 mirrored the same `awareness` collection across N hosts). That invariance under a brand-new multi-host consumer *is* the electricity proof. (R4.5.4's `activity` stream came later, as an additive *feature*, not a tax the multi-host consumer levied.)

Two boundary calls keep it honest:

- **The renderer is electricity ‚Äî but it already exists as OpenTUI.** A domain-agnostic Solid‚Üíterminal capability, so R4.5 *consumes* OpenTUI per *prefer-libraries* rather than minting a `@kolu/solid-tui`.
- **The aggregation join is a leaf reducer, not electricity.** It hides a bounded host-keyed merge, not a hard volatility (transport + provision + reconnect already belong to `@kolu/surface-nix-host`; the per-primitive mirror to `@kolu/surface`). So no `@kolu/fleet-mirror` ‚Äî that is exactly the "tidy generic helper" trap electricity warns against.

<Callout kind="good" title="The fleet glance view stays a TUI ‚Äî but the web twin earns its keep elsewhere (R4.8)">
For the *fleet glance view*, the TUI wins: a read-only aggregator has nothing to fold and nothing to persist, so a browser shell's stale-tab/reconnect story would be pure ceremony. The TUI ships fast, is dev-loop-native, and proved the aggregation core cheaply ‚Äî that judgement stands *for the glance view*. What it missed: kolu's own browser consumes this **same** surface over `websocketLink` + `surfaceClient`, and *that* leg needs a standalone proof. So **[pulam-web](#r48) (R4.8)** is not a prettier dashboard ‚Äî it is the electricity proof of kolu's *consumption shape*, the one thing the TUI's `stdioLink` can't exercise. A team-wide browser monitor is the real product that falls out; it stays pulam's own consumer, zero kolu-server, never composed into kolu ‚Äî a **different product** from kolu's own-canvas consume ([remote-terminals](remote-terminals.html) R8‚ÄìR9).

## Why Bun, contained to the viewer

OpenTUI is the production-grade TUI framework (it powers OpenCode and terminal.shop) and ships a **Solid reconciler** (`@opentui/solid`), so the dashboard is authored in the same fine-grained-reactive model kolu already lives in. The catch ‚Äî and the reason the runtime swap came first ‚Äî is that OpenTUI's renderer is a native **Zig** core loaded via `Bun.dlopen`: it needs Bun.

So the split is narrow and deliberate: **`pulam-tui` alone** becomes a Bun binary; everything else, including the `pulam` **daemon**, stays Node. The daemon is the half that travels over ssh and leans on `node:sqlite`, so it must stay Node (and stays cross-arch). The viewer is never provisioned over ssh, so the Bun runtime + per-arch Zig lib live only in the local closure. The packaging is reused, not new ground ‚Äî it is drishti's bun2nix recipe wholesale.

<Callout kind="warn" title="OpenTUI + Solid build gotchas (learned the hard way in R4.5.2)">

- **OpenTUI repaints only the CHANGED CELL.** Verify "is it live?" with **raw-PTY capture** (frame N+1 ‚âÝ frame N), never `snapshot | grep` or a glance ‚Äî both read a live UI as frozen.
- **Bundle `solid-js`; do NOT mark it `external`.** `@opentui/solid`'s plugin redirects solid-js to its reactive client build only when solid is loaded *during the build*. Marking it external ships the non-reactive *server* build and every live value silently freezes at runtime (invisible in dev, where the `bunfig.toml` preload redirects at load time). Use `external: ["@opentui/core"]` only.
- **Pre-bundle; don't run from source.** `build.ts` (`Bun.build` + `@opentui/solid/bun-plugin`) applies `babel-preset-solid` at build time; clean its `outdir` first (Bun.build leaves orphan chunks). The smoke that catches the freeze is running the **built `dist/bin.js`** and confirming a repaint.
- **SolidJS-canonical only** ‚Äî a live value is `createSignal` + `onMount(setInterval)` + `onCleanup(clearInterval)`, read in JSX. No fps caps, `requestRender`, or `process.stderr` monkeypatching.
- **Forwarded daemon stderr** over `--host` is quieted **at the daemon** (`warn` over `--stdio`) so it can't corrupt the alt-screen; the held-open fleet dials route their logs through `@kolu/surface-nix-host`'s `onLog` sink.

</Callout>

### R4.7 ‚Äî `pulam-tui fleet`: a live `git status` view (proving the `{seq}` shape)

The fleet board already shows each terminal's `repo¬∑branch` from the **`awareness` collection** ‚Äî the primitive R4.5 proved. R4.7 consumes the *other* arm of the surface, the one kolu's Code tab needs and **nothing had exercised**: the **`subscribeRepoChange` `{seq}` watcher stream** re-running the **`git.getStatus` procedure**. Each fleet **row** grows a **live working-tree cell** ‚Äî a changed-file count plus the branch's ahead/behind ‚Äî and selecting a row (‚Üë/‚Üì) and pressing Enter **drills in** to the full `git status`: the `staged ¬∑ modified ¬∑ untracked` summary and the changed-file list. *No file content, no diff* ‚Äî the changed-file **list** (paths + status codes), never file bodies; `git status` alone drives the exact pulse-plus-requery loop kolu fails on, so it is the proof.

To paint ahead/behind, `git.getStatus`'s `local` output grew a branch tracking header (`name ¬∑ upstream ¬∑ ahead ¬∑ behind`) and the working-tree section counts ‚Äî read off the *same* `git status` the file list already reads, so it costs no extra git call (simple-git already computes both and the code just stopped discarding them). The same change models the result as a discriminated union on `mode` and **drops the always-null `base`** from the `local` arm; removing a field a `0.3` viewer's schema still requires is a *breaking* reshape, so the workspace contract bumps `0.3 ‚Üí 1.0` (a **major**, not a minor) and the gate marks `0.3` and `1.0` mutually `skew` in both directions. This is the one place R4.7 extends the surface rather than purely consuming it ‚Äî and it stays in `kolu-git` + `@kolu/terminal-workspace`, touching no `@kolu/surface*` API, so it needs **no drishti gate**.

The consume loop is **`RepoWatchSet`**: per *distinct repo* across the fleet, subscribe to the `{seq}` pulse and re-query `getStatus` on each, keyed by repo root so repo-mates share one subscription and the last to leave tears it down. It runs over the real unix-socket / `stdioLink` link, so R4.7 answers one question: **does the `{seq}` source-arm stream + requery survive a real link, end to end?** Proven by an integration test over a real served socket (a working-tree change pulses ‚Üí the re-query reflects it) and by raw-PTY capture ‚Äî frame N+1 ‚âÝ frame N on a `touch` / `git add` ‚Äî *never directLink* (the in-process path that masked the bug).

So the *shape* is sound over a real wire. **But the consumer here is raw `for await` iteration over a `mirrorRemoteSurface` handle ‚Äî close kin to `directLink`, and *not* kolu's consumer.** kolu's browser (and the #1510 prototype that stuck) reads through `surfaceClient.streams.use()` + a Solid `reconcile` store ‚Äî a path R4.7 never touches. So two unknowns remain, both R4.8's: that consumer, and whether the surface should hand a raw `{seq}` pulse to a browser at all.

<a id="r48"></a>

### R4.8 ‚Äî `pulam-web`: the browser twin (kolu's `ws + surfaceClient` leg)

`pulam-web` is a [drishti](electricity.html)-shaped browser ‚Üî Bun app that shows pulam's awareness fleet + a **live `git status` view** ‚Äî but reaches the surface **the way kolu's browser will**, which is *not* the way `pulam-tui` does. That difference is the entire phase, so the plan states it precisely.

<Callout kind="warn" title="The trap ‚Äî do NOT port pulam-tui's consumer">
`pulam-tui` reads the `{seq}` stream by **raw `for await` iteration** over a `mirrorRemoteSurface` handle. That is structurally `directLink` ‚Äî it never touches the path that breaks. **kolu's browser** (and the #1510 prototype that stuck) reads through **`surfaceClients(websocketLink(ws))` ‚Üí `app.streams.<k>.use(inputFn)` ‚Üí a Solid `reconcile` store**. If pulam-web swaps pulam-tui's link to `websocketLink` and keeps the raw loop, it proves the ws *transport* and **dodges the actual suspect**. pulam-web's consumer MUST be `surfaceClient` + `reconcile` (a `useWatchedRead`-style requery), exactly kolu's. This is the lesson the earlier plan left implicit and the implementer must not miss.
</Callout>

**The decision R4.8 makes (resolve it first, grounded against the surface code).** There are two designs for git status reaching a browser, and #1510 picked the unproven one:

- **Design A ‚Äî raw pulse to the browser** (what #1510 did, and stuck): the browser subscribes terminal-workspace's raw `subscribeRepoChange` `{seq}` stream and re-queries `git.getStatus` *client-side*, over `surfaceClient` + `reconcile`.
- **Design B ‚Äî value-bearing, wrapped server-side** (what `koluSurface` *already* does and the Code tab proves daily): the server composes `{ read: getStatus, install: subscribeRepoChange, isEqual }` into one **value-bearing `gitStatus` stream** (`packages/server/src/surface.ts:271-277`) and serves the *value*; the browser consumes an ordinary value-bearing stream ‚Äî a shape proven over `surfaceClient` long ago.

**Lead hypothesis: Design B.** The raw `{seq}` pulse never needs to reach the browser; keep the requery server-side, where `koluSurface` already runs it. For a *remote* terminal (R9) the server wraps the **mirrored** `subscribeRepoChange` + the **forwarded** `getStatus` (R7) into the same value-bearing stream ‚Äî so the genuinely-new thing pulam-web must prove is **value-bearing git status sourced from a *mirror*, reaching a browser over `surfaceClient`, updating live**. If that holds, R8b's "ungraduated primitive" dissolves: it was an artifact of Design A. R4.8's job is to confirm B end-to-end (and, if useful, reproduce A's failure once to document *why* it was the wrong design).

**Architecture (mostly wiring shipped primitives).**
- **Bun backend** ‚Äî mirror N pulam daemons (reuse `pulam-tui`'s `RepoWatchSet` + the awareness-upsert/remove ‚Üí retain/release reconcile, `packages/pulam-tui/src/fleet.ts`); compose the **value-bearing `gitStatus`** stream from `subscribeRepoChange` + `getStatus` (the `koluSurface` descriptor is the template); **re-serve** to the browser over `/rpc/ws` via `implementSurfaces` + `WsRPCHandler` ‚Äî the exact serve stack `kolu-server` ships (`packages/server/src/{surface.ts,index.ts}`). Spread the fragment into the router; never hand it raw (double-prefix 404, pinned by `implementSurface.test.ts`).
- **Browser frontend** ‚Äî `new WebSocket('‚Ä¶/rpc/ws')` ‚Üí `websocketLink<contract>(ws)` ‚Üí `surfaceClients(link, surfaces)` ‚Üí `app.streams.gitStatus.use(() => ({ repoPath }))`; render the fleet rows + git cell + drill-in by porting `pulam-tui`'s pure `gitCell` / `gitDetail` projections (`packages/pulam-tui/src/render.ts`) to web DOM.

**Reuse vs build-new.** *Reuse:* `RepoWatchSet` ref-counting + repo-membership reconcile; the pure `gitCell` / `gitDetail` projections; the `terminalWorkspaceSurface` contract; the shipped serve stack (`implementSurfaces`, `WsRPCHandler`, `/rpc/ws`) and consume primitives (`websocketLink`, `surfaceClients`, `.use()`, `STREAM_RETRY`). *Build new:* the **value-bearing `gitStatus` wrap sourced from the mirror**, the **`surfaceClient` + reconcile** browser consumer (the point of the phase ‚Äî not pulam-tui's raw loop), the web UI shell, and the Bun re-serve entrypoint.

**The hermetic test.** There is **no in-memory `websocketLink` double** in `@kolu/surface` (only `directLink`, `stdioLink`, a real unix socket). Model the in-repo guard on R7's `mirrorRemoteSurface.test.ts` (`implementSurface ‚Üí directLink ‚Üí consume`), **but drive it through `surfaceClient` + a Solid `reconcile` store** ‚Äî asserting the reactive value re-notifies on the *second* `{seq}`/value delta. That catches a *reconcile-reactivity* break deterministically; a *ws-flush* break only appears over a real socket ‚Äî which is why **pulam-web over a real ws is the full proof** and the unit test is necessary-but-not-sufficient.

**Definition of done (so this PR cannot land half-proven).** (1) pulam-web's browser shows a fleet with a **live git-status cell + drill-in** that updates within ~1s of a `touch`/`git add` on a watched repo, captured as **video/screenshot evidence over the real ws** (not directLink, not a unit test alone). (2) The consumer is `surfaceClient.streams.*.use()` + `reconcile` ‚Äî confirmed by reading the code, not asserted. (3) The hermetic `@kolu/surface` test above is committed and green. (4) The note records **which design (A or B) was adopted and why**, as the explicit recipe R8b copies. (5) If Design A is adopted, the exact reconcile/ws fix that made it work is documented; if B, the value-bearing wrap is shown working **from a mirrored source**.

**Ground-truth file map (verified on master).** consumer-to-avoid + reuse: `packages/pulam-tui/src/fleet.ts` (`RepoWatchSet`, the raw pulse‚Üírequery loop), `render.ts` (`gitCell` ~693, `gitDetail` ~757). contract: `packages/terminal-workspace/src/surface.ts` (`subscribeRepoChange` ~135, `git.getStatus` ~165). value-bearing template: `packages/server/src/surface.ts:271-277` (`gitStatus { read, install, isEqual }`). serve: `packages/server/src/index.ts` (`/rpc/ws` `WsRPCHandler` ~427-510). consume (the path to copy): `packages/client/src/wire.ts` (`websocketLink` ‚Üí `surfaceClients` ‚Üí `.use()`), `packages/surface/src/solid/{surfaceClient,useStream}.ts`. test model: `packages/surface/src/mirrorRemoteSurface.test.ts`, `links/direct.test.ts`.

It ships **with** a hermetic `@kolu/surface` test as the deterministic CI guard: `implementSurfaces` (‚â• 2 sibling surfaces) ‚Üí `websocketLink` ‚Üí `surfaceClient.streams.<k>.use()`, asserting a watcher's frame N+1 ‚âÝ N. The unit test is the in-repo proof; pulam-web is the real second consumer that proves it is *electricity*, not just a green assertion.

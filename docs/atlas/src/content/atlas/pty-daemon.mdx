## Fresh approach — phased by hazard, not module

The plan is the output of a fresh `/lowy` (volatility-based decomposition) + `/hickey` (Simple Made Easy) pass over the whole feature, taking nothing about the discarded #1034 decomposition for granted. **The thesis:** decompose by concept and rank concepts most-stable → most-volatile — but let *build order* be set by a different fact: the process decouple is the only **one-way door**, the single step that changes runtime topology (IPC, a surviving process, single-instance, reattach). Everything is either before the door or part of it.

What's allowed to be a separate phase is decided by **hazard**, not by module or convenience. The door creates the hazards — a dead daemon (the empty-canvas lie), a wire-incompatible one (a bricked server), a live restart that can lose the session (#1034), an outdated daemon the user can neither see nor act on. Everything that defuses or honestly surfaces those hazards is hazard-coupled to the door and ships **with** it; nothing in that surface may drift to a follow-up — that drift is the bug #1034 paid for. Only what is neither hazardous nor meaningless-without-the-door can live before it: the in-process foundation. The one move that matters most: **identity/staleness is designed in as A2, before any daemon exists** — a typed, build-enforced wire+behaviour closure, so a wire change can't escape the staleness key by construction; currency is then a read-site derivation (`staleKey !== currentBuildId()`), never a stored boolean.

### The axes — most stable to most volatile

Both lenses converge on one spine: eight concerns ranked by how often they change. The survivor holds only the slow end; everything per-deploy rides kolu-server's lifecycle.

| Axis | Rate | Boundary that owns it |
| --- | --- | --- |
| PTY ownership (the survivor core) | rarely | `@kolu/pty-host` + the daemon process: node-pty + the `@xterm/headless` mirror + the raw VT taps; single-instance pid-gate; cgroup-escape spawn. Zero providers in the survivor. |
| Wire contract + compat rule + identity closure | per wire change | `ptyHostSurface` (+ `PTY_HOST_CONTRACT_VERSION` + `isPtyHostContractCompatible`) co-located with a typed `PtyHostIdentity`, all inside `@kolu/pty-host` — so its source hash provably covers the wire+behaviour closure. |
| Identity values: staleness key vs navigable commit | key: per wire change · commit: per deploy | `buildId.ts`: `currentBuildId` (staleKey) vs `currentCommitHash` (navigable ref), one typed pair — never conflated. |
| Connection liveness | per deploy | `DaemonHandle.state()`: liveness-for-RPC vs liveness-for-display kept distinct; currency must not be frozen onto the connection value. |
| Provider DAG (fresh detection) | per deploy — most-edited | `packages/server` `providers.ts`: a transport-agnostic `ProviderRecord` over plain channels seeded from the taps — never a `PtyHandle`. |
| Recovery sequence | per deploy | One orchestration unit of step-values (`captureSession → drainTerminals → respawn → finalize`), shared by the forced + user paths — neither destructive path can be written kill-then-pray. |
| Kill vs natural-exit discrimination | per deploy | One order-safe `killAndUnwatch(id)` (abort the exit tap *then* kill); every kill caller uses it. |
| Health → UI projection (honest state) | per UX tweak | One projection read by ChromeBar + DegradedCanvas + the restart dialog; the cell carries only the state enum, identity rides separately. |

### What this makes impossible by construction

The four production failures stop being review-time vigilance and become a failing unit test at the phase that owns the concept:

| Production failure | Lens diagnosis | Killed at | How |
| --- | --- | --- | --- |
| Mis-scoped staleness key (a wire change didn't flip it) | boundary leaked across packaging · concept inferred from a directory accident | A2 | The survivor's closure *is* the package; the staleKey hashes its source, and a build-time guard fails if the contract reaches a wire/behaviour module outside it. |
| Over-prompting (whole-binary hash nudged every deploy) | keyed on the wrong volatility axis | A2 | The closure-scoped key flips iff a restart gains terminal-host code; a server-only change leaving it unchanged is a falsifiable test. |
| Data-loss restart (kill-then-pray) | un-encapsulated sequence-volatility · a 7-concern god-function | B | One composed restart shared by the forced + user paths ships *with* the door — "kill before snapshot" is structurally impossible. |
| Empty-canvas lie (dead daemon → "you have no terminals") | failure presentation had no owning boundary | B | Honest liveness as a value + a single owned health→UI projection ship with the door — never a follow-up. |

Where the lenses disagreed, the resolutions: provenance is a one-shot `buildInfo` RPC into a signal, not a streaming cell (the health cell carries only `{state}`); recovery is one composed mechanism whose step-values parameterise what-to-preserve (the forced path's `captureSession` returns an empty context); `PtyHostIdentity` is the single emitted value and currency derives from its `staleKey` at the read site; `currentBuildId` is memoised/process-stable in B, with a genuinely runtime-changing key deferred to R-3.

## The three units — A1 · A2 · B all landed

No feature flags — each unit ships complete to master and leaves Kolu working. The boundary rule is asymmetric: hazard-coupled work must stay together (B is one PR despite its size), but hazard-free work may split however helps review (which is why A1/A2 is a clean cut).

### A1 — in-process structural foundation <Pill variant="ok">landed</Pill> <PrLink pr={1055} />

The three structural moves, all in-process and identity-free: **(1)** extract the multi-client `@kolu/pty-host` primitive (node-pty + the `@xterm/headless` mirror + the raw VT taps: cwd · title · command-run · exit · foreground); **(2)** define the wire contract and serve it in-process, both inside `@kolu/pty-host` — `ptyHostSurface` + the `major.minor` compat rule, fulfilled over `createPtyHost`; **(3)** decouple the provider DAG over `ProviderChannels` / `ProviderHooks` / `ProviderRecord` — never a `PtyHandle`. After the <PrLink pr={1059} /> link-family unification the serving is `servePtyHost(deps)` (transport-agnostic, returns the router) + `directLink(router)` (the no-wire member of `websocketLink / stdioLink / directLink`).

By serving the contract in-process, two of the door's scariest surfaces were absorbed early: the contract boundary itself (the consumer speaks `ptyHostSurface`, not `createPtyHost`, so it survives the transport swap untouched) and the sync→async spawn race. A1 also **reverted production to a pure in-process server**, surgically removing all of #1034's daemon machinery — there is no daemon in the codebase today. One observed cutover artifact: deploying A1 over a running #1034 daemon left it orphaned (it survives restarts by design, but A1 has no client for it) — reaped manually via `systemctl --user stop kolu-pty-host.service`.

### A2 — build/commit identity + the staleness-key hash <Pill variant="ok">landed</Pill> <PrLink pr={1063} />

All the identity/hashing, isolated for its own review — low-risk to ship but the single most load-bearing correctness mechanism in the plan. Three deliverables, as shipped:

- **`system.version` extended** with the typed `PtyHostIdentity { staleKey, navigableCommit }` (additive, optional — no contract bump).
- **Identity values + the closure-hash guard.** `currentBuildId` (the staleKey — the nix hash of the package's own source closure) and `currentCommitHash` (the navigable git ref) are nix-injected env (`KOLU_PTY_HOST_BUILD_ID` + `KOLU_COMMIT_HASH`); no dev fallback — kolu runs only under nix. A build-time test walks `index.ts`'s imports and fails if any reached wire/behaviour module is neither in-package nor on a small allowlist of stable deps — the #1034 mis-scope is a failing test from day one.
- **The consolidated `srv · pty` ChromeBar rail** — see the [rail prototype note](pty-daemon-chrome-bar.html). Built in its final two-column shape; in A2 the columns coincide (in-process) and that match *is* the acceptance signal; B only lets `pty` diverge.

Verify holds: mutating any pty-host source flips the staleKey; a server/client-only change doesn't (no over-prompt). A post-merge prod break (<Issue n={1068} />) surfaced an unrelated migration rule worth keeping: any *required* new field on a persisted+streamed schema needs a migration/default (the fix was `.default(0)`).

### B — the daemon, complete and safe <Pill variant="ok">shipped</Pill> <PrLink pr={1275} />

The foundation is in master, so B is one hazard-coupled unit: **the topology decouple, reattach-by-id, the composed #1034-proof recovery, and lighting up A2's dormant honest-state branches.** The user payoff is the headline — _local terminals survive a kolu-server deploy_ (the tmux/zmx replacement, <Issue n={671} />) — and the one genuinely new thing a user *sees* is honest state when a deploy leaves the daemon a build behind, or when it dies: never again the empty-canvas lie.

**The seam — one swap, the consumer untouched.** Today `packages/server/src/ptyHost.ts` exports `ptyHostClient = directLink(router)` — the no-wire in-process client — and the server *also* serves that same router over a unix socket for kolu-tui (<PrLink pr={1084} />). B inverts ownership: the **daemon** process owns `servePtyHostOverUnixSocket`, and the server becomes just another `unixSocketLink(getPtyHostSocketPath())` client of it — the exact role kolu-tui already plays. Both return the same `PtyHostClient` type, so `terminalBackend/local.ts` and the whole provider DAG below it compile unchanged; A1 moved the serving body to its final home for precisely this. `koluRoot`/`koluShellDir` (today computed in the server from its own `serverProcessId`) relocate to a shared module the daemon computes for itself. The daemon runs as `kolu --stdio` in its own cgroup (`systemd-run --user` on Linux, detached on macOS — spike-verified, see the design note below), single-instance-gated by a held-until-death pid file (`O_EXCL` + liveness probe), with the dev-flag exec-arg filter.

**Reattach-by-id.** The server already mints each PTY's UUID and passes it as `terminal.spawn({ id })`, so id ownership is unchanged; B adds the boot path — list the daemon's surviving PTYs, join the saved session by id with no respawn, and let the freshly-restarted provider DAG re-detect against the surviving taps. Metadata is briefly cold (a re-detection pass), never stale — the honest cost the survivor-is-pty-host-only split already accepted.

**The composed recovery — #1034 made impossible.** The sequence is shared verbatim by the two destructive paths (a user-clicked restart and a forced restart when a deploy finds the daemon a build behind), so neither can be written kill-then-pray.

It builds on primitives already live in master — the autosave throttle and `setSavedSession` (which cancels pending autosave) in `session.ts`, and the abort-first `killAll` in `terminalBackend/local.ts` — and adds only the missing #1034 pieces: an order-safe `killAndUnwatch(id)` wrapper, the `waitForPidGone` barrier (the step whose absence lost the #1034 session — the respawn raced the slow old-daemon exit), and load-aware timeouts sized for a thrashing prod box, not an idle dev one. Snapshot-and-save *before* any kill means a failed respawn leaves a restorable session, never a void. (The <Issue n={1041} /> preference storm that once contended with the autosave is already fixed — coalesced writes, <PrLink pr={1050} />.)

**The honest-state surface.** No re-layout: A2 shipped the `srv · pty` rail in its final two-column shape with the `pty` column and the `≡ in-process` tag **commented out** in `IdentityRail.tsx`; B uncomments them and feeds the `pty` column the read-site currency derivation. The currency key is the closure **staleKey**, not the commit: the server publishes the deployed build's expected staleKey (`ptyHostExpectedStaleKey`, its own `KOLU_PTY_HOST_BUILD_ID`) on the streaming `buildInfo` cell, and the rail compares the surviving daemon's relayed `ptyHost.staleKey` against it on every render — equal ⇒ `≡ current`, different ⇒ `⬆ update pending` (a server-/client-only deploy bumps the commit but not the closure, so it never over-prompts). A user-clicked **Restart pty-host daemon** command drives the composed recovery below; on success the server re-reads the fresh daemon's identity and republishes `buildInfo`, so the rail drops the nudge without a reload. A failed restart leaves the daemon degraded server-side (the column resolves to `—`) with the session saved, and the failure toast offers a retry — a browser reload would NOT recover a server-side handle, so it doesn't claim to.

B ships the first two rail states (`≡ current`, `⬆ update pending`) and the honest restart path (failure → `—` column + a retry, never an empty-canvas lie on a <em>known</em> failure). ## The #1034 postmortem — production failure and hard constraints

The first R4c-UI build (<PrLink pr={1034} />) shipped the build-mismatch "update pending" nudge + a *Restart local PTY daemon* command. On production (Linux/systemd, 2026-05-30) the nudge fired correctly on a deploy; clicking restart then **destroyed a live 20-terminal session and could not bring the daemon back**. From the journal: `killAll` drained all 20 terminals first; the old daemon (13.5h old, 25G RAM on a thrashing box) took ~2min to exit; the respawn timed out at 30s; the user was left with zero terminals, a dead daemon handle, and an **empty canvas indistinguishable from "you have no terminals."** No second daemon ever ran — the failure was the respawn losing the race against the slow old-daemon exit.

### Hard constraints for the redo

1. **Restart must be recoverable, never kill-then-pray.** Snapshot the session first; after respawn, auto-reattach/offer restore; if respawn fails, a loud recoverable degraded state with the session preserved — never an empty canvas.
2. **Wait for the old daemon to fully exit** — the single-instance lock fights the restart. Wait on actual process exit (`kill(pid,0)` → ESRCH) before spawning, with generous load-aware ceilings.
3. **Timeouts must fit a loaded production box**, not an idle dev one (20 heavy PTYs + a tsx cold-start under swap ≫ 30s).
4. **Never lie about state** — an explicit connecting/degraded UI, never silent emptiness.
5. **A persistent PTY status indicator, always visible** — connected · connecting · degraded/dead · update-pending (became A2's rail).
6. **Key staleness on pty-host, not the whole binary** — hash the `@kolu/pty-host` source closure so the nudge fires only when a restart actually gains something (became A2's staleKey).
7. **Back up the session before a destructive restart** (Export/Import session shipped via <PrLink pr={1046} />).
8. **Fix the preferences storm** (<Issue n={1041} />, fixed in <PrLink pr={1050} /> — coalesced writes).

The salvage of #1034 shipped constraints #1–#6 onto the PR and verified them live (recoverable ~600ms restart; honest DegradedCanvas; a deliberate server-only deploy did **not** nudge — the over-prompting bug fixed and confirmed). The fresh approach then discarded the lineage anyway: its defining mistake was letting the door's safety surface drift into a follow-up, and the redesign makes that structurally impossible.

### Execution lessons (process, not product)

From building + salvaging #1034 — pre-committed rules for the redo: typecheck every package before commit (a clean `nix build` is **not** proof CI passes — it never type-checks the server, <Issue n={1049} />); never `git stash` in the worktree (use `git show` / a throwaway worktree); grep for unresolved conflict markers before committing a merge; confirm every Edit actually applied; don't double-create GitHub issues on a flaky shell; never defer with a someday-issue — gaps go into the plan and get done in the PR; when the terminal contradicts itself, confirm via an independent path before "fixing" a phantom. Wire-shape rules that recurred: additive wire fields are **optional + no contract bump** (a required field force-restarts a surviving older daemon just to add a diagnostic); the identity surfaced to users must be the *navigable* one (commit SHA), with the source hash kept as the staleness key only.

## Design notes that carry forward

**The survivor is `@kolu/pty-host` only** — node-pty fds + the `@xterm/headless` mirror + the raw VT taps + a unix socket. A kolu-server restart re-runs the providers against the surviving PTYs, so detection is never stale while the PTYs persist. Honest cost (accepted): metadata is no longer "warm" across a restart — a brief re-detection pass, trading warm-on-reconnect for freshness-on-deploy. Only a pty-host contract change (rare) forces terminal loss; a provider change (frequent) restarts the cheap layer with PTYs untouched. This corrects #1031, which daemonized a survivor that *held* the providers.

**The cgroup mechanism (spike-verified).** On Linux/systemd the daemon spawns via `systemd-run --user --unit=kolu-pty-host` (gated on `INVOCATION_ID`), landing in its own transient cgroup — a plain detached/`setsid` child does **not** survive on cgroup-v2 (`KillMode=control-group` walks cgroup membership, not the session — the #1031 Linux failure). macOS's detached spawn already survives launchd. Caveats folded into B: linger must be on; absolute daemon path (minimal unit PATH); idempotent `--unit` reuse. Single-instance two ways: the unit name plus a pid-file (`O_EXCL` + liveness probe).

**tmux/dtach considered and rejected:** they only keep a PTY alive — no OSC-parsed taps, no headless snapshot for lazy-attach, no home for the provider DAG. You'd still build pty-host's streaming layer next to tmux and inherit its session model on top.

## History

- **R4a** <PrLink pr={1023} /> + **R4b** <PrLink pr={1028} /> shipped clean as pure in-process refactors.
- Two prototypes seeded the design: <PrLink pr={994} /> (remote) proved "providers in the agent", re-provisioned fresh per version; <PrLink pr={1010} /> was a PTY-only local daemon — the thin-survivor model. The redo is their correct combination.
- **R4c first attempt** <PrLink pr={1031} /> dropped after a macOS prod failure: it daemonized pty-host *plus* the provider DAG, so the survivor served stale detection on every deploy — and staleness keyed on the inert `pkgVersion` constant, so `outdated` was always false and a 20-hour-old daemon was reused forever.

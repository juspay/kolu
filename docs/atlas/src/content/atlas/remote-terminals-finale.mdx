**The last leg of <Issue n={951} /> ‚Äî kolu dials remote hosts.** R8 (<PrLink pr={1594} />) shipped the right **reader-join** (a terminal is `kolu.authored ‚ãà terminalWorkspace.awareness`, joined at the reader) ‚Äî *keep that*. But R8 also made **kolu serve awareness *in-process*** (the registry-projection backing of `serveTerminalWorkspace`), which **re-coupled awareness into kolu** ‚Äî the exact thing pulam ([R4](pulam.html)) exists to decouple. This note corrects course: **pulam is the single awareness authority per host; kolu *consumes* a local pulam.** Then the remote work (R9.3) is the *same* mirror pointed at a *remote* pulam ‚Äî **local becomes a special case of remote** ‚Äî and the R9a desync never arises. The parent [remote-terminals](remote-terminals.html) keeps the one-line rows; full model in [the terminal model](terminal-metadata-model.html).

<Callout kind="warn" title="The correction ‚Äî why R9a is discarded">
R8's "kolu serves its own awareness" left awareness with **two shapes** (kolu multiplexed vs pulam standalone) and **no single authority**. That forced R9a (<PrLink pr={1604} />) to teach pulam-web how to reach *into kolu* (a kolu-specific `connectTerminalWorkspace` slice) ‚Äî composing an *intermediary*, not pulam. The root fix is upstream: **one pulam per host owns awareness; every reader (kolu's Dock, pulam-web) mirrors that one pulam.** So **R9a is abandoned** ‚Äî its desync dissolves by construction here. The one real thing it surfaced ‚Äî a `@kolu/surface` membership bug (`broadcastKeys`) ‚Äî lands in its own PR.
</Callout>

<Callout kind="note" title="The backing is ALWAYS a pulam mirror ‚Äî local or remote">
R8's **reader-join** and the `serveTerminalWorkspace` **awareness-backing injection seam** are kept. Only the *backing* changes: instead of kolu's in-process sensors, kolu's awareness backing is a **mirror of a pulam** ‚Äî a **local** pulam for the local box (R9.0), a **remote** pulam for a remote host (R9.3), through the **same** mirror code path. One sensor set per host; the arrow always points **‚Üí pulam**. (Bonus: the local pulam owns the kaval byte-tap, so the old "kolu has no live `activity`" gap closes for free.)
</Callout>

## The sub-phases

<a id="r90"></a>

### R9.0 ‚Äî kolu consumes a local pulam (retire in-process awareness) ¬∑ *the decouple*

<Phase id="R9.0" name="kolu consumes a local pulam; pulam is the awareness authority" status="todo" needs={["R8", "R4"]} />

**The fault R8 left.** R8 made kolu run the awareness sensors **in-process** and serve them (the registry-projection backing). That re-coupled awareness into kolu and gave awareness two serving shapes (kolu multiplexed, pulam standalone) ‚Äî the asymmetry R9a then papered over with a kolu-aware mirror.

**The fix.** Run **one local pulam** ‚Äî it dials kolu's *own* kaval, runs the one sensor set, and serves `terminalWorkspaceSurface` **standalone**. kolu's awareness backing becomes a **mirror of that local pulam**, injected at the *same* `serveTerminalWorkspace` seam R9.3 uses for remote. kolu's Dock keeps the **reader-join unchanged**; pulam-web mirrors the **same** local pulam, uniform with every remote host ‚Äî no `connectTerminalWorkspace`, no multiplex knowledge, no second sensor set.

This **subsumes R9a**: one sensor set, all readers mirror it, so the `working`-vs-`idle` desync cannot occur.

<Svg svg={topology} caption="After R9.0, on the local box: kaval (owns the PTYs) and a local pulam (the one sensor set, R4) are each daemons on a unix socket, both spawned + supervised by kolu-server (R2 adoptOrEnsure). kolu-server dials kaval over a unix socket for PTY I/O, and mirrors the local pulam's terminalWorkspace.awareness over a unix socket as its awareness backing; the local pulam reads kaval's taps (unix socket) to run the sensors. The browser reads kolu-server over a websocket (/rpc/ws); pulam-web, if present, mirrors the SAME local pulam. A remote host (R9.3) is the identical shape, reached over ssh stdio instead of unix sockets." />

- **Decided:** the local pulam's lifecycle is **kolu-server's** ‚Äî it `adoptOrEnsure`s the local pulam via the [R2](pty-daemon.html) daemon-supervisor spine (boot brings up kaval ‚Üí local pulam ‚Üí mirror), the same supervision the kaval already gets.
- **Open:** *migration off R8's in-process backing* ‚Äî cut over, or flag the swap behind one seam and retire the in-process path once the mirror is proven.
- **Needs:** R8 (the reader-join + the injection seam) and R4 (the pulam daemon) ‚Äî **both shipped**.
- **Done:** kolu-server runs **no** awareness sensors; its awareness backing is a local-pulam mirror; kolu's Dock and pulam-web render identical state **by construction** (the old R9a differential, now free); local and remote awareness share **one** mirror code path.

### R9.2 ‚Äî ssh kaval driver ‚Üí a remote PTY tile

<Phase id="R9.2" name="ssh kaval driver ‚Äî remote PTY" status="todo" needs={["R9.1"]} />

Add the `{kind:"remote",hostId}` arm to the resolver: dial the remote host's **kaval** over `frontDaemonOverStdio` (the same `ssh <host> kaval --stdio` relay `kaval-tui --host` uses) ‚Üí a remote PTY on the canvas, **PTY only** (no awareness badge yet). Persist `location = {kind:"remote",hostId}`. Adoption reuses the local boot's `adoptOrEnsure` spine + the pid-gate's atomic `link(2)` (a dial *joins* a host's kaval, never races to kill it).

- **Needs:** R9.1.
- **Done:** dial a host, spawn a PTY, see an `ssh`-badged remote tile that echoes; it survives a network blip via re-attach. e2e: a remote PTY round-trips over ssh.

<a id="r93"></a>

### R9.3 ‚Äî remote awareness = the same mirror, a remote pulam

<Phase id="R9.3" name="remote awareness ‚Äî the same mirror, remote source" status="todo" needs={["R9.0", "R9.2", "R7"]} />

Because **R9.0 already made kolu mirror a (local) pulam** for its awareness backing, a remote host is the *same backing* pointed at the **remote** host's pulam ‚Äî dialed over a long-lived `getHostSession`, mirrored via R7's total dual. **Only the dial differs** (a local pulam in-process vs a remote pulam over ssh); the backing-injection, the reader-join, and disk persistence are identical. The remote tile gains the git/agent/PR badge + green dot; `activity` rides the mirror.

- **Needs:** R9.0 (the mirror backing exists), R9.2 (the remote PTY), R7 (the total dual).
- **Done:** a remote tile shows live awareness mirrored from the remote pulam, via the **same code path** as local; a differential test pins it equal to the remote pulam's own `awareness`.

### R9.4 ‚Äî persisted location, reconnect, adoption

<Phase id="R9.4" name="persisted location + reconnect + adoption" status="todo" needs={["R9.2"]} />

Harden the remote lifecycle: re-validate the persisted `location` on reconnect, and cover the adoption/reconcile cases.

| the remote kaval is‚Ä¶ | kolu does |
| --- | --- |
| absent | provision + spawn fresh (`adoptOrEnsure` ‚Üí ensure) |
| live ¬∑ wire-compatible | **adopt** ‚Äî connect, never kill; its PTYs reconcile in |
| live ¬∑ a build behind | adopt **+** an "update pending" nudge |
| live ¬∑ contract-skewed | recycle (kill ‚Üí respawn) ‚Äî only on a typed `DaemonContractSkewError` |
| unreachable | bounded retry, then **degraded ‚Äî never killed** |

- **Open (decide here):** *what* is re-validated on reconnect and the fail behavior. `location` is the host discriminator (`kolu.authored`); `cwd` is the observed dir (`terminalWorkspace.awareness`) ‚Äî different collections. Spell out the comparison and the fail-fast (crash loudly vs re-derive); don't invent it at implementation time.
- **Needs:** R9.2.
- **Done:** a kolu-server restart re-adopts remote PTYs; each adoption-table row has a test.

### R9.5 ‚Äî the Code-tab fs/git rewrite (procedure + pulse)

<Phase id="R9.5" name="Code-tab fs/git ‚Üí procedure + pulse" status="todo" needs={["R-pulamweb-4", "R9.3"]} />

The one **large client change**, and the only finale phase R-pulamweb-4 gates. The Code tab reads `koluSurface`'s **value-bearing** streams today (`CodeTab.tsx` ‚Äî `app.streams.gitStatus/fsListAll/gitDiff.use`). Rewrite it to the shared surface's **procedure + `{seq}` pulse**: call the procedure once, re-query on each pulse (the two wire shapes are in [surface live data](surface-live-data.html)). This deletes the last `koluSurface` fs/git reads and works for local and remote tiles alike.

- **Needs:** **R-pulamweb-4** (proves the procedure + pulse remote leg over `websocketLink` + `surfaceClient`), R9.3.
- **Done:** the Code tab shows live git status/diff for a **remote** tile via procedure + pulse; no `koluSurface` fs/git reads remain.

**The last leg of <Issue n={951} /> — kolu dials remote hosts.** R8 (<PrLink pr={1594} />) shipped the right **reader-join** (a terminal is `kolu.authored ⋈ terminalWorkspace.awareness`, joined at the reader) — *keep that*. But R8 also made **kolu serve awareness *in-process*** (the registry-projection backing of `serveTerminalWorkspace`), which **re-coupled awareness into kolu** — the exact thing pulam ([R4](pulam.html)) exists to decouple. This note corrects course: **kolu *consumes* a pulam instead of serving awareness itself** — a supervised local pulam for the local box (R9.0), a remote pulam for a remote host (R9.3, the same mirror). The arrow points **→ pulam** and **neither side reaches into the other**, so the `connectTerminalWorkspace` mud never arises. (Collapsing to *literally one* pulam per host — pulam-web reading kolu's pulam too — is a separate, optional step: [R9-converge](#r9-converge).) The parent [remote-terminals](remote-terminals.html) keeps the one-line rows; full model in [the terminal model](terminal-metadata-model.html).

<Callout kind="warn" title="The correction — why R9a is discarded">
R8's "kolu serves its own awareness" left awareness with **two shapes** (kolu multiplexed vs pulam standalone) and **no single authority**. That forced R9a (<PrLink pr={1604} />) to teach pulam-web how to reach *into kolu* (a kolu-specific `connectTerminalWorkspace` slice) — composing an *intermediary*, not pulam. The root fix is upstream: **kolu consumes a pulam instead of serving awareness itself**, so neither side reaches into the other and the mud is gone. So **R9a is abandoned** — its *approach* (pulam-web reaching into kolu) is exactly the mud we're removing; its narrow localhost desync is split out to the optional [R9-converge](#r9-converge). The one real thing it surfaced — a `@kolu/surface` membership bug (`broadcastKeys`) — lands in its own PR.
</Callout>

<Callout kind="note" title="The backing is ALWAYS a pulam mirror — local or remote">
R8's **reader-join** and the `serveTerminalWorkspace` **awareness-backing injection seam** are kept. Only the *backing* changes: instead of kolu's in-process sensors, kolu's awareness backing is a **mirror of a pulam** — a **local** pulam for the local box (R9.0), a **remote** pulam for a remote host (R9.3), through the **same** mirror code path. The arrow always points **→ pulam** — kolu never serves awareness itself. (Bonus: the local pulam owns the kaval byte-tap, so the old "kolu has no live `activity`" gap closes for free.)
</Callout>

## The sub-phases

<a id="r90"></a>

### R9.0 — kolu consumes a local pulam (retire in-process awareness) · *the decouple* <PrLink pr={1614} />

<Phase id="R9.0" name="kolu consumes a local pulam; pulam is the awareness authority" status="shipped" needs={["R8", "R4"]} links={[{ label: "#1614", href: "https://github.com/juspay/kolu/pull/1614" }]} />

**The fault R8 left.** R8 made kolu run the awareness sensors **in-process** and serve them (the registry-projection backing). That re-coupled awareness into kolu and gave awareness two serving shapes (kolu multiplexed, pulam standalone) — the asymmetry R9a then papered over with a kolu-aware mirror.

**The fix.** Run **one local pulam** — it dials kolu's *own* kaval, runs the one sensor set, and serves `terminalWorkspaceSurface` **standalone**. kolu's awareness backing becomes a **mirror of that local pulam**, injected at the *same* `serveTerminalWorkspace` seam R9.3 uses for remote. kolu's Dock keeps the **reader-join unchanged**. **pulam-web keeps its own pulam** (its existing ssh dial) — it does *not* reach into kolu, so the `connectTerminalWorkspace` mud never arises and the arrow points **→ pulam** for both sides.

This closes the **mud** — the architectural defect — by decoupling kolu, with neither side reaching into the other. It does **not** collapse the two pulam *instances* into one; that's the optional [R9-converge](#r9-converge). **The relaxed invariant for R9.0:** *kolu is decoupled, both kolu and pulam-web point at **a** pulam, with no cross-reach.*

<Svg svg={topology} caption="After R9.0, on the local box: kaval (owns the PTYs) and a local pulam (the one sensor set, R4) are each daemons on a unix socket, both spawned + supervised by kolu-server (R2 adoptOrEnsure). kolu-server dials kaval over a unix socket for PTY I/O, and mirrors the local pulam's terminalWorkspace.awareness over a unix socket as its awareness backing; the local pulam reads kaval's taps (unix socket) to run the sensors. The browser reads kolu-server over a websocket (/rpc/ws). pulam-web (if deployed) keeps its OWN pulam over ssh — neither side reaches into the other; collapsing those two pulam instances into one is the optional, deferred R9-converge. A remote host (R9.3) is the identical shape, reached over ssh stdio instead of unix sockets." />

<Callout kind="note" title="Lifecycle — kaval survives, pulam is ephemeral">
**kaval** owns the PTYs, so it must *survive* a kolu restart/redeploy: it's a detached daemon kolu **re-adopts** on boot (R2) — terminals untouched. **pulam** owns nothing irreplaceable (its awareness is *computed* from the kaval), so it is **ephemeral** — it dies with kolu, and a fresh one **re-derives** all awareness from the surviving kaval on boot. The only cost is a brief awareness gap during the restart; no work is lost. So `adoptOrEnsure` for the local pulam effectively always *ensures* (spawns fresh), never adopts.
</Callout>

- **Decided — supervision:** kolu-server owns the local pulam's lifecycle via the [R2](pty-daemon.html) `adoptOrEnsure` spine (ephemeral, per the callout above) — boot brings up kaval → local pulam → mirror.
- **Decided — migration:** a **hard cutover** — one PR deletes kolu's in-process sensors and points the awareness backing at the local-pulam mirror, with no flag and no fallback (the no-knobs rule; the worst case on the local awareness path is a briefly-stale badge, never lost work).
- **Needs:** R8 (the reader-join + the injection seam) and R4 (the pulam daemon) — **both shipped**.
- **Done:** kolu-server runs **no** in-process awareness sensors; its backing is a **mirror of its supervised local pulam**; `buildReServe` is lifted to a shared home (both kolu and pulam-web consume it); the arrow points **→ pulam** with **no cross-reach** (no `connectTerminalWorkspace`). *Out of scope* (→ [R9-converge](#r9-converge)): collapsing to one pulam *instance* / killing the localhost Dock-vs-pulam-web desync.

### R9.3 — remote awareness = the same mirror, a remote pulam

<Phase id="R9.3" name="remote awareness — the same mirror, remote source" status="todo" needs={["R9.0", "R9.2", "R7"]} />

Because **R9.0 already made kolu mirror a (local) pulam** for its awareness backing, a remote host is the *same backing* pointed at the **remote** host's pulam — dialed over a long-lived `getHostSession`, mirrored via R7's total dual. **Only the dial differs** (a local pulam in-process vs a remote pulam over ssh); the backing-injection, the reader-join, and disk persistence are identical. The remote tile gains the git/agent/PR badge + green dot; `activity` rides the mirror.

- **Needs:** R9.0 (the mirror backing exists), R9.2 (the remote PTY), R7 (the total dual).

### R9.4 — persisted location, reconnect, adoption

<Phase id="R9.4" name="persisted location + reconnect + adoption" status="todo" needs={["R9.2"]} />

Harden the remote lifecycle: re-validate the persisted `location` on reconnect, and cover the adoption/reconcile cases.

| the remote kaval is… | kolu does |
| --- | --- |
| absent | provision + spawn fresh (`adoptOrEnsure` → ensure) |
| live · wire-compatible | **adopt** — connect, never kill; its PTYs reconcile in |
| live · a build behind | adopt **+** an "update pending" nudge |
| live · contract-skewed | recycle (kill → respawn) — only on a typed `DaemonContractSkewError` |
| unreachable | bounded retry, then **degraded — never killed** |

- **Open (decide here):** *what* is re-validated on reconnect and the fail behavior. `location` is the host discriminator (`kolu.authored`); `cwd` is the observed dir (`terminalWorkspace.awareness`) — different collections. Spell out the comparison and the fail-fast (crash loudly vs re-derive); don't invent it at implementation time.
- **Needs:** R9.2.
- **Done:** a kolu-server restart re-adopts remote PTYs; each adoption-table row has a test.

### R9-converge — one pulam per host (optional, deferred)

<Phase id="R9-converge" name="pulam-web reads kolu's local pulam — one pulam instance per host" status="todo" needs={["R9.0"]} />

After R9.0 the arrow already points **→ pulam** for both kolu and pulam-web, with no cross-reach — *the mud is gone*. What remains is two pulam **instances** on the local box (kolu's supervised one + pulam-web's own), which can drift: the narrow `working`-in-pulam-web vs `idle`-in-Dock desync. This phase collapses them to **one** by pointing pulam-web at kolu's supervised local pulam.

**Why it's optional and off the critical path:**
- **Narrow benefit.** The desync only appears when pulam-web is *deployed* **and** *pointed at `localhost`* (a fleet dashboard watching its own box). The usual deployments — kolu alone locally, or pulam-web watching *remote* fleets — never hit it.
- **Chunky cost.** pulam-web is **ssh-locked**: its host registry, `HostSession`, and `pumpRemoteSurface` assume ssh-stdio reconnect (*"stdio links don't recover mid-stream"*). Dialing a local unix socket is **not** a config flip — it needs a **transport generalization** (a pluggable unix-socket | ssh session) inside `@kolu/surface-nix-host`.
- **Blocks nothing.** R9.2–R9.5 and R10 don't need it, and R9.3's remote mirror reuses the *existing* ssh path — not this generalization. So it can land any time after R9.0, or never.

- **Needs:** R9.0 (the local pulam exists) **+** the pulam-web transport generalization.
- **Done:** under `localhost`, pulam-web reads kolu's supervised local pulam (not its own); one pulam instance per host; the Dock-vs-pulam-web differential passes **by construction** (one source).

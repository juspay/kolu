**The last leg of <Issue n={951} /> вЂ” kolu dials remote hosts.** R8 (<PrLink pr={1594} />) shipped the right **reader-join** (a terminal is `kolu.authored в‹€ terminalWorkspace.awareness`, joined at the reader) вЂ” *keep that*. The question this note settles is how kolu gets the **awareness** half for a *remote* host, and the answer turns out to already be in the repo: the awareness capability is a **decoupled library** вЂ” `@kolu/pulam-library` вЂ” and kolu already consumes it **in-process**. The only real defect is that the library is a *bag of pieces* with **no single sensing assembly**, so the daemon and kolu each wire those pieces up themselves, divergently. This note corrects course: give the library **one shared per-terminal sensing leaf** (`watchTerminalAwareness`) and have **both homes drive it** вЂ” the pulam **daemon** by polling its kaval, **kolu** in-process from its own spawn/wake/adopt events (R9.0) вЂ” and **mirror the remote pulam daemon** for a remote host (R9.3). The decoupling is the **library boundary**, not a process boundary. The parent [remote-terminals](remote-terminals.html) keeps the one-line rows; full model in [the terminal model](terminal-metadata-model.html).

<Callout kind="warn" title="The earlier correction was itself a misdiagnosis вЂ” R9.0-as-a-process is CLOSED">
The prior plan (and PR #1614) said R8's mistake was that kolu *serves awareness in-process*, and proposed kolu run a **local pulam process** it mirrors over a socket. That was wrong on the diagnosis and the cure. **Grounded in the code:** `@kolu/pulam-library` already points its dependency arrow **out** (it names no kolu-app package; its lone host coupling is an injected logger) and already has six consumers вЂ” it *is* the decoupling. kolu already consumes it in-process (`server/src/terminalEndpoint/local.ts` runs the library's `startAwareness` against kaval's taps). So "decouple awareness from kolu in code" was solved before R9.0. The local-pulam-**process** bought no decoupling and introduced a fold-clobber blocker that is intrinsic to mirroring an *ephemeral* local process across a socket. **PR #1614 is CLOSED (never merged); the process approach is rejected.** The pulam daemon is kept for the *remote* case, where a process boundary is real.
</Callout>

<Callout kind="note" title="The real gap вЂ” two assemblers, assembled differently">
The library exposed the *parts* вЂ” the sensor engine (`startAwareness`) and the sink **interface** (`AwarenessSink`), the kaval-tap bridge (`bridgeKavalTaps`), the schema/seeds, the served-surface factory (`serveTerminalWorkspace`), the surface contract, the fs/git endpoint вЂ” but **not** the *whole*: "given a dialed kaval, produce a live, served terminalWorkspace surface." Before R9В·lib that **assembly** lived in the pulam **daemon** (`runPulamDaemon`), with two generic pieces stranded there вЂ” the per-terminal sink and the byte-tap activity tracker. **R9В·lib + R9.0 (below, shipped together) closed both halves**: the per-terminal SENSING is now ONE shared leaf, `watchTerminalAwareness` (bridgeKavalTaps в†’ cwd-persist в†’ startAwareness в†’ raw-output activity tap), and **both homes drive it** вЂ” the daemon by polling, kolu from its lifecycle events вЂ” so kolu's divergent in-line subset inside `local.ts` is gone. **Two assemblers became one leaf.** (The first cut shipped a maximal `createPulam` assembly the daemon wrapped; the re-review found that the *wrong* boundary вЂ” it complected "sense one terminal", genuinely shared, with each home's discovery / seed / sink / served-collection, which are legitimately per-home вЂ” and replaced it with the leaf; see [History](#history).)
</Callout>

## The sub-phases

### R9В·lib + R9.0 вЂ” the shared per-terminal sensing leaf, driven by both homes вЂ” *shipped* <PrLink pr={1617} />

<Phase id="R9В·lib" name="the shared sensing leaf (watchTerminalAwareness) + the library boundary" status="shipped" needs={["R6", "R4"]} links={[{ label: "#1617", href: "https://github.com/juspay/kolu/pull/1617" }]} />

**The gap, grounded.** The library (`@kolu/pulam-library`) is genuinely decoupled вЂ” its dependency arrow points **out** (`package.json`: `@kolu/surface`, kaval, the agent/git integrations; no kolu-app dependency) and it has six consumers. But it exported the sensing *parts* вЂ” the sensor engine `startAwareness`, the kaval-tap bridge `bridgeKavalTaps`, the per-terminal sink, the byte-tap tracker, the surface-skeleton factory `serveTerminalWorkspace` вЂ” never the *one thing that wires them per terminal*. So the **pulam daemon** assembled them one way (inside `runPulamDaemon`, with the sink + tracker stranded in the daemon package) and **kolu** assembled a divergent subset another way inside `local.ts`: its own inline four-channel build, its own `startAwareness` call, its own sink. **Two assemblers, assembled differently** вЂ” the root of the local-vs-remote divergence.

**The shape that shipped вЂ” ONE shared sensing leaf, two per-home loops.** The genuinely-shared volatility is "sense ONE terminal": given a dialed kaval + an already-seeded record + an injected sink + a per-terminal activity hook, wire `bridgeKavalTaps в†’ cwd-persist в†’ startAwareness в†’ raw-output activity tap`. That, and only that, is the library leaf:

```
watchTerminalAwareness({ kaval, id, record, sink, activity, log }) в†’ stop()
```

It bakes in **none** of what is legitimately per-home вЂ” *discovery*, *record-seeding*, the *sink implementation*, the *served collection*, the *activity-tracker ownership* вЂ” so each home injects those and drives the leaf from its **own** loop:

| concern | the pulam daemon (`awarenessLoop.ts`) | kolu-server (`local.ts`) |
| --- | --- | --- |
| discovery | **polls** `kaval.terminal.list` and reconciles | fires on its own **spawn / wake / adopt** events; departure on the **exit** tap (kolu-only вЂ” the daemon has no exit tap) |
| seed | `seedAwarenessValue(cwd)`, published before the leaf runs | the **restored registry** record (`getTerminal(id).awareness`) вЂ” never a fresh seed (the fold-clobber fix) |
| sink | the plain library `makeAwarenessSink` (publish the whole value) | its **richer fold sink** вЂ” the persisted/live `terminals:dirty` fence + `trackRecent` + screen-scrape вЂ” stays kolu-side |
| served collection | a cache-backed store it owns | the registry projection (the sink writes through `workspaceSurfaceCtx`) |
| activity tracker | one per host; served live via `liveActivity` | one per host on the endpoint; served live via `localTerminalActivity` |

The leaf **never re-seeds or over-publishes** the record it is handed вЂ” it only reacts to taps through the injected sink вЂ” so a restored cwd / git / agentSession survives untouched on adopt/wake/restore. That is the exact defect that killed the first local-pulam-process cut, now structurally impossible.

**Activity went live (R9.0), without double-sourcing the dot.** kolu's served `terminalWorkspace.activity` was a constant-empty `quietActivity`; now the leaf's raw-output tap feeds kolu's own per-host tracker, so the served stream is honest (`localTerminalActivity`). kolu's **client** green dot stays the local `useTerminalActivity` вЂ” it has a resize/reveal `suppress` window the server tap structurally cannot replicate, and it lights instantly off the local xterm sink. The two are **not** a double-source: kolu's client paints LOCAL tiles from `useTerminalActivity`; the served stream's consumers are REMOTE viewers (pulam-web's fleet board today, R9.3's mirror tomorrow) вЂ” a disjoint set. Retiring the client path is a separate later refactor that must first solve server-side resize-suppression.

<Callout kind="note" title="Design calls this phase settled (and why createPulam was dropped)">
The first cut shipped a maximal `createPulam(kaval)` that owned the *whole* per-terminal loop (seed в†’ sink в†’ tap-bridge в†’ sensors в†’ activity tap в†’ **reconcile**) and a two-step `start(collection)`; the daemon wrapped it. The re-review rejected that boundary: its poll-reconcile loop **complected** "sense one terminal" (shared) with "discover terminals + seed + which sink + which collection" (per-home). kolu could not reuse it вЂ” its sleeping tiles would be poll-removed, its restored awareness re-seeded/clobbered, its fold sink baked out. So the shared unit is the **leaf**, not `createPulam`; `createPulam` and its two-step are **gone**. Each home owns its own discovery loop and injects the rest. The earlier "how much does createPulam own" / "where does the sink write" / "does activity go live" scope calls are all settled by this shape; the rename rode this phase too.
</Callout>

- **Needs:** R6 (the library exists) and R4 (the pulam daemon) вЂ” **both shipped**.
- **Done:** there is **one** shared per-terminal sensing leaf in the library (`watchTerminalAwareness`); its internals (`startAwareness`, `bridgeKavalTaps`) are not even on the export surface; the daemon polls + leaf (`awarenessLoop.ts`, no awareness logic of its own); kolu fires the same leaf from spawn/wake/adopt (`local.ts`), its inline channels + `startAwareness` call deleted, its fold sink kept and injected; kolu's served activity is live; the reader-join + disk persistence are byte-unchanged. A **structural** guard (`pulam-library/src/sharedLeaf.assembler.test.ts`) walks **both** homes' import graphs and rejects *every* form that reaches a sensing primitive (named вЂ” keyed off the imported symbol, so an alias can't hide it вЂ” namespace, dynamic `import()`), over a denylist **derived from the library** so a new primitive can't drift past вЂ” and asserts the **differential** the original R9В·lib done-criterion wanted: *daemon path == kolu path == the one `watchTerminalAwareness` leaf*, neither re-rolling it. The daemon's real-kaval `daemon.test.ts` e2e (serve correct awareness end-to-end, reconcile on kill) stays green.

## History

- **2026-06 В· R9В·lib + R9.0 вЂ” createPulam в†’ the shared leaf** <PrLink pr={1617} />. The first cut of R9В·lib lifted the per-terminal assembly into a maximal `createPulam(kaval)` the daemon wrapped, returning `{ served, start(collection) }` with a built-in poll-reconcile loop. The perfection re-review found that the **wrong boundary**: `createPulam`'s loop complected "sense one terminal" (genuinely shared) with "discover terminals + seed the record + which sink + which served collection" (legitimately per-home вЂ” the daemon *observes* by polling; kolu *owns* via spawn/sleep/wake/adopt). kolu could not reuse it without its sleeping tiles being poll-removed and its restored awareness re-seeded. So the maximal assembly was **replaced** by the per-terminal **leaf** `watchTerminalAwareness`, and R9В·lib + R9.0 were finished **together**: each home owns its discovery loop (the daemon polls in `awarenessLoop.ts`; kolu fires from its lifecycle events in `local.ts`) and injects the seed/sink/collection/tracker, while the leaf owns only the sensing. `createPulam` + its two-step `start()` were deleted; the structural guard generalized to assert *both* homes rest on the leaf (the differential test the original criterion wanted, now that kolu is the second home).

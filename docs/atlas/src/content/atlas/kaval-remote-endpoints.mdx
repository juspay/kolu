**The forward half of [kaval-sessions](kaval-sessions.html).** That note carries the R-2 direction and the shipped spike (P0–P2.8, where `kaval-tui --host` already dials + provisions a remote kaval over ssh); this one is the implementation-ready plan for the two phases still ahead — **P3** (kolu-server itself dials a *remote* kaval, so a canvas tile can live on another host) and **P4** (the multiplex canvas on top). The substrate it builds on — *one backend, three surfaces (PTY · fs · git) bound to an endpoint*, reached by a unix-socket driver (local) or an ssh-stdio `HostSession` (remote) — is established in the parent; here it gets built.

## The remote process model (P3)

**kaval and kolu-watcher are two separate processes** — forced, not incidental. kaval is the durable survivor (it must outlive deploys); **kolu-watcher** (provider DAG + fs/git) re-runs fresh (it must be the current build's code — the #1031 line); **one process cannot be both.** Locally that role isn't even a separate process — the DAG + fs/git watchers run *inside* `kolu-server` (`local.ts`: *"the DAG runs HERE, in kolu-server"*), with kaval the separate daemon. Remotely, `kolu-server` isn't on the host, so it splits out as its own **kolu-watcher** process beside kaval.

And because **kaval serves a *unix socket*** (`serveOverUnixSocket`), reaching it over the network needs a remote *stdio* endpoint — and a `HostSession` is one ssh subprocess per host. So the remote host exposes **one** ssh endpoint — **kolu-watcher**: `kolu-server` dials it over one `HostSession` `stdioLink`, and kolu-watcher locally **fronts kaval** (its unix socket) for PTY attach/control + taps while serving its own fs/git + metadata surface (`mirrorRemoteCollection`). Concretely that is **one *composed* `@kolu/surface` router served over `serveOverStdio`** — kaval's `ptyHostSurface` contract *proxied through* (kolu-watcher is a client of kaval's socket) plus the watcher's own fs/git/metadata — **not** the contract-blind `frontDaemonOverStdio` byte-relay `kaval-tui` uses (P2/P2.5): that relay eats the whole stdio to reach kaval alone, leaving no room for fs/git on the same link, so kolu-watcher cannot use it. The closure is shipped first via `provisionAgent` (`nix copy --derivation` → remote `realise` → pin a GC root). **If the host already has a kaval running, it is adopted** — the same generic `adoptOrEnsure` spine (`endpoint.ts`), per endpoint, that the local boot uses. A duplicate is impossible by construction: a second launch hits the pid-gate's atomic `link(2)` EEXIST, liveness-probes the holder (`kill(pid,0)`), and exits 0 ("yielding to the live instance") **without binding a socket** — the running daemon is never disturbed. Re-provisioning is idempotent (`nix realise` no-ops if the closure is already present; an older build coexists and the moving GC root migrates to the new one). On dial:

| the remote kaval is… | kolu does |
| --- | --- |
| absent | provision + spawn fresh (`adoptOrEnsure` → ensure) |
| live · wire-compatible | **adopt** — connect, never kill; its PTYs reconcile in (older `startedAt`) |
| live · a build behind | adopt **+** the B3.4 "update pending" nudge (an orthogonal axis) |
| live · contract-skewed | recycle (kill → respawn) — only on a typed `DaemonContractSkewError`, no retry |
| unreachable (bad dial) | bounded retry, then **degraded — never killed** (a bad dial can't cost live PTYs) |

**One kaval per remote host, adopted.** Dialing a host *joins* its kaval — you reconnect to "the prod terminals" — and never clobbers it. The local per-port key (`kaval-<port>/`, <PrLink pr={1313} />), which isolates two dev servers under always-recycle, isn't needed across ssh: a remote host runs **one** shared kaval and dials *adopt* it (the table above), so two reconnects join the same terminals rather than racing to kill them. *(The pid-gate is host-local — pid + filesystem on that host — not a cross-host or cross-container lock.)*

## Implementing P3 — eight additive phases

P3 is the largest phase: a re-introduced dispatch seam, additive wire/record fields, a brand-new host-resident process (**kolu-watcher**), the remote endpoint binding, three remote-only seams, and remote adoption. **Ship it as eight sub-PRs, not one** — the cadence the P2.x series already proved. Each phase is *either a pure-additive change that keeps every local terminal byte-identical* **or** *a small user-facing increment*, and the whole sequence is strictly additive — a local tile behaves identically at every step. The seam is already cut: `TerminalEndpoint`'s own doc-comment names "Remote (P3)" as "the same shape, a different transport," and **P0 (<PrLink pr={1364} />) left three call sites commented `// P3 will select the endpoint…`**. P3.1 is the literal inverse of that commit.

- **P3.1–P3.2 are pure refactors** — byte-identical, the suite stays green for the same reason P0 landed standalone.
- **P3.3–P3.4 add new, isolated code** that changes no existing behavior (a contract type, a nix-config mirror, then the kolu-watcher process behind a throwaway spike) — additive, not yet user-reachable.
- **P3.5–P3.8 each deliver one small user-facing increment** — a remote tile, a correct remote shell, the Code-tab + pills lighting up, surviving a restart.

### The three remote seams

The "carries the remote seams" line, resolved concretely — each is small, and each is already de-risked by an existing pattern:

- **`composeSpawnInput` inversion (P3.6).** Today the rich spawn policy (`composeSpawnInput`, wrapped by `buildTerminalSpawnInput` in `server/src/ptyHost/index.ts`) layers env with the **local** kolu-server process winning over the daemon's `system.info` — correct on one machine, **wrong** over ssh (it would leak the local `$HOME`/`$PATH`/`/nix/store`). The remote sibling **inverts the precedence**: the remote daemon's `system.info` wins and only a presentation whitelist (`TERM`/`LANG`/`LC_*`/`COLORTERM`) rides through. This is exactly `buildRemoteCreateInput` + `REMOTE_ENV_PASSTHROUGH` from kaval-tui (P1/P2) — P3 lifts the pattern into kolu-server and routes by endpoint kind. The wire stays self-contained: no var refs, host-relative rc paths.
- **paste/upload `PRECONDITION_FAILED` guard (P3.6).** `pasteImage`/`uploadFile` (`router.ts`) save into a **local** scratch dir and bracketed-paste that local path into the PTY — unreachable from a remote shell (a silent `ENOENT` the agent would chase). P3's scope is the **guard, not the fix**: inspect the tile's recorded `location` (on the record after P3.2) and throw `PRECONDITION_FAILED` on a remote tile. The actual copy-to-remote-scratch is **P4**.
- **`drvPath` via `resolveSystem` (P3.3 + P3.5).** kolu-server must ship the right-arch kolu-watcher closure: `resolveSystem(host)` (the P2.8-memoized ssh arch probe) → a `{system → .drv}` map baked onto the server wrapper as `KOLU_AGENT_DRVS_JSON` (the exact mirror of kaval's `KAVAL_AGENT_DRVS_JSON`), fed to `getHostSession`. The pool key is `(host, binary)` — **one resolver per host**.

### Where P3 stops and P4 begins

P3 makes a remote terminal **correct and observable on the server**; P4 makes it **discoverable, controllable, and fully featured in the UI**. So P3 ends with `hostId` on the wire that the client *reads but renders nothing host-aware* — tiles, dock and chrome are visually **byte-identical** for local and remote. **P4** adds the active-kaval strip, the per-tile host hint, the ssh-config picker, the reconnect affordance (a button wired to `session.reconnect()`), and the real paste/upload-to-remote that turns P3.6's guard into a feature. The sharp line: anything that **renders** a host indicator, a picker, or a recovery control — or **transfers a file** to the remote host — is P4.

### Open calls — decide at implementation

Genuine design calls left to the implementer, each with a recommendation:

1. **Where fs/git lives** — kolu-watcher runs `kolu-git` natively + the DAG in-process; **kaval stays a pure PTY host** (co-locates agent detection with the JSONL files it reads). *Recommended.*
2. **Remote gate read** — an **RPC-baked `system.gateStatus()`** on the watcher contract, not an ssh-exec subprocess; bake it into the P3.3 contract so P3.8 needs no wire bump. *Recommended.*
3. **Pooling** — **one `HostSession` per remote host**, keyed `(host, 'kolu-watcher')`, shared across the PTY + fs/git + metadata pumps. *Recommended.*
4. **Create-time selection** — explicit `location` > parent inheritance > **default-local**; the interactive picker is P4. *Recommended.*
5. **Remote `list` failure** — **degrade, don't recycle** (a pure-dial remote may not be kolu's to own — unlike the local F3 "recycle on list failure", which assumes kolu owns the lifecycle). *Recommended; settle in P3.8.*

## P4 — the canvas: switch or multiplex

With P3 the server can hold N live endpoints; **P4 spends that on the UI — and settles the one open granularity call.** The same substrate answers either, so it is a **later, reversible** choice that the P1–P2 spike deliberately did not depend on:

| | switch (tmux-style) | multiplex (canvas) — leaning |
| --- | --- | --- |
| the canvas | one kaval in view | tiles span kavals |
| per-tile `location` | unneeded | additive — the R-1 discriminator, finally used |
| live endpoints | one, swapped | N concurrent |
| host hint | one *current* pointer | a light per-tile tag; ChromeBar shows the *active* kavals |
| why it fits kolu | simplest | the canvas is spatial + heterogeneous *by design* |

The lean is **multiplex**: tmux-switch is a *terminal-multiplexer* metaphor, but kolu is a *canvas* — "watch a local build beside a prod-ssh tile" is the point of a canvas, not a corner case. The cost over switch is modest and clean — per-tile `location` returns, but **additively** (no `RemoteTerminalBackend`, no screen mirror — just *which endpoint holds this tile's PTY*), and the ChromeBar shows only the kavals this canvas *actually has tiles on*.

**P4 builds multiplex, then validates the feel.** The UI work consumes exactly the server substrate P3 left: the **active-kaval strip** (local always present), the **per-tile host hint**, the **ssh-config picker** (how a user *chooses* a remote host — P3's selection is config/env-driven, not interactive), the **reconnect affordance** wired to `session.reconnect()` and tooltip'd from `lastError`, and turning P3.6's paste/upload `PRECONDITION_FAILED` guard into a **real feature** (copy the file to the remote scratch dir, bracketed-paste the *remote* path). Then the validation: **does N-at-once actually beat tmux-switch in daily use?** Fall back to switch if multiplex doesn't earn its keep — the granularity call lands **and is validated** here, on real use, not up front.

**kaval decoupled the PTY out of kolu into a standalone, minimal, durable daemon with a clean contract — independently shippable, independently driven by `kaval-tui`.** This note does the *same move one level up*, for the part kolu kept tangling: **terminal awareness** (git branch/dirty, PR status, agent detection, foreground). A new ephemeral daemon — **`arivu`** (Tamil *அறிவு*, "the faculty of knowing" — a sibling to `kaval`, *watch/guard*, and `odu`, *run*) — sits *on top of* kaval, runs the providers, and exposes its result as one typed `@kolu/surface` collection. Because a surface can be either *composed in-process* or *served over a link*, the same contract has two wirings chosen by where the terminal lives — and kolu becomes a pure consumer of awareness, never the place it tangles.

## The core model

One thesis: **awareness is a derived surface, and the thing that produces it is a standalone daemon on top of kaval.** Three claims follow, each verified against the shipped code.

- **The unit is the provider-authored *slice*, not the whole record.** `arivu` exposes `Collection<TerminalId, AwarenessValue>` where `AwarenessValue` is exactly the fields the providers compute — `cwd · git · lastAgentCommand · lastActivityAt` (server-persisted) plus `pr · agent · foreground` (live). That is a strict `.pick` of *existing*, partition-tested schemas: **zero new schema work**. It deliberately excludes `location` (no provider authors it; a remote tool can't know its own kolu-side `hostId`) and every client/UI field (`themeName`, `intent`, `canvasLayout`, …) — those are kolu-owned and unproducible host-side.
- **kaval stays byte-for-byte minimal.** `arivu` *dials* kaval as a `ptyHostSurface` client for the four taps + `getScreenText`; it adds **zero** awareness/git/gh logic to kaval. The decoupling is additive — a layer above, never a fork.
- **Ephemerality is the simplifier.** Unlike a PTY, awareness never has to survive a restart — it's re-derivable from live taps + current host fs. So `arivu` sheds *all* of kaval's durability machinery: no single-instance lock, no spawn/cleanup (it owns no PTYs), no persisted list (it borrows kaval's `terminal.list`), no adoption or reconnect-survival. Every (re)start just re-runs the providers and recomputes from now.

The payoff: **clients keep reading the one `terminalMetadata` collection they always did.** The awareness surface is an internal seam, not a new thing the browser learns.

## One contract, two wirings — and why remote needs a fold

The same surface contract is wired two ways, selected by locality. This is the honest version of *"local vs remote is only the transport"* — with one correction the design panel forced.

- **Local → compose, in-process.** The providers run *inside* kolu-server exactly as they do today, mutating the terminal's `meta` in place and publishing `terminalMetadata`. The daemon's surface is `implementSurface`'d and consumed via `directLink` — which only *formalizes the call boundary that already exists*. No socket, no serialization, no fold. **And local must stay this way:** the in-process providers are always the running build, so awareness is current *by construction*.
- **Remote → serve, mirror, fold.** `arivu` is Nix-provisioned over ssh (the same staleKey recipe kaval already uses), runs the *identical* providers, and **serves** the slice over `stdioLink`. kolu-server **mirrors** it with the already-shipped `mirrorRemoteCollection`, then **folds** each upsert into the terminal's `terminalMetadata` entry.

<Callout kind="warn" title="The mirror is a fold, not a write — and that's structural, not incidental">
`terminalMetadata`'s entry is **one mutable object** that publishes the *whole* record — server fields **and** kolu-owned client fields **and** the host-stamped `location`. So an arriving awareness slice can never whole-value-replace the entry (that would clobber the client's theme/layout and the location). It must be **merged in and split** across the persisted-vs-live fence: the live subset (`pr · agent · foreground`) through the quiet mutator, the persisted subset (`cwd · git · lastAgentCommand · lastActivityAt`) through the one that arms autosave. Lose that split and you either re-arm the ~150 ms agent firehose (a perf regression) or silently stop persisting. This fold is *irreducible domain logic* — it exists because `terminalMetadata` is co-owned, and no generic mirror can absorb it.
</Callout>

So the asymmetry is **encapsulated, not erased.** Local is a closure-mutate; remote is closure → frame → serve → mirror → fold. They share the *invariant providers and one schema* — but they are two paths, and pretending otherwise hides the one genuinely-new artifact (the host-side hooks impl that reifies each `mutate` closure into a serialized slice frame — a wire carries values, not closures). `location` is stamped kolu-side from the dial-list at the fold.

> **No `@kolu/surface` change is required.** The compose-or-serve duality already ships end-to-end and is proven by the `remote-process-monitor` example (*define the surface once, implement it twice* — `directLink` in-process, `stdioLink` over ssh, `mirrorRemoteCollection` to mirror). A future graduation — `mirrorRemoteSurface`, the spec-driven dual of `implementSurface` that drives every primitive of a served surface into a local ctx — is attractive *electricity* (the demo would delete its hand-rolled bridge loop), but it is **not** on the critical path and it does **not** dissolve the fold.

## The standalone tool — `arivu` + `arivu-tui`

Like kaval, the daemon is **a deliverable in its own right**, shipped and proven *before* kolu touches it. Where `kaval-tui` shows what's *running* in each PTY, `arivu-tui` shows what each terminal *is in* — a host-local "what is every agent doing, across every repo" dashboard with **zero kolu-server dependency**, dialable over ssh against a prod box.

Each slice ships and is provable on its own; the long pole is **P1 → P2 → P3**.

## P1 in detail — ship `arivu` standalone

P1 is the whole daemon, end to end, needing **no kolu-server at runtime** — it stands or falls on its own, exactly the way `kaval` did. (P1 makes exactly *one*, behavior-preserving change *to* kolu-server — re-pointing an import, below — so there's a single shared copy of the providers, never a duplicate; nothing kolu-server *does* changes until P2.) Two binaries fall out of one contract.

### What you get — user-facing

Two commands, mirroring `kaval` / `kaval-tui`:

- **`arivu`** — the daemon. Point it at a kaval socket and it serves live awareness for every terminal that kaval owns:
  ```
  arivu --kaval /run/kaval.sock           # serve on a local socket
  arivu --kaval /run/kaval.sock --stdio   # serve over stdio (what an ssh dial speaks to)
  ```
- **`arivu-tui`** — the viewer. Dial a local `arivu`, or one across ssh:
  ```
  arivu-tui list                # the dashboard — every terminal, one row each
  arivu-tui list --host prod    # … the same, on a remote box, over ssh
  arivu-tui watch a3f1          # follow one terminal live
  ```

The experience is the `arivu-tui` mockup above: per terminal — repo + branch, dirty or clean, the open PR and its checks, which AI agent and whether it's *working* or *waiting on you*, the foreground process — updating live. **Zero kolu-server, zero browser**; the runtime is just `node · git · gh`. It's a "what is every agent doing, across every repo" pane you can `ssh` into — useful the day it ships, before kolu depends on a line of it.

### How it's built — architecture + types

**One contract**, defined once in a tiny package both binaries import. The value type is *derived*, never authored:

```ts
// @kolu/arivu-contract — the single source of truth for all consumers
import { TerminalIdSchema, TerminalServerMetadataSchema } from "kolu-common/surface";
import { defineSurface, type SurfaceTypes } from "@kolu/surface/define";

// the provider-authored slice = server metadata minus the one field no provider owns
export const AwarenessValueSchema = TerminalServerMetadataSchema.omit({ location: true });
// ≡ { cwd, git, lastAgentCommand?, lastActivityAt, pr, agent, foreground }

export const arivuSurface = defineSurface({
  cells: { version: { schema: VersionSchema, default: DEFAULT_VERSION } }, // the P3 contract-version handshake
  collections: {
    awareness: { keySchema: TerminalIdSchema, schema: AwarenessValueSchema },
  },
});
export type AwarenessValue =
  SurfaceTypes<typeof arivuSurface.spec>["collections"]["awareness"]["Value"];
```

**The daemon** dials kaval as a plain `PtyHostClient` — kaval is untouched, `arivu` is just another client of it, like `kaval-tui` — and for each terminal in kaval's `terminal.list` runs the *existing, unchanged* providers:

```ts
import {
  startProviders,
  type ProviderRecord, type ProviderChannels, type ProviderHooks,
} from "@kolu/terminal-providers"; // the provider set, lifted out of kolu-server verbatim

function watchTerminal(id: TerminalId, kaval: PtyHostClient, ctx: ArivuCtx): () => void {
  const record: ProviderRecord = { pid, meta: seedAwareness(cwd), currentAgent: null };
  const channels: ProviderChannels = bridgeKavalTaps(kaval, id); // cwd·title·commandRun·foreground (git made internally)
  const publish = (m: TerminalServerMetadata) =>
    ctx.collections.awareness.upsert(id, AwarenessValueSchema.parse(m)); // drops location by construction
  const hooks: ProviderHooks = {
    // THE one seam that differs from kolu-server: instead of mutating kolu's entry.meta,
    // mutate this record's meta, then publish the (location-less) slice into the served collection.
    updateServerMetadata:     (r, mutate) => { mutate(r.meta); publish(r.meta); },
    updateServerLiveMetadata: (r, mutate) => { mutate(r.meta); publish(r.meta); },
    readScreenText: (n) => kaval.surface.terminal.getScreenText({ id, tailLines: n }),
  };
  return startProviders(record, id, channels, hooks, ctx.log); // logger injected — the one coupling severed in the lift-out
}
```

It then **serves** that surface — `serveOverStdio({ router, transport })` for the `--stdio`/ssh case, a local socket otherwise. **`arivu-tui`** is a thin surface *client*: it dials the daemon, reads `awareness` (snapshot-then-delta, per the framework), and renders rows.

Two things to notice, because they are exactly what P1 *proves*:

- **The reuse seam is the provider set, not the hooks signature — and it's a _move_, not a copy.** `startProviders` / `ProviderRecord` / `ProviderChannels` / `ProviderHooks` (and the kaval-tap→channel bridge) **relocate** from kolu-server into `@kolu/terminal-providers` — the only coupling to sever is the logger, injected as a param — and kolu-server re-points its own import at the package in the same change (behavior-preserving: it keeps running the providers in-process). So there is **one** copy of the freshness-critical provider computation, imported by both; P1 adds **no duplicate**. The only per-consumer code is the thin sink adapter — a `ProviderHooks` *impl*: publish-into-a-collection in `arivu` vs mutate-kolu's-`entry.meta` in kolu-server. That differing impl is the variation point the interface exists for, not duplication — and it is exactly what P3 reuses on the far side of the ssh link.
- **No fold here.** In P1 the daemon owns the *whole* `AwarenessValue` — there is no co-owned `terminalMetadata`, no client fields, no `location`. The fold (merge + persisted/live split) is *kolu-side* and first appears in P2. P1 is the clean, fold-free proof that the providers run correctly as a separate, kaval-dialing process — the single biggest unknown, retired in isolation.

**Deps & shape:** `@kolu/terminal-providers` already imports nothing from kolu-server; the daemon adds only the surface runtime + a `PtyHostClient`; agent-session reads use Node's built-in `node:sqlite` (no native addon). So `arivu` provisions and travels over ssh by the same staleKey recipe kaval uses — which is the whole reason to prove it here, before P3 introduces the wire.

## Load-bearing complexity & open decisions

The cleanest design still carries four irreducible costs — **all remote-only**, none dissolvable, and that confinement is the point:

1. **Closure → frame reification** — the one genuinely-new artifact: the host hooks-impl applies each `mutate` to its own record, then emits the resulting slice value as a snapshot/delta frame. The providers are untouched.
2. **The fold + persisted/live fence**, re-applied kolu-side on every remote upsert (see the callout). Frames must carry the split, or it degrades to runtime discipline.
3. **Freshness / semantic version-skew** — the deepest. In-process local providers are always current; a remote host runs a provisioned, possibly-stale binary. A version handshake catches *shape* skew, not *semantic* drift (a new agent state an old parser never emits). Bounded by re-provision-on-dial + ephemerality, but only re-provisioning truly closes it — which is *why* local must never route through the daemon.
4. **`location` stamped kolu-side** at the fold — a small permanent special-case at the seam.

*(Cold-start recompute storms and a two-hop key-set lag are operational, not architectural — bounded by idle-gating and the existing `pr:pending` / `agent:null` seed.)*

**Open decisions** (defaults in **bold**): the persisted/live split is carried by **a static field→half map applied kolu-side** (thinner wire) vs per-frame tags; **one `arivu` per host** (mirrors kaval's per-host socket, maximizes the dashboard) vs per-session; **staleKey-gated re-provision** (kaval's existing behavior) vs always-on-dial for the tightest semantic currency; build `mirrorRemoteSurface` **when a second served-surface consumer appears** vs now.

## Status

**Proposed — a clean-slate replacement for the awareness layer**, child of [kaval-sessions](kaval-sessions.html). It supersedes the earlier from-scratch plan (<PrLink pr={1398} />, closed) and the in-process provider extraction (<PrLink pr={1409} />, closed): both kept awareness *inside* kolu and split hairs over an in-process compose-vs-serve boundary that this design dissolves by moving the producer into a daemon. Built from an adversarial, cost-blind design panel (verdict: *sound, with the asymmetry-is-encapsulated and mirror-is-a-fold corrections folded in above*), grounded by reading the shipped code: `@kolu/terminal-providers`, `mirrorRemoteCollection`, the `remote-process-monitor` "define once, implement twice" demo, and kaval's provisioning recipe. The substrate it stands on — kaval (untouched), `location` as a first-class discriminated union — is already merged. The long pole is **P1 → P2 → P3**, and **P1 (the standalone `arivu` + `arivu-tui`) is the recommended first move**: independently useful, and it retires the biggest risk before kolu depends on a line of it.

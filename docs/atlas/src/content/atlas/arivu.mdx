**kaval decoupled the PTY out of kolu into a standalone, minimal, durable daemon.** This note makes the *same move one level up* for the part kolu kept tangling: **terminal awareness** (git branch/dirty, PR status, agent detection, foreground). A new daemon — **`arivu`** (Tamil *அறிவு*, "the faculty of knowing"; sibling to `kaval` = *guard* and `odu` = *run*) — sits *on top of* kaval, runs the sensors, and exposes the result as one typed `@kolu/surface` collection. **One sensor library, two homes:** locally kolu runs it in-process and writes `terminalMetadata` straight through the existing mutators (no surface, no fold); remotely the `arivu` daemon serves the same awareness over ssh, which kolu mirrors and folds in. Clients keep reading the one `terminalMetadata` collection they always did.

## The core model

Three claims, each verified against the shipped code:

- **The unit is a generic awareness value, not kolu's record.** `arivu` exposes `Collection<TerminalId, AwarenessValue>` — exactly `cwd · git · lastAgentCommand · lastActivityAt` plus `pr · agent · foreground`. That schema lives in `@kolu/terminal-awareness`, built from the vendor-neutral `anyagent` / `anyforge` / `kolu-git` shapes with **no kolu content** — not even `location` (no sensor can know its own kolu-side `hostId`). Kolu's `TerminalServerMetadata` is `AwarenessValue` **plus** `location` and UI fields: built *on* it, never carved *from* it.
- **kaval stays byte-for-byte minimal.** `arivu` *dials* kaval as a `ptyHostSurface` client for the four taps + `getScreenText`; it adds zero awareness/git/gh logic to kaval. A layer above, never a fork.
- **Ephemerality is the simplifier.** Awareness never has to survive a restart — it's re-derivable from live taps + current host fs. So `arivu` sheds all of kaval's durability machinery: no single-instance lock, no spawn/cleanup, no persisted list (it borrows kaval's `terminal.list`), no adoption or reconnect. Every (re)start just re-runs the sensors from now.

## One library, two homes — and why only remote folds

Local and remote share the *sensor library* and *one schema* — but they are genuinely different paths, not "only the transport." The surface and the fold exist only where awareness crosses a process boundary:

- **Local → in-process, no surface.** kolu-server runs the sensors and writes `terminalMetadata` directly. Same process = nothing to serve, nothing to merge; wrapping it in an in-process surface is the fictional boundary rejected in #1406. Local stays this way for good — in-process sensors are always the running build, fresh by construction. Even after remote ships, local does **not** switch to dialing a local arivu "for consistency" (pure ceremony to move data within one process). The consistency that matters — same library, schema, contract — is already there.
- **Remote → serve (arivu), then mirror + fold (kolu).** `arivu` is Nix-provisioned over ssh (kaval's staleKey recipe), runs the identical sensors, and **serves** the slice over `stdioLink`. The kolu-side **mirror + fold** is *not* arivu's concern — it is [remote-terminals](remote-terminals.html) **R7**'s, and the fold's costs live there.

So the asymmetry is **real and confined to the wire**: local is closure → mutate; remote is closure → frame → serve → mirror → fold. They share the invariant sensor library and one schema. The one new artifact is the host-side hooks impl that reifies each `mutate` closure into a serialized slice frame — a wire carries values, not closures.

> **`mirrorRemoteSurface` has graduated** (<PrLink pr={1497} />). It is now the one public mirror in `@kolu/surface` — the spec-driven dual of `implementSurface` that drives *every* primitive of a served surface into caller-supplied sinks; `mirrorRemoteCollection` is demoted to its private per-key engine ("you don't sell half a house"). The trigger was arivu growing a real **`activity` stream** (the green dot) — the second stream-bearing consumer the graduation waited for. It still does not dissolve kolu's R7 fold (irreducible domain logic); the fold simply consumes `mirrorRemoteSurface`. As a `@kolu/surface` API change it rode `surface.md`'s drishti merge-gate.

## The standalone tool — `arivu` + `arivu-tui`

Like kaval, the daemon is a deliverable in its own right, shipped and proven *before* kolu touches it. Where `kaval-tui` shows what's *running* in each PTY, `arivu-tui` shows what each terminal *is in* — a "what is every agent doing, across every repo" dashboard with **zero kolu-server**, dialable over ssh against a prod box.

## R4.5 — the fleet board

The user's framing made literal — *"what is every agent doing, across every repo, across every machine"* — as a **dashboard you leave open on a second monitor**: glance over from the game and a colour you can read across the room tells you whether any agent is **blocked on you**, and where. It stays a TUI, runs zero kolu-server, and shipped as four steps:

- **R4.5.1 — Bun runtime** <PrLink pr={1470} />. Re-platform `arivu-tui` onto **Bun** via [drishti](electricity.html)'s bun2nix recipe — pinned via npins, resolved with `builtins.getFlake` (the `odu` path, so kolu's zero-flake-input rule holds), `bun.lock` + autogenerated `bun.nix` kept outside the pnpm workspace, `@kolu/*` hydrated from the Nix store. Today's `list`/`watch` run unchanged. The foundation lands first so the OpenTUI bundle sits on proven ground.
- **R4.5.2 — OpenTUI render, one endpoint** <PrLink pr={1479} />. Retire `list`/`watch` + the `columnify` text; render a SolidJS-canonical OpenTUI list of one endpoint's terminals (the already-shipped `--socket` or single `--host`, reused verbatim — no new connection capability), with a **one-second clock** beside it as the liveness proof. The list itself is a one-shot snapshot; live deltas are R4.5.3's. `--json` dumps the awareness array.
- **R4.5.3 — the multi-host fleet board** <PrLink pr={1486} />. Fan the one-shot dial over N hosts, mirror each `awareness` collection into one aggregate keyed by `(host, terminalId)`, go **live** (repaint on any mirror delta), float every `awaiting_user` agent to the top across the fleet, and render per-host groups + a breathing alert strip + honest unreachable/skew/empty states.
- **R4.5.4 — the live green dot + full-surface mirror** <PrLink pr={1497} />. `@kolu/arivu-contract` grows its first **`activity` stream** (terminals moving bytes right now, from kaval's raw output tap); the board paints each a **live green dot** and drives the whole surface — version cell + `awareness` + `activity` — through one `mirrorRemoteSurface` call. Contract bumped `0.1 → 0.2` (additive); rode `surface.md`'s drishti merge-gate.

## Why this stays a TUI, and what is electricity vs. a leaf

The fleet board is the awareness analog of **drishti**. Per [electricity](electricity.html), `@kolu/surface` proved it was real electricity by graduating to a second consumer (drishti's fleet process-monitor). arivu's fleet is the exact same shape one layer up — a dashboard that mirrors N remote arivu daemons into one host-keyed view, proving arivu's **own** surface is itself a receptacle a different app plugs into. The clincher: the fan-out + aggregation needed **no contract change** to exist (R4.5.3 mirrored the same `awareness` collection across N hosts). That invariance under a brand-new multi-host consumer *is* the electricity proof. (R4.5.4's `activity` stream came later, as an additive *feature*, not a tax the multi-host consumer levied.)

Two boundary calls keep it honest:

- **The renderer is electricity — but it already exists as OpenTUI.** A domain-agnostic Solid→terminal capability, so R4.5 *consumes* OpenTUI per *prefer-libraries* rather than minting a `@kolu/solid-tui`.
- **The aggregation join is a leaf reducer, not electricity.** It hides a bounded host-keyed merge, not a hard volatility (transport + provision + reconnect already belong to `@kolu/surface-nix-host`; the per-primitive mirror to `@kolu/surface`). So no `@kolu/fleet-mirror` — that is exactly the "tidy generic helper" trap electricity warns against.

<Callout kind="good" title="It stays a TUI, not a surface-app">
A read-only aggregator has nothing to fold and nothing to persist, so a browser shell with a stale-tab/reconnect story is pure ceremony for a glance view. The TUI ships fast, is dev-loop-native, and proves the aggregation core cheaply. A web `arivu-board` (the literal drishti twin — browser ↔ Bun ↔ N ssh dials) is an attractive *follow-on*, explicitly second, never the headline. And a team-wide read-only monitor is a **different product** from kolu's own-canvas fold ([remote-terminals](remote-terminals.html) R7) — so it belongs as arivu's own consumer, with zero kolu-server, not folded into kolu.
</Callout>

## Why Bun, contained to the viewer

OpenTUI is the production-grade TUI framework (it powers OpenCode and terminal.shop) and ships a **Solid reconciler** (`@opentui/solid`), so the dashboard is authored in the same fine-grained-reactive model kolu already lives in. The catch — and the reason the runtime swap came first — is that OpenTUI's renderer is a native **Zig** core loaded via `Bun.dlopen`: it needs Bun.

So the split is narrow and deliberate: **`arivu-tui` alone** becomes a Bun binary; everything else, including the `arivu` **daemon**, stays Node. The daemon is the half that travels over ssh and leans on `node:sqlite`, so it must stay Node (and stays cross-arch). The viewer is never provisioned over ssh, so the Bun runtime + per-arch Zig lib live only in the local closure. The packaging is reused, not new ground — it is drishti's bun2nix recipe wholesale.

<Callout kind="warn" title="OpenTUI + Solid build gotchas (learned the hard way in R4.5.2)">

- **OpenTUI repaints only the CHANGED CELL.** Verify "is it live?" with **raw-PTY capture** (frame N+1 ≠ frame N), never `snapshot | grep` or a glance — both read a live UI as frozen.
- **Bundle `solid-js`; do NOT mark it `external`.** `@opentui/solid`'s plugin redirects solid-js to its reactive client build only when solid is loaded *during the build*. Marking it external ships the non-reactive *server* build and every live value silently freezes at runtime (invisible in dev, where the `bunfig.toml` preload redirects at load time). Use `external: ["@opentui/core"]` only.
- **Pre-bundle; don't run from source.** `build.ts` (`Bun.build` + `@opentui/solid/bun-plugin`) applies `babel-preset-solid` at build time; clean its `outdir` first (Bun.build leaves orphan chunks). The smoke that catches the freeze is running the **built `dist/bin.js`** and confirming a repaint.
- **SolidJS-canonical only** — a live value is `createSignal` + `onMount(setInterval)` + `onCleanup(clearInterval)`, read in JSX. No fps caps, `requestRender`, or `process.stderr` monkeypatching.
- **Forwarded daemon stderr** over `--host` is quieted **at the daemon** (`warn` over `--stdio`) so it can't corrupt the alt-screen; the held-open fleet dials route their logs through `@kolu/surface-nix-host`'s `onLog` sink.

</Callout>

## History

- The standalone story shipped end-to-end R4.1–R4.5: extract → daemon → remote viewer → Bun → OpenTUI → live fleet → live activity. Gauntlet + dogfooding added `--kaval <host>=<socket>` pinning, a responsive `repo·branch` layout, the `--host` stderr-bleed fix (`onLog` sink), and a darwin `bun install --backend=copyfile` fix.
- **Packaging co-located** (<PrLink pr={1491} />) — the viewer's Bun/Nix manifest moved from `nix/packages/` into `packages/arivu-tui/nix/` so the viewer is one self-contained directory (a pnpm-workspace exclusion keeps the bun manifest out of pnpm's glob).

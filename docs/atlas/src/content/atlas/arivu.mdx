**kaval decoupled the PTY out of kolu into a standalone, minimal, durable daemon with a clean contract — independently shippable, independently driven by `kaval-tui`.** This note does the *same move one level up*, for the part kolu kept tangling: **terminal awareness** (git branch/dirty, PR status, agent detection, foreground). A new ephemeral daemon — **`arivu`** (Tamil *அறிவு*, "the faculty of knowing" — a sibling to `kaval`, *watch/guard*, and `odu`, *run*) — sits *on top of* kaval, runs the sensors, and exposes its result as one typed `@kolu/surface` collection. Locally, kolu runs that sensor library in-process and writes `terminalMetadata` directly — no surface, no boundary. Remotely, the `arivu` daemon serves the same awareness over ssh, which kolu mirrors and folds in. **One sensor library, two homes** — and kolu becomes a consumer of awareness, never the place it tangles.

## The core model

One thesis: **awareness is a derived surface, and the thing that produces it is a standalone daemon on top of kaval.** Three claims follow, each verified against the shipped code.

- **The unit is a generic awareness value, not kolu's record.** `arivu` exposes `Collection<TerminalId, AwarenessValue>`, where `AwarenessValue` is exactly the fields the sensors compute — `cwd · git · lastAgentCommand · lastActivityAt` plus `pr · agent · foreground`. This schema is *generic*: it lives in `@kolu/terminal-awareness`, composed from the vendor-neutral `anyagent` / `anyforge` / `kolu-git` shapes, with **no kolu-specific content at all**. The one kolu concept — `location` (the endpoint discriminator) — is simply *absent* (no sensor authors it; a remote tool can't know its own kolu-side `hostId`). Kolu's own `TerminalServerMetadata` is then just `AwarenessValue` **plus** `location` and the client/UI fields (`themeName`, `intent`, …): the awareness value isn't *carved out of* kolu's record — kolu's record is *built on top of* it.
- **kaval stays byte-for-byte minimal.** `arivu` *dials* kaval as a `ptyHostSurface` client for the four taps + `getScreenText`; it adds **zero** awareness/git/gh logic to kaval. The decoupling is additive — a layer above, never a fork.
- **Ephemerality is the simplifier.** Unlike a PTY, awareness never has to survive a restart — it's re-derivable from live taps + current host fs. So `arivu` sheds *all* of kaval's durability machinery: no single-instance lock, no spawn/cleanup (it owns no PTYs), no persisted list (it borrows kaval's `terminal.list`), no adoption or reconnect-survival. Every (re)start just re-runs the sensors and recomputes from now.

The payoff: **clients keep reading the one `terminalMetadata` collection they always did.** The awareness surface is an internal seam, not a new thing the browser learns.

## One library, two homes — and why only remote folds

Local and remote share the *sensor library* and *one schema* — but they are genuinely different paths, not "only the transport." The surface, and the fold, exist only where awareness crosses a process boundary:

- **Local → the sensor library, in-process (no surface).** kolu-server runs the sensors in-process and writes `terminalMetadata` *directly* through the existing mutators — no surface, no `directLink`, no fold. The sensors and the record live in the same process, so there is nothing to serve and nothing to merge; wrapping them in an in-process surface would be the *fictional boundary* rejected in #1406. Local stays this way for good — the in-process sensors are always the running build, so awareness is current *by construction*.

> **Even after remote ships (P2), local does NOT switch to dialing a local `arivu` "for consistency."** That would trade freshness-by-construction and a plain function call for a separate process, per-delta serialization, and a fold — pure ceremony to move data within one process. The consistency that matters is already there: local and remote share the *same library, schema, and `AwarenessValue` contract*; only the transport differs, and it differs because the process boundary is real remotely and absent locally. `if local → in-process; else → mirror + fold` is the honest shape, with the cheap path on the common case. *(A pure-consumer **TUI-kolu** that runs no sensors itself is the one shape that would dial a local `arivu` — a different product, not kolu-server.)*
- **Remote → serve, mirror, fold.** `arivu` is Nix-provisioned over ssh (the same staleKey recipe kaval already uses), runs the *identical* sensors, and **serves** the slice over `stdioLink`. kolu-server **mirrors** it with the already-shipped `mirrorRemoteCollection`, then **folds** each upsert into the terminal's `terminalMetadata` entry.

<Callout kind="warn" title="The mirror is a fold, not a write — and that's structural, not incidental">
`terminalMetadata`'s entry is **one mutable object** that publishes the *whole* record — server fields **and** kolu-owned client fields **and** the host-stamped `location`. So an arriving awareness slice can never whole-value-replace the entry (that would clobber the client's theme/layout and the location). It must be **merged in and split** across the persisted-vs-live fence: the live subset (`pr · agent · foreground`) through the quiet mutator, the persisted subset (`cwd · git · lastAgentCommand · lastActivityAt`) through the one that arms autosave. Lose that split and you either re-arm the ~150 ms agent firehose (a perf regression) or silently stop persisting. This fold is *irreducible domain logic* — it exists because `terminalMetadata` is co-owned, and no generic mirror can absorb it.
</Callout>

So the asymmetry is **real, and confined to the wire.** Local is a closure-mutate straight to `terminalMetadata`; remote is closure → frame → serve → mirror → fold. They share the *invariant sensor library and one schema* — but they are two genuinely different paths, and pretending otherwise hides the one new artifact (the host-side hooks impl that reifies each `mutate` closure into a serialized slice frame — a wire carries values, not closures). `location` is stamped kolu-side from the dial-list at the fold.

> **No `@kolu/surface` change is required.** The remote serve + mirror path already ships end-to-end, proven by the `remote-process-monitor` example (a surface served over `stdioLink`, mirrored back via `mirrorRemoteCollection`). A future graduation — `mirrorRemoteSurface`, the spec-driven dual of `implementSurface` that drives every primitive of a served surface into a local ctx — is attractive *electricity* (the demo would delete its hand-rolled bridge loop), but it is **not** on the critical path and it does **not** dissolve the fold.

## The standalone tool — `arivu` + `arivu-tui`

Like kaval, the daemon is **a deliverable in its own right**, shipped and proven *before* kolu touches it. Where `kaval-tui` shows what's *running* in each PTY, `arivu-tui` shows what each terminal *is in* — a host-local "what is every agent doing, across every repo" dashboard with **zero kolu-server dependency**, dialable over ssh against a prod box.

Locally, kolu uses the sensor *library* directly and never touches the surface.

## P2 in detail — P2a (the standalone `--host`), then P2b (the kolu fold)

P2 is **two shippable slices on one provisioning substrate**, and they split the way P1 did — *the standalone tool ships before kolu depends on a line of it.* **P2a** gives `arivu-tui` a `--host <ssh>` dial: a zero-kolu-server clone of the **already-shipped** `kaval-tui --host`, ready to build now, that de-risks the cross-arch Nix provisioning before kolu ever consumes it. **P2b** is the irreducible kolu-side mirror + fold, and it is **gated on [remote-terminals](remote-terminals.html) R-2**: the fold stamps `location={kind:"remote",hostId}` onto a *remote* `terminalMetadata` entry, but kolu owns no remote terminals to fold onto until R-2 ships them (today `{kind:"remote"}` is constructed nowhere — only in docstrings; there is no `RemoteTerminalEndpoint` and no dispatch). So **P2a is the next move; P2b waits on R-2.**

### P2a — `arivu-tui --host` <Pill variant="ok">feature · ready to build</Pill>

**Goal:** `arivu-tui list --host <ssh>` (and `watch`) dials a *remote* `arivu` daemon over ssh, Nix-provisions it cross-arch, and renders the **same** awareness dashboard the local `--socket` path does. **Zero kolu-server.** It is a one-level-up clone of the shipped `kaval-tui --host` (`packages/kaval-tui/src/hostConnect.ts`), reusing `@kolu/surface-nix-host` (`getHostSession` · `resolveSystem` · `provisionAgent` · `nixCopy` · `arch`) **unchanged** — the reach + provision + supervise machinery is already electricity (it also ships in drishti). `arivu --stdio` (built in P1c) is the serve seam the ssh dial speaks to.

**Deliverables** (every path verified against the tree):

- **`packages/arivu-tui/src/hostConnect.ts`** *(new)* — clone of `kaval-tui/src/hostConnect.ts`'s `connectPtyHostViaHost`, with three swaps: the contract type → `typeof arivuSurface.contract`; the remote binary → `arivu --stdio`; the baked drv-map env var → `ARIVU_AGENT_DRVS_JSON`. It composes `getHostSession`, fires `markConnected()` itself, and returns the **same** `{ client, dispose }` `Connection` the local path returns — so every `cmd*()` in `bin.ts` is byte-identical over ssh.
- **`packages/arivu-tui/src/connect.ts`** *(edit)* — today `connectArivu(socketPath)` hardcodes `unixSocketLink<typeof arivuSurface.contract>` (`connect.ts:32`); add the `--host` branch that returns the same `Connection` from `hostConnect.ts`.
- **`packages/arivu-tui/src/bin.ts`** *(edit)* — add a `--host <ssh>` flag, **mutually exclusive with `--socket`**; dispatch `connectHost` vs the existing local connect.
- **`packages/arivu-tui/package.json`** *(edit)* — add the `@kolu/surface-nix-host` workspace dep. It is consumed **read-only and unchanged**, so **no drishti mirror PR** is needed (contrast P2b).
- **`packages/arivu-tui/src/hostConnect.test.ts`** *(new)* — mirror `kaval-tui/src/hostConnect.test.ts`: mock `getHostSession`/`resolveSystem` so `pin()` yields a real in-process `arivu` client, and assert the dashboard renders over the mocked transport.
- **`flake.nix`** *(edit)* — add `arivuDrvBySystem` + `arivuAgentDrvsJson` across **all 4 systems**, mirroring `kavalDrvBySystem`/`kavalAgentDrvsJson` (`flake.nix:37-40`). **Key the map to the arivu DAEMON drv** (`arivu` — it carries `git`+`gh`+sensors+`node:sqlite`), **not** the `arivu-tui` viewer drv: the remote box runs `arivu --stdio`.
- **`default.nix`** *(edit)* — on the `arivu-tui` wrapper (today `nodejs`-only, `default.nix:456-458`), `--set-default ARIVU_AGENT_DRVS_JSON '${arivuAgentDrvsJson}'` and `--prefix PATH` with `pkgs.openssh` + `pkgs.nix` — mirror the `kaval-tui` wrapper at `default.nix:412-415`.

**Traps:**

- **arivu has no `system.heartbeat`.** `kaval-tui/hostConnect.ts:124` does `await client.surface.system.heartbeat({})` before `markConnected()`; arivu's surface exposes only the `awareness` collection + a `version` cell (no `system` namespace), so copying that line 404s. Use a real **`version`-cell read** as the connectivity probe (the mini-ci pattern) — which doubles as exercising the seam P2b's version-skew gate later consumes.
- **One-shot, not long-lived.** P2a uses kaval-tui's one-shot dialer (pin → `markConnected` → `{client,dispose}`; destroy on any error, surfacing the first ssh/nix failure instead of spinning on `HostSession`'s reconnect loop). Do **not** build a long-lived `HostSession` variant — that is P2b's kolu-server composition (carrying `session`/`onState`), explicitly *not* this CLI `Connection` (`kaval-tui/hostConnect.ts:17`).
- **The drv map is built in `flake.nix`, not `default.nix`.** `default.nix` is single-system; a map built there carries only one system and silently breaks cross-arch dialing.

**Verify:** the unit test passes (mocked `getHostSession`/`resolveSystem`); and — the real proof — on a `pu`/prod box running `arivu` + `kaval`, `arivu-tui list --host <ssh>` renders the live awareness rows (capture a screenshot/transcript). **Retires:** that the cross-arch arivu-daemon Nix closure provisions and travels over ssh — the exact provisioning P2b reuses — and settles the `version`-cell-as-`markConnected` seam.

### P2b — kolu mirror + fold <Pill>feature · gated on remote-terminals R-2</Pill>

kolu-server dials a *remote* `arivu` over a **long-lived** `HostSession` (its own variant carrying `session`/`onState` for process lifetime + reconnect — **not** P2a's one-shot `Connection`), mirrors the `awareness` collection with the shipped `mirrorRemoteCollection`, and **folds** each `AwarenessValue` upsert into the co-owned `terminalMetadata` entry across the persisted/live fence (`cwd·git·lastAgentCommand·lastActivityAt` → `updateServerMetadata`; `pr·agent·foreground` → `updateServerLiveMetadata`), stamping `location={kind:"remote",hostId}`. A `version` handshake gates incompatible hosts into re-provision. The four irreducible costs (below) all live here. **The closure→frame reification is already done** — P1c's `makeAwarenessSink` (`packages/arivu/src/hooks.ts`) already publishes a serialized `AwarenessValue` slice and the framework frames it, so P2b is purely the kolu-side consumer. If its long-lived variant needs a new exported `@kolu/surface-nix-host` accessor (e.g. `onState`), that is an API change → a mandatory drishti mirror PR.

**Why it is gated:** the fold has nothing to fold *into* until [remote-terminals](remote-terminals.html) **R-2** ships remote terminals on the canvas — `{kind:"remote"}` is constructed nowhere today (only docstrings), and there is no `RemoteTerminalEndpoint` / dispatch. Until R-2, P2b could only land the mirror+fold *mechanics* against a synthetic entry, not a real remote tile.

**`mirrorRemoteSurface` stays P3**, not pulled forward: arivu serves a collection + a cell and **zero streams**, so arivu-as-second-consumer would exercise only one of that receptacle's three pump arms — a weak graduation proof. It graduates when a second served-surface consumer *with a stream* appears. The **fold itself is irreducible kolu domain logic** (it speaks the whole terminal domain + the autosave fence), so it stays inline in kolu-server — *not* electricity.

## Load-bearing complexity & open decisions

The cleanest design still carries four irreducible costs — **all remote-only**, none dissolvable, and that confinement is the point:

1. **Closure → frame reification** — the one genuinely-new artifact: the host hooks-impl applies each `mutate` to its own record, then emits the resulting slice value as a snapshot/delta frame. The sensors are untouched.
2. **The fold + persisted/live fence**, re-applied kolu-side on every remote upsert (see the callout). Frames must carry the split, or it degrades to runtime discipline.
3. **Freshness / semantic version-skew** — the deepest. In-process local sensors are always current; a remote host runs a provisioned, possibly-stale binary. A version handshake catches *shape* skew, not *semantic* drift (a new agent state an old parser never emits). Bounded by re-provision-on-dial + ephemerality, but only re-provisioning truly closes it — which is *why* local must never route through the daemon.
4. **`location` stamped kolu-side** at the fold — a small permanent special-case at the seam.

*(Cold-start recompute storms and a two-hop key-set lag are operational, not architectural — bounded by idle-gating and the existing `pr:pending` / `agent:null` seed.)*

**Open decisions** (defaults in **bold**): the persisted/live split is carried by **a static field→half map applied kolu-side** (thinner wire) vs per-frame tags; **one `arivu` per host** (mirrors kaval's per-host socket, maximizes the dashboard) vs per-session; **staleKey-gated re-provision** (kaval's existing behavior) vs always-on-dial for the tightest semantic currency; build `mirrorRemoteSurface` **when a second served-surface consumer appears** vs now.

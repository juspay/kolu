**kaval decoupled the PTY out of kolu into a standalone, minimal, durable daemon with a clean contract — independently shippable, independently driven by `kaval-tui`.** This note does the *same move one level up*, for the part kolu kept tangling: **terminal awareness** (git branch/dirty, PR status, agent detection, foreground). A new ephemeral daemon — **`arivu`** (Tamil *அறிவு*, "the faculty of knowing" — a sibling to `kaval`, *watch/guard*, and `odu`, *run*) — sits *on top of* kaval, runs the sensors, and exposes its result as one typed `@kolu/surface` collection. Locally, kolu runs that sensor library in-process and writes `terminalMetadata` directly — no surface, no boundary. Remotely, the `arivu` daemon serves the same awareness over ssh, which kolu mirrors and folds in. **One sensor library, two homes** — and kolu becomes a consumer of awareness, never the place it tangles.

## The core model

One thesis: **awareness is a derived surface, and the thing that produces it is a standalone daemon on top of kaval.** Three claims follow, each verified against the shipped code.

- **The unit is a generic awareness value, not kolu's record.** `arivu` exposes `Collection<TerminalId, AwarenessValue>`, where `AwarenessValue` is exactly the fields the sensors compute — `cwd · git · lastAgentCommand · lastActivityAt` plus `pr · agent · foreground`. This schema is *generic*: it lives in `@kolu/terminal-awareness`, composed from the vendor-neutral `anyagent` / `anyforge` / `kolu-git` shapes, with **no kolu-specific content at all**. The one kolu concept — `location` (the endpoint discriminator) — is simply *absent* (no sensor authors it; a remote tool can't know its own kolu-side `hostId`). Kolu's own `TerminalServerMetadata` is then just `AwarenessValue` **plus** `location` and the client/UI fields (`themeName`, `intent`, …): the awareness value isn't *carved out of* kolu's record — kolu's record is *built on top of* it.
- **kaval stays byte-for-byte minimal.** `arivu` *dials* kaval as a `ptyHostSurface` client for the four taps + `getScreenText`; it adds **zero** awareness/git/gh logic to kaval. The decoupling is additive — a layer above, never a fork.
- **Ephemerality is the simplifier.** Unlike a PTY, awareness never has to survive a restart — it's re-derivable from live taps + current host fs. So `arivu` sheds *all* of kaval's durability machinery: no single-instance lock, no spawn/cleanup (it owns no PTYs), no persisted list (it borrows kaval's `terminal.list`), no adoption or reconnect-survival. Every (re)start just re-runs the sensors and recomputes from now.

The payoff: **clients keep reading the one `terminalMetadata` collection they always did.** The awareness surface is an internal seam, not a new thing the browser learns.

## One library, two homes — and why only remote folds

Local and remote share the *sensor library* and *one schema* — but they are genuinely different paths, not "only the transport." The surface, and the fold, exist only where awareness crosses a process boundary:

- **Local → the sensor library, in-process (no surface).** kolu-server runs the sensors in-process and writes `terminalMetadata` *directly* through the existing mutators — no surface, no `directLink`, no fold. The sensors and the record live in the same process, so there is nothing to serve and nothing to merge; wrapping them in an in-process surface would be the *fictional boundary* rejected in #1406. Local stays this way for good — the in-process sensors are always the running build, so awareness is current *by construction*.

> **Even after remote ships (P2), local does NOT switch to dialing a local `arivu` "for consistency."** That would trade freshness-by-construction and a plain function call for a separate process, per-delta serialization, and a fold — pure ceremony to move data within one process. The consistency that matters is already there: local and remote share the *same library, schema, and `AwarenessValue` contract*; only the transport differs, and it differs because the process boundary is real remotely and absent locally. `if local → in-process; else → mirror + fold` is the honest shape, with the cheap path on the common case. *(A pure-consumer **TUI-kolu** that runs no sensors itself is the one shape that would dial a local `arivu` — a different product, not kolu-server.)*
- **Remote → serve (arivu's side), then mirror + fold (kolu's side).** `arivu` is Nix-provisioned over ssh (the same staleKey recipe kaval already uses), runs the *identical* sensors, and **serves** the slice over `stdioLink` — that serve side is arivu's. The kolu-side **mirror + fold** — mirroring with the already-shipped `mirrorRemoteCollection`, then folding each upsert into the terminal's `terminalMetadata` entry across the persisted/live fence, with the host-stamped `location` — is **not** arivu's concern; it is [remote-terminals](remote-terminals.html) **R-2**'s, and the fold's structural costs live in that note.

So the asymmetry is **real, and confined to the wire.** Local is a closure-mutate straight to `terminalMetadata`; remote is closure → frame → serve (and then, kolu-side, mirror → fold). They share the *invariant sensor library and one schema* — but they are two genuinely different paths, and pretending otherwise hides the one new artifact (the host-side hooks impl that reifies each `mutate` closure into a serialized slice frame — a wire carries values, not closures).

> **No `@kolu/surface` change is required.** The remote serve + mirror path already ships end-to-end, proven by the `remote-process-monitor` example (a surface served over `stdioLink`, mirrored back via `mirrorRemoteCollection`). A future graduation — `mirrorRemoteSurface`, the spec-driven dual of `implementSurface` that drives every primitive of a served surface into a local ctx — is attractive *electricity* (the demo would delete its hand-rolled bridge loop), but it is **not** on the critical path and it does **not** dissolve the fold; that graduation is now tracked in [remote-terminals](remote-terminals.html).

## The standalone tool — `arivu` + `arivu-tui`

Like kaval, the daemon is **a deliverable in its own right**, shipped and proven *before* kolu touches it. Where `kaval-tui` shows what's *running* in each PTY, `arivu-tui` shows what each terminal *is in* — a host-local "what is every agent doing, across every repo" dashboard with **zero kolu-server dependency**, dialable over ssh against a prod box.

Locally, kolu uses the sensor *library* directly and never touches the surface.

## Shippable slices

## P2 — `arivu-tui --host` (shipped) <Pill variant="ok">feature</Pill> <PrLink pr={1439} />

`arivu-tui --host <ssh>` dials a *remote* `arivu` daemon over ssh, Nix-provisions it cross-arch, and renders the **same** awareness dashboard the local `--socket` path does — **zero kolu-server**, the read-only viewer the only local process. A one-level-up clone of the shipped `kaval-tui --host`, riding `@kolu/surface-nix-host`'s reach + provision machinery; `arivu --stdio` (built in P1c) is the serve seam the ssh dial speaks to. The provisioned arivu **discovers** the host's kaval (the same `discoverPtyHostSockets` scan `kaval-tui` uses — a standalone daemon, or a **kolu-server** namespaced by listen port), so `--host` lands on a real remote kolu's terminals with no flag; `--kaval <path>` pins one when several run.

The review gauntlet pulled the shared one-shot **reach → provision → `pin → probe → markConnected → dispose`** composition — until then duplicated between `kaval-tui` and `arivu-tui` — *up* into `@kolu/surface-nix-host` as a new **`dialAgentOnce`** primitive. Each CLI is now a ~10-line wrapper supplying only its volatile axes: the binary, the drv-map env var, and a connectivity `probe` (`system.heartbeat` for kaval; a **`version`-cell read** for arivu, which has no `system` namespace). The receptacle that already encapsulated the *hard* ssh/reconnect/provision volatility now owns the one-shot socket every `--host` consumer needs, instead of each one re-wiring it on top. *(Because this adds public API to the drishti-shared `@kolu/surface*` packages — `dialAgentOnce` + its types, the new `@kolu/surface/first-frame` export, and the `HostSession` state/options changes — `.claude/rules/surface.md` and `packages/AGENTS.md` require a paired **drishti PR opened before or alongside merge and green before "done"**, linked from this PR. Mechanically the mirror is small — an npins `kolu` pin-bump plus hydrating the new zero-dep `@kolu/shell-quote` leaf; drishti's source imports none of the new symbols, so it is a zero-source-change, green-CI confirmation — but it is a merge gate, not a post-merge follow-up.)*

**Retired:** that the cross-arch arivu-daemon Nix closure provisions and travels over ssh (the substrate P3's fleet dial reuses), and the `version`-cell-as-`markConnected` seam — proven by mocked unit tests plus a live capture: `arivu-tui --host <pu box>` provisioning arivu onto a leased pool box and rendering the box's kaval terminals (branch · foreground, with the gh sensor degrading gracefully where unauthenticated).

## P3 — arivu-tui fleet, the ambient multi-host agent dashboard <Pill variant="ok">feature</Pill>

P2 made **one** remote host's awareness reachable. P3's end state is the user's framing made literal — *"what is every agent doing, across every repo, across every machine"* — as a **dashboard you leave open on a second monitor**: glance over from the game on the main screen and the board tells you, at a colour you can read across the room, whether any agent is **blocked on you** and where. It stays a TUI, runs **zero kolu-server**, and ships as **three PRs** — the Bun runtime, then the OpenTUI render swap on one host, then the multi-host fleet:

- **PR1 — Bun runtime, via bun2nix.** Re-platform `arivu-tui` from the shared `tsx` wrapper onto **Bun**, adopting [drishti](electricity.html)'s Bun-on-Nix recipe *now* rather than deferring it: **bun2nix pinned via npins** and resolved with `builtins.getFlake` keyed off that pin — *not* a flake input, the same way `odu` is consumed, so kolu's zero-flake-input rule holds and the dev shell never forces it — an isolated `bun.lock` + autogenerated `bun.nix` dep cache kept **outside** the pnpm workspace, and the `@kolu/*` viewer closure **hydrated from the Nix store** (never bun-installed — `terminal-awareness`'s full manifest drags the sensor spine, so only its zod-only `/schema` reach matters). Today's `list` / `watch` (the [`columnify`](https://www.npmjs.com/package/columnify) text) run unchanged — behaviour-preserving in the output, no OpenTUI, no Solid, no new *runtime* npm deps. The standalone step: it stands up a real `bun2nix` foundation so PR2a's `bun build` bundle lands on proven ground, not a wrapper that would be ripped out. *Proof:* `list` / `watch` (local *and* `--host`) emit identical output, now under Bun. kaval-tui stays on `tsx`; the `arivu` **daemon** stays Node. **Shipped** — <PrLink pr={1470} />.
- **PR2a — OpenTUI render swap, one endpoint.** On the Bun runtime, **retire `list` / `watch`** and their [`columnify`](https://www.npmjs.com/package/columnify) text, and render a **simple, SolidJS-canonical OpenTUI list** of one endpoint's terminals. This adds **no new connection capability** — it dials the **already-shipped** local `--socket` (P1c) or a single `--host` (P2's `dialAgentOnce`), reused verbatim; the only new axis (dialing N hosts at once) is PR2b's. Beside it, a **clock that ticks every second** is the deliberate *liveness proof* — `createSignal` + `onMount(() => { const id = setInterval(…); onCleanup(() => clearInterval(id)) })`, read in JSX, repainting only the one changed cell — so the whole **Bun → OpenTUI (Zig core via `Bun.dlopen`) → Solid** pipeline is proven reactive behind the smallest possible UI. The **terminal list itself is a one-shot snapshot** at render — it does *not* yet auto-update on awareness changes; wiring the mirror deltas that make it live is PR2b's job. `arivu-tui --json` dumps the flat awareness JSON. This adds `@opentui/*` + `solid-js`, the `build.ts` bundle (`Bun.build` + `@opentui/solid/bun-plugin`), and the `bunfig.toml` dev preload — the only real packaging change, touching only the viewer wrapper. *Proof:* the OpenTUI list renders a host's terminals, and the one-second clock is confirmed live by **raw-PTY capture** (frame N+1 ≠ frame N), not a snapshot-and-grep. (PR2a can lift `render.ts`'s projection + tones, the data-read helpers, and `bin.ts`'s output-mode dispatch from the `feat/arivu-tui-bun-opentui` branch.)
- **PR2b — the multi-host fleet board.** On PR2a's OpenTUI render, fan it across hosts: fan P2's one-shot per-host dial over N hosts, `mirrorRemoteCollection` each `awareness` collection into one aggregate keyed by `(host, terminalId)`, make the list go **live** (re-render on any mirror delta — the dynamism PR2a deferred), float **every `awaiting_user` agent to the top of the whole fleet**, and render the per-host groups + the breathing alert strip + the honest unreachable/skew/empty states. *Proof:* a live board over 2–3 real boxes, liveness checked by raw-PTY capture; kill one box and watch its header flip to "unreachable" while the others keep updating.

<Callout kind="note" title="Deferred to follow-up PRs (surfaced building/testing PR2a)">

Items surfaced while building and testing PR2a (<PrLink pr={1479} />), each tracked for its **own PR** rather than folded into the render swap. The first two are **arivu *daemon* concerns, not the OpenTUI render** — the render faithfully shows what the daemon serves; the third is a packaging-layout cleanup:

- **Idle terminals show no `repo·branch`.** arivu is *ephemeral*: it seeds each terminal's cwd from kaval's **spawn cwd** and only updates on an OSC 7 emitted *after* it connects (`daemon.ts` `seedAwarenessValue(entry.cwd)` + the `signals.cwd` stream). So a terminal that `cd`'d into its repo *before* arivu started, then went idle, reports no cwd → the git sensor never resolves → an empty (`—`) `REPO·BRANCH`. This is **pre-existing P1c behaviour, identical under the old `list`** — the render swap didn't introduce it. Fix (a daemon PR): on seeding a terminal, read its **live** cwd (e.g. `/proc/<pid>/cwd` via a kaval tap) instead of only the spawn cwd, so an idle terminal resolves immediately.
- **`--host` forwarded stderr bleeds into the alt-screen.** Over `--host`, the daemon's forwarded stderr (its pino JSON logs) plus the viewer's connection-teardown lines (`agent exited`, `connection: disconnected`) write to the tty *during* the alt-screen render, corrupting its right edge. (`--host`-only; the local `--socket` path has no forwarded stream.) Fix (a daemon PR, the approach this note already prescribes): quiet the daemon to **`warn` over `--stdio`** so the info-level logs never cross the wire — *not* a viewer-side `process.stderr` capture hack. The **closed `feat/arivu-tui-bun-opentui` branch (PR <PrLink pr={1453} />) carries a reference implementation** to lift from (its stderr handling, treated as reference only — its TUI was otherwise superseded).
- **Encapsulate the viewer's Nix packaging into the package.** The viewer's Bun derivation + bun manifest live under `nix/packages/arivu-tui/` (`bun/{package.json,bun.lock,bun.nix,bunfig.toml}` + `default.nix`), away from `packages/arivu-tui/` — so the package is *not* self-contained. A follow-up should co-locate them (e.g. `packages/arivu-tui/nix/`) so the whole viewer is one directory. The tension to resolve, which is *why* it was split out: the bun manifest is deliberately kept **out** of `packages/**` so pnpm (whose workspace glob is `packages/**`) never fights bun over a second `package.json`. So the move needs a pnpm-workspace **exclusion** (`!packages/arivu-tui/nix/**` or similar), not a naive relocation — a structure-only PR, no behaviour change.

</Callout>

<Callout kind="warn" title="Build constraints — OpenTUI + Solid (land in PR2a, hold through PR2b)">

- **OpenTUI repaints only the CHANGED CELL** (a cursor move + one character), never the whole string. Verify "is it live?" with **raw-PTY capture** (or a `testRender` assertion that frame N+1 ≠ frame N) — **never** `kaval-tui snapshot | grep` or a glance, which miss the one ticking character and read a live UI as "frozen".
- **Pre-bundle; don't run from source.** `build.ts` (`Bun.build` + `@opentui/solid/bun-plugin`) applies `babel-preset-solid` at build time, so the runtime needs no `@babel` / `browserslist`. Clean `build.ts`'s `outdir` before each build (`Bun.build` doesn't, so orphan chunks linger).
- **Bundle solid-js — do NOT mark it `external`.** `@opentui/solid`'s plugin rewrites `solid-js/dist/server.js` → the reactive `solid.js` client build (Bun resolves solid-js to its **SSR/server** build under the `node` condition, whose signals don't track), and that redirect only fires when solid is loaded *during the build*. So `packages: "external"` ships the **non-reactive server build** and every live value silently **freezes at runtime** — while `bun run`/dev still works, because the `bunfig.toml` preload does the same redirect at load time (so the bug is invisible until you run the *Nix bundle* with no preload). Use `external: ["@opentui/core"]` (only the native package; bundling it is bundle-hostile) — solid-js + `@opentui/solid` get bundled, sharing one reactive instance. This cost a full debug cycle in PR2a (<PrLink pr={1479} />); the smoke that catches it is running the **built `dist/bin.js`** (not source) and confirming a repaint.
- **SolidJS-canonical only.** A live value is `createSignal` + `onMount(() => { const id = setInterval(…); onCleanup(() => clearInterval(id)) })`, read in JSX. No fps caps, `requestRender`, externally-driven timers, or `process.stderr` monkeypatching.
- **Forwarded daemon stderr** over `--host` is quieted **at the daemon** (`warn` over `--stdio`), never with a viewer-side capture hack, so it can't corrupt the alt-screen.

</Callout>

### Why Bun, contained to the viewer

OpenTUI is the production-grade TUI framework — it powers **OpenCode** and **terminal.shop** — and it ships a **Solid reconciler** (`@opentui/solid`), so the dashboard is authored in the *same* fine-grained-reactive model kolu already lives in: the mirrored `awareness` collection becomes Solid signals, `<box>`/`<text>` components re-render only the cells that changed, and a `useTimeline` hook drives animation. The catch — and the whole reason the runtime swap (**PR1**) comes first — is that OpenTUI's renderer is a native **Zig** core loaded via **`Bun.dlopen`**: it needs **Bun**.

So the decision is narrow: **`arivu-tui` *alone* becomes a Bun binary.** The rest of kolu stays Node/tsx, and so does the `arivu` **daemon** — it is the half that travels over ssh and leans on Node's built-in `node:sqlite`, so it must stay Node, and neither PR touches it. Only the **local viewer** re-platforms, and that blast radius is exactly what makes the cost acceptable: the viewer is never provisioned over ssh, so the Bun runtime + the per-arch Zig lib live only in the *local* closure. arivu-tui's existing deps (`@kolu/surface`, `@kolu/surface-nix-host`, the contract) are Node-API code that runs unmodified under Bun's Node compatibility.

Packaging is **not** new ground: **reuse drishti's Bun-on-Nix recipe.** [drishti](electricity.html) already ships a Bun server packaged with Nix (and already vendors `@kolu/surface` out of *this* repo), so "a Bun binary built cross-arch under Nix" is solved in the kolu orbit — **PR1** adopts that recipe wholesale: `bun2nix`'s `fetchBunDeps` reads a committed `bun.nix` (per-tarball FODs, no network in the sandbox), `@kolu/*` is hydrated post-install exactly as drishti hydrates `@kolu/surface`, and the bun manifest lives **outside** the pnpm workspace so the two package managers never fight over one `package.json` (the viewer keeps its pnpm manifest for typecheck + vitest). **PR2a** then re-renders the same `AwarenessValue` fields `render.ts` shows today through OpenTUI on one host — behaviour-preserving in the *data*, an upgrade in *presentation* — and **PR2b** fans that render across the fleet.

<Callout kind="warn" title="One Bun binary in a Node monorepo — contained on purpose">
A deliberate, *bounded* runtime split, not a drift. It honours volatility-locality: the thing that forces Bun (OpenTUI's FFI renderer) is a **local-viewer** concern, so the Bun runtime stays inside the viewer's wrapper and never leaks into a shared package, the daemon closure, or anything that crosses ssh. The daemon — the piece with the hard portability constraint (`node:sqlite`, cross-arch travel) — stays Node. The Bun-ness is one wrapper, one binary: easy to read, easy to undo, and the packaging already graduated (it's drishti's) the day a second Bun consumer appears.
</Callout>

Default grouping is **by host** (a header per box carrying its terminal count and a reachable/skew/unreachable badge); within and across the fleet the rows sort **"needs-you" first** — every `awaiting_user` agent bubbles to the top regardless of which box it's on, so the operator's eye lands on *which agent is blocked on me, and where*. **How it composes the arivu surface** — four moves, each reusing a shipped seam; the only genuinely-new code is the cross-host join.

- **Discover hosts** (bin layer, no new transport). Resolve from explicit `--host` flags, an `~/.ssh/config` parse (drop wildcard `*` stanzas), or `fleet.toml`; merge `local` unless `--no-local`. No host-discovery protocol is invented — lean on the box list the way the `.gitignore`/source-of-truth-reuse instinct demands.
- **Dial + provision per host** (reuse P2 *verbatim*). For each remote host call the **same** one-shot `connectArivuViaHost(host)` P2 builds in `arivu-tui/src/hostConnect.ts` — `getHostSession` → `nix copy` the cross-arch `arivu` daemon closure → `ssh <host> arivu --stdio` → a transport-blind `Connection`, with the **`version`-cell read as the connectivity probe** (P2's `markConnected` seam; arivu has no `system.heartbeat`). For `local`, call the existing `connectArivu(socketPath)`. P3 adds only the **fan-out**: `Promise.allSettled` over the host list, so a cold-provisioning box (a multi-minute `nix copy`) and a fast one connect in parallel and one dead box can't sink the others.
- **Mirror each host's awareness collection.** For every live `Connection`, run the shipped `mirrorRemoteCollection<TerminalId, AwarenessValue>` against that client's `awareness` collection — the *same* primitive kolu's R-2 fold will consume and the `remote-process-monitor` demo already proved. Its upserts/removes write into one in-process aggregate `Map<HostKey, Map<TerminalId, AwarenessValue>>`, so agent-state transitions flow live across every box with no polling.
- **Aggregate / key-by-host.** Host is stamped onto each mirrored row **at the dial site** — the fleet knows which connection a row came from, and the daemon never authors a host id, exactly as arivu's core model requires (`AwarenessValue` deliberately carries no `location`/`hostId`; a daemon can't know its own kolu-side key). This is the awareness-layer echo of kolu's kolu-side `location` stamp. The render layer reads the aggregate keyed by `(host, terminalId)`, groups under per-host headers, applies the cross-fleet needs-you sort, and repaints on any delta.

**Why this is arivu-native** — it is the awareness analog of **drishti**, and that parallel is the whole point. Per [electricity](electricity.html), `@kolu/surface` proved it was real electricity *not by being extracted* but by graduating to a **second consumer** in drishti — a fleet process-monitor (browser ↔ Bun ↔ ssh-agent) that mirrors per-host *process* surfaces and keys by host, with none of kolu's domain. arivu's P3 is the exact same shape one layer up: a fleet dashboard that mirrors N remote *arivu* daemons into one host-keyed view, proving arivu's **own** surface (the `awareness` collection + the `version` cell) is itself a receptacle a different app plugs into. Crucially, **nothing in `@kolu/arivu-contract` changes for P3 to exist** — that invariance under a brand-new multi-host consumer *is* the electricity proof (test ③ of [electricity](electricity.html) made real for the arivu surface). It is a genuinely different app from the single-host viewer (one host → N, vertical record → aggregated board), and the contract never notices it went from 1 host to N. And it is the honest home for the user's *"remote kolu monitor / agent-status dashboard"*: a team-wide, read-only monitor is a **different product** from kolu's own-canvas fold ([remote-terminals](remote-terminals.html) R-2's job) — so it belongs as arivu's own consumer, with zero kolu-server, *not* folded into kolu.

Two boundary calls keep this honest. The **renderer** *is* electricity — a domain-agnostic Solid→terminal capability — but it already exists as **OpenTUI**, so P3 *consumes* that receptacle rather than minting a `@kolu/solid-tui` of its own (a Solid-on-Node renderer was the road not taken once Bun was on the table). The **aggregation join**, by contrast, is a **leaf reducer, not electricity** — it hides a bounded host-keyed merge, not a hard volatility (transport + provision + reconnect is already owned by `@kolu/surface-nix-host` and `mirrorRemoteCollection`). So PR2b must **not** extract a `@kolu/fleet-mirror`: a generic host-keyed merge is exactly the "tidy generic helper" trap [electricity](electricity.html) warns against.

**Fleet proof** — dial 2–3 real boxes and see one aggregated fleet.

- **Real proof** (the bar every arivu phase holds): on 2–3 `pu`/prod boxes each running `kaval` + `arivu` with a live agent or two, `arivu-tui fleet --host <a> --host <b> --host <c>` renders every host's terminals in one screen, with any `awaiting_user` agent sorted to the top across hosts, and live `working`→`awaiting` transitions repainting without a re-dial — capture a screenshot/transcript of the multi-host board, plus the `--by agent` fleet-wide "who is waiting on input" view.
- **Negative proof matters too** (the no-fallback rule — *a caught dial error must surface, not collapse to empty*): kill one box and confirm its header flips to "unreachable" while the other host's rows keep updating (partial observability is honest, never a vanished section); point at a version-skew box to confirm the degraded "skew" header.
- **Unit proof** (cheap, CI): the mocked two-fake-client test asserts the aggregate is correctly `(host, terminalId)`-keyed (two boxes' identical terminal ids stay distinct), the needs-you sort, and that one rejecting dial degrades to a per-host error row rather than sinking the board — the same mock harness P2's `hostConnect.test.ts` established.

**Retires:** that arivu's served surface is genuinely a **second-order receptacle** — a different app (the fleet aggregator) plugs into the same contract the single-host viewer uses, with zero kolu-server and no contract change, the awareness-layer analog of how drishti retired the same risk for `@kolu/surface`.

It supersedes the earlier from-scratch plan (<PrLink pr={1398} />, closed) and the in-process provider extraction (<PrLink pr={1409} />, closed): both kept awareness *inside* kolu and split hairs over an in-process compose-vs-serve boundary that this design dissolves by moving the producer into a daemon. Built from an adversarial, cost-blind design panel (verdict: *sound, with the asymmetry-is-encapsulated and mirror-is-a-fold corrections folded in above*), grounded by reading the shipped code: the in-server sensor set (now `@kolu/terminal-awareness/sensors.ts`), `mirrorRemoteCollection`, the `remote-process-monitor` "define once, implement twice" demo, and kaval's provisioning recipe. The substrate it stands on — kaval (untouched), `location` as a first-class discriminated union — is already merged. The long pole is **P1a → P1b → P1c → P2**, and **P1a (the behavior-preserving extraction) is the recommended first move**, then **P1b** (finish the leaf rename), then **P1c** (the standalone `arivu` + `arivu-tui`): independently useful, and it retires the biggest risk before kolu depends on a line of it. (P3 — the `arivu-tui fleet` dashboard — fans P2 out to N hosts; the kolu mirror + fold lives in [remote-terminals](remote-terminals.html) R-2.) There is no local-compose phase — locally kolu runs the sensor library in-process (settled at P1a); the surface and the fold are remote-only (P2).

**P1a shipped** as <PrLink pr={1413} /> — `@kolu/terminal-awareness` is extracted, the schema home inverted, kolu-server re-pointed at it, and the package's own surface renamed to the **sensor · signal · sink** vocabulary (`startProviders` → `startAwareness`), behaviour-identical. (The `kavalChannels.ts` kaval-dial bridge listed above moved to **P1c**: it is the daemon's connective tissue — kolu-server builds its sensor signals from its own pty-host taps in `local.ts`, so P1a never needed it.) **P1b shipped** as <PrLink pr={1419} /> — it finished the `provider` → `adapter` rename through the `anyagent`/`anyforge` leaves (a single adapter spine, no `probe`; the `Watcher` handle and the wire `provider` discriminant stayed), behaviour-preserving. **P1c shipped** (<PrLink pr={1428} />, merged 2026-06-19) — the standalone `arivu` (the daemon) + `arivu-tui` (the viewer), built on the lifted sensor set: arivu dials kaval as a plain `ptyHostSurface` client, runs the unchanged sensors per terminal, and serves each slice into the `@kolu/arivu-contract` `awareness` collection; ephemeral (no gate, no PTY ownership), zero kolu-server at runtime. `--stdio` is the seam P2's ssh dial speaks to. **P2 shipped** (<PrLink pr={1439} />) — `arivu-tui --host`, the standalone remote viewer; the gauntlet lifted the shared one-shot dial into `@kolu/surface-nix-host`'s `dialAgentOnce`, so kaval-tui and arivu-tui are thin wrappers over one primitive. **P3 · PR1 shipped** (<PrLink pr={1470} />) — the Bun re-platform of the viewer, via **bun2nix** adopted now rather than deferred: bun2nix pinned via **npins** and resolved with `builtins.getFlake` (no flake input — the `odu` path, so the zero-flake-input rule holds), an isolated `bun.lock` + `bun.nix` dep cache kept outside the pnpm workspace, and the `@kolu/*` closure hydrated from the store (the heavy sensor manifest never bun-installed — only `terminal-awareness`'s zod-only `/schema` reach matters). `list` / `watch` (local *and* `--host`) emit identical output, now under Bun; kaval-tui stays `tsx`, the daemon stays Node. **PR2 is split in two. P3 · PR2a is in review** (<PrLink pr={1479} />) — the OpenTUI render swap against one endpoint at a time (the already-shipped local `--socket` or single `--host`, reused verbatim — no new connection capability; `list`/`watch` and the text path dropped; the awareness table read once from the daemon, with a one-second header clock as the liveness proof; `--json` dumps the awareness array; the rows stay a one-shot snapshot), proving the Bun → OpenTUI(native Zig via `Bun.dlopen`) → Solid pipeline on PR1's bun2nix base. **Next: P3 · PR2b** wires the multi-host fleet board (fan-out + mirror + the cross-fleet needs-you sort + live deltas) on top. Viewer-only Bun throughout; the daemon stays Node. See the **P3** section above for the PRs and the OpenTUI/Solid build constraints. The **kolu mirror + fold** is *not* arivu's job — it is how [remote-terminals](remote-terminals.html) **R-2** consumes arivu (long-lived `HostSession`, `mirrorRemoteCollection`, the persisted/live fold), and it lives in that note, gated on R-2.

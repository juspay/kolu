## The core model

One thesis: **awareness is a derived surface, and the thing that produces it is a standalone daemon on top of kaval.** Three claims follow, each verified against the shipped code.

- **The unit is the provider-authored *slice*, not the whole record.** `arivu` exposes `Collection<TerminalId, AwarenessValue>` where `AwarenessValue` is exactly the fields the providers compute тАФ `cwd ┬╖ git ┬╖ lastAgentCommand ┬╖ lastActivityAt` (server-persisted) plus `pr ┬╖ agent ┬╖ foreground` (live). That is a strict `.pick` of *existing*, partition-tested schemas: **zero new schema work**. It deliberately excludes `location` (no provider authors it; a remote tool can't know its own kolu-side `hostId`) and every client/UI field (`themeName`, `intent`, `canvasLayout`, тАж) тАФ those are kolu-owned and unproducible host-side.
- **kaval stays byte-for-byte minimal.** `arivu` *dials* kaval as a `ptyHostSurface` client for the four taps + `getScreenText`; it adds **zero** awareness/git/gh logic to kaval. The decoupling is additive тАФ a layer above, never a fork.
- **Ephemerality is the simplifier.** Unlike a PTY, awareness never has to survive a restart тАФ it's re-derivable from live taps + current host fs. So `arivu` sheds *all* of kaval's durability machinery: no single-instance lock, no spawn/cleanup (it owns no PTYs), no persisted list (it borrows kaval's `terminal.list`), no adoption or reconnect-survival. Every (re)start just re-runs the providers and recomputes from now.

The payoff: **clients keep reading the one `terminalMetadata` collection they always did.** The awareness surface is an internal seam, not a new thing the browser learns.

## One contract, two wirings тАФ and why remote needs a fold

The same surface contract is wired two ways, selected by locality. This is the honest version of *"local vs remote is only the transport"* тАФ with one correction the design panel forced.

- **Local тЖТ compose, in-process.** The providers run *inside* kolu-server exactly as they do today, mutating the terminal's `meta` in place and publishing `terminalMetadata`. The daemon's surface is `implementSurface`'d and consumed via `directLink` тАФ which only *formalizes the call boundary that already exists*. No socket, no serialization, no fold. **And local must stay this way:** the in-process providers are always the running build, so awareness is current *by construction*.
- **Remote тЖТ serve, mirror, fold.** `arivu` is Nix-provisioned over ssh (the same staleKey recipe kaval already uses), runs the *identical* providers, and **serves** the slice over `stdioLink`. kolu-server **mirrors** it with the already-shipped `mirrorRemoteCollection`, then **folds** each upsert into the terminal's `terminalMetadata` entry.

<Callout kind="warn" title="The mirror is a fold, not a write тАФ and that's structural, not incidental">
`terminalMetadata`'s entry is **one mutable object** that publishes the *whole* record тАФ server fields **and** kolu-owned client fields **and** the host-stamped `location`. So an arriving awareness slice can never whole-value-replace the entry (that would clobber the client's theme/layout and the location). It must be **merged in and split** across the persisted-vs-live fence: the live subset (`pr ┬╖ agent ┬╖ foreground`) through the quiet mutator, the persisted subset (`cwd ┬╖ git ┬╖ lastAgentCommand ┬╖ lastActivityAt`) through the one that arms autosave. Lose that split and you either re-arm the ~150 ms agent firehose (a perf regression) or silently stop persisting. This fold is *irreducible domain logic* тАФ it exists because `terminalMetadata` is co-owned, and no generic mirror can absorb it.
</Callout>

So the asymmetry is **encapsulated, not erased.** Local is a closure-mutate; remote is closure тЖТ frame тЖТ serve тЖТ mirror тЖТ fold. They share the *invariant providers and one schema* тАФ but they are two paths, and pretending otherwise hides the one genuinely-new artifact (the host-side hooks impl that reifies each `mutate` closure into a serialized slice frame тАФ a wire carries values, not closures). `location` is stamped kolu-side from the dial-list at the fold.

> **No `@kolu/surface` change is required.** The compose-or-serve duality already ships end-to-end and is proven by the `remote-process-monitor` example (*define the surface once, implement it twice* тАФ `directLink` in-process, `stdioLink` over ssh, `mirrorRemoteCollection` to mirror). A future graduation тАФ `mirrorRemoteSurface`, the spec-driven dual of `implementSurface` that drives every primitive of a served surface into a local ctx тАФ is attractive *electricity* (the demo would delete its hand-rolled bridge loop), but it is **not** on the critical path and it does **not** dissolve the fold.

## The standalone tool тАФ `arivu` + `arivu-tui`

Like kaval, the daemon is **a deliverable in its own right**, shipped and proven *before* kolu touches it. Where `kaval-tui` shows what's *running* in each PTY, `arivu-tui` shows what each terminal *is in* тАФ a host-local "what is every agent doing, across every repo" dashboard with **zero kolu-server dependency**, dialable over ssh against a prod box.

## Load-bearing complexity & open decisions

The cleanest design still carries four irreducible costs тАФ **all remote-only**, none dissolvable, and that confinement is the point:

1. **Closure тЖТ frame reification** тАФ the one genuinely-new artifact: the host hooks-impl applies each `mutate` to its own record, then emits the resulting slice value as a snapshot/delta frame. The providers are untouched.
2. **The fold + persisted/live fence**, re-applied kolu-side on every remote upsert (see the callout). Frames must carry the split, or it degrades to runtime discipline.
3. **Freshness / semantic version-skew** тАФ the deepest. In-process local providers are always current; a remote host runs a provisioned, possibly-stale binary. A version handshake catches *shape* skew, not *semantic* drift (a new agent state an old parser never emits). Bounded by re-provision-on-dial + ephemerality, but only re-provisioning truly closes it тАФ which is *why* local must never route through the daemon.
4. **`location` stamped kolu-side** at the fold тАФ a small permanent special-case at the seam.

*(Cold-start recompute storms and a two-hop key-set lag are operational, not architectural тАФ bounded by idle-gating and the existing `pr:pending` / `agent:null` seed.)*

**Open decisions** (defaults in **bold**): the persisted/live split is carried by **a static fieldтЖТhalf map applied kolu-side** (thinner wire) vs per-frame tags; **one `arivu` per host** (mirrors kaval's per-host socket, maximizes the dashboard) vs per-session; **staleKey-gated re-provision** (kaval's existing behavior) vs always-on-dial for the tightest semantic currency; build `mirrorRemoteSurface` **when a second served-surface consumer appears** vs now.

## Status

**Proposed тАФ a clean-slate replacement for the awareness layer**, child of [kaval-sessions](kaval-sessions.html). It supersedes the earlier from-scratch plan (<PrLink pr={1398} />, closed) and the in-process provider extraction (<PrLink pr={1409} />, closed): both kept awareness *inside* kolu and split hairs over an in-process compose-vs-serve boundary that this design dissolves by moving the producer into a daemon. Built from an adversarial, cost-blind design panel (verdict: *sound, with the asymmetry-is-encapsulated and mirror-is-a-fold corrections folded in above*), grounded by reading the shipped code: `@kolu/terminal-providers`, `mirrorRemoteCollection`, the `remote-process-monitor` "define once, implement twice" demo, and kaval's provisioning recipe. The substrate it stands on тАФ kaval (untouched), `location` as a first-class discriminated union тАФ is already merged. The long pole is **P1 тЖТ P2 тЖТ P3**, and **P1 (the standalone `arivu` + `arivu-tui`) is the recommended first move**: independently useful, and it retires the biggest risk before kolu depends on a line of it.

**pulam-web is [drishti](electricity.html)'s twin, pointed at the terminal-workspace surface.** Where drishti monitors per-host *processes* in a browser, pulam-web monitors per-host *terminals* — the fleet you already see in `pulam-tui`, but in a browser, **auto-provisioned over ssh**. It exists for one engineering reason on top of being a real product: it is **kolu's own browser-consumption leg, minus kolu** — `websocketLink` → `surfaceClient` → Solid `reconcile` — so it graduates the path [remote-terminals **R-compose-2**](remote-terminals.html#r8) is blocked on, in a build whose console you can actually see.

**It is the same `@kolu/surface` code kolu and drishti run — that is the whole point.** Serve side: `implementSurface` → oRPC `RPCHandler` (`@orpc/server/ws`) → `handler.upgrade(ws)` on a `/rpc/ws` socket. Consume side: `websocketLink` → `surfaceClient`/`surfaceClients` → the reactive `.use()` hooks (`surfaceClient.ts` owns snapshot-then-deltas + `reconcile`). drishti proves this trio is electricity; kolu's Code tab proves `.streams.use()` for a **value-bearing** `gitStatus` stream. pulam-web is the first *standalone* app to put those together — value-bearing git status, over `.streams.use()`, **sourced from a mirror** — which is precisely R-compose-2's path.

**Node, matching kolu's own stack.** The surface leg is identical `@kolu/surface` code on any runtime — drishti even writes its server Node-style (`@hono/node-server` + the `ws` package) and runs it on Bun *only* for its `Bun.build` client bundler. We go **Node**, like kolu-server, and bundle the SolidJS frontend with **Vite + `vite-plugin-solid`** (kolu's own client toolchain) rather than drishti's `Bun.build` — so pulam-web is kolu's stack end-to-end.

<Callout kind="note" title="Common it up — what graduates to the surface family, what stays app-local (grounded)">
The directive was *don't replicate drishti's plumbing into pulam-web — upstream the common stuff*. Grounding both apps against the installed code drew the line precisely:

- **What graduates — the per-host mirror-bridge + registry → `@kolu/surface-nix-host`.** drishti's `bridgeAgentToParent` (a `makeClientCursor` reconnect loop re-issuing `mirrorRemoteSurface` per spawn + a live-procedure forwarding holder) and its `hostRegistry` (`Map<host,{session,handler}>`) are ~600 lines pulam-web would otherwise copy verbatim. They graduate as `pumpRemoteSurface` + `buildHostRegistry` — the **N-host, consume-side companions** to nix-host's *existing* `makeClientCursor` + `getHostSession`, with **zero new deps** (the registry stays generic over the handler type, so no `@orpc/server/ws` in nix-host). Both apps consume one copy.
- **What stays app-local — the ws-serve upgrade block.** It is *not* the right thing to extract: `@kolu/surface` core has no `ws` dep (putting a `WebSocketServer` helper there inverts its transport-agnostic boundary — a defect per [electricity](electricity.html)), and drishti's real handler validates `?host=` and `destroy()`s the raw socket *before* `handleUpgrade`, plus dispatches an `__admin__` sentinel to a second handler — neither fits a generic `onConnection(ws)` seam. So each app keeps its ~18-line upgrade block, composed from the **already-shared** `gateWsOrigin` (`@kolu/surface/ws-origin`) + `gateStaleSocket` + `startWsHeartbeat` (`@kolu/surface-app/server`).

So the extraction is drishti's own proven code lifted up — drishti's adoption is a near-mechanical swap whose green CI proves the lift was behaviour-preserving (the same way `gateStaleSocket` was already extracted from drishti).
</Callout>

<Callout kind="note" title="The design fork R-compose-2 rides on — resolve it here, grounded">
Two ways git status reaches a browser. **Design A** (what #1510 did, and stuck): ship terminal-workspace's raw `{seq}` pulse to the browser and re-query `getStatus` client-side via `useWatchedRead`. **Design B** (what `koluSurface` *already* does — `packages/server/src/surface.ts:271-277`, `{read:getStatus, install:subscribeRepoChange, isEqual}` — and the Code tab proves daily): wrap it into one **value-bearing** stream *server-side*; the browser consumes an ordinary value-bearing stream via `.streams.use()`. **Lead: Design B.** For a remote host the backend wraps the **mirrored** `subscribeRepoChange` + **forwarded** `getStatus` (R7) into that value-bearing stream — so the raw `{seq}` pulse never crosses to the browser, and the "ungraduated primitive" dissolves as an artifact of A. pulam-web confirms B end-to-end from a mirrored source; that becomes R-compose-2's recipe.
</Callout>

## Implementation details

Three steps, each isolating **one** risk. **R-pulamweb-1 (in drishti)** graduates the reactive *stream* consumer — kolu's failing leg — on its own. **R-pulamweb-2** stands up the *entire framework* (provision · dial · fan-out · mirror · re-serve · browser-consume) rendering **only a terminal list** — no features, so a plumbing failure has nowhere to hide. **R-pulamweb-3** layers the user-facing features (live git status, drill-in), combining R-pulamweb-2's proven framework with R-pulamweb-1's proven consumer. **R-pulamweb-1 and R-pulamweb-2 are independent** (one's a drishti change, the other a fresh app); **R-pulamweb-3 needs both**; **R-compose-2 gates on R-pulamweb-3**.

### R-pulamweb-1 — the reactive stream consumer, in drishti ✅ shipped

<Phase id="R-pulamweb-1" name="reactive stream consumer (drishti)" status="shipped" blocks={["R-pulamweb-3"]} links={[{ label: "drishti #72", href: "https://github.com/srid/drishti/pull/72" }]} />

**Shipped** ([drishti #72](https://github.com/srid/drishti/pull/72)). Before pulam-web exists, prove its riskiest piece — a reactive stream consumer that survives delta accumulation — where the browser ⇄ ssh ⇄ mirror stack already runs: **drishti**. drishti's `processesSnapshot` (`packages/common/src/surface.ts`) is mirrored per host and was consumed **imperatively** — `streamCall(app.rpc.surface.processesSnapshot.get)` + `for await` + a teardown controller. This graduates that one call site to a declarative reactive subscription. The exercise surfaced the trap that **is** R-compose-2's lesson:

**`.streams.use()` is for *value-bearing* streams; `processesSnapshot` is *delta-accumulate*.** The first cut switched the call site to `app.streams.processesSnapshot.use()` and copied the latest frame into a `reconcile` store from a coarse `createEffect`. That **silently drops same-shape delta frames**: `.streams.use()` writes each frame into a `reconcile` store, and a coarse reader (`const msg = sub()`) only re-fires for the paths it actually reads — so two consecutive deltas differing only in a nested field (a hot PID's `cpuPct` ticking under an unchanged key set) coalesce and the second is lost; live values freeze in steady state. `processesSnapshot` is snapshot-**then-delta** — each frame is a *change*, not the full state — so the consumer must accumulate.

**The fix — and the rule it pins.** Drop one level to **`createSubscription` + `reduce`**: the reducer folds *every* frame in the subscription's own `for await` loop (no frame can be coalesced away), and the table renders **fine-grained** off the accumulated value (`processes()[pid].cpuPct`), so an in-place `reconcile` leaf update re-notifies exactly that cell. The hermetic test drives a **same-shape delta** (cpu `10 → 20 → 30`) and asserts a fine-grained reader observes `30` — the exact case the coarse copy dropped. Because drishti's CI is typecheck + nix only (no test lane), the review gauntlet, not CI, is what caught this — which is precisely why graduating it in drishti *before* R-compose-2 earns its keep. A drishti PR — independent of R-pulamweb-2's framework, the thing **R-pulamweb-3**'s git-status consumer rides on.

### R-pulamweb-2 — the framework (terminal list only) ✅ <PrLink pr={1524} />

<Phase id="R-pulamweb-2" name="framework — provision · fan-out · list" status="shipped" blocks={["R-pulamweb-3"]} links={[{ label: "#1524", href: "https://github.com/juspay/kolu/pull/1524" }]} />

R-pulamweb-2 stands up **everything hard** — auto-provision + dial N pulam over ssh, mirror each, fan out, re-serve to the browser over `/rpc/ws`, and consume in the browser — but renders **only a plain list of terminals** (grouped by host). No git status, no drill-in, no features. The point is to prove the *plumbing* with a payload so small a failure has nowhere to hide.

- **Backend (Node)** — per host, `buildEntry(host)`: `getHostSession({ binary:"pulam", resolveDrvPath, … })` (the pooled, **reconnecting** session — `getHostSession` + `makeClientCursor`, *not* one-shot `dialAgentOnce`, so a transient ssh drop self-heals), a per-host re-serve of the **whole** `terminalWorkspaceSurface` via `implementSurface` (`implementSurface` fail-fast-throws on any unimplemented primitive, so the re-serve folds `version` + `awareness` from the mirror and **forwards** the rest — `fs.*`/`git.*` procedures via the live-procedure holder, the `activity`/repo/file streams via the live client — never a degraded stub), and `pumpRemoteSurface(session, makeSink)` (the shared reconnect-mirror loop) folding the agent's frames in. The N entries live in `buildHostRegistry({ buildEntry })`; one `RPCHandler.upgrade(ws)` per `/rpc/ws?host=<id>`, dispatched by an **app-local** upgrade block (`gateWsOrigin` + `gateStaleSocket` + `startWsHeartbeat`, all already shared). `pulam-tui`'s `hostConnect.ts` is the dial template.
- **Browser (SolidJS + Vite)** — per host a `surfaceClient(terminalWorkspaceSurface, websocketLink(ws?host=<id>))` → **`app.collections.awareness.use({})`** → render a list of terminals per host (`cwd` · foreground · agent), plus a coarse `connecting…` overlay until the first frame. This is the **`.collections.use()`** consumer (which drishti already proves); the *value-bearing stream* consumer (`.streams.use()`) is deferred to R-pulamweb-3. (For R-pulamweb-2 the host roster is a static `/api/hosts` boot list; dynamic add + the fine `copying…` provision state ride a small parent fleet surface in a follow-up — `terminalWorkspaceSurface` itself carries no connection cell.)

**Done when:** the browser shows a **live terminal list across N auto-provisioned pulam hosts** — its terminals appear and come/go live (awareness deltas) — all over the real ws. The mirror → re-serve → browser-store path is proven deterministically by the hermetic test (below). No features yet. That alone proves the framework: provision · dial · reconnect · fan-out · ws-serve · browser collection-consume.

### R-pulamweb-3 — the user-facing features (live git status + drill-in) ◀ do next

<Phase id="R-pulamweb-3" name="features — live git status · drill-in" status="next" needs={["R-pulamweb-1", "R-pulamweb-2"]} blocks={["R-compose-2"]} links={[{ label: "repaint prep #1534", href: "https://github.com/juspay/kolu/issues/1534" }, { label: "deselect #1533", href: "https://github.com/juspay/kolu/issues/1533" }]} />

With R-pulamweb-2's framework proven and R-pulamweb-1's stream consumer proven, R-pulamweb-3 just **layers the features** onto the pipeline:

- **Live git status** — wrap `subscribeRepoChange` + `getStatus` into a **value-bearing `gitStatus` stream** server-side (Design B), then consume it via **`.streams.use()` read fine-grained**: each frame *is* the full status, so the value-bearing hook fits directly — but paint the cell by reading its fields off the subscription, never coarse-read-and-copy (R-pulamweb-1's rule, or same-shape frames coalesce). This is the leg R-compose-2 rides on.
- **Drill-in + the polished UI** — port `pulam-tui`'s pure `gitCell` / `gitDetail` projections (`render.ts`) to web DOM; selection + the drill-in pane (the mockup above). The `RepoWatchSet` + repo-membership reconcile from `pulam-tui/src/fleet.ts` carry over (renderer-agnostic — `fleet.ts:6-7`) via a `FleetSink` over a Solid `createStore`; only the renderer is new.

<Callout kind="warn" title="The real R-compose-2 risk is the renderer, not the data (learned from #1510)">
The discarded #1510 reached **green** keeping the `{seq}`-pulse consumer — its months-long block was **one layer up**: `@pierre/trees` (the file-tree component kolu's Code tab uses) swallows the first controller emit after every re-subscribe, so a live change landing in that window updates the model but **never repaints the DOM** (value right, screen stale). master's old stream hid this by *fully remounting* the tree; a persistently-mounted, live-updating view exposes it. So R-pulamweb-3 must render its git view into a **mounted** (not remounted) tree and prove it **repaints on a same-shape change** — carrying #1510's vendored **`@pierre/trees` patch** (always bump the revision per emit) + the `reconcile`-into-store write (`useWatchedRead`). The hermetic test asserts a **DOM repaint**, not just a store re-notify. That is exactly the thing R-compose-2 needs proven — so pulam-web's git view should mirror kolu's Code-tab renderer, not a throwaway list.
</Callout>

<Callout kind="warn" title="The trap — three consumers, only one is kolu's">
`pulam-tui` reads streams by **raw `mirrorRemoteSurface` iteration**. Copy drishti for the *backend mirror + provision + ws-serve* and for *cells/collections* (`.use()`), but the **git-status stream is value-bearing (Design B) — consume it via `.streams.use()` read fine-grained** (R-pulamweb-1's rule: read the cell's fields off the subscription, never coarse-read-and-copy, or same-shape frames coalesce and freeze). That consumer is the reason pulam-web exists.
</Callout>

**The hermetic test.** No in-memory `websocketLink` double exists (only `directLink`/`stdio`/unix-socket). Model on R7's `mirrorRemoteSurface.test.ts`: stand up a real `terminalWorkspaceSurface` agent via `implementSurface`, connect it with `directLink`, drive pulam-web's `buildReServe` fold path (the mirror sink) with that client, then consume the **re-served** surface through a second `surfaceClient` + a Solid `reconcile` store — asserting the re-served value re-notifies on the second delta (agent → mirror → re-serve → browser-store). `buildReServe` is split so the mirror step takes an injected client, which is exactly what makes it driveable without ssh. Per **R-pulamweb-1's correction**, R-pulamweb-3's value-bearing git-status consumer must read **fine-grained** and the test must drive a **same-shape** second frame (the coalescing regression a coarse copy-into-store silently drops); R-pulamweb-2's awareness consumer asserts the simpler key re-notify. That catches a coalescing/reconcile break deterministically; a ws-flush break only shows over a real socket — so **pulam-web over real ws is the full proof**, the unit test necessary-but-not-sufficient.

**Definition of done — by phase.** **R-pulamweb-1 (drishti):** ✅ shipped ([drishti #72](https://github.com/srid/drishti/pull/72)) — `processesSnapshot` rendered live via `createSubscription` + `reduce`, read fine-grained, guarded by a same-shape-delta regression test. **R-pulamweb-2:** a live terminal list across N auto-provisioned hosts over the real ws — the *framework*, no features. **R-pulamweb-3:** a live git-status cell + drill-in updating within ~1s of a working-tree change (**video/screenshot over the real ws**), the consumer confirmed *in code* as `.streams.use()` read fine-grained + `reconcile`, the **file tree repainting on a same-shape change** (DOM, not just store) with #1510's `@pierre/trees` patch carried, the hermetic test green, and the recipe recorded for **R-compose-2**.

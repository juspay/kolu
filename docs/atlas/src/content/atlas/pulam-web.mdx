**pulam-web is [drishti](electricity.html)'s twin, pointed at the terminal-workspace surface.** Where drishti monitors per-host *processes* in a browser, pulam-web monitors per-host *terminals* — the fleet you already see in `pulam-tui`, but in a browser, **auto-provisioned over ssh**. It exists for one engineering reason on top of being a real product: it is **kolu's own browser-consumption leg, minus kolu** — `websocketLink` → `surfaceClient` → Solid `reconcile` — so it graduates the path [remote-terminals **R8b**](remote-terminals.html#r8) is blocked on, in a build whose console you can actually see.

**It is the same `@kolu/surface` code kolu and drishti run — that is the whole point.** Serve side: `implementSurface` → oRPC `RPCHandler` (`@orpc/server/ws`) → `handler.upgrade(ws)` on a `/rpc/ws` socket. Consume side: `websocketLink` → `surfaceClient`/`surfaceClients` → the reactive `.use()` hooks (`surfaceClient.ts` owns snapshot-then-deltas + `reconcile`). drishti proves this trio is electricity; kolu's Code tab proves `.streams.use()` for a **value-bearing** `gitStatus` stream. pulam-web is the first *standalone* app to put those together — value-bearing git status, over `.streams.use()`, **sourced from a mirror** — which is precisely R8b's path.

**Node, matching kolu's own stack.** The surface leg is identical `@kolu/surface` code on any runtime — drishti even writes its server Node-style (`@hono/node-server` + the `ws` package) and runs it on Bun *only* for its `Bun.build` client bundler. We go **Node**, like kolu-server, and bundle the SolidJS frontend with **Vite + `vite-plugin-solid`** (kolu's own client toolchain) rather than drishti's `Bun.build` — so pulam-web is kolu's stack end-to-end. drishti's *server / provision / mirror* code ports to Node verbatim (it is already Node-shaped); only the bundler differs. **There is no `serveOverWebSocket` helper** — the ~18-line ws-serve block is copied from the surface example (`packages/surface/example/src/server/main.ts:94-111`) or kolu-server (`packages/server/src/index.ts:427-510`).

<Callout kind="note" title="The design fork R8b rides on — resolve it here, grounded">
Two ways git status reaches a browser. **Design A** (what #1510 did, and stuck): ship terminal-workspace's raw `{seq}` pulse to the browser and re-query `getStatus` client-side via `useWatchedRead`. **Design B** (what `koluSurface` *already* does — `packages/server/src/surface.ts:271-277`, `{read:getStatus, install:subscribeRepoChange, isEqual}` — and the Code tab proves daily): wrap it into one **value-bearing** stream *server-side*; the browser consumes an ordinary value-bearing stream via `.streams.use()`. **Lead: Design B.** For a remote host the backend wraps the **mirrored** `subscribeRepoChange` + **forwarded** `getStatus` (R7) into that value-bearing stream — so the raw `{seq}` pulse never crosses to the browser, and the "ungraduated primitive" dissolves as an artifact of A. pulam-web confirms B end-to-end from a mirrored source; that becomes R8b's recipe.
</Callout>

## Implementation details

Three steps, each isolating **one** risk. **Step 0 (in drishti)** graduates the reactive *stream* consumer — kolu's failing leg — on its own. **R4.8a** stands up the *entire framework* (provision · dial · fan-out · mirror · re-serve · browser-consume) rendering **only a terminal list** — no features, so a plumbing failure has nowhere to hide. **R4.8b** layers the user-facing features (live git status, drill-in), combining R4.8a's proven framework with Step 0's proven consumer. **Step 0 and R4.8a are independent** (one's a drishti change, the other a fresh app); **R4.8b needs both**; **R8b gates on R4.8b**.

### Step 0 (in drishti) — graduate the reactive stream consumer

Before pulam-web exists, prove its riskiest piece — **`.streams.use()` + Solid `reconcile` for a *mirrored, value-bearing* stream** — where the browser ⇄ ssh ⇄ mirror stack already runs: **drishti**. drishti's `processesSnapshot` (a value-bearing snapshot-then-delta stream it mirrors per host, `packages/common/src/surface.ts:279`) is consumed today **imperatively** — `streamCall(app.rpc.surface.processesSnapshot.get)` + `for await` + a teardown controller (`packages/app/src/client/App.tsx:749,879`). Switch that one call site to **`app.streams.processesSnapshot.use(() => ({}))`** + render `.value()`. That proves the exact R8b consumer — a mirrored value-bearing stream → re-served → browser `.streams.use()` + `reconcile` — in a real browser with a visible console, and *deletes* the hand-rolled controller. It is the **R7 pattern again** (drishti graduated forwarded procedures via "Kill process"; here it graduates the reactive *stream* consumer via its process list) — a genuine drishti improvement (declarative replacing imperative), not a throwaway. **It de-risks the consumer; the git-status wrap (Design B) and the fleet UI stay pulam-web's.** A drishti PR — independent of R4.8a's framework, and the thing **R4.8b**'s git-status consumer rides on.

### R4.8a — the framework (the whole pipeline, rendering only a terminal list)

R4.8a stands up **everything hard** — auto-provision + dial N pulam over ssh, mirror each, fan out, re-serve to the browser over `/rpc/ws`, and consume in the browser — but renders **only a plain list of terminals** (grouped by host). No git status, no drill-in, no features. The point is to prove the *plumbing* with a payload so small a failure has nowhere to hide.

- **Backend (Node)** — per host: `getHostSession({ binary:"pulam", … })` + `provisionAgent` (auto-provision + dial over ssh), `mirrorRemoteSurface` the `awareness` collection, keyed in a `Map<host,{session,handler}>`, one `RPCHandler.upgrade(ws)` per `/rpc/ws?host=<id>`; surface the `copying→connecting→connected` lifecycle via a `connection` cell. drishti's `hostRegistry.ts` + `router.ts` are the templates; `pulam-tui`'s `connectArivuViaHost` is the provision wrapper. The ws-serve is the ~18-line block from the surface example.
- **Browser (SolidJS + Vite)** — `surfaceClients(websocketLink(ws))` → **`app.collections.awareness.use({})`** → render a list of terminals per host, plus the `connection`-cell `copying…` overlay. This is the **`.collections.use()`** consumer (which drishti already proves); the *stream* consumer is deferred to R4.8b.

**Done when:** the browser shows a **live terminal list across N auto-provisioned pulam hosts** — adding a host shows `copying→connected` and its terminals appear; terminals come and go live (awareness deltas) — all over the real ws. No features yet. That alone proves the framework: provision · dial · reconnect · fan-out · ws-serve · browser collection-consume.

### R4.8b — the user-facing features (the R8b proof + the product)

With R4.8a's framework proven and Step 0's stream consumer proven, R4.8b just **layers the features** onto the pipeline:

- **Live git status** — wrap `subscribeRepoChange` + `getStatus` into a **value-bearing `gitStatus` stream** server-side (Design B), consume it via **`.streams.use()` + `reconcile`** (the Step 0-proven hook — *not* `pulam-tui`'s raw `mirrorRemoteSurface` loop, *not* drishti's `for await`), and paint the row's git cell. This is the leg R8b rides on.
- **Drill-in + the polished UI** — port `pulam-tui`'s pure `gitCell` / `gitDetail` projections (`render.ts`) to web DOM; selection + the drill-in pane (the mockup above). The `RepoWatchSet` + repo-membership reconcile from `pulam-tui/src/fleet.ts` carry over (renderer-agnostic — `fleet.ts:6-7`) via a `FleetSink` over a Solid `createStore`; only the renderer is new.

<Callout kind="warn" title="The trap — three consumers, only one is kolu's">
`pulam-tui` reads streams by **raw `mirrorRemoteSurface` iteration**; drishti reads streams by **raw `for await` over `streamCall`**. *Neither* exercises the `.streams.use()` + Solid `reconcile` hook — the path kolu's Code tab uses and #1510 stuck on. Copy drishti for the *backend mirror + provision + ws-serve* and for *cells/collections* (`.use()`), but the **git-status stream must go through `.streams.use()` + `reconcile`, value-bearing (Design B)** — that consumer is the reason pulam-web exists.
</Callout>

**The hermetic test.** No in-memory `websocketLink` double exists (only `directLink`/`stdio`/unix-socket). Model on R7's `mirrorRemoteSurface.test.ts` (`implementSurface → directLink → consume`) **but drive it through `surfaceClient` + a Solid `reconcile` store**, asserting the reactive value re-notifies on the second delta. That catches a reconcile break deterministically; a ws-flush break only shows over a real socket — so **pulam-web over real ws is the full proof**, the unit test necessary-but-not-sufficient.

**Definition of done — by phase.** **Step 0 (drishti):** `processesSnapshot` rendered live via `.streams.use()`, its hand-rolled controller deleted. **R4.8a:** a live terminal list across N auto-provisioned hosts over the real ws — the *framework*, no features. **R4.8b:** a live git-status cell + drill-in updating within ~1s of a working-tree change (**video/screenshot over the real ws**), the consumer confirmed *in code* as `.streams.use()` + `reconcile`, value-bearing, the hermetic test green, and the adopted design recorded as **R8b's recipe**.

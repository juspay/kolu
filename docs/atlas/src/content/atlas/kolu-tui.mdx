**kolu itself, in the terminal.** kolu already has two *raw* single-daemon CLIs — [kaval-tui](pty-daemon-tui.html) reads PTY **bytes**, [pulam-tui](pulam-tui.html) reads **awareness** — and each is a scriptable face of one daemon. kolu-tui is a different animal: it **embeds the [terminal substrate](sutra.html)** (own process, own state — like kaval-tui is its own process, *not* a thin client of a running kolu-server) and lays out **kolu's own two-pane shape** — the **Dock rail on the left**, the **live terminal you type into as the main panel** — as a **full-screen TUI** on [OpenTUI](https://opentui.com). It's the **second consumer that proves the substrate graduated**, and it's built **remote-first** as the de-risk milestone for the [finale](remote-terminals-finale.html).

The layout is **kolu's own, in the terminal**: the **[Dock](dock-repo-identity.html) rail on the left** — every terminal you own, grouped by host, each with its live agent badge — and the **main panel is the live terminal you type into**. The same two-pane shape as kolu-web (Dock + canvas), rendered by OpenTUI. You don't leave one view to reach the other; both are on screen at once.

<Workbench />

You **switch between terminals** from the rail: `↑`/`↓` (or `j`/`k`) moves the selection and the **main pane swaps to that terminal**; `⇥` toggles focus between navigating the rail and typing into the terminal, so keystrokes go where you mean. The host headers make it *kinda like tmux sessions* — except a session can live on an ssh remote, and the rail's badge tells you which terminal is **awaiting you** while you work in another. A detached terminal keeps running on its host; the rail just shows it dimmer.

The badges are not a re-implementation: they are the **shared `agentProjection` folds** the web Dock and [pulam-web](pulam-web.html) already use — `agentBucket` (`working` · `awaiting` · `waiting`), with the **order≠colour** split intact (a just-finished `waiting` agent *ranks* idle but *paints* awaiting). kolu-tui is a **new renderer of the [Dock ⇄ fleet-mirror contract](dock-repo-identity.html)**, importing the folds, never re-deriving a `switch` over agent states.

<Callout kind="note" title="Typing is raw passthrough — and it must not steal the inner agent's chords">
When the main pane has focus, keystrokes are forwarded to that PTY **raw** (kolu-tui reuses kaval-tui's attach machine — the escape machine + reply filter + terminal restore, via `@kolu/terminal-protocol`). kolu-tui owns only the rail + its own chrome; the terminal pane is total passthrough. So its focus/switch chords must respect `input/prohibitedKeybinds.ts` — it never claims a chord an inner tool needs (Claude Code's `Ctrl+B`/`Ctrl+J`); the rail-focus toggle and the ssh-style line-start escape stay clear of them, exactly the reserved-keybinding rule the web client already follows.
</Callout>

**Create on a remote host** — `c`, pick a host; it's the substrate's `create(location)`, one keystroke and one code path, and the new terminal lands in the rail focused in the main pane:

<Callout kind="note" title="What it deliberately is NOT — the scope line that keeps it honest">
kolu-tui is the Dock rail + the terminal pane + **basic awareness**. It has **no Code-tab fs/git browser, no binary/image preview, no paste/upload, no transcript export, no recency clock of its own** (recency is kolu's remembered fact, absent from the memoryless snapshot a dial serves). Those are *full-client surface* work that lands in [F-REMOTE](remote-terminals-finale.html#fremote), not here. The smaller surface is the point: prove the *hard transport/lifecycle volatility* without the *web rendering* riding along.
</Callout>

## Architecture — embed the substrate · render with OpenTUI · reuse the Dock's folds

kolu-tui is **its own process**, composed of three parts that map cleanly onto kolu's existing boundaries:

1. **The embedded [substrate](sutra.html)** owns the lifecycle — `create(location)`, `attach(key)`, `forEachHost` at boot — exactly as kolu-server uses it. **This is Option B and it is load-bearing**: if kolu-tui were a thin renderer of a running kolu-server, nothing would graduate and it would be a second UI, not a second consumer. Because it embeds and dials, it *is* the proof.
2. **[OpenTUI](https://opentui.com) renders the two-pane layout** — specifically its **SolidJS reconciler `@opentui/solid`** (compile-time JSX → renderables, no virtual DOM, the same Solid model kolu's web client already uses), so the Dock rail and the terminal pane can be expressed in the framework kolu is built in. OpenTUI is the right reuse: it's the SolidJS-native TUI framework (the lib that powers opencode), and pulam-tui already road-tested it in-repo.
3. **The shared `agentProjection` folds** give the rail its meaning — `agentBucket` / `agentUrgency` / `agentPaintClass` from `@kolu/terminal-workspace`, imported (never re-derived), making kolu-tui the next renderer of the Dock ⇄ fleet-mirror contract.

<Callout kind="warn" title="The pulam-dial requirement — the proof condition, not an afterthought">
kolu-tui **must** dial the host's pulam for the rail's awareness badges, and that is deliberate. Awareness **and** fs/git are procedures on the *same* `terminalWorkspaceSurface`, served by the *same* `pulam` binary, reached through the *same* `getHostSession({ host, binary: "pulam" })` pool key (<Cite file="packages/surface-nix-host/src/hostSession.ts" lines="911-929" />; pulam-tui dials it the same way, <Cite file="packages/pulam-tui/src/hostConnect.ts" lines="56-64" />). So a kolu-tui that exercises the pulam dial for the Dock **has by construction exercised the exact transport fs/git rides** — same ssh provision, same `pulam --stdio` seam, same typed link. Even with no Code tab, kolu-tui proves the fs/git *transport*; what F-REMOTE adds is the *rendering*. Skip the pulam dial (e.g. fold only the in-process local producer) and the TUI stops being a de-risk — it becomes a second UI. This condition is what makes the milestone worth more than a product.
</Callout>

The three terminal-side clients, kept distinct:

| | kaval-tui | pulam-tui | **kolu-tui** |
| --- | --- | --- | --- |
| Dials | kaval, `binary: kaval` — `ptyHostSurface` | pulam, `binary: pulam` — `terminalWorkspaceSurface` | **embeds the substrate**, which dials *both* |
| Sees | raw PTY **bytes** | derived **meaning** (agent/git/PR), no bytes | a **live terminal pane** *and* the Dock rail |
| Renderer | raw passthrough (Node) | scriptable text/NDJSON (Node, *no OpenTUI*) | **full-screen OpenTUI** two-pane (`@opentui/solid`, Bun) |
| Proves | PTY transport over ssh | awareness producer over ssh | the **substrate orchestration** + both dials, end-to-end |

<Callout kind="note" title="pulam-tui shed OpenTUI; kolu-tui is where it earns its place">
[pulam-tui](pulam-tui.html) *was* a full OpenTUI dashboard, then **reverted to a thin Node CLI** (<PrLink pr={1582} />) once [pulam-web](pulam-web.html) became the browser fleet board — a raw `status`/`watch` snapshot has no place for a full-screen renderer. kolu-tui is the opposite case: a two-pane Dock+terminal you switch and type in genuinely needs a layout/widget/reactivity engine, and hand-rolling one would violate the repo's "prefer external libraries" rule. So OpenTUI comes back here, where it's load-bearing rather than redundant.
</Callout>

## What it proves vs skips

The milestone is honest only if the path it *exists to prove* is the one a done-criterion can only meet by hitting it — so name it exactly.

**Proves** (transfers verbatim to the full client):

- The **sealed substrate end-to-end** — every op routed by `TerminalKey`, no host-seam bypass possible.
- **Remote PTY** into the terminal pane over the kaval dial; **reconnect / blip survival** (`HostSession` backoff + the durable `kaval --stdio` daemon outliving the link).
- **Boot re-dial-re-adopt** and **adoption** across hosts — `forEachHost` + the branded complete-host sweep, exercised on a *real* second host.
- The **composite `TerminalKey`** under genuine pressure: two hosts whose bare PTY ids can collide, kept distinct by `{host, ptyId}`.
- The **rail fed over the pulam dial** — which, per the condition above, proves the **awareness *and* fs/git transport** the full client inherits, and exercises the shared `agentProjection` as a TUI renderer.

**Skips** (deliberately — full-client surface, lands in F-REMOTE): the fs/git **Code tab** UI, the binary-preview **byte route**, **paste/upload**, **transcript export**, **rich awareness rendering** + restore UX. kaval-tui already proves raw PTY; kolu-tui's added value is the *kolu orchestration*, the *two-pane Dock+terminal*, and the *dial coverage* — never the rendering F-REMOTE owns.

<Callout kind="accent" title="The de-risk, in one line">
kolu-tui lights the hardest, least-reversible third of remote terminals — transport, reconnect, host-lifecycle, adoption, the composite key — behind kolu's real Dock, with no web rendering, and ships as a usable product while doing it. The full client then inherits a *proven* substrate and only has to add its own pixels.
</Callout>

## Implementation & phases — OpenTUI/Bun, the agentProjection reuse, K1–K3

The library decisions are **made** (so an implementer doesn't have to guess):

- **Renderer: OpenTUI via `@opentui/solid`.** SolidJS reconciler, so the Dock rail and terminal pane are authored in the same framework as the web client; reuse the presentation-neutral `agentProjection` folds outright, and reuse the web Dock's *component logic* where it's portable (the structure + the folds port; DOM/CSS specifics do not — confirm the component-reuse depth at K1).
- **Runtime: Bun** — OpenTUI's native Zig core needs it (the same reason pulam-tui ran under Bun before reverting). The substrate dials kaval/pulam *daemons* over sockets/ssh rather than spawning native PTYs in-process, so the Bun process carries no native PTY dependency; the kaval/pulam daemons stay as they are.
- **Honest caveat:** OpenTUI is **pre-1.0** ("not ready for production" upstream). It's the right SolidJS-native bet and has in-repo precedent, but pin a version and treat a renderer-API break as a real K1 risk — the substrate/awareness work behind the Dock is unaffected either way.

Three increments, each shippable, each remote-from-the-start. They come **after** the [substrate](sutra.html) extraction.

- **K1 — embed + the Dock.** The skeleton that proves Option B: kolu-tui *embeds* the substrate (own process, under Bun) and renders the two-pane Dock rail + terminal on OpenTUI; `create`/`kill`, **switch-between-terminals** (rail selection swaps the main pane), and raw typing into the focused pane work across local and ssh hosts through the kaval dial. Reuses kaval-tui's raw attach machine ([the escape machine + reply filter + restore](pty-daemon-tui.html#phase-2--the-attach-loop-shipped-), via `@kolu/terminal-protocol`). *Done: a remote terminal appears in the rail, is selected/typed-into/killed through the substrate; the PTY survives a detach.*
- **K2 — survive the link.** Reconnect, boot re-dial-re-adopt, adoption — the substrate's hard volatility under real ssh conditions. *Done: kill the link mid-session → the pane reconnects and the remote PTY is intact; restart kolu-tui → it re-dials each host and re-adopts surviving PTYs; two hosts with colliding bare ids stay distinct.*
- **K3 — live rail badges over the pulam dial.** Dial each host's pulam, fold the snapshot stream through the shared `agentProjection` into each rail row's bucketed badge + live `●`. *Done: a remote agent's `working → awaiting` shows in the rail while you work in another pane — proving the pulam transport (and thus fs/git's) that F-REMOTE's Code tab will ride.*

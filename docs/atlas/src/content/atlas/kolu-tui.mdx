**kolu itself, in the terminal.** kolu already has two *raw* single-daemon CLIs — [kaval-tui](pty-daemon-tui.html) reads PTY **bytes**, [pulam-tui](pulam-tui.html) reads **awareness** — and each is a scriptable face of one daemon. kolu-tui is a different animal: it **embeds the [sutra](sutra.html) substrate** (own process, own state — like kaval-tui is its own process, *not* a thin client of a running kolu-server) connected to **one host**, and lays out **kolu's own two-pane shape** — the **Dock rail on the left**, the **live terminal you type into as the main panel**, and a **host selector** to switch which host you're on. A tmux-for-agents that *is the kolu UI*, minus the Code browser. It's the **second consumer that proves sutra graduated**, and it's built **remote-first** as the de-risk milestone for the [finale](remote-terminals-finale.html).

The layout is **kolu's own, in the terminal**: the **[Dock](dock-repo-identity.html) rail on the left** — **the selected host's** terminals, each with its live agent badge — and the **main panel is the live terminal you type into**. The same two-pane shape as kolu-web (Dock + canvas), rendered by OpenTUI. At the top of the rail sits a **host selector**: you're on **one host at a time** (*kinda like tmux sessions*), and switching hosts (`h`) swaps the whole rail to that host's terminals.

<Workbench />

Two kinds of switching, cleanly separated: `↑`/`↓` (or `j`/`k`) moves the **terminal** selection *within the current host* and the main pane follows; the **host selector** (`h`) changes *which host* you're on. `⇥` toggles focus between navigating the rail and typing into the terminal, so keystrokes go where you mean. Because a surface is scoped to one host, a terminal's id is just its **bare `ptyId`** — no composite key, no host threaded through every action.

The badges are not a re-implementation: they are the **shared `agentProjection` folds** the web Dock and [pulam-web](pulam-web.html) already use — `agentBucket` (`working` · `awaiting` · `waiting`), with the **order≠colour** split intact (a just-finished `waiting` agent *ranks* idle but *paints* awaiting). kolu-tui is a **new renderer of the [Dock ⇄ fleet-mirror contract](dock-repo-identity.html)**, importing the folds, never re-deriving a `switch` over agent states.

<Callout kind="note" title="One host at a time — the cross-host glance is pulam-web's">
kolu-tui shows one host's terminals, switched by the selector — it does **not** mix local and remote terminals in one rail. That's deliberate, and it's what lets the id stay a bare `ptyId` (no composite key). The cross-host *"which agent needs me across ALL my hosts, at once"* view already exists in the right place: [pulam-web](pulam-web.html), the browser fleet board that spans every host, sorted needs-you-first. kolu-tui is the **workbench for the selected host** (tmux-style); the fleet glance is pulam-web's.
</Callout>

**Create on the current host** — `c`; it's sutra's `create(opts)` on the connected host, one keystroke, and the new terminal lands in the rail focused in the main pane:

<Callout kind="note" title="What it deliberately is NOT — the scope line that keeps it honest">
kolu-tui is the Dock rail + the terminal pane + **basic awareness**, one host at a time. It has **no Code-tab fs/git browser, no binary/image preview, no paste/upload, no transcript export, no recency clock of its own** (recency is kolu's remembered fact, absent from the memoryless snapshot a dial serves). Those are *full-client surface* work that lands in [F-REMOTE](remote-terminals-finale.html#fremote), not here. The smaller surface is the point: prove the *hard transport/lifecycle volatility* without the *web rendering* riding along.
</Callout>

## Architecture — embed the substrate · render with OpenTUI · reuse the Dock's folds

kolu-tui is **its own process**, composed of three parts that map cleanly onto kolu's existing boundaries:

1. **The embedded [sutra](sutra.html) substrate**, `connectSutra(host)`, owns the lifecycle for the selected host — `create(opts)`, `attach(id)`, dial + reconnect + boot re-adopt — exactly as kolu-server uses it. **This is Option B and it is load-bearing**: if kolu-tui were a thin renderer of a running kolu-server, nothing would graduate and it would be a second UI, not a second consumer. Because it embeds and dials the host itself, it *is* the proof.
2. **[OpenTUI](https://opentui.com) renders the two-pane layout** — specifically its **SolidJS reconciler `@opentui/solid`** (compile-time JSX → renderables, no virtual DOM, the same Solid model kolu's web client already uses), so the Dock rail and the terminal pane can be expressed in the framework kolu is built in. OpenTUI is the right reuse: it's the SolidJS-native TUI framework (the lib that powers opencode), and pulam-tui already road-tested it in-repo.
3. **The shared `agentProjection` folds** give the rail its meaning — `agentBucket` / `agentUrgency` / `agentPaintClass` from `@kolu/terminal-workspace`, imported (never re-derived), making kolu-tui the next renderer of the Dock ⇄ fleet-mirror contract.

<Callout kind="warn" title="The pulam-dial requirement — the proof condition, not an afterthought">
kolu-tui **must** dial the connected host's pulam for the rail's awareness badges, and that is deliberate. Awareness **and** fs/git are procedures on the *same* `terminalWorkspaceSurface`, served by the *same* `pulam` binary, reached through the *same* `getHostSession({ host, binary: "pulam" })` pool key (<Cite file="packages/surface-nix-host/src/hostSession.ts" lines="911-929" />; pulam-tui dials it the same way, <Cite file="packages/pulam-tui/src/hostConnect.ts" lines="56-64" />). So a kolu-tui that exercises the pulam dial for the Dock **has by construction exercised the exact transport fs/git rides** — same ssh provision, same `pulam --stdio` seam, same typed link. Even with no Code tab, kolu-tui proves the fs/git *transport*; what F-REMOTE adds is the *rendering*. Skip the pulam dial (e.g. fold only the in-process local producer) and the TUI stops being a de-risk — it becomes a second UI. This condition is what makes the milestone worth more than a product.
</Callout>

The three terminal-side clients, kept distinct:

| | kaval-tui | pulam-tui | **kolu-tui** |
| --- | --- | --- | --- |
| Dials | kaval, `binary: kaval` — `ptyHostSurface` | pulam, `binary: pulam` — `terminalWorkspaceSurface` | **embeds sutra**, which dials *both* for the connected host |
| Sees | raw PTY **bytes** | derived **meaning** (agent/git/PR), no bytes | a **live terminal pane** *and* the Dock rail |
| Renderer | raw passthrough (Node) | scriptable text/NDJSON (Node, *no OpenTUI*) | **full-screen OpenTUI** two-pane (`@opentui/solid`, Bun) |
| Proves | PTY transport over ssh | awareness producer over ssh | the **sutra orchestration** + both dials, end-to-end |

<Callout kind="note" title="pulam-tui shed OpenTUI; kolu-tui is where it earns its place">
[pulam-tui](pulam-tui.html) *was* a full OpenTUI dashboard, then **reverted to a thin Node CLI** (<PrLink pr={1582} />) once [pulam-web](pulam-web.html) became the browser fleet board — a raw `status`/`watch` snapshot has no place for a full-screen renderer. kolu-tui is the opposite case: a two-pane Dock+terminal you switch and type in genuinely needs a layout/widget/reactivity engine, and hand-rolling one would violate the repo's "prefer external libraries" rule. So OpenTUI comes back here, where it's load-bearing rather than redundant.
</Callout>

## What it proves vs skips

The milestone is honest only if the path it *exists to prove* is the one a done-criterion can only meet by hitting it — so name it exactly.

**Proves** (transfers verbatim to the full client):

- The **sealed sutra substrate end-to-end** — every op routed by bare `ptyId` through the connected host's endpoint, no host-seam bypass possible.
- **Connecting to a real ssh host** and driving its **remote PTY** into the terminal pane over the kaval dial; **reconnect / blip survival** (`HostSession` backoff + the durable `kaval --stdio` daemon outliving the link).
- **Boot re-dial-re-adopt** and **adoption** — reconnecting to a host and re-adopting its surviving PTYs, exercised on a *real* remote host.
- **Host-switching** — the selector tears down one `connectSutra` and opens another (local ↔ remote), the connection-context model the whole plan rests on.
- The **rail fed over the pulam dial** — which, per the condition above, proves the **awareness *and* fs/git transport** the full client inherits, and exercises the shared `agentProjection` as a TUI renderer.

**Skips** (deliberately — full-client surface, lands in F-REMOTE): the fs/git **Code tab** UI, the binary-preview **byte route**, **paste/upload**, **transcript export**, **rich awareness rendering** + restore UX. kaval-tui already proves raw PTY; kolu-tui's added value is the *kolu orchestration*, the *two-pane Dock+terminal*, and the *dial coverage* — never the rendering F-REMOTE owns.

<Callout kind="accent" title="The de-risk, in one line">
kolu-tui lights the hardest, least-reversible third of remote terminals — dial, transport, reconnect, host-lifecycle, adoption, host-switching — behind kolu's real Dock, with no web rendering, and ships as a usable product while doing it. The full client then inherits a *proven* substrate and only has to add its own pixels.
</Callout>

## Implementation & phases — OpenTUI/Bun, the agentProjection reuse, K1–K3

The library decisions are **made** (so an implementer doesn't have to guess):

- **Renderer: OpenTUI via `@opentui/solid`.** SolidJS reconciler, so the Dock rail and terminal pane are authored in the same framework as the web client; reuse the presentation-neutral `agentProjection` folds outright, and reuse the web Dock's *component logic* where it's portable (the structure + the folds port; DOM/CSS specifics do not — confirm the component-reuse depth at K1).
- **Runtime: Bun** — OpenTUI's native Zig core needs it (the same reason pulam-tui ran under Bun before reverting). sutra dials kaval/pulam *daemons* over sockets/ssh rather than spawning native PTYs in-process, so the Bun process carries no native PTY dependency; the kaval/pulam daemons stay as they are.
- **Honest caveat:** OpenTUI is **pre-1.0** ("not ready for production" upstream). It's the right SolidJS-native bet and has in-repo precedent, but pin a version and treat a renderer-API break as a real K1 risk — the sutra/awareness work behind the Dock is unaffected either way.

Three increments, each shippable, each remote-from-the-start. They come **after** the [sutra](sutra.html) extraction.

- **K1 — embed + the Dock.** The skeleton that proves Option B: kolu-tui *embeds* sutra (own process, under Bun) and renders the two-pane Dock rail + terminal + host selector on OpenTUI; `create`/`kill`, **switch-between-terminals** (rail selection swaps the main pane), and raw typing into the focused pane work on a connected host — local **and** a remote you can select. *Done: a remote terminal appears in the rail, is selected/typed-into/killed through sutra; the PTY survives a detach.*
- **K2 — survive the link.** Reconnect, boot re-dial-re-adopt, adoption, host-switching — sutra's hard volatility under real ssh conditions. *Done: kill the link mid-session → the pane reconnects and the remote PTY is intact; restart kolu-tui → it re-dials the host and re-adopts surviving PTYs; the host selector switches local ↔ remote and each shows its own terminals.*
- **K3 — live rail badges over the pulam dial.** Dial the connected host's pulam, fold the snapshot stream through the shared `agentProjection` into each rail row's bucketed badge + live `●`. *Done: a remote agent's `working → awaiting` shows in the rail while you work in another pane — proving the pulam transport (and thus fs/git's) that F-REMOTE's Code tab will ride.*

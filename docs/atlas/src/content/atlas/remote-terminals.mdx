## User-facing description

<RemoteCanvas />

You switch hosts from the **ChromeBar** — *kinda like tmux sessions* — and a remote host's terminals sit on the canvas **beside** your local ones. A remote tile wears an `ssh` badge and carries the *same* live awareness a local tile does — git branch/dirty, PR status, agent state, and a **green dot** when it's moving bytes — and it survives a network blip or a kolu-server deploy with full scrollback. One local host is always present; remotes are added on demand, auto-detected from your ssh config.

## Architecture-level changes

<Svg svg={arch} caption="The multi-host shape. Every host runs the same hashed kaval daemon; kolu-server keys its terminal endpoint by hostId behind one TerminalEndpoint seam. The local driver (R2, shipped) spawns + supervises a kaval over a unix socket; the ssh driver (R3) reaches + provisions the same kaval closure over ssh stdio; the awareness fold (R4) mirrors a host-side arivu. Local awareness runs in-process; only remote crosses a process boundary." />

**One endpoint, bound to a dialed kaval.** A kaval is a daemon you *dial* — local over a unix socket, remote over ssh — so remote is **not a second backend**. There is *one* `TerminalEndpoint`, bound to a `HostLocation`-resolved kaval endpoint (`{kind:"local"}` today, `{kind:"remote",host}` in R3). That resolver is the **sole** place a tile maps to its kaval; everything downstream talks to the backend and never asks "which kind?". No `RemoteTerminalEndpoint` — R3 adds the ssh *driver* as an additive sibling of R2's local driver, never a retrofit.

Two volatility axes stay open (R1/R2 closed the rest):

| Axis — changes for its own reason | Encapsulated by |
| --- | --- |
| Where a terminal's state lives — this machine vs an ssh host | the `HostLocation`-resolved kaval endpoint — one impl + a discriminator, not a second backend |
| How the backend reaches its agent — transport · framing · reconnect | `HostSession` + `@kolu/surface/links/stdio` — unix socket today, ssh in R3 |

**Zero domain knowledge crosses the transport** (Hickey's cut): the agent runs the unmodified sensors and the boundary ships their *values*. An earlier per-domain design (`RemoteGitInfoProvider`, …) was transport adapters wearing domain costumes; the single-endpoint seam dissolved them.

<Callout kind="warn" title="The awareness fold is domain logic, not electricity">
R4's fold merges each remote `AwarenessValue` into the **co-owned** `terminalMetadata` entry and **splits** it across the persisted/live fence — `pr·agent·foreground` through the quiet live mutator, `cwd·git·lastAgentCommand·lastActivityAt` through the one that arms autosave. Lose the split and you either re-arm the agent-firehose (a perf regression) or stop persisting. It speaks the whole terminal domain *and* the autosave fence, so it stays **inline in kolu-server** — never a `@kolu/*` package.
</Callout>

**Already paid for, so R3–R5 are narrow.** `@kolu/surface-nix-host` ships `HostSession`, `provisionAgent` (ships the *derivation* — a darwin parent drives a linux remote with no cross-builder), and `resolveSystem`; `@kolu/surface` ships `mirrorRemoteSurface` (the per-primitive mirror). R2 already shipped the host-count-agnostic spine: `@kolu/surface-daemon-supervisor` and the per-connect `system.version` handshake. R3 is a driver behind a shipped seam; R4 a fold behind a shipped mirror; R5 one wire signal.

### R3 — remote over ssh

Defined in [kaval-sessions](kaval-sessions.html) (R3.1–R3.8). The spike shipped (R3.1–R3.6): a kaval is now dialable local-or-ssh from `kaval-tui`, with the warm ssh path provisioned, deduped, and connection-multiplexed (<PrLink pr={1373} /> · <PrLink pr={1377} /> · <PrLink pr={1378} />). What's left: **R3.7** — kolu-server's N-endpoint registry + the ssh driver (the `{kind:"remote",host}` arm, the `(host,id)` join, a persisted `location`), dialing the host's kaval directly via the shipped `frontDaemonOverStdio` plus a thin remote fs/git surface for the Code tab (awareness comes from arivu — R4); **R3.8** — the canvas **multiplex** (a tile per kaval, local always present) and validating the feel against tmux-switch.

### R4 — remote awareness fold

R4.1–R4.5 are defined in [arivu](arivu.html); **R4.6, the kolu fold, is defined here** (it is kolu-server glue, with no child note). The awareness daemon shipped standalone (R4.1–R4.5): the sensor library, `arivu` + `arivu-tui`, `--host`, and the live multi-host fleet board (<PrLink pr={1428} /> · <PrLink pr={1439} /> · <PrLink pr={1497} />). **R4.6** is the kolu consumer: dial a *remote* arivu over a long-lived `HostSession`, mirror its `awareness` collection with `mirrorRemoteSurface`, and fold each `AwarenessValue` into `terminalMetadata` across the persisted/live fence (see the callout above), stamping `location` kolu-side. Four irreducible, remote-only costs: closure→frame reification (already shipped, R4.3), the fold + fence split, semantic version-skew (bounded by re-provision-on-dial), and the kolu-side `location` stamp. **Gated on R3** — there's no remote tile to fold into until R3 ships one.

<a id="r5-defn"></a>

### R5 — overflow-recovery loop

**Defined here** (kolu/kaval-side, no child note). ~130 LoC w/ tests. kaval sheds a slow attach subscriber by *ending* its iterator — today indistinguishable on the wire from a PTY exit, so the client treats the drop as terminal and silently freezes scrollback (`onOverflow` exists but is unwired). Thread a typed *overflow → re-attach* signal through the attach contract so the client re-attaches for a **fresh snapshot** (reusing the existing `onRetry` scroll-lock reset), distinct from PTY-exit. *Open:* whether a drop is reachable on the local socket at all, or only across an ssh hop — if the latter, R5 collapses into an R3 companion.

## User-facing description

<RemoteCanvas />

You switch hosts from the **ChromeBar** — *kinda like tmux sessions* — and a remote host's terminals sit on the canvas **beside** your local ones. A remote tile wears an `ssh` badge and carries the *same* live awareness a local tile does — git branch/dirty, PR status, agent state, and a **green dot** when it's moving bytes — and it survives a network blip or a kolu-server deploy with full scrollback. One local host is always present; remotes are added on demand, auto-detected from your ssh config.

## Architecture-level changes

<Svg svg={arch} caption="The multi-host shape. Every host runs the same hashed kaval daemon; kolu-server keys its terminal endpoint by hostId behind one TerminalEndpoint seam. The local driver (R2, shipped) spawns + supervises a kaval over a unix socket; the ssh driver (R9) reaches + provisions the same kaval closure over ssh stdio; the remote mirror (R9) reaches a host-side pulam's awareness + fs/git, composed into one surface (R8). Local awareness runs in-process; only remote crosses a process boundary." />

**One endpoint, bound to a dialed kaval.** A kaval is a daemon you *dial* — local over a unix socket, remote over ssh — so remote is **not a second backend**. There is *one* `TerminalEndpoint`, bound to a `HostLocation`-resolved kaval endpoint (`{kind:"local"}` today, `{kind:"remote",host}` in R9). That resolver is the **sole** place a tile maps to its kaval; everything downstream talks to the backend and never asks "which kind?". No `RemoteTerminalEndpoint` — R9 adds the ssh *driver* as an additive sibling of R2's local driver, never a retrofit.

Two volatility axes stay open (R1/R2 closed the rest):

| Axis — changes for its own reason | Encapsulated by |
| --- | --- |
| Where a terminal's state lives — this machine vs an ssh host | the `HostLocation`-resolved kaval endpoint — one impl + a discriminator, not a second backend |
| How the backend reaches its agent — transport · framing · reconnect | `HostSession` + `@kolu/surface/links/stdio` — unix socket today, ssh in R9 |

**Zero domain knowledge crosses the transport** (Hickey's cut): the agent runs the unmodified sensors and the boundary ships their *values*. An earlier per-domain design (`RemoteGitInfoProvider`, …) was transport adapters wearing domain costumes; the single-endpoint seam dissolved them.

<Callout kind="note" title="kolu keeps its own copy of awareness — R8 deletes it">
Today the sensors write awareness into kolu's **own** `terminalMetadata` record — a private copy. R8 has them write the shared `terminalWorkspaceSurface.awareness` instead, and kolu (and the browser) just **read** it through one seam (`awarenessFor`). One copy, one writer. Detail in **R8**.
</Callout>

**Already paid for, so R5–R10 are narrow.** `@kolu/surface-nix-host` ships `HostSession`, `provisionAgent` (ships the *derivation* — a darwin parent drives a linux remote with no cross-builder), and `resolveSystem`; `@kolu/surface` ships `mirrorRemoteSurface` (now a total dual). R2 already shipped the host-count-agnostic spine: `@kolu/surface-daemon-supervisor` and the per-connect `system.version` handshake. So R9 is an ssh driver behind shipped seams; R8 makes kolu serve the one shared surface; R7 *completed* the surface mirror (<PrLink pr={1505} />); R6 grew pulam's fs/git (<PrLink pr={1506} />); R5 added the one wire signal that distinguishes a slow-subscriber drop from a PTY exit (<PrLink pr={1591} />).

### R5 — overflow-recovery loop

<Phase id="R5" name="overflow-recovery loop" status="shipped" links={[{ label: "#1591", href: "https://github.com/juspay/kolu/pull/1591" }]} />
*Independent — needed nothing, blocked nothing; landed on its own.*

**Shipped** (<PrLink pr={1591} />). kaval sheds a slow attach subscriber by *ending* its iterator, which was indistinguishable on the wire from a PTY exit — so the client treated the drop as terminal and froze scrollback. The attach contract (`ptyHostSurface`, bumped to 4.0 — a new *emitted* union variant is breaking for an older client, which the version predicate would otherwise wave through) now carries a typed `overflow` control frame, emitted as the stream's last frame when the host drops a lagging subscriber, distinct from a PTY `exit`. The web tier (`terminalEndpoint/local.ts`) reads it and **re-attaches for a fresh snapshot** — reset-then-snapshot so the repaint replaces stale rows rather than double-painting — instead of freezing; kaval-tui's existing re-attach loop reads it to skip writing a dataless frame; pulam ignores it (activity is best-effort). The drop is reproducible on the *local* socket, so it is CI-tested today (`inProcessPtyHost.test.ts`, `reattachingDeltas.test.ts`), and the hardened contract is a clean prerequisite for the remote reconnect R8/R9 lean on. [herdr vs kolu](herdr-vs-kolu.html) flagged the same gap.

### R8 — kolu serves the one surface (`terminalWorkspaceSurface`)

<Phase id="R8" name="kolu composes terminalWorkspaceSurface; koluSurface deleted" status="todo" needs={["R6"]} blocks={["R9", "R-pulamweb-4"]} links={[{ label: "renderer non-issue #1534", href: "https://github.com/juspay/kolu/issues/1534" }]} />

Today kolu invents its **own** surface: `koluSurface` carries the fs/git value-bearing streams (`gitStatus`, `fsListAll`, …) and keeps awareness in its own `terminalMetadata` record (a private copy) — duplicating the `terminalWorkspaceSurface` the `pulam` daemon already serves. R6 unified only the *implementation* (`createTerminalWorkspaceEndpoint`), not the surface. **R8 deletes the duplicate: kolu serves and reads `terminalWorkspaceSurface`, the same surface a remote host serves.**

Two things move onto the shared surface, in one phase:

- **awareness** — the sensors write `terminalWorkspaceSurface.awareness`; kolu reads it through one seam (`awarenessFor`) instead of keeping its own copy in `terminalMetadata`. kolu keeps only what it *authors* (id · restore-cwd · theme · layout · sleeping-state). One writer per fact.
- **fs/git** — the value-bearing `gitStatus`/`fsListAll` streams move onto `terminalWorkspaceSurface`, declared there and implemented once in `serveFsGit.ts` via the framework `pollOnEvent` arm (`{ read: getStatus, install: subscribeRepoChange, isEqual }`). kolu's Code tab reads them there; the `koluSurface` copies are deleted.

It ships **local-only, directly in kolu** — kolu already serves value-bearing streams and reads awareness in-process, so nothing needs proving first. The payoff is **R9**: because kolu already serves the one surface, dialing a remote host is just pointing that surface's per-terminal dispatch at the host's mirror — no private copy to maintain, no second data path.

(The `@pierre/trees` renderer once feared here is a non-issue — measured harmless in kolu, [#1534](https://github.com/juspay/kolu/issues/1534). [pulam-web R-pulamweb-4](pulam-web.html#r-pulamweb-4) then consumes this exact surface over the mirror, proving the remote leg ahead of R9.)

### R9 — kolu dials remotes

<Phase id="R9" name="kolu dials remotes" status="todo" needs={["R7", "R8", "R-pulamweb-4"]} blocks={["R10"]} />

The linchpin — it **needs R7** (the total mirror) and **R8** (kolu already serves the one `terminalWorkspaceSurface`). kolu-server gains the **N-endpoint registry + the ssh driver** — the `{kind:"remote",host}` arm, the `(host,id)` join, a persisted `location` — reusing the *same* `getHostSession` the spike (R3) built. It dials the host's **kaval** for the PTY (via the shipped `frontDaemonOverStdio`) → a remote tile; and dials its **pulam** over a *long-lived* `HostSession`, **mirroring the whole `terminal-workspace` surface** (R7) into one local handle. Because R8 already made that surface the one kolu serves the browser, R9 just **points the remote terminal's dispatch at the mirror** instead of the in-process impl. **No private copy to maintain** (R8 removed it), no second data path. The mirror's **streams** — awareness `activity` + the fs/git `{seq}` watchers — re-serve to the browser over the *same* `websocketLink` + `surfaceClient` leg **R-pulamweb-4** proves (and R9 turns the quiet `activity` stream live on it), so R-pulamweb-4's [pulam-web](pulam-web.html) graduation gates R9. `location` is persisted kolu-side, re-validated on reconnect against the remote kaval's live `cwd`. (Paste/upload is a `TerminalEndpoint` concern — the same seam, the `PRECONDITION_FAILED` guard.) The result: a remote terminal as capable as a local one — PTY, awareness badge, green dot, working Code tab. The shape is the **lead diagram** above.

**Adoption — a dial joins, never clobbers.** If the host already runs a kaval it is **adopted** (the same `adoptOrEnsure` spine the local boot uses); a duplicate is impossible by construction (the pid-gate's atomic `link(2)` EEXIST). On dial:

| the remote kaval is… | kolu does |
| --- | --- |
| absent | provision + spawn fresh (`adoptOrEnsure` → ensure) |
| live · wire-compatible | **adopt** — connect, never kill; its PTYs reconcile in |
| live · a build behind | adopt **+** an "update pending" nudge (orthogonal) |
| live · contract-skewed | recycle (kill → respawn) — only on a typed `DaemonContractSkewError` |
| unreachable (bad dial) | bounded retry, then **degraded — never killed** |

One kaval per remote host: dialing *joins* its terminals ("the prod terminals"), never racing to kill them.

<a id="r9a"></a>

**R9a — localhost dedup (the desync the dashboard shows).** Under `PULAM_WEB_HOSTS=localhost`, pulam-web spawns its **own** pulam, so two independent sensor sets observe the same terminal and never reconcile — the `working`-in-pulam-web vs `idle`-in-Dock split. The fix is R9's backing-swap applied locally: don't spawn a second pulam — point pulam-web's `awarenessFor` at kolu's served awareness. One sensor, two readers. **Prereq:** R8 must *serve* `terminalWorkspaceSurface.awareness` cross-process (not only read it in-process), or there's nothing for the second process to read.

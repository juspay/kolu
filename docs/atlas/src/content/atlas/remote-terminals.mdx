## User-facing description

<RemoteCanvas />

You switch hosts from the **ChromeBar** — *kinda like tmux sessions* — and a remote host's terminals sit on the canvas **beside** your local ones. A remote tile wears an `ssh` badge and carries the *same* live awareness a local tile does — git branch/dirty, PR status, agent state, and a **green dot** when it's moving bytes — and it survives a network blip or a kolu-server deploy with full scrollback. One local host is always present; remotes are added on demand, auto-detected from your ssh config.

## Architecture-level changes

<Svg svg={arch} caption="The multi-host shape. Every host runs the same hashed kaval daemon; kolu-server keys its terminal endpoint by hostId behind one TerminalEndpoint seam. The local driver (R2, shipped) spawns + supervises a kaval over a unix socket; the ssh driver (R8) reaches + provisions the same kaval closure over ssh stdio; the fold (R8) mirrors a host-side arivu's awareness + fs/git. Local awareness runs in-process; only remote crosses a process boundary." />

**One endpoint, bound to a dialed kaval.** A kaval is a daemon you *dial* — local over a unix socket, remote over ssh — so remote is **not a second backend**. There is *one* `TerminalEndpoint`, bound to a `HostLocation`-resolved kaval endpoint (`{kind:"local"}` today, `{kind:"remote",host}` in R8). That resolver is the **sole** place a tile maps to its kaval; everything downstream talks to the backend and never asks "which kind?". No `RemoteTerminalEndpoint` — R8 adds the ssh *driver* as an additive sibling of R2's local driver, never a retrofit.

Two volatility axes stay open (R1/R2 closed the rest):

| Axis — changes for its own reason | Encapsulated by |
| --- | --- |
| Where a terminal's state lives — this machine vs an ssh host | the `HostLocation`-resolved kaval endpoint — one impl + a discriminator, not a second backend |
| How the backend reaches its agent — transport · framing · reconnect | `HostSession` + `@kolu/surface/links/stdio` — unix socket today, ssh in R8 |

**Zero domain knowledge crosses the transport** (Hickey's cut): the agent runs the unmodified sensors and the boundary ships their *values*. An earlier per-domain design (`RemoteGitInfoProvider`, …) was transport adapters wearing domain costumes; the single-endpoint seam dissolved them.

<Callout kind="warn" title="The awareness fold is domain logic, not electricity">
R8's fold merges each remote `AwarenessValue` into the **co-owned** `terminalMetadata` entry and **splits** it across the persisted/live fence — `pr·agent·foreground` through the quiet live mutator, `cwd·git·lastAgentCommand·lastActivityAt` through the one that arms autosave. Lose the split and you either re-arm the agent-firehose (a perf regression) or stop persisting. It speaks the whole terminal domain *and* the autosave fence, so it stays **inline in kolu-server** — never a `@kolu/*` package.
</Callout>

**Already paid for, so R5–R9 are narrow.** `@kolu/surface-nix-host` ships `HostSession`, `provisionAgent` (ships the *derivation* — a darwin parent drives a linux remote with no cross-builder), and `resolveSystem`; `@kolu/surface` ships `mirrorRemoteSurface` (the per-primitive mirror). R2 already shipped the host-count-agnostic spine: `@kolu/surface-daemon-supervisor` and the per-connect `system.version` handshake. So R8 is a driver + fold behind shipped seams; R7 *completed* the surface mirror (<PrLink pr={1505} />); R6 grows arivu's fs/git; R5 is one wire signal.

### R5 — overflow-recovery loop

~130 LoC w/ tests; the next step. kaval sheds a slow attach subscriber by *ending* its iterator, today indistinguishable on the wire from a PTY exit, so the client treats the drop as terminal and freezes scrollback (`onOverflow` exists but is unwired). Thread a typed *overflow → re-attach* signal through the attach contract so the client re-attaches for a **fresh snapshot** (reusing the existing `onRetry` reset), distinct from PTY-exit. **Ships independently of the remote work**: a slow subscriber is reproducible on the *local* socket, so it is CI-testable today, and the fix hardens the attach contract for *both* transports — a clean prerequisite for the remote reconnect R8 leans on. [herdr vs kolu](herdr-vs-kolu.html) flagged the same gap.

<a id="r6"></a>

### R6 — `@kolu/terminal-workspace`: one library, one surface, two homes

The host-side library that knows a terminal's working directory becomes **`@kolu/terminal-workspace`** — the shipped `@kolu/terminal-awareness` (R4.1) grown to cover fs/git too, run *in-process* locally (kolu) and hosted by *arivu* remotely: the same two homes the sensors already proved. It serves **one `@kolu/surface` surface**; the primitives differ only in *kind*:

- `awareness` (collection) + `activity` (stream) — the derived status badge + the green-dot liveness. **Shipped.**
- `fsListAll` · `fsReadFile` · `gitStatus` · `gitDiff` (procedures) + `subscribeRepoChange` / `FileChange` (watcher streams) — the Code tab's raw reads + live deltas. **New in R6**, lifted from kolu-server's `localEndpoint` (a thin wrapper over `kolu-git` + `serve-dir`) into the shared library.

This kills the duplication wart: without it, arivu would **reimplement** fs/git and kolu would carry *two* fs/git impls (its own local, arivu's remote). Instead there is **one** — in-process locally, via arivu remotely. Only the thin fs/git wrapper moves; the terminal-endpoint *orchestration* around it (spawn · adopt · the registry) stays kolu's and gets a remote impl in R8. (One surface; R8 mirrors it whole — via R7's total mirror — folding the `awareness` collection into `terminalMetadata` and *forwarding* the fs/git procedures. Nothing is folded into `AwarenessValue` — procedures were never part of it.)

<a id="r7"></a>

### R7 — `mirrorRemoteSurface` is now a total dual — *shipped* <PrLink pr={1505} />

`mirrorRemoteSurface` is the consume-side **dual of `implementSurface`** — and a dual must mirror *every* primitive its producer wires. `implementSurface` wires procedures too, but the mirror's `SurfaceSink` carried only `cells`/`collections`/`streams`/`events`: a **leaky half-mirror**, harmless only because every surface mirrored so far had **zero procedures**. R7 closed the gap by kind: **streaming state stays PUSH** (subscribed and folded into the sink); a **procedure is PULL** (a local call runs on the remote and returns), so it can't sit in the push-only sink — it comes back as a **forwarding stub**. The return is therefore `{ procedures, done }`: `procedures.<ns>.<verb>(input)` relays to the remote (total — every procedure gets a stub, since a passthrough has no standing cost), and `done` is the former settle-on-link-close promise. Graft the stubs into a re-serve and `serve ∘ mirror ≈ identity` — the location-transparency the whole epic rests on — **pinned in-repo** by a hermetic unit test (`implementSurface` → `directLink` → mirror → re-serve → call). A `@kolu/surface` change → the **drishti merge-gate**.

**Proven by drishti — the canonical external second consumer.** [drishti](https://github.com/srid/drishti) is the browser ↔ Bun ↔ ssh process-monitor that already mirrors per-host *process* surfaces (the graduation that first proved `@kolu/surface` is electricity), and it was **read-only**. R7's external proof is drishti gaining a **"Kill process"** action — the first *forwarded procedure* on a mirrored surface, exercised end-to-end (browser → Bun → ssh → remote agent → `kill(pid)`), **zero kolu**. That proof is the **merge-gate** for this kolu PR, not optional decoration: the return shape changed from `Promise<void>` to `{ procedures, done }`, and drishti `master` still `await`s the old shape on its streaming sinks ([drishti #70](https://github.com/srid/drishti/pull/70) adopted the #1497 `mirrorRemoteSurface`, all CI green). So the paired [drishti R7 PR #71](https://github.com/srid/drishti/pull/71) — audit that `await` to `.done`, pin drishti's kolu source to this R7 revision, go green — is now **open and gates #1505**: its full CI is green against the new `{ procedures, done }` API — all 18 odu nodes (every lane on both `aarch64-darwin` and `x86_64-linux`), the "Kill process" forwarded-procedure action landing on top of the migrated mirror — exactly the lane set #70 passed to gate #1497. Because that proof lives entirely in drishti's own surface, **R7 stands alone — it needs neither R6 nor kolu**; R8 then consumes a *proven* total mirror for arivu's fs/git, not a speculative one.

<a id="r8"></a>

### R8 — kolu dials remotes + the fold

The linchpin — it **needs R6** (the shared fs/git library) **and R7** (the total mirror). kolu-server gains the **N-endpoint registry + the ssh driver** — the `{kind:"remote",host}` arm, the `(host,id)` join, a persisted `location` — reusing the *same* `getHostSession` the spike (R3) built. It dials the host's **kaval** for the PTY (via the shipped `frontDaemonOverStdio`) → a remote tile; and dials its **arivu** over a *long-lived* `HostSession`, **mirroring the whole `terminal-workspace` surface** (R7) into one local handle. Because the mirror is total, kolu binds the remote `TerminalEndpoint` to that single handle and runs the **same** logic as against the in-process local surface — never asking "which kind?":

- the `awareness` collection + `activity` stream are subscribed into kolu's **sink**, which *folds* each `AwarenessValue` into the tile's co-owned `terminalMetadata` across the persisted/live fence (see the callout) and lights the **green dot**;
- the fs/git **procedures** (`fsReadFile`/`gitDiff` + the watcher deltas) are **forwarded** on demand for the Code tab.

`location` is co-owned with `terminalMetadata`, persisted kolu-side, re-validated on reconnect against the remote kaval's live `cwd`. (Paste/upload is a `TerminalEndpoint` concern — served behind the same seam with the `PRECONDITION_FAILED` guard — not awareness, not fs/git.) The result: a remote terminal as capable as a local one — PTY, awareness badge, green dot, working Code tab. The shape is the **lead diagram** above.

**Adoption — a dial joins, never clobbers.** If the host already runs a kaval it is **adopted** (the same `adoptOrEnsure` spine the local boot uses); a duplicate is impossible by construction (the pid-gate's atomic `link(2)` EEXIST). On dial:

| the remote kaval is… | kolu does |
| --- | --- |
| absent | provision + spawn fresh (`adoptOrEnsure` → ensure) |
| live · wire-compatible | **adopt** — connect, never kill; its PTYs reconcile in |
| live · a build behind | adopt **+** an "update pending" nudge (orthogonal) |
| live · contract-skewed | recycle (kill → respawn) — only on a typed `DaemonContractSkewError` |
| unreachable (bad dial) | bounded retry, then **degraded — never killed** |

One kaval per remote host: dialing *joins* its terminals ("the prod terminals"), never racing to kill them.

<a id="r9"></a>

### R9 — canvas multiplex + validate

The UI: an active-kaval strip (local always present), a per-tile host hint, an ssh-config host picker — *kinda like tmux sessions*. Then **validate the feel** on real use: does N-kavals-at-once beat tmux-style one-at-a-time switching? Fall back to switch if multiplex doesn't earn its keep. The granularity call lands here, validated, not up front.

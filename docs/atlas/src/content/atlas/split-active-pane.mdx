Split a tile and you get two stacked terminals — a main pane on top, a sub-panel
below. You click into one, you start typing… and nothing on screen tells you
which one is listening. The split already *knows* — it routes your keystrokes to
the right pane and even restores that choice across sessions — but it keeps the
secret to itself. This note graduates the Sundry item **"Distinguish the active
terminal in a split"** into a plan: surface that hidden signal as a quiet visual
cue.

## User-facing description

The cue is a thin **teal rail** down the inside edge of whichever pane has focus —
teal is kolu's established "you are here" colour. It moves the instant focus moves,
so a glance answers "where do my keystrokes land?" without reading anything.

It shows only inside the **tile you're working in** and only while the split is
**open** — a collapsed split has just one terminal, so there's nothing to
distinguish. The active sub-*tab* already has its own highlight in the tab bar;
this is the missing, coarser signal: **main vs. sub**.

### The treatment is a chooser

Three quiet, static languages — pick one during plan review. All reuse the same
teal accent and add no motion, so `prefers-reduced-motion` needs no special case.

The rail is the least intrusive — it never dims live terminal output (rules out
**C**, which also collides with the canvas's existing parked-tile dim) and never
draws a box around the xterm grid (**B**). It's a single edge in the one colour
kolu already means by "focus."

## Architecture-level changes

There is almost nothing structural to change — and that's the point. The split
**already tracks which pane has focus**: `focusTarget: "main" | "sub"` lives in the
per-tile `useSubPanel` store, is read on every keystroke to route input, and is
even persisted for session restore. The work is to *render* that existing signal,
not to invent a new one.

<Svg svg={flow} caption="One existing signal drives the cue. focusTarget already exists; TerminalContent already gates on whether the tile is focused and the split expanded. We add only the last hop: mark the active pane and let CSS paint it." />

- **Reuse the source of truth, don't fork it.** The cue reads `focusTarget` — the
  same field that routes keystrokes (<Cite file="packages/client/src/terminal/useSubPanel.ts" />).
  A parallel "which pane looks active" flag, or a CSS `:focus-within` shortcut that
  re-derives the same fact a second way, would be two sources that can disagree.
  One signal, two consumers (input routing + the cue).
- **A presentational leaf, not electricity.** This hides no hard volatility
  (transport, reconnect, GPU-context loss) — it's a bounded bit of styling inside
  the terminal module. It stays in `TerminalContent` + the global stylesheet; it
  earns no `@kolu/*` package.
- **No knob, no fallback.** The cue is always on whenever a split is open and
  focused — no preference toggle (an override is a defect, not a feature) and no
  degraded path. It either reflects `focusTarget` or, if that signal were ever
  absent, the split itself would already be broken upstream.
- **Paint over, never reflow.** The cue is a CSS overlay drawn *above* the pane,
  exactly like the canvas tile-aura (<Cite file="packages/client/src/canvas/CanvasTile.tsx" />)
  — `pointer-events: none`, no border or padding that changes the pane's box. That
  matters: any size change to a pane forces xterm to refit its grid and resize the
  PTY. The cue must cost zero layout.

## Implementation details

Small and contained — one component gains a derived flag, the stylesheet gains a
rule, and the Sundry row graduates to this note.

1. **Mark the active pane** in <Cite file="packages/client/src/terminal/TerminalContent.tsx" />.
   It already computes `shouldFocusMain()` / `shouldFocusSub()` from
   `props.focused`, `isExpanded()`, and `focusTarget()` — the cue keys off the same
   conditions. Tag each of the two `Resizable.Panel`s with a `data-pane-active`
   marker (the focused pane only), so the live pane is addressable from CSS.
2. **Paint the rail** in <Cite file="packages/client/src/index.css" /> — a
   `data-pane-active` rule using `--color-accent`, drawn as a non-layout overlay
   (mirroring the `.tile-aura` block already in that file). Static; no
   `prefers-reduced-motion` branch needed.
3. **Graduate the Sundry item** — remove the row from
   <Cite file="docs/atlas/src/content/atlas/sundry.mdx" /> (per its own
   "Graduating an item" rule) and leave a one-line pointer here.
4. **Docs + changelog** — this note is the design of record; add an `Added` line to
   the changelog. No README/marketing surface lists split-pane focus behaviour, so
   the doc-sync is this note plus the changelog.

**Test (feature, so test-first):** an e2e scenario that opens a split, focuses the
main pane then the sub-panel, and asserts `data-pane-active` lands on the right
pane each time — reusing the existing split/sub-terminal steps
(<Cite file="packages/tests/step_definitions/sub_terminal_steps.ts" />).

**Risks, and why they're small**

- *xterm refit on a layout change* — avoided by construction: the cue is an overlay,
  not a border, so no pane box changes size.
- *Two "active" signals competing* — the cue is gated to the **focused tile**, so it
  never lights a pane in a tile you're not in; the canvas's active-tile outline and
  this inner rail live on different surfaces and can't be confused.
- *Mobile* — `TerminalContent` is shared by the canvas and the mobile tile view, so
  the cue appears in both with no extra work; evidence will cover the mobile layout.

### Open decision — leaning noted

- **Scope: focused tile only (leaning) vs. every open split.** Showing it only in
  the tile you're in keeps the canvas calm and removes any "is teal = active tile?"
  ambiguity. Showing it everywhere would let you read each split's pending pane at a
  glance, but adds visual noise. Lean: **focused tile only**; revisit if the
  always-on version proves more useful in practice.

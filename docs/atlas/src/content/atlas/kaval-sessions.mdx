**A reframe of [remote-terminals](remote-terminals.html)'s R-2, proposed now kaval ships.** Old R-2 made *remote* a heavyweight per-tile **backend** (a `RemoteTerminalBackend`, a parent-side screen mirror, a bespoke agent). This proposes a lighter substrate: **a kaval is a daemon you dial** — local over a unix socket, remote over ssh — and Kolu can put a tile on any kaval. Whether the canvas shows **one kaval at a time** (tmux-style switch) or **multiplexes several** (a tile per kaval, side by side) is a UI choice on the *same* substrate — and because kolu is a *canvas*, it leans multiplex.

## The substrate — a kaval is a daemon you dial

kolu-server already keys its pty-host endpoint + status by `hostId` (a `Map`, instantiated as one — `local` — today). This instantiates more keys: a registry of kaval endpoints, each reached by a driver — unix socket (local, shipped) or ssh stdio (`HostSession`, R-2's only new transport, the same closure shipped by `@kolu/surface-nix-host`).

**There is one backend, not many** — and one backend means one *shape*, not one host. A backend is **three surfaces bound to an endpoint** — PTY · fs · git — and local vs remote is only the transport each is dialed over. The PTY surface already works this way (kaval at a unix socket since B2, or ssh stdio); fs and git extend the **same** pattern — served at the host, mirrored over the same `HostSession` link (`mirrorRemoteCollection`, the R-1.5 remote-process-monitor pattern). **No reach-over-ssh per op, no polling.**

| At the endpoint | local | remote — same `HostSession` link |
| --- | --- | --- |
| **PTY** — kaval (fds + OSC taps) | unix socket | ssh stdio |
| **provider DAG + fs/git watchers** — run where the files are | in kolu-server | in the provisioned agent → stream metadata |
| **Code-tab fs** — browse · read · *watch* | node fs (native) | remote fs surface · snapshot-then-delta |

The **awareness** providers (cwd · title · foreground · command · agent) are *tap-fed*, so they are transport-free. kaval stays the durable survivor; the watcher tier **re-runs fresh** — the #1031 line, now applied remote-side. So R-1's `TerminalBackend` *interface* anticipated a second (remote) *implementation* the kaval model dissolves: not two backends, one backend whose surfaces bind to an endpoint. Adding a kaval is **adoption you already shipped** — `adoptOrEnsure` (B3.3) adopts a live survivor's PTYs (else provisions + spawns), now run per endpoint — and multiple *local* kavals fall out for free (kaval is namespaced per kolu-server port, `kaval-<port>/`, <PrLink pr={1313} />).

## Switch or multiplex?

The open question — and the same substrate answers either, so it is a **later, reversible** UI call that the spike below does not depend on:

| | switch (tmux-style) | multiplex (canvas) — leaning |
| --- | --- | --- |
| the canvas | one kaval in view | tiles span kavals |
| per-tile `location` | unneeded | additive — the R-1 discriminator, finally used |
| live endpoints | one, swapped | N concurrent |
| host hint | one *current* pointer | a light per-tile tag; ChromeBar shows the *active* kavals |
| why it fits kolu | simplest | the canvas is spatial + heterogeneous *by design* |

The lean is **multiplex**: tmux-switch is a *terminal-multiplexer* metaphor, but kolu is a *canvas* — "watch a local build beside a prod-ssh tile" is the point of a canvas, not a corner case. The cost over switch is modest and clean — per-tile `location` returns, but **additively** (no `RemoteTerminalBackend`, no screen mirror — just *which endpoint holds this tile's PTY*), and the ChromeBar shows only the kavals this canvas *actually has tiles on*.

## Phases

| Phase | Ships | Note |
| --- | --- | --- |
| **P0** · backend cleanup | collapse R-1's anticipated backend polymorphism — a backend is **PTY · fs · git surfaces bound to an endpoint** (no `RemoteTerminalBackend`); byte-identical locally, the seam P3 fills with the remote impls | kolu-server only · low-risk · grounds P3 · can lead or run in parallel |
| **P1** · `kaval-tui create` | spawn a plain shell — the attach-needs-something prerequisite | thin over `terminal.spawn` |
| **P2** · `kaval-tui --host` | **reach + provision in one PR** (ssh), hosts auto-detected from ssh config | **the spike** — *is* the production `sshDriver`; the validation long-pole |
| **P3** · kolu dials remotes | N-endpoint registry + the ssh driver; the **remote agent** (kaval + provider DAG + fs/git surface) mirrored over the `HostSession` link; `location` additive on records | builds on P0 + P2 · carries the remote seams (`composeSpawnInput` inversion · paste/upload `PRECONDITION_FAILED` guard · `drvPath` via `resolveSystem`) |
| **P4** · the canvas UI | multiplex (or switch) — the active-kaval strip (local always present), per-tile host hint, ssh-config picker | the granularity call lands here |

**P1–P2 — the spike — in detail.** [kaval-tui](pty-daemon-tui.html) is the minimal client; it proves the transport before any Kolu UI, and because it and kolu-server dial the same socket through the same client, the spike **is** the production driver.

- **`kaval-tui create`** *(prereq)* — spawn a **plain shell** (a thin subcommand over `terminal.spawn`; B0 made spawn fully-specified, the tests already carry the minimal `spawnInput()`). kaval-tui today is `list` / `snapshot` / `attach` — its own source says *"spawn / kill land later."* A freshly-provisioned remote kaval has **zero** terminals, so `create` comes first.
- **`kaval-tui --host <ssh>`** *(reach + provision, one PR)* — `provisionAgent` (`nix copy --derivation` → realise) **then** dial over `HostSession`'s ssh `stdioLink`. Both halves already ship in `@kolu/surface-nix-host`, so it is one PR wiring two proven primitives.

<Callout kind="note" title="The spike is the substrate — and it doesn't depend on switch-vs-multiplex">
ssh <em>reach</em> + provision is the one genuinely-new transport, shared by both granularities and R-2's biggest unknown. Proving it in kaval-tui builds the production driver and lets real feedback <em>settle</em> switch-vs-multiplex later — so it is the right validation step even though the canvas model isn't finalized. (P0 is independent kolu-server cleanup; it can land first in parallel without waiting on the spike.)
</Callout>

## Open questions & status

- **Resolved — host source:** auto-detected from ssh config; no registry to curate.
- **Leaning multiplex,** but validate the *feel* with a prototype once the substrate lands (does N-at-once actually beat tmux-switch in daily use?).
- **Resolved — fs/git/providers:** surfaces at the host, mirrored over the `HostSession` link — the watcher tier runs in the provisioned agent, awareness is tap-fed, no reach-over-ssh per op (see the substrate table). The one inherent cost is remote file *bytes* crossing ssh (physics), and they ride snapshot-then-delta.
- **Open — resize arbitration** across a remote hop, and the multiplex *feel* (does N-at-once beat tmux-switch in daily use?). *([herdr vs. kolu](herdr-vs-kolu.html) flagged multi-client resize as already-latent.)*

**Status — accepted (the approach).** R-2 in [remote-terminals](remote-terminals.html) is committed to this kaval-endpoint substrate — *one* backend, no complecting. Execution starts with the transport spike (`kaval-tui create` → `kaval-tui --host`), which validates the one genuinely-new piece; P0 (backend cleanup) can land first in parallel; the switch-vs-multiplex UI is a later, reversible call.

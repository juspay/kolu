**A reframe of [remote-terminals](remote-terminals.html)'s R-2, proposed now kaval ships.** Old R-2 made *remote* a per-tile property — every terminal carries a `location`, the grid mixes hosts, host-knowledge smears onto ~5 surfaces. This proposes the opposite: **a kaval is a daemon you dial, and Kolu switches between kavals like tmux sessions** — local over a unix socket, remote over ssh, one kaval in view at a time.

<KavalSwitcher />

> The conflation it dissolves: old R-2 fused *"more than one kaval can exist"* (real, valuable) with *"tiles mix hosts"* (what made it heavy). Keep the first, drop the second.

## The model — dial a kaval, switch like tmux

kolu-server already keys its pty-host endpoint + status by `hostId` (a `Map`, instantiated as one — `local` — today). This just instantiates more keys: a registry of kaval endpoints, each reached by a driver — unix socket (local, shipped) or ssh stdio (`HostSession`, R-2's only new transport). A *current* pointer picks which kaval the canvas renders.

<D2
  caption="kolu-server holds a registry of kaval endpoints keyed by hostId + a 'current' pointer. The canvas renders ONE current kaval. Remote is just the ssh driver; more local kavals are just more unix-socket endpoints. Switching repoints 'current' and adopts."
  code={`direction: down
ui: "kolu canvas — kaval switcher\\n(one current kaval in view)"
server: "kolu-server" {
  switch: "switch(hostId) = repoint 'current' + adoptOrEnsure"
  registry: "endpoint registry — Map<hostId, endpoint>" {
    epWork: "work (local) — unix socket"
    epProd: "prod (remote) — ssh stdio"
  }
}
kWork: "kaval @ work — survives deploys"
kProd: "kaval @ prod — survives (remote)"
ui -> server.switch: "pick a kaval"
server.registry.epWork -> kWork: "dial unix socket"
server.registry.epProd -> kProd: "dial over ssh (stdioLink)"
`}
/>

Switching is cheap because it is **adoption you already shipped**: `adoptOrEnsure` (B3.3) runs at boot against `local`; here it runs *on demand* against the chosen endpoint. Switch away and the kaval survives — the client just detaches. So host-knowledge collapses to one ambient pointer instead of smearing across the UI:

| | per-tile `location` (old R-2) | per-kaval switch (this) |
| --- | --- | --- |
| "which host" lives | every record + ~5 surfaces | one *current kaval* pointer |
| the grid | mixes hosts | one kaval in view |
| persistence | `location` threaded per record | per-kaval session + endpoint registry |
| the switch | — | `adoptOrEnsure` — already shipped |
| remote is… | a per-terminal kind | a transport (the ssh driver) |

Multiple *local* kavals fall out for free — kaval is already namespaced per kolu-server port (`kaval-<port>/`, <PrLink pr={1313} />), so several coexist today; the new capability is one server *switching among* them (isolation: a daemon per project).

## Prototype the reach in kaval-tui first

[kaval-tui](pty-daemon-tui.html) is the minimal client — no switcher, no chips, it just dials a socket and drives a PTY. Teaching it to dial a *remote* kaval isolates the **transport** from the **UI**, and because kaval-tui and kolu-server dial the same socket through the same client, the spike **is** the production `sshDriver`, not a throwaway.

Split *remote* into its two hard parts and prove the unknown one first:

- **Reach** — dial a remote kaval's socket over an ssh `stdioLink` (`HostSession`). *Never actually run yet.* Prove it first, against a kaval you **hand-start** on the box — fewest moving parts.
- **Provision** — `nix copy --derivation` the kaval closure, realise, run (`provisionAgent`). Well-trodden (how odu ships `odu-runner`, drishti its agent) — automate *after* reach works.

<Callout kind="note" title="The spike is on the critical path either way">
Per-tile or per-kaval, the ssh-<em>reach</em> transport is shared by both and is R-2's biggest unknown. Proving it in kaval-tui de-risks the part both models need — and lets real feedback <em>settle</em> the granularity before any switcher UI is designed. So "prototype in kaval-tui first" is the right first step even though the model above isn't accepted yet.
</Callout>

## Trade-off & status

**You give up the mixed grid** — a local editor tile beside a prod-ssh tile in one saved layout. Right trade for v1: it deletes R-2's dominant complexity, and it is **recoverable** — the `TerminalLocation` discriminator (`{ kind: "local" }`) still lives at the `getTerminalBackendFor` seam, so a later mixed-grid feature threads `location` additively, on a working switch. Open for the spike to answer:

- Does one-kaval-at-a-time chafe, or is tmux-style switching enough?
- Does the existing KAVAL rail column generalize to the switcher, and does a remote kaval's `DisconnectedOverlay` belong on the whole canvas, not per-tile?
- Endpoint registry: how are kavals named/added (ssh config? explicit list?), and does the per-kaval session partition compose with the restore-card flow?

**Status — proposed.** Gate: the kaval-tui ssh-*reach* spike (reach before provision). Lands clean and one-kaval-at-a-time holds → this supersedes per-tile R-2 in [remote-terminals](remote-terminals.html) and flips to `accepted`. Surfaces a real need for the mixed grid → per-tile returns, transport already proven. Either way the spike is the next concrete step.

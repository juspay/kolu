**Assume we throw away both <PrLink pr={1385} /> *and* its design, and start from the goal.** One way to ship it is a *resequencing* of #1385 — reverse-engineering safe horizontal *layers* out of the code we already have, which keeps the watcher in the PTY path, keeps `hostId`-absent-means-local threaded through the contract, keeps persisted remote-terminal records, and merely *re-times* the cold-boot orphan reconcile (the `fbc06c31` 10-second settle band-aid). This note does the opposite: it designs **vertically and risk-first against a cleaner core**, and instead of re-timing the boot race it **removes the race's inputs**. It stands on the shipped substrate (kaval, `getHostSession`, the `TerminalEndpoint` seam — all merged and validated) and rebuilds only the part #1385 got wrong: how kolu-the-app *consumes* that substrate.

## The core model

One thesis, and everything follows: **the host is the sole source of truth for its own live terminals, and kolu persists only a dial-list** — the set of `hostId`s to re-dial, never a remote-terminal record. A remote tile is just a local `TerminalEndpoint` (the shipped PTY·fs·git seam) whose surfaces are bound to an ssh transport; *local vs remote is only the transport*.

*The host agent runs **where the files live**, and only the **wiring** varies. Locally the providers **compose directly into kolu-server's surface** — a `{ collections, procedures }` + deps fragment spread in, exactly how `@kolu/surface-app` already composes into kolu-server (one router, no client, no wire, no Nix). The diagram shows the remote topology; remotely the host agent serves the **same** surface fragment over `stdioLink` and kolu-server **mirrors** it. Same contract, same provider code — only compose-vs-serve+mirror changes.*

The load-bearing principles:

- **Existence comes from one atomic pull, never a stream.** Boot, restore, and reconnect collapse into *one* routine: dial the host, `await` a single complete kaval `terminal.list`, **synchronously seed** a `terminalMetadata` entry (cwd + location) for every listed id so the client's membership filter (`meta.keys()`) reaches its full set in one tick, then project. There is no shadow tile and no empty→partial→full enumeration for the boot race to mis-settle on.
- **Persist only a dial-list.** Persist a record only for an endpoint that *dies with you* (the local PTY). The durable remote kaval outlives you, so you persist only `Set<hostId>` to re-dial — and that dial-list is the *sole* boot-dial authority. A remote-located terminal can never enter `SavedSession.terminals` (unit-tested), so no autosave can ever prune it.
- **Endpoint is first-class.** `location` is a non-optional **discriminated union** — `{ kind: "local" }` or `{ kind: "remote", hostId }` — so `local` is an explicit *variant*, not a reserved host-id string (a remote host aliased `local` in `~/.ssh/config` can't collide with the in-process endpoint). A dropped location is a compile error, not a silent local respawn. *(Shipped — <PrLink pr={1402} />.)*
- **Awareness is split by where its inputs live** — the one place a naïve clean-slate (and #1385's own framing) goes wrong. See below.

## What we stand on, what we throw away

### Stand on (the shipped foundation is correct)

- The **`TerminalEndpoint` PTY·fs·git seam** — the one correct shape; local and remote are the same interface over different transports.
- **`getHostSession` / `HostSession`** — the genuinely hard, already-solved physics: ssh reach + Nix provision + the ref-counted reconnect/give-up machine. The host-as-truth model leans on its reconnect loop to re-pull `terminal.list`.
- **kaval + its atomic `terminal.list`** — a durable remote PTY with *zero* kolu logic on the host, plus the complete-list pull that is the real canvas-existence signal.
- The **synchronous `seedMeta` upsert** at spawn — the client paints a tile only when its metadata has arrived, so this is *kept* and generalized to seed every id from the list.

### Throw away (this is the #1385 design being rejected)

- **The watcher forwarding the PTY.** <Pill variant="hi">reject</Pill> Putting the PTY data path through the most-volatile host process. Dial kaval *directly* for the PTY; the host agent keeps the PTY out of its path.
- **Persisted remote-terminal records + the dual boot-dial authority.** <Pill variant="hi">reject</Pill> Replaced by the dial-list, which *supersedes* the `savedRemoteTerminals → bootDialHosts` branch.
- **`preRegisterSavedTerminals` → shadow tiles + the settle-window orphan reconcile** <Commit sha="fbc06c31" />. <Pill variant="hi">reject</Pill> Deleted, not re-timed — replaced by the one atomic `terminal.list` + synchronous seed.
- **Optional `hostId` ("absent ⇒ local") threaded through the contract.** <Pill variant="hi">reject</Pill> Replaced by the non-optional `location` *sum* on the terminal surfaces — `{kind:"local"}` ∣ `{kind:"remote",hostId}` (P1, shipped <PrLink pr={1402} />); the activity feed's `hostId` is finished off in P2b.

### Honest cost scorecard

Calling every slice "thin" was wrong. The honest split:

- **Cheap, regardless of which remote path ships — P1 · P2 · P2b (+ P6).** Pure local-world refactors/UX; do them first — they turn the #1385 bug classes into compile errors and failing tests with no remote code.
- **Moderate — P4a, P4w, P3, P5, P9.** Real but contained. **P4w** is the local consumption refactor — extract `@kolu/terminal-providers` as a surface fragment and **compose it into kolu-server's surface** (no client, no mirror, no `directLink`, no ssh, no Nix); composition dissolves the *mirror* and the stream-replay/lost-update window the `directLink` boundary would otherwise have dragged in — the awareness→`terminalMetadata` fold itself stays, exercised locally first.
- **Fat — the cost centre — P4d.** *Go remote*: the `bin`/`serveOverStdio` ssh entrypoint + `captureLoginPath`, the **Nix closure** (real deps `@kolu/terminal-providers`, `kolu-git`, `gh`, `node`, guarded by a watcher `buildId.closure.test.ts`), the cross-arch drv-map, provisioning, and **serving the fragment over `stdioLink` + the remote mirror**. Two honest qualifiers: the Nix half is **not novel/scary** — it's a near-verbatim clone of the shipped, tested **kaval** packaging (same shared pnpmDeps FOD, same `hashString` staleKey, a closure-test that is *the analog of kaval's*); and the **mirror** — the cross-process copy of the awareness collection, with its genuine lost-update concern — that P4w's composition *avoids* genuinely reappears here, because remote is the only place there's a real wire to mirror (the fold runs in **both** wirings; only the mirror feeding it is remote-only).

> **Caveat:** **P1 is now on master** (<PrLink pr={1402} />) — landed directly on trunk as the recommended first move, *not* cherry-picked from #1385, and with a better shape than this note first proposed (a discriminated union, not a reserved `LOCAL_HOST_ID` string). P2–P4d are still ahead and map to commits on the open <PrLink pr={1385} />, so "merges independently and stays green" is a per-slice *goal to re-validate* for those.

## The four bug classes, dissolved by construction

The point of designing afresh is that #1385's bugs become *unreachable*, not *fixed*:

| #1385 bug class | The construction that makes it impossible |
| --- | --- |
| **Cold-boot re-adoption race** (shadows wiped on an empty/partial frame; autosave prunes the session; boot stops re-dialing) | Existence comes from **kaval's atomic `terminal.list`** (dialed directly), *never* the watcher's incremental mirror — so there's no empty→partial→full enumeration to mis-settle, and the watcher only *enriches* existing keys. kolu persists only a dial-list (no record to prune); the client gate waits per host so a remote-only boot shows "dialing", never a blank canvas. (#1385's settle window existed precisely *because* it routed existence through the watcher's incremental enumeration — this design removes that cause, it doesn't re-time it.) |
| **Env leak** (remote PTY gets a restricted PATH) | Spawn is `{argv,env,initFiles}` fully-specified; env is built fresh from `system.info` — *and* the host **login PATH** is captured host-side (moved into kaval's `system.info`), so a kaval-direct dial reports it. An e2e asserts the PTY's PATH carries the user's profile dirs. |
| **Remote-repo prune** (recent-repos stat'd against local fs) | Reachability is an endpoint question: every record carries its `hostId`, and prune-stat routes through `endpointFor(record.hostId).fs`, tolerant of a down host. A unit test asserts a remote record is never stat'd against local fs. |
| **Restore-to-wrong-endpoint** (saved `hostId` dropped → respawn locally) | `location` is non-optional (a drop is a compile error); restore is *dial + adopt the host's list*, never read-record-and-spawn; the client respawn-by-`hostId` branch is neutered. |

## The honest correction: awareness is split

The seductive clean-slate idea — and the one the *adversarial review killed* — was "the first remote slice needs no host agent at all, and the providers just run in kolu-server over kaval's taps." Half true, and the half that's false is exactly the half #1385's existence already proves.

<Callout kind="warn" title="Why git/PR/agent awareness MUST run host-side">
The providers split cleanly by **where its inputs live**:

- **PTY-tap signals** — cwd, title, foreground, command — ride kaval's wire taps. These *can* run in kolu-server, and the kaval-direct shell slice (P4a) ships them with **no host agent**. That is a genuine surface-sharing win.
- **fs-derived signals** — git branch/dirty state, PR status, agent detection — read the host's own filesystem and home dir (`git/resolve.ts` runs `simpleGit` on the terminal's cwd; the claude/codex/opencode detectors read `~/.claude` under `os.homedir()`). Run *in kolu-server* against a remote cwd/pid, they inspect the **wrong machine** — empty git state, no PR, no agent, or a collision where a remote pid matches a local session file.

So git/PR/agent awareness *must* run on the host (P4w, the host agent) — which is **exactly why kolu-watcher exists**. The fresh design keeps that role and renames it the **host agent**, dropping only its PTY-forwarding. The honest scope: the host agent is the *awareness host* and the *fs/git host*, never the PTY path and never the canvas-existence signal. A from-scratch design that skipped this would re-learn #1385's lesson the hard way.
</Callout>

## One contract, two wirings — compose (local) vs serve+mirror (ssh)

The awareness providers are a **surface** — a typed contract — but *in the same process you don't dial a surface, you compose it.* `@kolu/surface` supports this directly, and `@kolu/surface-app` already does it against kolu-server (`surfaceAppSurfaceWith` + `surfaceAppServer`): a sub-surface ships as two spreadable fragments — a **contract** fragment (`{ collections, procedures }`, spread into the parent `defineSurface`) and a **deps** fragment, spread into the parent's `implementSurfaces`. One router, one transport, **no client**.

So `terminal-providers` does the same, and the only thing that varies is the wiring:

```
local   →  compose: spread { collections, procedures } + watcherDeps into kolu-server's surface; in-process fold → terminalMetadata   // one router · no wire · no client · no mirror
remote  →  host serves the SAME fragment over stdioLink; kolu-server mirrors the collection, then the SAME fold → terminalMetadata     // a real wire — only the mirror is new
```

- **P4w composes.** The host-derived providers run in-server and publish to the awareness collections through kolu-server's *own* ctx; a thin in-server **fold** reads those and writes `terminalMetadata` via the existing `updateServerMetadata`/`updateServerLiveMetadata`. What composition removes is **`directLink`, the client, and the mirror** — the awareness surface is *spread* into kolu-server's `defineSurface`/`implementSurfaces`, not dialed — and because the fold reads the awareness collection **synchronously from the shared ctx** (not over a `.get()` stream), **the lost-update window and the `firstPersisted` replay reconcile vanish**: those were artifacts of the stream hop a `directLink` dragged in, never of the fold. The fold **stays** — it projects the host-derived awareness subset (git/PR/agent) onto the unified `terminalMetadata` record (which also carries the in-server PTY-tap fields cwd/title/foreground), so it is *not* an identity even locally. It is the **sole** writer of those host-derived fields, so the persisted-vs-live `terminals:dirty` fence is preserved (it lives in *which* mutator the fold calls), and it **seeds from the restored record** so an adopted survivor's `lastActivityAt`/`lastAgentCommand` is never clobbered (B3.3). One carried-over detail: collections have no `connect` hook, so the providers' publish binds to the parent ctx via the existing `surfaceCtx` late-bind holder (the same `#1005` cycle-break kolu already uses) — kept kolu-local, so **no drishti PR**.
- **P4d serves + mirrors.** The *same* contract fragment is wrapped in its own served router over `stdioLink` on the host (the `bin`/`serveOverStdio` half); kolu-server's remote endpoint holds the client and **mirrors** the awareness collection, then runs the **same fold** P4w established. The **mirror** — and its lost-update concern, now real over an RPC-wide wire — lives **only here**, alongside the genuinely-remote surface a real box exercises: cross-arch `.drv`, ssh reconnect/recheck/`ControlMaster`, `nix copy` over ssh-ng.

This is the honest version of *"local vs remote is only the transport"*: **same contract fragment, same provider code, same fold** — locally the awareness collection is *composed* in and the fold reads it in-process (no link at all); remotely the host *serves* the collection and kolu-server *mirrors* it before the same fold runs. The boundary is real only where the processes actually are. (The earlier draft routed the local case through `directLink` too — a *fictional* in-process boundary that dragged in a mirror and a stream-replay/lost-update window; composing dissolves those — the fold itself was never the problem and stays.)

## Versus the resequencing

A layered *resequencing* of #1385's existing code and this from-scratch design are two real options, and they differ in kind:

- **Resequencing** (finish #1385's design in safe slices) is *layer-completion of the existing design*: lower review risk, faster to start (the code exists), but it preserves the worst coupling — the watcher in the PTY path, persisted remote records, the settle-window band-aid, optional-`hostId` everywhere. You ship the same architecture in safe slices.
- **from-scratch** is *dissolution-by-construction on a cleaner core*: more up-front rewrite, but the four bug classes become unreachable, the PTY is dialed straight at kaval, and the boot model is one atomic pull. Every remote PR is demoable from the first one, and the load-bearing awareness split is isolated and honestly scoped.

A pragmatic blend is available: the structural bug-proofing here (**P1/P2/P2b** — endpoint-first-class, dial-list, host-aware feed) is cheap, local-only, and *worth doing regardless of which path ships the remote code* — they convert the bug classes into compile errors and failing tests on the world we already have. **P4w earns its place in that set too**: extracting the provider pipeline into `@kolu/terminal-providers` and composing it as a surface fragment is better modularity and testability *on its own* — a good refactor even if remote never ships (the composed surface is also the seam that lets remote reuse the same contract).

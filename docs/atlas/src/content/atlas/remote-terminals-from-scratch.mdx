**Assume we throw away both <PrLink pr={1385} /> *and* its design, and start from the goal.** One way to ship it is a *resequencing* of #1385 — reverse-engineering safe horizontal *layers* out of the code we already have, which keeps the watcher in the PTY path, keeps `hostId`-absent-means-local threaded through the contract, keeps persisted remote-terminal records, and merely *re-times* the cold-boot orphan reconcile (the `fbc06c31` 10-second settle band-aid). This note does the opposite: it designs **vertically and risk-first against a cleaner core**, and instead of re-timing the boot race it **removes the race's inputs**. It stands on the shipped substrate (kaval, `getHostSession`, the `TerminalEndpoint` seam — all merged and validated) and rebuilds only the part #1385 got wrong: how kolu-the-app *consumes* that substrate.

## The core model

One thesis, and everything follows: **the host is the sole source of truth for its own live terminals, and kolu persists only a dial-list** — the set of `hostId`s to re-dial, never a remote-terminal record. A remote tile is just a local `TerminalEndpoint` (the shipped PTY·fs·git seam) whose surfaces are bound to an ssh transport; *local vs remote is only the transport*.

The load-bearing principles:

- **Existence comes from one atomic pull, never a stream.** Boot, restore, and reconnect collapse into *one* routine: dial the host, `await` a single complete kaval `terminal.list`, **synchronously seed** a `terminalMetadata` entry (cwd + location) for every listed id so the client's membership filter (`meta.keys()`) reaches its full set in one tick, then project. There is no shadow tile and no empty→partial→full enumeration for the boot race to mis-settle on.
- **Persist only a dial-list.** Persist a record only for an endpoint that *dies with you* (the local PTY). The durable remote kaval outlives you, so you persist only `Set<hostId>` to re-dial — and that dial-list is the *sole* boot-dial authority. A remote-located terminal can never enter `SavedSession.terminals` (unit-tested), so no autosave can ever prune it.
- **Endpoint is first-class.** `location` is non-optional on the terminal surfaces; `local` (`LOCAL_HOST_ID`) is an explicit value, not the *absence* of a `hostId`. A dropped location becomes a compile error, not a silent local respawn.
- **Awareness is split by where its inputs live** — the one place a naïve clean-slate (and #1385's own framing) goes wrong. See below.

## What we stand on, what we throw away

### Stand on (the shipped foundation is correct)

- The **`TerminalEndpoint` PTY·fs·git seam** — the one correct shape; local and remote are the same interface over different transports.
- **`getHostSession` / `HostSession`** — the genuinely hard, already-solved physics: ssh reach + Nix provision + the ref-counted reconnect/give-up machine. The host-as-truth model leans on its reconnect loop to re-pull `terminal.list`.
- **kaval + its atomic `terminal.list`** — a durable remote PTY with *zero* kolu logic on the host, plus the complete-list pull that is the real canvas-existence signal.
- The **synchronous `seedMeta` upsert** at spawn — the client paints a tile only when its metadata has arrived, so this is *kept* and generalized to seed every id from the list.

### Throw away (this is the #1385 design being rejected)

- **The watcher forwarding the PTY.** <Pill variant="hi">reject</Pill> Putting the PTY data path through the most-volatile host process. Dial kaval *directly* for the PTY; the host agent keeps the PTY out of its path.
- **Persisted remote-terminal records + the dual boot-dial authority.** <Pill variant="hi">reject</Pill> Replaced by the dial-list, which *supersedes* the `savedRemoteTerminals → bootDialHosts` branch.
- **`preRegisterSavedTerminals` → shadow tiles + the settle-window orphan reconcile** <Commit sha="fbc06c31" />. <Pill variant="hi">reject</Pill> Deleted, not re-timed — replaced by the one atomic `terminal.list` + synchronous seed.
- **Optional `hostId` ("absent ⇒ local") threaded through the contract.** <Pill variant="hi">reject</Pill> Made explicit and non-optional on the terminal surfaces (and finished off in the activity feed).

## The four bug classes, dissolved by construction

The point of designing afresh is that #1385's bugs become *unreachable*, not *fixed*:

| #1385 bug class | The construction that makes it impossible |
| --- | --- |
| **Cold-boot re-adoption race** (shadows wiped on an empty/partial frame; autosave prunes the session; boot stops re-dialing) | Existence is one atomic `terminal.list` + a synchronous seed (no streamed enumeration to mis-settle); kolu persists only a dial-list (no record to prune); the client gate waits per host so a remote-only boot shows "dialing", never a blank canvas. |
| **Env leak** (remote PTY gets a restricted PATH) | Spawn is `{argv,env,initFiles}` fully-specified; env is built fresh from `system.info` — *and* the host **login PATH** is captured host-side (moved into kaval's `system.info`), so a kaval-direct dial reports it. An e2e asserts the PTY's PATH carries the user's profile dirs. |
| **Remote-repo prune** (recent-repos stat'd against local fs) | Reachability is an endpoint question: every record carries its `hostId`, and prune-stat routes through `endpointFor(record.hostId).fs`, tolerant of a down host. A unit test asserts a remote record is never stat'd against local fs. |
| **Restore-to-wrong-endpoint** (saved `hostId` dropped → respawn locally) | `location` is non-optional (a drop is a compile error); restore is *dial + adopt the host's list*, never read-record-and-spawn; the client respawn-by-`hostId` branch is neutered. |

## The honest correction: awareness is split

The seductive clean-slate idea — and the one the *adversarial review killed* — was "the first remote slice needs no host agent at all, and the provider DAG just runs in kolu-server over kaval's taps." Half true, and the half that's false is exactly the half #1385's existence already proves.

<Callout kind="warn" title="Why git/PR/agent awareness MUST run host-side">
The provider DAG splits cleanly by **where its inputs live**:

- **PTY-tap signals** — cwd, title, foreground, command — ride kaval's wire taps. These *can* run in kolu-server, and the kaval-direct shell slice (P4a) ships them with **no host agent**. That is a genuine surface-sharing win.
- **fs-derived signals** — git branch/dirty state, PR status, agent detection — read the host's own filesystem and home dir (`git/resolve.ts` runs `simpleGit` on the terminal's cwd; the claude/codex/opencode detectors read `~/.claude` under `os.homedir()`). Run *in kolu-server* against a remote cwd/pid, they inspect the **wrong machine** — empty git state, no PR, no agent, or a collision where a remote pid matches a local session file.

So git/PR/agent awareness *must* run on the host (P4b) — which is **exactly why kolu-watcher exists**. The fresh design keeps that role and renames it the **host agent**, dropping only its PTY-forwarding. The honest scope: the host agent is the *awareness host* and the *fs/git host*, never the PTY path and never the canvas-existence signal. A from-scratch design that skipped this would re-learn #1385's lesson the hard way.
</Callout>

## Versus the resequencing

A layered *resequencing* of #1385's existing code and this from-scratch design are two real options, and they differ in kind:

- **Resequencing** (finish #1385's design in safe slices) is *layer-completion of the existing design*: lower review risk, faster to start (the code exists), but it preserves the worst coupling — the watcher in the PTY path, persisted remote records, the settle-window band-aid, optional-`hostId` everywhere. You ship the same architecture in safe slices.
- **from-scratch** is *dissolution-by-construction on a cleaner core*: more up-front rewrite, but the four bug classes become unreachable, the PTY is dialed straight at kaval, and the boot model is one atomic pull. Every remote PR is demoable from the first one, and the load-bearing awareness split is isolated and honestly scoped.

A pragmatic blend is available: the structural bug-proofing here (**P1/P2/P2b** — endpoint-first-class, dial-list, host-aware feed) is cheap, local-only, and *worth doing regardless of which path ships the remote code*. They convert the bug classes into compile errors and failing tests on the world we already have.

## Status

**Proposed — a clean-slate alternative to finishing <PrLink pr={1385} /> in layers, and a child of [kaval-sessions](kaval-sessions.html).** This design was built from the goal, grounded in the shipped foundation by reading the real repo, judged by a panel, and adversarially verified — the verification is what forced the awareness split (the draft's "no host agent, full awareness" thesis was provably wrong) and the seed-from-`terminal.list` existence model. It is *not* committed; it is the "if we started over" answer to weigh against finishing what we have. The recommended first move either way is the **P1/P2/P2b** structural trio — pure local-world refactors that make the #1385 bug classes unreachable before any remote code is written.

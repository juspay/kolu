You asked for Sleep: leave a Claude Code terminal blocked for days, **sleep** it
(its PTY, xterm, WebGL context, and agent all released — gone, like closing it),
and **wake** it later in place with the agent resumed. A first cut shipped the
mechanism — but built the sleeping tile as a **parallel silo** (its own
component, its own dock section, a list it was *excluded* from). It worked, but
it was wrong, and a structural review (**lowy** volatility + **hickey**
simplicity) said why: there is **no first-class "tile"** in
kolu. A tile *is* a live terminal. So a tile with no live terminal has nowhere to
exist, and the only move left was to bolt one on beside the real ones.

This note is the re-plan. It fixes the wart first, then sleep falls out almost
for free — and a future **planned terminal** comes for free after that. Four
phased PRs.

## The wart — a tile *is* a live terminal

The canvas calls itself "domain-agnostic — tiles, content injected via render
props." It isn't, quite. Four identities are fused into "terminal", so a tile
cannot exist without a live one:

- `tileIds === terminalIds` — the canvas iterates the live terminal list.
- `tile.layout === TerminalMetadata.canvasLayout` (`surface.ts:245`) — a tile's
  position is a field *on the terminal*.
- `activeTile === activeId`, a *terminal* id — selection is keyed to terminals.
- dock rows rank `terminalIds` — a tile that isn't a live terminal is invisible.

Proposed (right): a first-class Tile registry owns identity, layout, active/selection, and MRU plus a TileContent union (terminal-only today); the live list feeds it, and the canvas, dock, and selection all read the one source. Sleeping and planned become union variants, not silos." />

<Callout kind="warn" title="The silo, named — both lenses independently flagged the same fingerprints">
The first attempt proved the wart by reproducing it: a second `<For>` loop, a
second layout store inside the sleeping record, and a split-brained reconciler
(`layoutOf`/`persistLayout` branching on `sleeping.getLayout(id) !== undefined`).
That branch is the smell — drag/resize/active should *never* know a tile's kind.
Both lenses also caught real latent bugs born of the parallel registry:
`pendingLayouts.dropEvicted` never evicts a sleeping tile's pending layout (it
isn't in the live set), and `terminalCount + sleeping.records().length` leaks the
two-registry split into the canvas-mode decision. Fragmentation, not
decomposition.
</Callout>

## A first-class Tile <Pill variant="run">PR 1</Pill>

The fix is **not** an electricity (no `@kolu/*` package — a tile is app-domain
state, hiding a bounded algorithm, not a transport-class volatility; the same
verdict `electricity.mdx` already records for `@kolu/geometry`). It's an in-app
**decomplect**: separate *tile presence* from *terminal liveness*.

A **Tile registry** (`useTileStore`, fed by the existing terminal store) becomes
the one source of truth for what the canvas, dock, and selection read:

| The **Tile** owns | The **Terminal** owns |
|---|---|
| identity (`TileId`), layout, active/selection, MRU | PTY · xterm · agent · attach stream |
| a `TileContent` discriminated union | repo/branch identity, agent state, the live body |

`TileContent` is `{ kind: "terminal", terminalId }` — and that is the **only**
variant in PR 1. The canvas renders **one** `<For>` over `tileStore.tileIds()`,
dispatching on `tileStore.contentOf(id).kind`; `activeId` becomes a `TileId`;
`focusedId` narrows `TileId → TerminalId` only when `kind === "terminal"`;
`mode()` keys off `tileStore.tileCount()`. The reconciler disappears. **Zero
behavior change** — Terminal is the only tile there is.

<Callout kind="accent" title="Why this is the load-bearing PR">
This is the whole point of the re-plan: once a tile is a real thing with its own
identity, layout, selection, and a *content union*, "sleeping" is a new union
variant — it inherits drag, resize, focus, active, dock ordering, and
persistence **for free**, because those operate on the content-agnostic registry.
The silo becomes unrepresentable. A future *planned terminal* is then a third
variant, not a third silo.
</Callout>

## Sleeping is a tile state <Pill variant="run">PR 2</Pill> · <Pill variant="run">PR 3</Pill>

With the registry in place, a sleeping terminal is a tile whose content is
`{ kind: "sleeping", record }` and whose liveness is false. It behaves **no
differently from a live terminal**: same dock row (selectable — clicking
**focuses** it, it can be the *active* tile, it never auto-wakes), same draggable
and resizable canvas tile, same MRU. The only differences are a dormant body and
one added verb — **Wake** (re-spawn PTY + resume the agent, flipping the content
back to `terminal`).

### A frozen screenshot for the dormant body <Pill variant="run">PR 3</Pill>

The dormant body shouldn't be a bare card — it should be the terminal as you left
it. Just before sleeping, capture a **screenshot** of the tile, write it to the
state dir, and serve it back so the sleeping tile shows its **frozen** last frame
(dimmed, with ☾ and Wake). It reuses the existing `screenshotTerminal` path; the
new surface is a small **served image route** from `KOLU_STATE_DIR` (the
`serve-dir` / iframe-preview route is the precedent), and the screenshot rides
the sleeping record so it survives a restart.

## Persisted and restored — tiles, not just terminals

Tiles must survive a reload and a restart, live or sleeping — that's the whole
point of sleeping for days. The model makes this explicit:

- **Live tiles** persist + restore exactly as today: the `session` cell holds
  each terminal's layout/theme/intent/agent, and session restore respawns them.
- **Sleeping tiles** persist in their own on-disk `sleepingTerminals` store
  (disjoint from `session`, so the live-session autosave can't clobber them) and
  rehydrate on boot **as sleeping** — never auto-woken. The frozen screenshot is
  part of that record.
- On boot the Tile registry **merges both sources** into one tile list, so the
  canvas/dock see a single, ordered set of tiles — some live, some asleep —
  restored to their saved positions.

<Svg svg={lifecycle} caption="One Tile, content states over time. A planned tile (later) has only a future — activate → start fresh. A live tile sleeps (snapshot + release) and wakes (respawn + resume). All three are variants of one TileContent union on one Tile — never separate objects." />

<Callout kind="accent" title="Layout's home — decided with the lenses">
hickey argued `canvasLayout` should move off `TerminalMetadata` onto the tile
record; lowy argued defer that schema migration (the client registry split needs
no schema change — `SleepingTerminal.id` is already a stable tile id). **We take
lowy's sequencing:** PR 1 introduces the client-side Tile registry with no schema
change (live-tile layout still reads from terminal metadata, sleeping-tile layout
from the record — the registry hides which); a server-side unified tile store is
a *later, optional* migration, only if a second consumer of "persisted tile
position independent of liveness" appears. Don't move the schema on spec.
</Callout>

## The four phases

Each phase is one reviewable PR; each leaves `master` shippable.

| Phase | PR | What lands | Why separable |
|---|---|---|---|
| **1 — Tile** | <PrLink pr={1435} /> | `useTileStore` + `TileContent` union (terminal-only), one canvas `<For>` over `tileIds()` dispatching on `contentOf(id).kind`, `TileId` selection, `mode()` on `tileCount()`, layout read/write through the registry | Pure decomplect, **zero behavior change** — reviewable on its own, de-risks everything after |
| **2 — Sleeping state** | <PrLink pr={1441} /> | `{kind:"sleeping"}` variant + sleep/wake RPCs + the persisted store + restore-as-sleeping; dock row + canvas tile via the shared paths | The user-facing core; no silo, no new dock section |
| **3 — Frozen screenshot** | feature | capture-on-sleep, served image route from the state dir, frozen body | Cohesive, isolated surface (a route + a capture) |
| **4 — Louder alerts & parity** | feature | the *awaiting/alert* signals get louder everywhere (an escape hatch earns it), switcher parity, the 48h idle-nudge | Polish; keeps PR 2 tight |

<Callout kind="good" title="Why the order is the point">
Sleep was never the hard part — the machinery shipped in a day. The hard part was
that there was nowhere clean to *put* a dormant tile. Phase 1 builds that place.
Everything after is a variant, a body, or a signal — small, because the structure
finally fits.
</Callout>

The open decisions, now narrow: whether the served screenshot route reuses
`iframePreviewRoute`'s static-serve path or gets its own; whether a focused
sleeping tile shows a hint that typing will offer to wake it; and the eventual
server-side tile-store migration (deferred above). None block Phase 1.

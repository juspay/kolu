**A terminal-side client of kolu-server's pty-host.** kolu-server owns its PTYs _in-process_ today; `kaval-tui` connects to that same in-process pty-host over a local unix socket (speaking `ptyHostSurface`) and gives you `list` / `create` / `attach` / `kill` from the terminal вЂ” no browser, **no daemon**. It is the CLI face of kolu and the seed of a `tmux`/`zmx` replacement (<Issue n={671} />), shipped in phases as a beta.

_Scoped to kaval-tui-on-the-in-process-server. The parent plan's Phase B вЂ” moving the pty-host into a surviving daemon вЂ” is a separate plan this later dovetails with (see [Later](#later)). Ancestry: child of the [pty-daemon](pty-daemon.html) plan, sibling of the [chrome-bar rail](pty-daemon-chrome-bar.html); cf. [Ghostex vs. kolu remote-terminals](ghostex-vs-remote-terminals.html)._

## Architecture вЂ” everything stays in-process

No new process. kolu-server keeps owning the PTYs exactly as it does now; it just exposes the _same_ in-process router over an additional local socket so a second client вЂ” the CLI вЂ” can reach it. **The web path is byte-identical.**

**The split by richness.** The browser is the _rich_ client вЂ” it speaks kolu-server's public contract and gets the whole world (PTY plus the provider DAG: git context, agent state, PR status, session grouping). `kaval-tui` is the _raw_ client вЂ” it speaks `ptyHostSurface` directly and gets the bare multiplexer (PTY plus the VT taps), exactly the `tmux` altitude. Both attach to the one in-process pty-host.

<Callout kind="accent" title="The only server change is one additive socket">
`servePtyHost(deps).router` already exists and is already consumed in-process via `directLink`. Phase 1 adds `serveOverStdio({ router })` on a unix socket beside it вЂ” the same router, a second transport. The in-process web path doesn't change; spawning, killing, and metadata all still flow through `LocalTerminalEndpoint` as today. _No PTY moves, no process is added, nothing survives a restart that didn't before._
</Callout>

### Why a second client at all

Two reasons, both standing on their own without any reference to daemons.

- **It's a feature.** A persistent-while-the-server-runs terminal you drive from the shell вЂ” the CLI face of kolu, the start of the `tmux` replacement <Issue n={671} /> has always pointed at.
- **It proves the contract is consumer-agnostic.** A1 (<PrLink pr={1055} />) asserts `ptyHostSurface` is a clean seam вЂ” kolu-server talks to the pty-host through a contract, not through `node-pty` internals. The only real proof of that is a _second, independent consumer_. If writing a terminal client against the surface is clean, the seam is at the right altitude (the "framework needs an end-to-end demo, not just unit tests" lesson вЂ” parent plan, lesson #3). If it's awkward, that's a finding worth having early.


<Callout kind="warn" title="Honest scope вЂ” client-side detach, not server survival">
Because the pty-host is in-process, restarting kolu-server (a deploy) **kills** its PTYs вЂ” `kaval-tui` does not make terminals survive a _server_ restart. What it does give is **client-side** detach/reattach: the CLI can come and go (close the terminal, reopen, re-attach) while kolu-server keeps running and holds the PTY. Surviving a server restart is the separate daemon plan (Phase B), not this.
</Callout>

## The surface вЂ” ~850 LOC in `packages/kaval-tui`

A small node CLI in its own package, `packages/kaval-tui`, that dials the unix socket through `unixSocketLink` (the local-IPC member of `@kolu/surface`'s link family вЂ” same base64+newline framing the daemon's ssh stdio path uses later) and gets a typed `ptyHostSurface` client. No new wire, no new framing вЂ” each accepted connection is pumped through the already-tested `serveOverStdio`.

| Subcommand | Surface call | Behaviour |
| --- | --- | --- |
| `kaval-tui list` | `terminal.list()` | One-shot вЂ” print each live PTY's id В· pid В· idle В· cmd В· cwd (`cmd` = the OSC title or the foreground command), with the full entry under `--json`. The pty-host's inventory. _Shipped: the entry was enriched with `title` + `foregroundProcess` (contract 2.1) so this is one round-trip, not per-row tap fetches._ |
| `kaval-tui attach <id>` | `terminalAttach.get({id})` (stream) + `terminal.write` / `terminal.resize` + `exit.get({id})` | Put the local tty in raw mode. First yield is the **scrollback snapshot** в†’ paint to stdout; deltas stream after. Forward stdin via `write`; `SIGWINCH` в†’ `resize({id, cols, rows})`. A line-start `~.` escape detaches вЂ” the CLI exits, kolu-server keeps the PTY. When the deltas end, the `exit` stream discriminates "PTY died" (report the real code, quit with it) from "stream dropped" (re-attach; the fresh snapshot repaints). _The full client-side loop вЂ” reply filter, deterministic restore, sizing вЂ” shipped with Phase 2 below._ |
| `kaval-tui create [-- cmd]` | `terminal.spawn({id, argv, cwd, env, initFiles})` | Mint a UUID client-side and spawn a plain `$SHELL` (or the given `[-- cmd]`), then print its id and exit вЂ” it does _not_ auto-attach. The whole spawn input is composed client-side (the host derives nothing of its own from B0); `--json` emits the full result for scripting. |
| `kaval-tui snapshot <id>` | `terminal.getScreenText({id})` | One-shot, non-interactive: dump the current scrollback as **plain rendered text** (not the `terminalAttach` first frame вЂ” that's serialized VT screen state for late attach, which would replay control sequences and defeat `grep`) + a trailer line to stderr, then exit. The scriptable primitive the [headless test](#headless-test) asserts on. |
| `kaval-tui kill <id>` | `terminal.kill({id})` | Out-of-band termination вЂ” exercises the order-safe kill path (abort the exit tap _then_ kill) from a non-browser client. |

Of the taps, attach consumes `exit` вЂ” stream-end discrimination plus the real exit code, which tombstones server-side so it's retrievable even after the deltas end. The metadata taps (`cwd` / `title` / `foreground`) feed **no persistent status line**: that early sketch contradicted the raw-passthrough decision and was dropped (see the Phase 2 decisions below); they stay on the contract for a later richer UX. `--json` makes `list` machine-readable; `--socket <path>` points at a non-default server.

<Callout kind="note" title="Contract reality check">
The subcommand в†’ surface mapping is shorthand; these are the exact shapes the _current_ `ptyHostSurface` exposes (verified against `@kolu/pty-host` after A1 <PrLink pr={1055} /> and the link-family unification <PrLink pr={1059} />). **All three phases have now landed** вЂ” the socket + list-metadata items in Phase 1 (<PrLink pr={1084} />), and the spawn-command item in Phase 3 (the B0 client-composed spawn, <PrLink pr={1370} />).

- **`terminal.list` now carries the metadata вЂ” resolved in <PrLink pr={1084} />.** The entry was `{ id, pid, cwd, lastActivity }` with no title; Phase 1 enriched it with `title` + `foregroundProcess` (contract 2.1, additive В· optional), so a one-shot `list` shows a `cmd` column without per-row tap subscriptions. `list --json` emits a top-level array (`jq '.[]'`-friendly).
- **`terminal.spawn` now carries the command via `argv` вЂ” resolved by B0 (<PrLink pr={1370} />).** The input is `{ id?, argv, cwd, env, initFiles, cols?, rows?, scrollback? }`: `argv` is the fully resolved program + args (`argv[0]` is the shell, or the `[-- cmd]` passed verbatim), and the host neither chooses the shell nor appends flags вЂ” the client composes the whole input (`buildCreateInput`) and the host derives nothing of its own. So `create [-- cmd]` needed no extra field: a passed command becomes `argv`, an absent one defaults to `[$SHELL]`. The earlier "no command field" gap is closed.
- **The taps are host-method names; on the wire they're streams.** `subscribeCwd`/`subscribeTitle`/`subscribeForeground`/`exitPromise` are `PtyHost` methods below the seam вЂ” a CLI client consumes the surface _streams_ `surface.cwd.get({id})` / `surface.title.get({id})` / `surface.foreground.get({id})` / `surface.exit.get({id})`. `exit` crosses as a one-shot _stream_ (yields `{ exitCode }` once, then ends), not a `Promise`.
- **The socket exists at a stable path вЂ” resolved in <PrLink pr={1084} />.** `servePtyHostOverUnixSocket` chose `$XDG_RUNTIME_DIR/kolu/pty-host.sock` (distinct from the per-process `kolu-<serverProcessId>/` scratch dir), with a fixed, `$TMPDIR`-independent `/tmp/kolu-$UID/pty-host.sock` fallback off systemd вЂ” deliberately NOT `os.tmpdir()`, which differs by launch context so server and CLI would miss each other вЂ” and a `--socket` override (the same flag name on server and CLI). `getPtyHostSocketPath` is the one resolver both share; a live-vs-stale probe stops a second server hijacking a live socket.
</Callout>

### Decisions recorded here (to be echoed in code comments)

| Decision | Choice | Why |
| --- | --- | --- |
| **Home** | a tiny separate `packages/kaval-tui` | An independent CLI package, not a `bin` folded into `kaval` вЂ” keeps the kaval daemon package focused on the contract + primitive, and keeps the CLI's own deps (raw-tty handling, arg parsing) out of the closure the A2 staleKey hashes. |
| **Render fidelity** | raw VT passthrough | Write the pty-host's bytes straight to stdout; do _not_ re-render through a second `@xterm/headless` mirror. Simplest and truest, and it avoids a second rendering path drifting from the server's. _Recorded as a design-decision comment at the passthrough site._ |
| **Detach / escape** | ssh-style line-start `~` escape вЂ” **never `Ctrl+B`** | A passthrough multiplexer must not steal any control char the inner tools need вЂ” `Ctrl+B` and `Ctrl+J` are reserved by Claude Code (`input/prohibitedKeybinds.ts`). The escape is the unambiguous ssh model: `~` recognised _only_ immediately after a newline. `~.` detach В· `~~` literal tilde В· `~?` help; configurable via `--escape` (a single character). The `kill` *subcommand* shipped (Phase 3, <PrLink pr={1462} />), but the in-attach `~k` escape is still later вЂ” killing is a standalone `kaval-tui kill <id>` today, not a keystroke from inside attach. A literal `~` mid-line passes through untouched, so _every_ inner-app chord reaches the program unmodified. |

### Phase 2 вЂ” the attach loop <Pill variant="ok">shipped</Pill> <PrLink pr={1255} />

Recon verdict before implementation, borne out by the ship: **no contract bump, no server change** вЂ” `terminalAttach` / `write` / `resize` / `exit` all exist at contract 2.1 (and the web path already consumes the same `host.attach` as just another subscriber on the per-PTY broadcast channel, so a CLI attach violates no exclusivity). Attach is a pure consumer; no drishti-mirror PR. The whole phase is ~300вЂ“400 LOC in `packages/kaval-tui` plus tests. _One delivery note: the reply filter first shipped as a move of the browser's `terminalResponseFilter` into `kolu-common`; a blocking structural review then promoted the whole protocol policy вЂ” grammars + stripper, the headless forward/drop rule, the answered/silent device-query matrix, paste delimiters, and the snapshot-reciprocal TTY reset вЂ” into a dedicated zero-dep leaf, **`@kolu/terminal-protocol`**, that the browser, the pty-host, and kaval-tui all import. The pty-host's staleKey hashes it (a protocol change is observable daemon behaviour), and its device-query matrix is executed as contract tests against a real headless._

The sequence: connect в†’ version gate (both shipped in Phase 1) в†’ **resize-then-attach** в†’ raw mode в†’ one-shot `в†» snapshot restoredвЂ¦` notice в†’ paint snapshot в†’ pump until detach or stream end. Five decisions carry the design weight (echoed as code comments at their sites, per this note's convention):

| Decision | Choice | Why |
| --- | --- | --- |
| **Device-query replies** | filter the tty's auto-answers out of the stdinв†’`write` path | The snapshot/deltas carry queries (DA1 В· DSR/CPR В· XTVERSION вЂ¦) that the user's _real_ terminal auto-answers on stdin вЂ” but the headless mirror already answered them server-side, so forwarding the duplicate reply corrupts the inner program's stdin (the yazi-class bug). Mirror the browser path's suppression predicates, preserving the client-suppressed в‡’ server-answered invariant. |
| **Terminal restore** | one deterministic reset on _every_ exit path | The snapshot replays modes onto the local terminal вЂ” alt-buffer (`?1049h`), mouse tracking, bracketed paste (`?2004h`), app cursor keys. Detach, PTY exit, SIGTERM/SIGHUP, and crash all run `setRawMode(false)` **+ a fixed reset string** (alt-buffer off, mouse off, paste-wrap off, cursor visible). Restore is much more than un-raw-ing stdin. |
| **Sizing** | resize-then-attach; last-resize-wins across clients | The snapshot serializes at the server-side grid (the browser's last size, or the 80Г—24 default); resizing _first_ renders it at the local dimensions. The contract has no size-change tap, so a concurrently-attached browser tile may show wrap artifacts until its own next resize вЂ” accepted and documented; a size tap would be contract 2.2, out of Phase 2 scope. |
| **Status lines** | one-shot notices only вЂ” **no persistent footer** | A live taps-fed footer needs scroll-region ownership, which violates the raw-passthrough render-fidelity decision above (the earlier sketch self-contradicted). One line before the paint (`в†» snapshot restoredвЂ¦`), one after restore on detach; the CLI owns zero pixels while attached. |
| **Stream end & retry** | discriminate via the `exit` stream; never auto-retry | The deltas iterator ends identically on PTY exit, server-side abort, and the silent slow-consumer drop (bounded 10k queue). On end, ask `exit`: code present в†’ report it, quit with it; PTY still live в†’ re-attach, and the fresh snapshot repaints вЂ” exactly the right recovery for the drop case. Stream auto-retry would replay the snapshot mid-session, so attach opts out; if the server itself goes away the CLI restores the tty and prints an honest one-liner вЂ” manual re-dial is the only reconnect. |

The escape machine's fine print: it runs on **bytes**, decoding via `string_decoder` only at the `write` boundary so multibyte characters split across stdin chunks survive; session start counts as line-start (ssh behaviour); recognition is suspended inside bracketed-paste brackets so a pasted `\n~.` cannot detach; in raw mode `Ctrl+C` arrives as byte `0x03` and is _forwarded_ like everything else вЂ” only external SIGTERM/SIGHUP trigger the restore-and-exit path.

**Testing.** The escape machine, reply filter, and reset emission are pure functions вЂ” unit-tested with no tty (the `render.test.ts` pattern). The loop itself is factored over read/write streams plus a tty-ish interface and integration-tested against the real-socket harness (the `serveOverSocket.test.ts` pattern: in-process pty-host + `unixSocketLink` on a temp socket). The home-manager VM test keeps its `list` smoke вЂ” attach is infeasible headless.

## The user flow

The interactive loop a user walks, then the same loop scripted as a headless test. The running example: a PTY `3f9aвЂ¦c21` in `~/code/kolu` that becomes a Claude Code session.

### Headless test вЂ” client-death detach/reattach, no browser

The detach/reattach loop scripted: it asserts that the CLI client can die and a fresh one re-joins the same PTY (the server runs throughout) вЂ” fast, no browser.

```bash
#!/usr/bin/env bash
set -euo pipefail

# 1 В· create a PTY with a unique marker in its scrollback (create never attaches)
id=$(kaval-tui create --json -- bash -c 'echo "MARK-$$"; exec sleep 1d' | jq -r .id)
pid_before=$(kaval-tui list --json | jq -r ".[] | select(.id==\"$id\").pid")

# 2 В· (no kaval-tui process is attached вЂ” simulating client death)

# 3 В· a fresh client re-attaches by id and asserts the state is intact
kaval-tui snapshot "$id" | grep -q "MARK-"            # scrollback preserved
pid_after=$(kaval-tui list --json | jq -r ".[] | select(.id==\"$id\").pid")
[ "$pid_before" = "$pid_after" ]                     # same PTY, not a respawn

echo "вњ“ client came and went; server held the PTY (pid $pid_after unchanged)"
```

This is the kaval-tui-scoped reattach test вЂ” the client can come and go while the server holds the PTY. (Surviving a _server_ restart is a different assertion that belongs to the daemon plan, not here.)

## Later вЂ” how this dovetails with the daemon plan (out of scope)

Stated only so we don't trip over it: when the parent plan's Phase B eventually moves the pty-host out of kolu-server into a surviving `kolu --stdio` daemon, `kaval-tui`'s socket target shifts from kolu-server to the daemon вЂ” **with no contract change**, because both serve the same `ptyHostSurface`. At that point the CLI's terminals gain server-restart survival for free, and the raw-vs-rich client split is unchanged. _But that is a separate plan; nothing in kaval-tui's phases above depends on it, and kaval-tui ships and is useful entirely on the in-process server._

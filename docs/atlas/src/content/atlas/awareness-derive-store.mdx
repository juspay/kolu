**The de-entanglement that makes R9 mostly evaporate.** Every hard thing in [the finale](remote-terminals-finale.html) — the two assemblers, the createPulam boundary, the fold-clobber — traces to **one** complect: *deriving* awareness is fused with *storing* it. This note proposes splitting them (S1: a pure engine; S2: a host-owned fold), shows the **types and API** so the boundary can be evaluated with a fine-tooth comb, and shows how R9·lib · R9.0 · R9.3 collapse once it lands. It is grounded in the current `@kolu/terminal-workspace` (the library the finale renames to pulam-library); every claim cites the code it restructures.

<Callout kind="note" title="Where this sits">
This **replaces** the createPulam / shared-leaf direction of the finale's R9·lib + R9.0. The thesis: don't share *the assembly* — split *derive* from *store* and the assembly disappears, leaving two pure functions both homes call. R9.2 / R9.4 / R9.5 / R10 are untouched.
</Callout>

## The knot today — derive is fused with store

The sensor engine does **not** return awareness. It takes a host `AwarenessSink` + a per-terminal `AwarenessRecord`, **mutates `record.meta` through the sink**, and **reads `record.meta` back** as its own prior state. `sensors.ts:139-162` calls this the load-bearing *apply-and-publish* contract — a sink that publishes *without* mutating the record "would silently defeat every dedup/transition gate."

So `record.meta` plays **two roles at once**:

- the engine's **prior-state** for its gates — the command dedup `if (record.meta.lastAgentCommand !== normalized)` (`sensors.ts:410`), the publish-if-changed `agentInfoEqual(record.meta.agent, next)` (`sensors.ts:516`), the recency decision `shouldBumpRecencyForAgentChange(record.meta.agent, next, record.meta.lastActivityAt)` (`sensors.ts:519`), the #1495 session dedup `agentSessionToPersist(record.meta.agentSession, next)` (`sensors.ts:526`);
- the host's **stored value** — kolu aliases `record.meta = getTerminal(id)!.awareness`, the registry entry mutated in place (`local.ts:700-704`); `pulam` points it at its own cache (`daemon.ts:209-217`).

<Svg svg={before} caption="Today: the sensor set and the host's storage are the SAME object. record.meta is both the engine's prior-state (read back by the dedup/recency/session gates) and the host's stored value (mutated in place via the sink and fanned to the served collection). Two sinks implement the one interface — kolu's folds + persists + trackRecent; pulam's publishes to a cache. Because deriving writes the store, a fresh derivation can clobber remembered state, and 'share the assembly' is the only way two homes can reuse the sensing." />

That fusion is the cause of everything downstream:

- **Two assemblers.** Because the engine is wired *to a host's storage*, kolu and the daemon each wire their own (`local.ts:278` vs `pulam/src/hooks.ts:39`) — the duplication R9·lib chases.
- **The `AwarenessSink` foot-gun.** "Mutate synchronously *before* you publish or you defeat the gates" (`sensors.ts:139-162`) is a hazard that exists *only* because prior-state and store are one object.
- **The fold-clobber class.** Deriving *writes the store*, so a fresh/ephemeral derivation can overwrite remembered history — exactly the defect that killed the local-pulam-process (#1614) and that the createPulam loop re-risked.

## S1 — the engine becomes a pure producer

Split the two roles. The engine **owns its own internal working state** for its gates and **emits events**; it never receives or mutates a host record.

```ts
// What the engine emits. The two patch kinds ARE the persisted/live write-fence
// (narrowed patch types = the same compile-time fence the two sink methods give today).
// recentRepo/recentAgent are cross-terminal MRU side-effects the host may consume.
type AwarenessEvent =
  | { kind: "persisted"; patch: Partial<AwarenessPersistedFields> }   // → host folds + ARMS autosave
  | { kind: "live";      patch: Partial<AwarenessLiveFields> }         // → host folds, stays quiet
  | { kind: "recentRepo"; root: string; name: string }                // MRU bump (git resolved)
  | { kind: "recentAgent"; command: string }                          // MRU bump (LIVE command mark only)

// The one start-time seed the engine needs: the remembered values its STICKY gates
// suppress against. NOT the whole record — only the three gate fields. kolu passes its
// remembered persisted half; an ephemeral host (pulam) passes the zero seed.
interface AwarenessGateSeed {
  lastActivityAt: number;          // recency restore-caveat (don't re-bump on re-detection)
  agentSession?: AgentSessionRef;  // #1495 same-conversation suppression
  lastAgentCommand?: string;       // command dedup
}

interface EngineInputs {
  pid: number;
  cwd: string;                                          // spawn-time cwd (read once; later cwd flows via signals)
  seed: AwarenessGateSeed;                              // remembered gate-state (the load-bearing addition)
  signals: AwarenessSignals;                            // the four taps — UNCHANGED (cwd · title · commandRun · foreground)
  readScreenText?: (tailLines: number) => Promise<string>;  // pure input for screen-scrape promotion (#905)
  log: Logger;
}

// Pure producer: owns currentAgent + the sensor-to-sensor wires + the last-emitted
// live mirror + the seeded sticky-gate state. Touches no host store. Returns a stop fn.
function startAwarenessEngine(
  id: TerminalId,
  inputs: EngineInputs,
  emit: (event: AwarenessEvent) => void,
): () => void;
```

**What moves to engine-internal** (was riding the host record, is genuinely the engine's): `record.currentAgent` (the command→detector wire, `sensors.ts:406,446`), the last-published `agent`/`foreground` mirrors (`sensors.ts:516,220` — the engine is their sole writer, re-seeded null each start), and the three sticky-gate priors (seeded once from `inputs.seed`, owned thereafter).

**What's deleted:** the `AwarenessSink` interface and *both* `makeAwarenessSink` implementations, and with them the apply-and-publish read-back contract. The git→PR internal channel and the agent-adapter process-scoped registry stay exactly as they are (they were always engine-internal).

<Callout kind="warn" title="The subtle invariant — the engine is seeded, not fresh (validated against the code)">
A naive "fresh engine + partial patches" **regresses three sticky gates**. They suppress writes based on the *remembered* value:
- **recency** — `isReDetectionAfterRestore = prev===null && next!==null && currentLastActivityAt>0` (`agentRecency.ts:31-33`) keeps a restored tile's `lastActivityAt` from being re-stamped to *now* on re-detection. A zero-seeded engine would **bump** it.
- **session (#1495)** — `agentSessionToPersist(remembered, next)` returns null on a same-conversation tick (`agentSession.ts:20-32`). Un-seeded, it re-persists → re-arms autosave every wake.
- **command dedup** — `lastAgentCommand !== normalized` (`sensors.ts:410`); a replayed `commandRun` mark re-emits without the seed.

So `inputs.seed` carries exactly those three. The *re-derived* persisted fields (`git`, `cwd`) need **no** seed — the partial-patch shape protects the remembered value until the engine re-resolves them. This is the precise line between "seed it" (sticky gates) and "let partial-patch protect it" (re-derived fields).
</Callout>

## S2 — the host owns the fold + the fence

The host seeds a value, runs the engine, folds each event onto its store, and reacts to the event's kind. Both homes run the **same** recipe; only the seed source, the store, and the reactions differ.

```ts
// The fold is a one-line pure merge — clobber-free because the patch is PARTIAL.
function foldPatch(current: AwarenessValue, ev: { patch: object }): AwarenessValue {
  return { ...current, ...ev.patch };
}

// The host recipe (both homes). Differences are the host's, never the engine's.
function watch(id, terminal): () => void {
  // SEED: kolu = { ...rememberedPersisted, ...seedAwarenessLive() }; pulam = seedAwarenessValue(cwd)
  let current = seed(terminal);
  store.set(id, current);                       // kolu: registry entry.awareness; pulam: its cache — both fan out to the collection

  const stop = startAwarenessEngine(id, {
    pid, cwd,
    seed: rememberedGateSeed(terminal),         // kolu = the remembered three; pulam = { lastActivityAt: 0 }
    signals, readScreenText, log,
  }, (ev) => {
    switch (ev.kind) {
      case "persisted":
        current = foldPatch(current, ev);
        store.upsert(id, current);
        host.armAutosave();                      // kolu fires terminals:dirty; pulam no-op  ← the write-fence, as a host reaction to the tag
        break;
      case "live":
        current = foldPatch(current, ev);
        store.upsert(id, current);               // no autosave — the ~150ms agent firehose can't reach disk
        break;
      case "recentRepo": host.trackRecentRepo?.(ev.root, ev.name); break;   // kolu MRU; pulam ignores
      case "recentAgent": host.trackRecentAgent?.(ev.command); break;
    }
  });
  return () => { stop(); /* host ignores any late event — the post-teardown no-op guard */ };
}
```

<Callout kind="note" title="Clobber-free = partial patch ∧ seeded gates (both halves required)">
The persisted half survives a restart because (1) the engine emits **partial** patches, so a field it hasn't re-derived is never written, **and** (2) the engine is **seeded** so the sticky gates make the same suppression decisions they do today. Partial-patch alone is *not* sufficient — `lastActivityAt` *is* touched; the seed is what keeps it from re-stamping. The fold writes the **registry `entry.awareness`** (the source of truth `wake` / `beginSleep` / `snapshotSession` read — `local.ts:922,871`, `terminals.ts:86`), not only the served collection.
</Callout>

<Callout kind="note" title="The write-fence, faithfully">
Today the fence is *route + gate*: `updateServerMetadata` fires `terminals:dirty` (persisted), `updateServerLiveMetadata` doesn't (live) — `metadata.ts:124-159`. S2 reproduces the **route** as the event's `kind` (host arms autosave iff `persisted`) and keeps the **gate** in the engine (a persisted event is emitted *only when a sticky gate actually fires*). The narrowed `Partial<…>` patch types preserve the compile-time half-fence. `trackRecentRepo`/`trackRecentAgent` become explicit events (not derivable from patches — `recentAgent` fires per **live** command mark regardless of the dedup, and must skip replays). `cwd` is relayed by the engine as a `persisted` `{ cwd }` patch (it already consumes `signals.cwd` for git), retiring the per-host cwd-persist copy. The **green-dot activity tracker** stays a separate per-host primitive (`activity.ts`) fed by the raw-output byte tap — not part of the engine; whether kolu serves it live or keeps client-side `useTerminalActivity` is the one remaining sub-decision.
</Callout>

<Svg svg={foldflow} caption="S1+S2, local (R9.0). The host seeds current from its remembered persisted half, starts the pure engine (seeded with the three sticky-gate fields), and folds each emitted partial patch onto the registry entry.awareness. A `cd` emits a persisted {git} patch (folded, autosave armed); an agent tick emits a live {agent} patch (folded, quiet) and — only if a sticky gate fires — a persisted {lastActivityAt} or {agentSession} patch. ## How R9 collapses

A "host" is now just *dial a kaval → per terminal: seed, run the engine, fold its events onto your store, serve*. kolu is a host with an authored half + a durable registry; the `pulam` daemon is the **degenerate** host — no authored half, ephemeral cache. Same recipe.

<Callout kind="warn" title="R9.3 honestly — what is shared vs genuinely remote">
The awareness **collection carries whole `AwarenessValue`s**, not field-deltas (`surface.ts:119-124`; verbs `keys/get/upsert/delete`, each frame a full value — confirmed by `pulam-tui`'s wholesale `upsert`). So the *shared* win is the pure **engine** (one derivation, both homes) and the persisted/live **fence**; the *local* fold merges partial engine events; the *remote* path mirrors **full** values and needs `reconcileRemoteSnapshot(remembered, daemonSnapshot)` — keep the remembered persisted fields the ephemeral daemon hasn't derived (`lastActivityAt>0`, `lastAgentCommand`, `agentSession`), take its live half + any derived persisted. That reconcile runs **only after** the `version` handshake passes (`surface.ts:48-55`, `daemon.ts:130-137`). This is the one genuinely-remote piece — exactly where the finale always said the fold lives — and it is small and bounded, not a second engine or sink.
</Callout>

```ts
// R9.3's only new function — the remote reconcile (runs once per (re)connect, after the version gate).
function reconcileRemoteSnapshot(
  remembered: AwarenessValue,   // kolu's durable memory for this remote terminal
  snapshot: AwarenessValue,     // the ephemeral daemon's current served value
): AwarenessValue {
  return {
    ...snapshot,                                  // take the daemon's live half + whatever it has derived
    lastAgentCommand: snapshot.lastAgentCommand ?? remembered.lastAgentCommand,
    agentSession:     snapshot.agentSession     ?? remembered.agentSession,
    lastActivityAt:   Math.max(snapshot.lastActivityAt, remembered.lastActivityAt),
    git:              snapshot.git ?? remembered.git,
  };
}
```

## Phasing & migration

<a id="s1"></a><a id="s2"></a><a id="collapse"></a>

- **S1 → S2 → R9** in that order. S1 is the one real lift — rewriting the freshness-critical engine from mutate-via-sink to emit-events, with the existing sensor tests as the safety net. It **deletes** the sink, both impls, the read-back contract, and the clobber — net-negative LoC. S2 is small (the fold + the host recipe). After both, R9.0 and the daemon are each an ~8-line loop, and R9.3 is mirror + one reconcile.
- **Hard cutover, no knobs.** kolu and the daemon move onto the engine in one change each; no flag, no fallback (the fail-fast rule). The persisted/live fence is kept *by type* (the narrowed patch kinds) and *by the seed* (the sticky gates), so a restored badge is never lost.
- **What this retires:** the createPulam / shared-leaf design; the `AwarenessSink` interface; the "two homes, two assemblers" framing (there is one engine and one fold, called by N hosts); and the fold-clobber as a *class* (deriving no longer writes the store). It keeps #1614 dead for the right reason — running the engine in kolu's process is safe because it can no longer clobber, so no local pulam *process* is ever needed.

## Open questions for review

- **The seed boundary.** Is seeding the engine with the three sticky-gate fields (option A here) cleaner than moving those three pure decisions (`shouldBumpRecencyForAgentChange`, `agentSessionToPersist`, the command dedup) into the host fold where `current` is in hand (option B)? A keeps the gates beside the sensors; B makes the engine *fully* stateless-per-event. This note picks A; B is the alternative.
- **`recentRepo`/`recentAgent` as events** vs. leaving them as optional host callbacks the engine still calls. Events keep the engine push-pure; callbacks are fewer types. (This note picks events.)
- **Activity source** after the byte-tap is a per-host primitive: does kolu serve it live (retire client-side `useTerminalActivity`) or keep client-side? Orthogonal to S1+S2, decide at R9.0.
- **Wire shape for R9.3.** Keep the collection carrying whole values (this note) — simplest, one reconcile — or teach it field-deltas so the remote path folds exactly like local (more wire surface, no reconcile). This note keeps whole values.

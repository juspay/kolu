## The spine: one command unlocks two features

kolu's two onboarding goals — **pin it as an app** and **reach it from
anywhere** — look separate. They are the same action.

A self-hosted kolu is almost always served over **plain HTTP** on a LAN
(`192.168.x.x`) or Tailscale (`100.x`) address. Browsers refuse to install a
PWA from there: installation requires a **secure context**, and the only
non-HTTPS origins that qualify are loopback / `localhost`. So the blocker for
*remote install* and the enabler for *remote access* are resolved by the very
same `tailscale serve` — it gives kolu a real `https://…ts.net` origin. Tell it
as **one story with one payoff**, never two features.

### Verified facts (the plan rests on these)

Four load-bearing claims, each adversarially fact-checked against current
(2025–26) primary sources.

<Callout kind="danger" title="No install over plain HTTP — and this is the #1 trap">
Installability requires a **secure context**
([W3C Secure Contexts §3.1](https://www.w3.org/TR/secure-contexts/)). The only
HTTP origins that qualify are loopback (`127.0.0.0/8`, `::1`) and the
`localhost` name family. A `192.168.x.x` LAN IP or a Tailscale `100.x` IP
(CGNAT `100.64.0.0/10`) over `http://` is **not** secure: `beforeinstallprompt`
never fires and Chromium shows **no install affordance at all**
([MDN](https://developer.mozilla.org/en-US/docs/Web/Progressive_web_apps/Guides/Making_PWAs_installable)).
**Never render an Install button or "install me" tip on an insecure origin** —
it would be a dead control. The one correct runtime gate is
`window.isSecureContext === true`.
</Callout>

- **Tailscale Serve = a genuine secure context.** `tailscale serve` reverse-proxies a local port and terminates TLS at `https://machine.tailnet.ts.net` with a real **Let's Encrypt** cert (DNS-01 against `*.ts.net`; private key never leaves your box) — browsers accept it with no warnings, so install + service workers unlock. <Pill variant="ok">confirmed</Pill> Caveats: enable **MagicDNS + HTTPS Certificates** once in the admin console; the **full FQDN** is mandatory (the bare hostname and the `100.x` IP are still plain HTTP); Serve is **tailnet-only**; the FQDN lands in public Certificate-Transparency logs, so don't bake secrets into machine names. ([Serve KB](https://tailscale.com/kb/1312/serve), [HTTPS KB](https://tailscale.com/kb/1153/enabling-https))
- **`beforeinstallprompt` is Chromium-only.** Chrome/Edge/Brave/etc. fire it; **Firefox and Safari (desktop *and* iOS) never do** — and Chrome/Edge *on iOS* run on WebKit, so they don't fire it either. A custom JS Install button must start **hidden** and reveal only on the event. <Pill variant="ok">confirmed</Pill>
- **iOS has no install API.** The only path is the manual **Share → Add to Home Screen**, which works regardless of any event (since iOS 16.4 from Chrome/Edge/Firefox too, not just Safari). On iOS you render *instructions, not a button*. Requires HTTPS + `display: standalone` + an `apple-touch-icon`; avoid the deprecated `apple-mobile-web-app-capable` meta. <Pill variant="ok">confirmed</Pill>

<Callout kind="note" title="Manifest enrichment — in scope for this work, not 'later'">
The manifest is served `standalone` with 192/512 icons, now assembled by
`@kolu/surface-app`'s `installPwaManifest` (<PrLink pr={1154} />, in `master`),
whose `...extra` passthrough already accepts the missing fields — so enriching it
is just passing **`description`**, **`screenshots`**, and a **maskable** 512px
icon at the call site (<Cite file="packages/server/src/index.ts" lines="223" />).
That richer manifest upgrades both Chromium's native install card and the library
preview below. Validate via **DevTools → Application → Manifest** — Lighthouse's
PWA audit is deprecated.
</Callout>

## The in-app welcome

**Revamp `EmptyState` in place** rather than add a modal. The patterns worth
stealing: VS Code **Walkthroughs** (a short, re-openable checklist — *not* a
forced tour; it explicitly warns against excessive steps), Warp (one welcome
surface, success measured as *habit* not completion), Zed (empty-state-as-
welcome, re-openable from the palette), and Raycast's `EmptyView` (the empty
state carries the onboarding *with actions*).

**The moments (cap ~3–5, hard).** The three cards above — **Pin it**,
**Reach it anywhere**, **Run agents** — sit above the existing restore card and
shortcut list. Each is one verb + one line + one chip; depth lives behind the
"Full guide" link, not inline. Lean on kolu's auto-detection (recent repos,
recent agents in <Cite file="packages/client/src/wire.ts" />) so the welcome
shows *the user's own data*, reinforcing zero-setup.

**Re-openable via the palette.** `EmptyState` only auto-shows at zero
terminals, so the welcome would vanish forever once you're working. Add a
stable palette command — labelled **"Tutorial"** (alias "Welcome") — that
re-summons it anytime, mirroring VS Code's `openWalkthrough`. Wire it as an
action in <Cite file="packages/client/src/input/actions.ts" /> and surface it
through the registry in <Cite file="packages/client/src/commands.tsx" />.

**No dismiss state — by design.** Zero terminals *always* shows the welcome.
The way you "dismiss" it is to **open a terminal**; open zero again and it comes
right back. There is no `welcomeSeen` flag, no "Don't show again," nothing to
persist, nothing to go stale — the empty canvas *is* the trigger, full stop.
The only addition is the **"Tutorial"** palette command, which re-summons the
same welcome (as an overlay) even while you're working. Register a
feature-discoverability **tip** pointing at it (per
<Cite file=".claude/rules/conventions.md" />) so the re-open is discoverable.

**Gate order** (apply top-down — the first match wins):

```
1. !window.isSecureContext      → "needs HTTPS" hint → link to Tailscale section
2. already installed            → render nothing
     matchMedia('(display-mode: standalone)').matches
     || navigator.standalone === true
3. secure + beforeinstallprompt → real one-click Install button (Chromium)
4. secure, no event             → per-platform manual instructions (table below)
```

**Per-platform instruction set** (render the literal glyph as inline SVG next to
numbered steps — visual beats prose):

| Platform / browser | What kolu shows |
| --- | --- |
| Chromium desktop / Android, event captured | Real **Install** button → `prompt()` in the click handler |
| Android Chrome (no event) | Menu ⋮ → **Install app** |
| Android Firefox | Menu → **Install** (no event, but install works) |
| iOS — Safari / Chrome / Edge / Firefox (16.4+) | Illustrated **Share ⬆ → Add to Home Screen** (instructions, not a button) |
| Safari desktop (macOS Sonoma / Safari 17+) | **File → Add to Dock** (works on any page; no event) |
| Firefox desktop | Native install not shipped yet (experimental *Taskbar Tabs*) — suggest Chrome/Edge |
| Any non-secure origin (`http://`) | The HTTPS/Tailscale hint — never an Install button |

<Callout kind="good" title="Fact-check (June 2026): non-Chromium still installs — just not via JS">
The "`beforeinstallprompt` is Chromium-only" claim is accurate **about the
event** — Firefox and Safari never fire it — but that is *not* "no install," so
kolu must **instruct per browser** rather than dead-end on "use Chrome":

- **Safari desktop** (macOS Sonoma / Safari 17+): **File → Add to Dock** turns any page into a web app — no manifest or event needed. ([Apple](https://support.apple.com/en-us/104996))
- **Safari iOS / iPadOS**: **Share → Add to Home Screen** (since iOS 16.4 also from Chrome / Edge / Firefox).
- **Firefox Android**: menu → **Install**.
- **Firefox desktop**: no native install yet — *Taskbar Tabs* is experimental in Nightly (`browser.taskbarTabs.enabled`), not GA; suggest Chrome/Edge. ([MDN](https://developer.mozilla.org/en-US/docs/Web/Progressive_web_apps/Guides/Installing))

The `@khmyznikov/pwa-install` component already renders the Safari/iOS
instruction screens, so the detection branch picks the right one.
</Callout>

<Callout kind="accent" title="Library decision: use @khmyznikov/pwa-install, wrap thinly in Solid">
A ~28 kB, MIT, framework-agnostic web component (~41k weekly downloads, kept
current for iOS 18+) that already solves the parts we'd otherwise re-implement:
the iOS/macOS Share-sheet instruction screens, localization, standalone
detection, one-shot `prompt()`, and the `appinstalled` listener. This matches
kolu's "prefer well-maintained libraries over custom code" rule
(<Cite file=".claude/rules/conventions.md" />). Wrap the custom element in a
SolidJS component, feed it `manifest-url` / `icon` / `install-description`, hand
it the captured event via `externalPromptEvent`, and read
`isInstallAvailable` / `isUnderStandaloneMode`. Hand-rolling is only worth it
for Chromium-only with fully bespoke branding — which is not the goal. Ship the
wrapper as a **new package, `packages/solid-pwa-install`** (`@kolu/solid-pwa-install`),
a focused SolidJS adapter alongside `solid-pierre` / `solid-markdown` /
`solid-fileview` — so the install card is reusable by other surface apps
(drishti included), not buried in kolu's client.
([repo](https://github.com/khmyznikov/pwa-install))
</Callout>

The existing tip-gate (<Cite file="packages/client/src/settings/useTips.ts" lines="14" />)
already filters `amb-pwa-install` when installed; extend that to also drop it on
an insecure origin — a passive "install me" tip is just as dead as a button on
`http://`.

Every in-app card and empty-state hint **deep-links to the matching anchor**
here; this page stays the single source of truth, kept in sync with in-app copy
and the README (per <Cite file=".claude/rules/workflow.md" />).

<Callout kind="good" title="The Tailscale walkthrough (the one to get exactly right)">
1. Install Tailscale on the dev box **and** the phone/laptop; sign both into the **same account**.<br/>
2. Admin console → DNS: enable **MagicDNS**, then **HTTPS Certificates** (one-time; acknowledge the public-ledger notice).<br/>
3. On the dev box: <code>tailscale serve --bg &lt;PORT&gt;</code> (<code>--bg</code> survives reboots).<br/>
4. Open the printed <code>https://&lt;machine&gt;.&lt;tailnet&gt;.ts.net</code> on the phone — install the PWA from there.

Default to **Serve** (tailnet-only), never **Funnel** (public, no auth — a
footgun; a single flag flips a port public). Keep the dev server on
`127.0.0.1` and let Serve be the only exposed surface. Manage with
`tailscale serve status` / `reset` / `off`. Decision aid: Serve = "just my
devices"; Cloudflare Tunnel + Access = "public + identity-gated"; ngrok =
"throwaway public URL". Rule out Caddy local-CA / self-signed — its root isn't
trusted on a remote phone, so no secure context.

**Frictionless payoff:** when kolu can resolve its own Serve URL, surface it
in-app with a **copy button** so handing it off is one tap.
</Callout>

## Building it

### Architecture — through the electricity lens

Read against <Cite file="docs/atlas/src/content/atlas/electricity.mdx" label="electricity.mdx" />:
infrastructure that ① is domain-agnostic, ② hides a hard volatility, and ③
graduates to a second consumer earns its own `@kolu/*` package; everything else
stays kolu domain. That splits this work into three layers.

**① `@kolu/surface-app`** <PrLink pr={1154} /> — the app-shell electricity, now
in `master`. Already owns the **manifest** (`installPwaManifest`, whose `...extra` passthrough
absorbs the enrichment above) and the **headless "relationship-to-server" model**
(`useSurfaceApp()`: connection lifecycle, build-skew). *Installability* —
`window.isSecureContext` + standalone display-mode — is a sibling
environment fact, so the `canInstallPwa` / `isInstalled` signals belong in that
same headless model. Graduates today (drishti is the second consumer).

**② `@kolu/solid-pwa-install`** — a *new* electricity. A focused SolidJS adapter
wrapping `@khmyznikov/pwa-install`, sibling to `solid-pierre` / `solid-markdown` /
`solid-fileview`. Behind one socket ("install this web app") it owns the
cross-browser install volatility — `beforeinstallprompt` capture, the
per-platform instruction screens, `appinstalled` — reading installability from ①.
Passes all three tests: agnostic (no terminal/canvas/git), hides a genuinely hard
volatility (every browser installs differently), graduates (any surface app).
**`@kolu/surface-app` may itself depend on `@kolu/solid-pwa-install`** and
compose the card with its own installability signals — so a consumer wires
install exactly once. *Supersedes the earlier "wrap it inside the kolu client"
idea.*

**③ kolu app** — domain, *not* electricity. The welcome itself is kolu's story:

| Area | File | Change |
| --- | --- | --- |
| In-app welcome | <Cite file="packages/client/src/EmptyState.tsx" /> | The 3 moments + the `@kolu/solid-pwa-install` card, above restore/shortcuts |
| Re-open command | <Cite file="packages/client/src/input/actions.ts" />, <Cite file="packages/client/src/commands.tsx" /> | New "Tutorial" action + palette entry — re-summons the welcome as an overlay |

| Full guide | new `website/src/pages/welcome.astro` | The skill-ladder page |

No new server contract; no service worker (kolu has none by design).

### Pitfalls

<Callout kind="warn" title="Onboarding anti-patterns">

- **Dead Install buttons on HTTP.** The #1 kolu-specific trap — gate on `isSecureContext` first, always.
- **Default-visible custom Install button.** Safari/Firefox/iOS never fire the event; hide by default, reveal on event.
- **Treating the iOS card as a button.** It's instructions; it can't trigger anything.
- **Long forced tours.** kolu's audience hates hand-holding — one win, then links. (VS Code warns against this explicitly.)
- **Replaying onboarding for returning users** — "the airplane safety video every flight." Persist + detect returning + offer re-entry.
- **Funnel by default, or the bare hostname / `100.x` IP.** Funnel is public+unauthenticated; the bare host and IP are plain HTTP. Always the full `ts.net` FQDN, Serve not Funnel.
- **Reproducing docs in the panel.** Verb + one line + one action inline; depth links out, or the two copies drift.

</Callout>

### Decisions

- **No mobile welcome.** Onboarding is desktop-only — mobile already opts out of ambient tips via `showsAmbientTips` (<Cite file="packages/client/src/capabilities.ts" lines="34" />).
- **No persistence.** Zero terminals always shows the welcome; *opening a terminal* is the dismissal, and it returns whenever you're back at zero. No `welcomeSeen`, nothing to go stale.
- **Tailscale: instruct, never auto-run.** Even when `tailscale` is on `PATH`, the welcome *documents* the command — it does not offer to run `tailscale serve` for you.
- **No QR.** A copy button on the `ts.net` URL is enough.
- **Install ownership.** `@kolu/solid-pwa-install` owns the card UI + cross-browser install volatility; `@kolu/surface-app` owns the manifest + installability signals and **may depend on `solid-pwa-install`** to compose the card — a surface app wires install once.
- **Builds on #1154 (merged).** `@kolu/surface-app` is in `master`; layers ① and ② extend its `installPwaManifest` and `useSurfaceApp()` directly.

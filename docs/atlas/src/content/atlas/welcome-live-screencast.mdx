The <Cite file="website/src/pages/welcome.astro" label="kolu.dev/welcome" />
guide tells you what kolu does in prose and static terminal mockups. This note
adds the thing prose can't carry: **short, looping clips of a real kolu actually
doing the thing** — open a repo, launch an agent, watch the dock pulse — embedded
inline next to the section that describes it, and (later) promoted to the homepage
hero.

The capture mechanism already exists — it's the
<Cite file="docs/atlas/src/content/atlas/video-evidence.mdx" label="video-evidence" />
substrate (`KOLU_EVIDENCE=1` → Playwright `recordVideo` on the Cucumber harness).
The work here is **not a new recorder**; it's reusing that capture and **branching
at transcode** for a different audience. PR evidence optimizes for *proof* and
degrades to a 12 fps GIF; a marketing clip optimizes for *crispness* and must
never touch GIF. One load-bearing constraint drives every decision below:
**the clip has to look sharp — no low-quality nonsense.**

## The pipeline — reuse evidence capture, fork at transcode

Capture is a *solved problem* in this repo. <Cite file="packages/tests/support/hooks.ts" />
already gates `recordVideo` on `KOLU_EVIDENCE` and pins `recordVideo.size` to the
evidence viewport, and the ~880-line step library drives every interaction. A
marketing clip reuses all of it: author (or pick) a scenario, select it by
`--name`, get a video out. No parallel harness, no hand-driven clicks.

What changes is everything *downstream* of the raw capture:

| | PR evidence (<Cite file="docs/atlas/src/content/atlas/video-evidence.mdx" label="existing" />) | Marketing clip (this note) |
| --- | --- | --- |
| Audience | a reviewer, once | every visitor, repeatedly |
| Final form | inline **GIF** + Pages player | `<video>` loop, **mp4 + webm** |
| Priority | "proves it works" | **crisp + small + autoplay** |
| Hosting | `evidence-assets` release | committed under `website/public/` |
| Lifetime | throwaway | regenerated when the UI moves |

So the two pipelines **share the capture stage and diverge at transcode.** That
keeps the new surface area tiny — a transcode recipe and an embed — and means a
single seeded scenario can feed *both* a PR-evidence GIF and a welcome clip.

## Crisp by construction

This is the section the "no low-quality nonsense" rule lives in. There are two
quality ceilings, and only one is fixable in post.

**The transcode is the easy half.** Never ship GIF (H.264 is 10–50× smaller for
the same frames). Emit two web assets from the source, audio-free:

```bash
# Universal, crisp, fast-start. crf 18 ≈ visually lossless for screen content.
ffmpeg -i demo.src.mkv \
  -c:v libx264 -pix_fmt yuv420p -crf 18 -preset slow \
  -vf "scale=trunc(iw/2)*2:trunc(ih/2)*2" -an -movflags +faststart demo.mp4

# Smaller modern path, served first. VP9 CRF ~31 ≈ x264 crf 18 — NOT 18.
ffmpeg -i demo.src.mkv -c:v libvpx-vp9 -b:v 0 -crf 31 -row-mt 1 -an demo.webm

# Exact first frame as the poster (identical to frame 1 → no LCP flash).
ffmpeg -i demo.src.mkv -frames:v 1 -update 1 -q:v 2 poster.webp
```

<Callout kind="warn" title="The real ceiling: Playwright's recordVideo is hardcoded VP8 @ ~1 Mbit/s">
Playwright's `recordVideo` encodes **VP8 at a fixed ~1 Mbit/s, 25 fps, no quality
knob** — the flags don't exist ([videos docs](https://playwright.dev/docs/videos),
issues [#17217](https://github.com/microsoft/playwright/issues/17217) /
[#31424](https://github.com/microsoft/playwright/issues/31424)). At 720p of dense
xterm.js text, 1 Mbit/s VP8 *shimmers* — and **no ffmpeg pass can recover detail
the VP8 stage already threw away.** A `crf 18` mp4 transcoded from a 1 Mbit/s
webm just faithfully preserves blur. So for a marketing-grade clip the source
must be higher fidelity than `recordVideo` can produce.
</Callout>

Two levers make it genuinely crisp:

- **Capture above display size.** The clip renders at maybe ~720–900px wide on the
  page; capture at **2× device scale factor** (`deviceScaleFactor: 2`) and crop to
  the relevant region (one tile + dock, not a full 1280×720 desktop). Downscaling a
  2560-wide source to an 800px element is what "retina" looks like. Keep
  `recordVideo.size == viewport` regardless — the default silently downscales to fit
  800×800 and blurs text ([pitfall #1](https://playwright.dev/docs/videos)).
- **For the shipped asset, capture lossless frames, not VP8.** Swap `recordVideo`
  for **CDP per-frame capture** (`Page.captureScreenshot` PNG per tick, or
  `page.screencast` on Playwright ≥1.59 / the chrome-devtools MCP `screencast_start`)
  → assemble with ffmpeg from PNGs → `crf 18`. A lossless source removes the only
  ceiling we can't transcode around. <Pill variant="ok">recommended for ship</Pill>

<Callout kind="note" title="Decision: prototype on recordVideo, ship on frame-capture">
Use the **existing `recordVideo` path for the prototype** — zero new plumbing,
proves the scenario and embed end-to-end. **Judge the result at 100% on a retina
display.** If xterm text shimmers (it likely will at 720p VP8), promote that one
clip to the **frame-capture source**. Don't pre-build frame-capture before seeing
the VP8 fail — but don't ship VP8 to a marketing page if it does.
</Callout>

## The embed — a poster-first looping video

The asset lives at `website/public/demo/<clip>.{mp4,webm}` + `poster.webp` and is
referenced from <Cite file="website/src/pages/welcome.astro" /> with the pattern
every serious dev-tool site (Zed, Linear, Warp) converges on:

```html
<video autoplay muted loop playsinline preload="metadata" poster="/demo/first-agent.webp">
  <source src="/demo/first-agent.webm" type="video/webm" />  <!-- VP9 first: smaller -->
  <source src="/demo/first-agent.mp4"  type="video/mp4"  />  <!-- H.264 fallback -->
</video>
```

Non-negotiables, each a real failure mode if dropped:

- **`muted`** — browsers block autoplay for anything with sound; muted is the exemption ([MDN Autoplay](https://developer.mozilla.org/en-US/docs/Web/Media/Autoplay_guide)).
- **`playsinline`** — without it, iOS Safari forces the clip *fullscreen* on play ([WebKit](https://webkit.org/blog/6784/new-video-policies-for-ios/)).
- **WebM `<source>` first** — the browser picks the first decodable source top-to-bottom; mp4-first makes Chrome/Firefox fetch the bigger file.
- **Poster = frame 1** — exported above, so first paint is instant and the swap to video is seamless (no flash). Below the fold, switch to `preload="none"` + an `IntersectionObserver`.

<Callout kind="warn" title="Reduced motion is mandatory, not a nicety">
An autoplaying loop is motion. Gate it on `prefers-reduced-motion: reduce`: when
set, render the **poster only** (no autoplay) and a visible play control. The
welcome page is plain Astro HTML — a tiny inline `<script>` reading
`matchMedia('(prefers-reduced-motion: reduce)')` to strip `autoplay` is enough
(no SolidJS / `@solid-primitives/media` needed here — that primitive is for the
in-app client, not the static site). Also call `.play().catch()` so iOS
low-power-mode rejection falls back to the poster gracefully.
</Callout>

## Reproducible, not hand-recorded

A clip recorded by hand goes stale silently and can't be refreshed when the UI
moves. Make it a **build artifact**. The capture rides a seeded scenario; five
levers keep it byte-stable enough to regenerate:

1. **A seeded demo scenario** reusing the step library (deterministic worktree
   names, fake-agent output, repo fixtures — the harness already scaffolds these).
2. **Freeze time** — Playwright `Clock.setFixedTime`, installed **before**
   `page.goto()` ([clock docs](https://playwright.dev/docs/clock)).
3. **Motion policy is inverted vs. evidence.** `hooks.ts` injects
   `prefers-reduced-motion: reduce` for test determinism — a marketing clip is
   *about* the motion (the dock pulse), so the demo run must **skip that
   animations-off init** (same carve-out the video-evidence note flags), while
   still pacing with wait-for-condition, not `sleep`.
4. **Seed randomness** — pin `Math.random`/UUID sources; `crypto.randomUUID()` is a
   top cause of run-to-run drift.
5. **Pin viewport / DPR / fonts** and run in the harness's standard environment so
   rendering matches.

Wrap it as a `just` recipe pair mirroring the house pattern —
`just website::demo-preview` (capture + transcode to a scratch dir for eyeballing)
and `just website::demo-update` (regenerate + stage `website/public/demo/`).

<Callout kind="note" title="Decision: commit the clips, don't release-host them">
PR evidence lives on the `evidence-assets` release because GitHub won't embed a
committed mp4 in a comment. The website has no such constraint — Astro serves
`public/` directly, and the clips are part of the page. **Commit them** (a few
small mp4/webm under `website/public/demo/`), so the site builds reproducibly with
no external fetch. Keep each clip in the **single-MB** range; if a clip ever blows
past that, revisit hosting then.
</Callout>

## Prototype — the thinnest end-to-end slice

One clip, the "First 5 minutes" section: *open a repo → launch an agent → the dock
pulses.* Smallest path that proves capture, transcode, and embed together.

### Pitfalls

<Callout kind="warn" title="The traps, condensed">

- **Transcoding VP8 and calling it crisp.** 1 Mbit/s VP8 is the ceiling; ffmpeg can't recover it. Capture lossless if it shimmers.
- **Default `recordVideo` size.** Pin it to the viewport or text blurs from the 800×800 downscale.
- **VP9 CRF copied from x264.** Use ~31, not ~18, or the webm bloats past the mp4.
- **Missing `muted` / `playsinline`.** No autoplay anywhere / forced fullscreen on iOS.
- **mp4-first `<source>`.** Wastes bytes — webm first.
- **Animations-off bleed-through.** The evidence harness suppresses motion; the marketing run must skip that init or you film a still.
- **Real-time frame drift.** `recordVideo`/screencast capture in real time, so smoothness varies run-to-run; only reach for `beginFrame`/virtual-time capture if a clip *must* be frame-perfect.

</Callout>

### Decisions

- **Reuse the Cucumber harness for capture** — same scenario can feed PR evidence *and* the welcome clip; no second recorder.
- **Fork at transcode, never GIF** — H.264 mp4 + VP9 webm + WebP poster.
- **Prototype on recordVideo, ship on frame-capture if VP8 shimmers** — don't pre-build the crisp path; don't ship the blurry one.
- **Commit clips under `website/public/demo/`** — not the evidence release; the site needs them at build time, single-MB each.
- **Static-site embed, not the SolidJS client** — a plain inline `<script>` handles reduced-motion; no `@solid-primitives` dependency on the marketing page.

## Architecture — capture engine vs. recordings (the electricity call)

Read through <Cite file="docs/atlas/src/content/atlas/electricity.mdx" label="electricity.mdx" />,
this work falls on **two sides of the domain line**, and conflating them is the
trap to avoid.

- **The capture engine is a graduation candidate, not yet electricity.**
  ① *agnostic* — "drive a headful browser at 2× under Xvfb, `x11grab` the
  framebuffer, transcode to web assets" names no terminal/canvas/git; it would
  screencast any web app. ② *hard volatility* — Xvfb lifecycle, app-mode launch,
  the fps-vs-resolution wall, `x11grab`/ffmpeg flags, SIGINT finalize, the
  `ffmpeg-full` nix gotcha — not a 3-line helper. ③ *graduates* — **aspirationally
  only**: drishti (surface's real 2nd consumer) or any project could make its own
  clips, but no second consumer exists *yet*. Verdict, per the note's own traps:
  isolate it into **its own folder with the dependency arrow pointing out** (nix
  deps + `.ts` runner), flagged <Pill variant="run">publish?</Pill> like
  `createSharedRoot` — mint `@kolu/web-screencast` and claim electricity only when
  a second consumer is real. Do **not** bury it in the app (trap #2: location *is*
  structure); do **not** call it electricity on agnosticism alone (trap #3).

- **The recordings are kolu domain.** Each recording's script
  (*new terminal → git clone → claude*) and its display tweaks (*hide the right
  panel*, *collapse the dock* — enacted via kolu shortcuts/prefs) reference kolu's
  own UI. They stay in kolu, depending on the engine — never the reverse.

The interface that keeps the line clean: the engine exposes
`capture({ chrome, pace, size }, drive)` where `drive(page)` is a **kolu-domain
closure**. A recording's kolu-specific knobs (`hideRightPanel`, `collapseDock`)
are applied *inside* `drive` via kolu shortcuts — they are **recording**
properties, never **engine** options. So the engine stays agnostic while each
recording still carries its declarative display properties.

**Folder layout (isolate the concept, per your call):**

- `packages/screencast/` — the agnostic engine + its **own nix** (`ffmpeg-full`,
  `xvfb`, playwright) + `.ts` runner. Zero kolu imports; arrow points out. *(Until
  it earns a package, it can live as a self-contained `screencast/` module beside

- A `just record <name>` / `record-all` recipe drives a recording through the engine.

## Log — build journal

A running record of *this work's* research, dead-ends, and decisions — raw
material for a future blog post. Newest at the bottom.

<Callout kind="note" title="The sections above predate the pivot">
The plan above (sections 1–5) captures the *original* design — reuse `recordVideo`,
fork at transcode. The log below is where it actually went: through a hard
fps-vs-resolution wall, a second research pass, and out the other side to an
**OS-level x11grab capture in Chrome app-mode**, then a **`Recording` subsystem**.
The body will be revised to match once the subsystem lands.
</Callout>

**1 · Plan + prior-art sweep.** <Pill variant="ok">workflow #1</Pill> Picked up the
`kolu.dev/welcome` thread from <Cite file="docs/atlas/src/content/atlas/welcome-revamp.mdx" label="welcome-revamp" />.
A fan-out research workflow (terminal recorders vs browser capture · embed
patterns · CI determinism), adversarially verified, concluded: PTY recorders
(vhs, asciinema) **can't** capture kolu's xterm-on-canvas pixels — it's a raster
GUI, not a replayable ANSI stream — so **pixel capture via the existing
Cucumber + Playwright evidence harness** is the path. Embed pattern: poster-first
`autoplay/muted/loop/playsinline`, mp4 + webm, reduced-motion.

**2 · The crispness constraint.** "Video needs to be crisp — no low-quality
nonsense." This became the spine. The evidence path records Playwright
`recordVideo` = **VP8 @ ~1 Mbit/s, 720p** — fine for PR proof, too soft for a
marketing page.

**3 · Spike A (VP8).** Reused the harness (`KOLU_EVIDENCE`) on a real
`worktree-agent` scenario → 1280×720 VP8 → ffmpeg H.264. Smooth, but the text
shimmered. Verdict: "could be crispier."

**4 · The fps-vs-resolution wall** (the interesting part — good blog material).
Chasing 2×, every in-Chrome capture API hit a structural limit:

- CDP `Page.startScreencast` **ignores `deviceScaleFactor`** → caps at CSS pixels (1280); can't do true 2×.
- CDP `Page.captureScreenshot` *can* do 2× (via `clip.scale`), but each 2560×1440 grab is **compositor-bound at ~260 ms** → ~3 fps **slideshow**.
- `page.screenshot()` gives true 2× but **serialises behind the driver's actions** → ~10 frames over a 4 s scenario.
- JPEG q92 instead of PNG sped the encode, but the scenario is only ~4 s → still ~13 frames.
- Bonus bug: the capture loop started *before* the app loaded, so frame 0 was a blank white flash that strobed on every loop.

**The lesson:** real-time frame capture is *either* smooth-but-low-res/lossy *or*
high-res-but-too-slow-to-be-smooth. You can't win it from inside Chrome.

**5 · Reset + second research pass.** <Pill variant="ok">workflow #2</Pill>
Reverted B to the smooth (supersampled-1280) screencast and ran a second fan-out
workflow on *existing solutions* (virtual-time `beginFrame` · OS-level capture ·
purpose-built recorders · native Playwright options), adversarially verified.
**Verdict: move the capture boundary outside Chrome** — `Xvfb` + `ffmpeg
-f x11grab` of a **headful 2× browser** samples the framebuffer on its own fixed
clock in physical pixels, so it's *structurally* smooth **and** crisp at once.
Also surfaced: **no top dev tool ships an auto-captured hero clip — they're
"designed, not recorded"** (Screen Studio etc.). We're deliberately taking the
reproducible-over-cinematic path.

**6 · x11grab, working.** Implemented `KOLU_X11CAP`: Xvfb at 2560×1440, headful
Chrome at `--force-device-scale-factor=2`, `ffmpeg -f x11grab` at 30 fps, **SIGINT**
to flush the moov atom. Gotcha that cost a cycle: nixpkgs `ffmpeg` is built
`--disable-xlib --disable-libxcb` (**no x11grab device** — "Unrecognized option
'draw_mouse'") → switched the **e2e devShell** to `ffmpeg-full`. Result:
**2560×1440, 30 fps, smooth + crisp.** Verdict: "B looks good."

**7 · The browser-chrome realisation.** x11grab records the *whole window* — the
clip showed Chrome's tab strip + `localhost:38425` address bar. Reads as "a
browser tab," not "the app." Fix: **Chrome app-mode (`--app=<url>`)** = the
chromeless window an installed PWA uses. Driven via `launchPersistentContext`;
the steps' own `page.goto("/")` re-navigation re-applies the harness init scripts.

**8 · Scope crystallised into a subsystem.** Decided: **drop A**, one clip per
slot, **human pace**, and **N recordings — each its own file** with declarative
properties (hide right panel, collapse dock, app/browser chrome, pace): a
lowy/hickey-clean **`Recording`** abstraction (data + script, separate from the
capture pipeline). First two: ① **PWA install/setup**, ② **installed-PWA demo**
(new terminal → `git clone` → `cd` → `claude`). Confirmed the pieces exist:
kolu's in-page install card is automatable (<Cite file="packages/client/src/WelcomeMoments.tsx" />,
`data-testid="welcome-install"` / `welcome-install-manual`), and the terminal /
git / palette steps for ② already exist in the step library — no new step code.

**9 · Subsystem shipped.** <PrLink pr={1213} /> The agnostic engine
(<Cite file="packages/tests/screencast/engine.ts" />, nix deps in
`screencast/shell.nix` — top-level flake devShells untouched), the `Recording`
modules (<Cite file="packages/tests/screencast/recordings" />), a
`When I record "<name>"` dispatcher, and a `just record` recipe. Both clips
captured: ② **app-mode** (chromeless), `git clone` kolu into `~/demo` →
`cd` → `ls` → **`claude` launches with its trust prompt** (15.6 s, 2560×1440);
① **browser chrome**, the palette **Tutorial** command summons the welcome with
the **Pin it** card + expanded install steps (8.6 s). Captured **locally**, not
on pu: `kolu-ci-1` works (egress OK) but has **no authenticated `claude`**, so
the demo's climax only renders on a machine already running Claude Code — a
deliberate, documented tradeoff (quality > box-purity; reproducibility lives in
the recipe + modules, which *do* run on pu).

**10 · Refined to one hero clip.** Dropped the **pwa-install** recording — an
in-clip browser→app transition isn't reachable (bare Xvfb has no window manager,
so F11 / the Fullscreen API can't drop Chrome's chrome; it would need two
concatenated segments), and the app-mode demo already shows the chromeless app.
The demo became the real loop: clone kolu → launch `claude --dangerously-skip-
permissions --model sonnet` → accept the trust gate → ask *"Explain this project,
in 5 lines"* → wait for the **dock bucket to flip `working` → `awaiting`** (the
high-level state is `data-bucket`, not the raw `data-agent-state`) → **glow the
awaiting dock row** and hold so the status change is unmissable. Two harness
fixes made the dock track the live agent at all: (a) under `KOLU_X11CAP`, omit
the `KOLU_CLAUDE_*_DIR` overrides so kolu watches the **real** `~/.claude/projects`
(the mock-harness temp dirs hid the launched claude); (b) a 240s timeout on the
`I record` step (a real agent query exceeds the default budget). Result: a tight
**~27s / ~1.3 MB** clip. Reusable bits factored into `helpers.ts`
(`setupSingleTerminal`, `launchAgentAndAsk`, theme-aware `newTerminal`) for the
next single-terminal demo.

## What the reader sees

A footnote marker becomes a button. Click (or tap) the small `[1]` and its
definition opens in a card anchored just under the marker, where your eye
already is. Click anywhere else, press <kbd>Esc</kbd>, or click the marker
again, and it closes. The gesture is identical on a trackpad and on a phone â€”
there is no hover step, so nothing is stranded on touch.

The details that matter to a reader:

- **The bottom "Footnotes" section stays exactly as it is.** It is still the
  printable, copyable, screen-reader-navigable record of every note â€” and it is
  where the popover reads its content from. The popover is an *additional* way
  in, not a replacement.
- **Footnote bodies can be rich.** A note may run to several paragraphs or carry
  a link, a `[[wikilink]]`, inline code, or an image. The panel renders the full
  body and scrolls if it is tall; any link inside it still opens the right way,
  because it rides the preview's existing resolvers (relative links open the
  file, wikilinks resolve vault-wide, images load through the repo route).
- **The back-reference `â†©` is unaffected.** It still lives in the bottom section
  and still jumps back to the marker; it never opens a popover.

<Callout>
Only the **document** preview is in scope â€” and that is by construction, not a
limitation. The compact/inline chat slots strip the whole footnotes
`<section>` during sanitization, so a footnote definition does not exist in
their DOM and there is nothing to pop. The feature naturally no-ops there.
</Callout>

## Architecture â€” a leaf that reuses the wikilink seam

<Svg svg={reuse} caption="A footnote click threads the same three seams alerts and wikilinks already use. The package gains one callback; the client renders the overlay with the hook the wikilink-disambiguation menu already uses." />

This is a **leaf**, not a new `@kolu/*` package. Apply the electricity test: a
footnote popover hides only *bounded* logic â€” pair a marker to its definition
`<li>` by id, then place a panel beside it. It hides no hard volatility â€”
no transport, no reconnect, no persistence, no GPU-context loss â€” so it fails
all three extraction tests. Nothing to receptacle; nothing to install.

**The boundary it must respect is the dependency arrow.** `@kolu/solid-markdown`
depends only on `solid-js`, `marked*`, `dompurify`, `shiki`, `yaml`, and two
tiny leaf utils â€” it has *zero* knowledge of the client app, of Corvu, or of
`Portal`, and the arrow points outward (client â†’ package). The package already
hands hard, host-specific decisions back across that arrow rather than owning
them: `onNavigateRelative` and `onNavigateWikilink` exist precisely because
*resolving* a link is host volatility. **Footnote popovers split along the same
seam:**

- **Inside the package** (`@kolu/solid-markdown`): the delegated `onClick` in
  `bindInteractions` already inspects every clicked anchor and already does the
  `el.querySelector('#â€¦')` lookup that scrolls an in-page anchor into view. It
  grows one branch â€” recognise a footnote marker, find its definition node, and
  fire a **new `onFootnote(anchor, definition)` callback**. That callback is the
  package's *entire* new surface.
- **Inside the client** (`BrowseFileDispatcher`): the host catches `onFootnote`
  and renders the popover with `useAnchoredPopover` + `<Portal>` â€” the *exact*
  hook the wikilink-disambiguation menu already uses to anchor a panel to a
  clicked marker in this same preview. Overlay rendering â€” and any positioning
  library â€” stays on the client side of the arrow, where it already lives.

**The verdict is contingent on the click model (Â§ the Architectureâ‡„Implementation
loop).** Because we open on *click*, the existing `useAnchoredPopover` fits with
no new dependency â€” it is built for click-to-open, outside-click/Escape-dismiss
panels. Had we chosen a *hover* hovercard, we would have needed hover-intent
timing plus `floating-ui`-style `autoUpdate` scroll-tracking that
`useAnchoredPopover` does not have, which would have pulled a positioning
dependency (likely `@corvu/tooltip`) into play and pressured this boundary. The
"leaf, no new package, no new dep" verdict *is* the click decision; a different
trigger model would have produced a different shape.

## Implementation â€” the seams, in order

It threads the same render â†’ sanitize â†’ interact â†’ host-overlay path that alerts
(`data-md-alert`) and wikilinks (`data-md-wikilink`) already proved. Three PRs,
test-first.

### PR1 â€” pin the contract

DOMPurify is brutal on footnote markup: `class`, every `data-footnote-*`, and
`aria-describedby` are all stripped, so the only thing that survives a forward
reference is `<sup><a id="md-footnote-ref-1" href="#md-footnote-1">1</a></sup>`
(ids and `#`-hrefs get an `md-` namespace prefix). The whole feature stands on
that shape, and `marked-footnote` is pinned only as `^1.2.4` â€” a minor bump
could change the markup silently. So land a sanitize-level test
(`packages/solid-markdown/src/sanitize.test.ts`, happy-dom) first: feed footnote
HTML through `sanitizeHtml` and assert `id="md-footnote-1"` + `href="#md-footnote-1"`
survive while `data-footnote-ref` / `aria-describedby` / `class` are gone. A
markup change then fails loudly instead of quietly breaking the popover.

### PR2 â€” mint the marker and the callback (the package's only new surface)

- **`render.ts` â€” `rewriteFootnotes(html)`**, a sibling of the existing
  `rewriteAlerts` (`render.ts:291`). It runs on the *pre-sanitize* string, where
  `marked-footnote`'s `class="footnote-ref"` / `data-footnote-ref` markers still
  exist, and rewrites the forward-ref anchor to carry an allowlist-safe
  `data-md-footnote` attribute (the back-ref `â†©` is deliberately *not* tagged).
  This is the same move alerts make, and it sidesteps the two traps of detecting
  the ref by structure after sanitization: a back-ref `â†©` link also points at
  `#md-footnote-â€¦`, and a heading literally titled "Footnote 1" would mint the
  same `id`. An explicit parser-minted marker is unambiguous.
- **`sanitize.ts`** â€” add `data-md-footnote` to `DOCUMENT_ATTR` (one line, beside
  `data-md-wikilink` at `sanitize.ts:197`; security-reviewed, since it widens the
  allowlist).
- **`Markdown.tsx` â€” `bindInteractions`** grows a footnote branch in `onClick`,
  placed **before** the generic `#`-anchor scroll branch (`Markdown.tsx:98`):
  `target.closest('[data-md-footnote]')` â†’ resolve the definition via the same
  `el.querySelector('#' + CSS.escape(â€¦))` the scroll branch uses â†’ `preventDefault`
  â†’ `props.onFootnote(anchor, definition)`. The back-ref `â†©` keeps the existing
  scroll-up behaviour. Add the `onFootnote(anchor, definition)` prop (mirror
  `onNavigateWikilink`, `Markdown.tsx:153`).

### PR3 â€” render the popover in the client (reuse `useAnchoredPopover`)

- **`solid-fileview/src/renderers/markdown.tsx`** â€” thread `onFootnote` through
  `MarkdownRenderer`'s props, mirroring `onNavigateWikilink` (`markdown.tsx:29`).
- **`BrowseFileDispatcher.tsx`** â€” a `createSignal<{ anchor, definition }>()` set
  in `onFootnote` (mirror `wikiMenu`, `BrowseFileDispatcher.tsx:147`), and a
  `FootnotePopover` rendered via `<Portal>` + `useAnchoredPopover({ triggerRef:
  () => fn()?.anchor, anchor: "bottom-start", flip: true, onDismiss })` â€” the same
  recipe as the `OptionMenu` disambiguation list (`BrowseFileDispatcher.tsx:462`).
  Panel content is `definition.cloneNode(true)` with **every** `a[href*="-ref-"]`
  back-ref removed and the `id` stripped (a re-cited footnote has several `â†©`, and
  the clone must not duplicate a live `id`); its already-sanitized `innerHTML` is
  re-injected and the same delegated click handling bound, so inner
  relative-links / wikilinks / images route through the host resolvers the
  preview already wires. Reuse `surface()` for chrome (as `OptionMenu` does) with
  a `max-height` + scroll for tall bodies.

### The risks that bite

- **`useAnchoredPopover` doesn't track scroll.** It repositions only on
  open/trigger-change (`useAnchoredPopover.ts:135`), and the preview sits inside
  *nested* `overflow:auto` containers â€” so a panel anchored with `position:fixed`
  drifts off the marker as the reader scrolls, and `flip` is vertical-only (a
  marker near the right edge of the reading column can overflow). Decide one:
  **dismiss-on-scroll** (smallest, keeps the hook untouched â€” the likely first
  cut) or **extend `useAnchoredPopover`** with a scroll/resize reposition. This is
  the low â†’ low-medium risk bump in the verdict.
- **The comment overlay.** The preview is wrapped in `CommentTextSurface` with a
  `MutationObserver` over its subtree; an *in-flow* popup would trip it. The
  `Portal`-to-body route avoids it entirely â€” which is what `useAnchoredPopover`
  already does, and another reason to reuse it rather than mount in place.
- **Touch + the drawer.** Click-to-open already covers touch (no hover step). On
  mobile the right panel can mount inside a `@corvu/drawer` (`modal: true` sets
  `body { pointer-events: none }`); `useAnchoredPopover` already re-enables
  `pointer-events: auto` on its panel (`useAnchoredPopover.ts:154`), so the
  popover stays tappable there.
- **Nested footnotes.** A footnote body can cite another footnote, so a marker
  can appear *inside* a popover. Keep it simple: markers inside a popover body are
  inert (they carry no live handler in the cloned subtree), rather than stacking
  popovers.

<Callout kind="accent" title="Verdict ‚Äî the diagnosis is correct, with one sharpening: CI already catches this">
At the time of this diagnosis a standalone `typecheck` node ran in the CI DAG, so a server type error *could not* reach `master` through CI even then; <PrLink pr={1056} /> has since replaced that node with the cached `checks.*.typecheck` flake output realized by the `nix` devour-flake node (<Cite file="ci/mod.just" lines="86-90" />). The real, narrower gap was **local false confidence**: a standalone `nix build` exits 0 with broken types (incident #1034), and `nix build` is therefore not the proof people treat it as. The fix the issue asks for ‚Äî make `nix build` itself a type-proof ‚Äî is worth doing, and it also lets us *delete* a CI node rather than add one.
</Callout>

## Recommendation ‚Äî a `checks.typecheck` derivation

Add a typecheck *derivation* to the root `flake.nix` that reuses `pnpmDeps`, runs `pnpm -r typecheck`, and `touch $out` on success. Three things fall out of this:

- **It becomes a hard, un-skippable gate.** The CI `nix` node builds *all* flake outputs via devour-flake (<Cite file="ci/mod.just" lines="63-64" />). A new `checks.*` output is picked up automatically ‚Äî no new CI recipe needed.
- **It's content-addressed.** The derivation only re-runs when its source changes, and the result is shared through `cache.nixos.asia/oss`. The current `ci::typecheck` recipe re-runs `pnpm typecheck` unconditionally every pipeline.
- **It lets us delete a node.** Once typecheck is a flake output covered by the `nix` node, the standalone `typecheck` recipe and its DAG edge are redundant ‚Üí remove them. Net: **3 typecheck loci ‚Üí 3** (package.json script, `just check`, nix derivation), one fewer CI node, all still delegating to the one shared `pnpm -r typecheck` mechanism.

<Callout kind="good" title="Why a separate derivation and not .#default's buildPhase">
"Produce a runnable artifact" and "prove types are sound" are independent axes of change ‚Äî incident #1034 *is* the proof they diverge. Folding `tsc` into `.#default` would complect them: a type-only edit would bust the artifact cache, and `nix run .` / `smoke` / `e2e` would pay a typecheck tax for nothing. The build needs node-gyp + Vite + font symlinks + commit stamping; typecheck needs only `tsc` + node_modules. Keep them as causally independent outputs sharing `pnpmDeps`. *(Lowy: correct seam, Nix layer is where the burned consumer lives.)*
</Callout>

### The trap to avoid ‚Äî fileset drift <Pill variant="bad">must-fix</Pill>

The build's `src` fileset (<Cite file="default.nix" lines="15-38" />) is a hand-maintained allowlist of package dirs ‚Äî and it **omits `packages/tests`**. If the typecheck derivation reused that fileset, it would typecheck a *different, smaller* set than `just check` (which sees the whole working tree). A package added to `pnpm-workspace.yaml` but forgotten in the fileset would type-error while the derivation exits 0 ‚Äî reintroducing the exact false-green #1049 is about, just relocated.

<Callout kind="warn" title="Resolution (chosen): give the typecheck derivation its own broad fileset">
Source it from `./packages` wholesale (plus root `tsconfig.base.json`, `package.json`, `pnpm-workspace.yaml`, lockfile) rather than the narrow shipping allowlist. There is then **no per-package list to drift** ‚Äî `pnpm -r typecheck` sees exactly the workspace, by construction. This is simpler than the alternative (a second `checks.fileset-completeness` derivation diffing the workspace against the allowlist), and the broader source is *correct* for typecheck anyway: it *should* re-run whenever any typechecked package changes. The build keeps its tight allowlist for cache reasons; the two filesets have opposite goals, so they should differ.

**As built, this was reversed.** <PrLink pr={1056} /> has the typecheck derivation (`nix/pnpm-typecheck.nix`) reuse the build's narrow allowlist `src` + `pnpmDeps`, and holds off the drift trap with an INVARIANT comment on the fileset instead (<Cite file="default.nix" lines="21-26" />): every workspace package with a `typecheck` script must be listed, and `packages/tests` ‚Äî the one member absent ‚Äî has no typecheck script, so it is outside the gate's scope either way.
</Callout>

### Pin to one platform <Pill variant="warn">cost</Pill>

devour-flake builds `checks.${system}` for all three systems. TypeScript typecheck is platform-independent, so attaching it to all three triples CI cost for identical results. Pin it to one ‚Äî following the existing precedent <Cite file="nix/home/example/flake.nix" lines="70" /> (`checks.${linuxSystem}.vm-test`) and <Cite file="nix/home/example/flake.nix" lines="114" /> (`checks.${darwinSystem}`).

**As built, this was reversed too.** <PrLink pr={1056} /> runs the type gates on *every* system, deliberately: the build environment (nodejs/pnpm and the platform-resolved deps `pnpmConfigHook` installs) differs per platform, so each platform's `tsc`/`astro check` is its own proof ‚Äî a darwin-only type error wouldn't surface from a linux-only check (rationale comment in <Cite file="flake.nix" lines="66-72" />).

## Concrete change (single PR ‚Äî infra, not user-visible, no phasing)

### 1 ¬∑ `flake.nix` ‚Äî add the check output

```diff
+# typecheck the whole workspace as a cached, devour-flake-gated proof.
+# Own broad fileset (all of ./packages) ‚Äî NOT default.nix's narrow shipping
+# allowlist ‚Äî so a new package can never silently escape the gate (#1049).
+# Pinned to one system: tsc is platform-independent (cf. nix/home/example).
+checks.${linuxSystem}.typecheck = import ./nix/typecheck.nix { inherit pkgs; };
```

A small `nix/typecheck.nix` leaf: `mkDerivation` reusing `pnpmDeps` + `pnpmConfigHook`, `buildPhase = "pnpm -r typecheck"`, `installPhase = "touch $out"`. No node-gyp rebuild ‚Äî `tsc --noEmit` needs `.d.ts`, not the compiled `.node`.

### 2 ¬∑ `ci/mod.just` ‚Äî delete the now-redundant recipe

```diff
-default: nix home-manager e2e smoke fmt typecheck biome unit surface-example-build pnpm-hash-fresh
+default: nix home-manager e2e smoke fmt biome unit surface-example-build pnpm-hash-fresh

-typecheck: install
-    {{ nix_shell }} pnpm typecheck
```

And one comment at the `nix:` recipe (<Cite file="ci/mod.just" lines="63" />) noting that `checks.typecheck` (built here via devour-flake) replaced the standalone recipe ‚Äî so the temporal coupling ("the `nix` node is now load-bearing for typecheck") is documented, not folklore.

### 3 ¬∑ `default.nix` ‚Äî one comment at the buildPhase

```diff
 buildPhase = ''
   runHook preBuild
   ...
+  # NOTE: this does NOT typecheck ‚Äî only the Vite client bundle is built and
+  # the server runs under tsx. The type gate is `nix build .#checks‚Ä¶typecheck`
+  # (run by CI's `nix` node). A green `nix build .#default` is not a type-proof.
   pnpm --filter kolu-client build
   runHook postBuild
 '';
```

Leave `just check` (<Cite file="justfile" lines="67-68" />) exactly as-is ‚Äî it stays the fast local inner loop. No pre-push hook needed: making `nix build` a type-proof and keeping the CI gate is sufficient; a hook would be a fourth locus for the same concept.

## What I deliberately rejected

- **Folding `tsc` into `.#default`.** Complects artifact-production with type-proof; taxes `nix run` / smoke / e2e; busts the build cache on type-only edits.
- **A separate `checks.fileset-completeness` guard.** Hickey's fix for the drift trap is valid, but the broad-fileset approach above kills the same failure mode *by construction* with no new abstraction ‚Äî preferred unless a narrow fileset must be kept for caching (it needn't be).
- **Docs-only ("just rely on the existing CI `typecheck` node").** Honest and zero-abstraction, but it leaves `nix build` a non-proof ‚Äî which is the issue's actual ask ‚Äî and forgoes the cache win. The buildPhase comment ships regardless; the derivation is the part that closes the gap.

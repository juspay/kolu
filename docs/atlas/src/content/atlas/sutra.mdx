**`@kolu/sutra` — the receptacle the [finale](remote-terminals-finale.html) was missing.** *sutra* (Sanskrit: the connecting thread) is the thread that binds you to **one host's** terminals — the sibling of [kaval](pty-daemon.html), [pulam](pulam.html) and [drishti](https://github.com/srid/drishti) in kolu's graduated-electricity family. Every hard thing left in remote terminals — dialing a host, routing a lifecycle op to *its* kaval/pulam, reconnecting an ssh link, re-adopting surviving PTYs at boot — is **one volatility**: *the terminal workflow on a host you may be reaching over ssh*. Today it's not a boundary, it's a **convention** smeared across kolu-server: the local endpoint is `export`ed and directly reachable, so "hard-pin local / bypass the host router" is a *writable expression* a missed seam forgets (the round-2 type-enforcement debate verified the smear; `debates/pr1637/`). This note lifts that volatility into **one `@kolu/*` package both UIs embed** — and, crucially, scopes it to **one host at a time**, which is what turns ssh-remote from a cross-cutting rewrite into a connection you dial.

Depends OUTWARD; neither UI is depended on. The cross-host fleet glance stays pulam-web's job." />

<Callout kind="note" title="Why this note exists — naming the receptacle, not the organ">
A live debate proposed graduating a **kolu-TUI** as the electricity. That names an *organ*: a TUI is a product surface (panes, keybindings, attach) — it fails electricity [test ①](electricity.html) (domain-agnostic) on sight, exactly the "history stack vs. browser" trap. The electricity is the **substrate** the TUI would *consume*. The TUI's role is the other half of the proof: it's the **second consumer** that shows the substrate graduated — the way [drishti](https://github.com/srid/drishti) proved `@kolu/surface`, [kaval-tui](pty-daemon-tui.html) proved kaval's `ptyHostSurface`, and [pulam-tui](pulam-tui.html) proved pulam's awareness surface. Decided across `debates/pr1637/` + `debates/sutra-reconsidered/`; the substrate-not-TUI call and the single-host scoping both came out of that 4-agent record.
</Callout>

## What it owns — and why it's electricity, not the server's internals renamed

Run the three [electricity tests](electricity.html) honestly:

- **① Domain-agnostic?** sutra's interface is *the terminal workflow on a host*, shared verbatim by a browser GUI and a TUI — no Solid, no canvas, no Code-tab, no xterm leaks across it. The thing that fails ① is the TUI, not this.
- **② A real volatility?** Yes, and a *different* one than the pieces already graduated. kaval owns **PTY transport**; pulam owns **awareness + fs/git observation**; `@kolu/surface-nix-host` owns the **ssh/Nix dial**. None owns kolu's **terminal *lifecycle* over a (possibly remote) connection** — create/attach/kill/sleep/wake/adopt, dialing the host's kaval + pulam, surviving a dropped link, re-adopting at boot. That orchestration is the smear the round-2 debate found — a transport-grade volatility (reconnect, adoption, dial), not a bounded algorithm.
- **③ Graduates, second consumer plugs in?** It ships as `@kolu/sutra` and **kolu-tui embeds it** ([the next note](kolu-tui.html)) — the next work on the roadmap.

<Callout kind="warn" title="The bar this must clear — or it's just a rename">
The substrate must encapsulate a *genuine* volatility — the lifecycle/dial/reconnect/adoption **orchestration** — not "kolu-server internals given a package name." Two falsifiable conditions hold it there: (1) **kaval-tui already proves the raw *PTY* transport** over ssh, so sutra's value is explicitly the *kolu orchestration* both consumers share, never the transport (that's kaval's); (2) **kolu-tui must EMBED it and dial the host itself** — if the TUI were a thin renderer of a running kolu-server, nothing graduates and it's a second UI, not a second consumer. Both are load-bearing in the phase plan.
</Callout>

**What it owns** (per connection — one host):

| Capability | What it encapsulates | Grounded in today's code |
| --- | --- | --- |
| **Sealed id-keyed dispatch** | per-terminal ops as `op(id)` that resolve the **connected host's** endpoint *internally*; the endpoint instance + resolver + `*Local*` helpers are **private**, so no op can hard-pin the *local* endpoint when the connection is remote | seals the resolver-bypass at <Cite file="packages/server/src/terminalEndpoint/resolve.ts" lines="39-53" /> + the 9 `*Local*` escape hatches at <Cite file="packages/server/src/terminalEndpoint/local.ts" lines="1143-1283" /> |
| **Dial + reconnect + adoption** | connect to the host's kaval + pulam over ssh, survive a dropped link, re-dial-re-adopt the host's surviving PTYs at boot | `adoptSurvivingSession` <Cite file="packages/server/src/terminalEndpoint/reattach.ts" lines="61-174" /> + the pure `reconcile` <Cite file="packages/server/src/reconcile.ts" lines="68-96" />; `getHostSession` backoff + durable `kaval --stdio` |
| **Host-local session / restore** | which terminals exist on this host, which are sleeping, their restore targets — the live registry for the connection | the registry `Map<PtyId, …>` at <Cite file="packages/server/src/terminal-registry.ts" lines="70" /> (bare id — no rekey) |

**What it consumes** (deps point out): `kaval` (PTY, remote over ssh), `pulam` (awareness + fs/git, dialed as one surface), `@kolu/surface-nix-host` (`getHostSession` — the ssh/Nix dial), `@kolu/surface` (the mirror). sutra depends on these; **neither UI is depended on by sutra.**

## One host at a time — the decision that deletes the composite key

The pivotal call, settled by a 4-agent debate (`debates/sutra-reconsidered/`): **a kolu surface shows ONE host's terminals at a time.** Host is **connection context** — you pick it with a dropdown (*kinda like tmux sessions*), and switching hosts tears down one sutra connection and opens another. There is **no surface that mixes local and remote tiles.**

That one decision is the biggest honest simplification in the whole plan, because it makes **host stop being a per-terminal property**:

- **No composite `TerminalKey`.** Within a single-host connection, a terminal's id is just its bare kaval `ptyId` (`z.string().uuid()`, <Cite file="packages/terminal-workspace/src/schema.ts" lines="48" />). Two hosts' PTYs never share a surface, so they can't collide — the cross-host aggregation key (R9.3's `HostTerminalKey`) simply isn't needed.
- **No `HostLocation` threaded through every op.** The host lives at the connection boundary (dialed once), not on each `create`/`attach`/`sleep` call. `resolveTerminalEndpoint(entry.meta.location)` per terminal collapses to "the connection's endpoint."
- **No cross-host registry / `forEachHost` / branded complete-host sweep.** There is one host per connection; boot adoption is that host's `reconcile`, not a sweep over a map. The round-1 "save after adopting one host of three" bug can't be *written* because there is no multi-host save.

<Callout kind="good" title="What's lost, and why it's fine — the fleet glance is pulam-web's job">
The cost is real: you can't put a local tile and a `nix@prod` tile **side by side in one canvas**, and the Dock rail shows one host at a time. But the thing that loss would hurt — *"which agent needs me, across ALL my hosts, at once"* — **is already built, in the right place**: [pulam-web](pulam-web.html) is the browser fleet board, spanning every host, sorted needs-you-first, and it does multi-host as *N single-host connections* with host identity living only at the transport layer (verified across the 4-agent debate). [pulam-tui](pulam-tui.html) is likewise single-daemon and defers the fleet view to pulam-web. So kolu is the **workbench for the selected host** (tmux-style); the **cross-host glance is pulam-web's**. Nothing kolu actually does — a terminal, its Code tab, its awareness, its restore — is cross-host; it's all *intra*-host. The composite key was paying for a requirement kolu didn't have.
</Callout>

## The API both embedders call

You **connect** sutra to a host, then drive terminals by **bare id**. The endpoint, the resolver and the local impls are package-private, so no op can reach around the connection to a hardcoded local endpoint.

```ts
// @kolu/sutra — a SINGLE-HOST terminal-workflow surface. Connect it to ONE host; everything
// below is scoped to that connection. Switching hosts = connect a fresh one.
// PRIVATE (not exported): TerminalEndpoint, resolveTerminalEndpoint, the LocalTerminalEndpoint
// instance, every *Local* helper — so "hard-pin the local endpoint" has no symbol to write.

function connectSutra(host: HostLocation): TerminalSubstrate;   // dials host's kaval + pulam

export interface TerminalSubstrate {
  // —— per-terminal ops: BARE ptyId; the connected host's endpoint resolves INTERNALLY ——
  create(opts: PtySpawnOpts): TerminalInfo;                 // spawns on the connected host
  attach(id: PtyId, signal?: AbortSignal): Promise<TerminalAttachment>;
  sendInput(id: PtyId, data: string): void;
  resize(id: PtyId, cols: number, rows: number): void;
  screenText(id: PtyId, extent?: ScreenExtent): Promise<string>;
  kill(id: PtyId): Promise<TerminalInfo | undefined>;
  sleep(id: PtyId): Promise<void>;         // was beginSleepLocal + saveSession + releaseSleptLocalPty
  wake(id: PtyId): TerminalInfo | undefined;
  discardSleeping(id: PtyId): boolean;
  restoreSleeping(record: SavedSleepingTerminal): boolean;

  // —— awareness: one framed stream per terminal (folded by the consumer) ——
  subscribeTerminal(id: PtyId): AsyncIterable<TerminalFrame>;

  // —— connection-scoped session ops (this host only — no cross-host sweep) ——
  killAll(): Promise<void>;                 // the connected host's terminals
  adoptOnConnect(): Promise<void>;          // re-dial-re-adopt this host's survivors at boot
}
```

*(`PtyId` is the bare kaval uuid. Types like `TerminalFrame` / `ScreenExtent` are illustrative — an implementer derives them from the existing `TerminalEndpoint` + awareness code; the shape that matters is the sealing + the bare id.)*

This is not new surface area — it's **today's call sites with the escape hatches removed and the host lifted to the connection.** Every op already exists; it's just reached three incompatible ways the facade collapses into one:

| Op | How it routes today | Through sutra |
| --- | --- | --- |
| `attach` · `kill` | the resolver, per-terminal — `resolveTerminalEndpoint(entry.meta.location).attach(…)` <Cite file="packages/server/src/router.ts" lines="183-186" /> | `op(id)` — the connection's endpoint, resolved once |
| `resize` · `sendInput` · `screenText` | a raw registry lookup + handle verb — `getActiveTerminal(id)?.handle.write(…)` <Cite file="packages/server/src/router.ts" lines="118-125" /> | `op(id)` — the registry lookup moves *inside* |
| `sleep` · `wake` · `discardSleeping` · `restoreSleeping` | **direct `*Local*` calls that bypass the resolver entirely** — `wakeLocalTerminal(id)` <Cite file="packages/server/src/router.ts" lines="242-253" /> | `op(id)` — the bypass becomes **unspellable** |
| `create` | `crypto.randomUUID()` then a constant-pinned `localEndpoint = resolveTerminalEndpoint(LOCAL_LOCATION)` <Cite file="packages/server/src/terminals.ts" lines="49-117" /> | `create(opts)` — spawns on the connected host, no `location` param |

<Callout kind="accent" title="The guarantee: a missed host-seam is a COMPILE ERROR, not a code-review catch">
Because `TerminalEndpoint`, `resolveTerminalEndpoint`, the `localTerminalEndpoint` instance and all nine `*Local*` helpers become **package-private**, the offending expressions have *no symbols to write*:

- `discardLocalSleeping(id)` → **`Cannot find name 'discardLocalSleeping'`**
- `localEndpoint.killAllTerminals()` → **`Cannot find name 'localEndpoint'`**
- `resolveTerminalEndpoint(LOCAL_LOCATION).sleep(id)` → **`Cannot find name 'resolveTerminalEndpoint'`**

Strictly stronger than a branded "routed" type: a brand rejects a value that still *exists*; removing the producer means there's nothing to form. The compiler does the seam-hunt that the manual lifecycle-threading prep did by hand — which is why that prep **dissolves into this one**. Single-host makes the guarantee *smaller*: sutra seals **one** endpoint (the connected host's), not a map of them.
</Callout>

**Both embedders call this identically** — connect a host, then drive by bare id:

```ts
// kolu-server — the web client's oRPC handlers (packages/server):
const sutra = connectSutra(selectedHost);   // "local" by default; the dropdown switches it
await sutra.attach(id, signal);
sutra.create(opts);

// kolu-tui — the Dock's attach action + create command (the SAME package, embedded):
const sutra = connectSutra(selectedHost);   // kolu-tui dials its own host
await sutra.attach(id, signal);             // byte-for-byte the same call
sutra.create(opts);
```

sutra doesn't know or care which embedder it's inside, or whether its host is local or an ssh remote — that ignorance is test ① passing.

## Slimming kolu-server — and the session split

The extraction is the move that makes remote terminals *small*. Today the lifecycle volatility is ~**half of kolu-server's source**, threaded by hand at every site. Lift it into sutra and the remote host is just **a different `connectSutra` argument** — a dial, not a cross-cutting edit.

| Migrates **into** `@kolu/sutra` (the volatility) | LOC | Stays in kolu-server (web/HTTP/product) | LOC |
| --- | --- | --- | --- |
| `terminalEndpoint/*` — the lifecycle engine, sleep/wake, adopt, the resolver | 1981 | `index.ts` — the Hono HTTP server, oRPC handlers, static serving, TLS | 500 |
| `ptyHost/*` — daemon supervision, dial, handshake, restart policy | 670 | `state.ts` — on-disk persisted-state + migrations | 539 |
| `terminal-registry.ts` · `terminals.ts` (lifecycle verbs) · `reconcile.ts` | 595 | `surface.ts` · `router.ts` — the surface/contract serving + oRPC glue | 838 |
| | | `iframePreviewRoute.ts`, `pwaIdentity.ts`, `tls.ts`, … web-only | rest |

That's **~3246 LOC ≈ 52%** of kolu-server's non-test `src/` (a hair less once `terminals.ts`'s client-chrome setters — netted out below — stay behind), plus its matching tests (~1491 LOC).

<Callout kind="note" title="Q1 — who owns session/restore: sutra owns host-local, kolu keeps the cross-host durable save">
The current saved session is **braided** — terminal-restore facts and web/UI chrome live together (<Cite file="packages/common/src/surface.ts" lines="313-335" /> for the authored server fields; client fields like `canvasLayout`/`rightPanel`/`intent` alongside). Unbraid it along the connection line:
- **sutra owns the host-local restore** — the connected host's live registry, sleeping records, restore targets, adoption. This is memoryless-producer + snapshot-half fold; the **two remembered facts** (recency, launch line) stay on `kolu.authored` (`terminalEndpoint/metadata.ts` splits: commit-to-registry is sutra, publish-to-kolu-surface is kolu). **kolu still owns memory.**
- **kolu keeps the durable CROSS-host session** — *which hosts* the user had open and which terminals on each, so a kolu restart re-`connectSutra`s each saved host and re-adopts. That list spans hosts, so it's kolu's, not any one sutra's. `session.ts` / `state.ts` stay in kolu.
- **Each UI keeps its own view chrome** — the web canvas layout is kolu-server's; the TUI has no canvas.

So the boundary the whole [awareness design](awareness-derive-store.html) rests on is unchanged: the producer is memoryless, kolu remembers.
</Callout>

## The extraction phase

One PR: **extract `@kolu/sutra` and seal the dispatch.** A pure **local-preserving refactor + a package move** — every terminal is on the local connection today, so every `op(id)` resolves to the same local endpoint and create/attach/kill/sleep/wake/adoption stay byte-identical. It **absorbs the old manual lifecycle-threading prep** (there is no seam-hunt — the missed seam is a compile error) and it is the foundation both later phases build on.

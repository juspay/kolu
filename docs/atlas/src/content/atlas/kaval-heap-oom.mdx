## What crashed

Two processes, one cascade. The **kaval PTY host** — the daemon that holds the live PTY file descriptors for every terminal (<Cite file="packages/kaval/src/bin.ts" label="kaval/src/bin.ts" />) — exhausted its V8 JavaScript heap and self-aborted. Losing its only PTY host, the **kolu server** then fail-fast-exited, and systemd restarted it.

| Time (EDT) | Event | Evidence |
| --- | --- | --- |
| 14:25:49 | kaval GC pinned at the ceiling: `Mark-Compact (reduce) 4083.2 → 4058.0 MB`, then `FATAL ERROR: Ineffective mark-compacts near heap limit — JavaScript heap out of memory` | kaval unit journal |
| 14:25:49 | kaval aborts: `node::OOMErrorHandler → abort → raise` | coredump stack, thread 329839 |
| 14:26:08 | Coredump captured — 732 MB compressed (`Signal 6 / ABRT`); kaval socket closes | `coredumpctl info 329839` |
| 14:26:09.04 | kolu server sees `[@kolu/surface/links/stdio] outbound write error: read ECONNRESET`, then a storm of `pty-host tap subscription failed` / `terminal.spawn failed` | `kolu.service` journal |
| 14:26:09.07 | kolu `FATAL … uncaught exception` → `Error: write ECANCELED` | `kolu.service` journal |
| 14:26:09 | systemd: `Main process exited, status=1/FAILURE` → `Failed with result 'exit-code'` → `Scheduled restart job, restart counter is at 1` | `systemd[1274]` |
| 14:26:09 | New server (`serverId 1010bc7d`) up with a fresh kaval on the same socket — **healthy** | `systemctl --user show` (`NRestarts=1`) |

<Callout kind="good" title="The kolu-side cascade is working as designed">
The server's exit is **not** a second bug. When kaval dies the pty-host stdio socket closes; a pending write rejects with `ECANCELED`, escapes as an uncaught exception, and the process exits `status=1` — exactly the repo's *fail-fast, crash-loudly* contract. systemd restarts it cleanly. **The fix belongs in kaval (stop leaking) and in detection — never in swallowing the kolu-side error**, which would mask a dead PTY host. Catching `ECANCELED` to "keep kolu alive" is a fallback, and a fallback here is a defect.
</Callout>

This was **not** a kernel OOM-kill (signal is 6/ABRT, a userspace self-abort — not 9/SIGKILL; `journalctl -k` had zero OOM lines; the unit's `MemoryMax=infinity`), **not** a deploy (`status=1/FAILURE` + scheduled restart is a crash, not a clean stop/start), and **not** disk-full (`/` at 78 %, 190 GB free). The kolu exit is causally tied to the kaval death — the `ECONNRESET`/`ECANCELED` both originate on the pty-host link and fire in the same ~30 ms the kaval socket closes.

## Root cause: live terminals accumulate, each pinning a 50 K-line mirror

**Reproduced and confirmed** by driving `createPtyHost` in isolation on a clean box (`naiveintent`). kaval keeps, *per live PTY*, an `@xterm/headless` screen mirror sized at `DEFAULT_SCROLLBACK = 50_000` lines (<Cite file="packages/common/src/config.ts" label="config.ts" />, passed at <Cite file="packages/server/src/ptyHost/index.ts" label="ptyHost/index.ts" />). The heap is **linear in live-terminal count** and **flat under everything else**:

| Driver | Heap behaviour |
| --- | --- |
| 1 terminal, unbounded `yes` output, 90 s | oscillates 14–64 MB — **bounded** (scrollback caps it) |
| attach + abort a subscription, tight loop | flat — **bounded** (abort cleanup works) |
| spawn + write + kill, 8 000× | flat — **bounded** (`teardown` is complete) |
| **spawn terminals, never kill** | **linear** — ~10 MB/terminal at 10 K scrollback; **~62 MB/terminal at the production 50 K** |

Under a 1 GB heap cap at the production 50 K scrollback, the host dies with the **exact production signature** — `FATAL ERROR: Reached heap limit Allocation failed - JavaScript heap out of memory` — at **~54 fully-scrolled terminals** (heap climbing linearly 358 → 970 MB; ~18 MB V8 heap + ~44 MB ArrayBuffer-backed cells each). A one-terminal heap snapshot is ~10 000 each of `ArrayBuffer` / `Uint32Array` / xterm `BufferLine` — the scrollback grid, nothing else.

So the **operative answer**: heap is proportional to **live terminal count** (× each terminal's scrollback fill). A single busy terminal, or terminal/subscription churn, is bounded — *activity* alone doesn't grow it. The ~4 GB ceiling is reached at a couple hundred fully-scrolled terminals (fewer in practice, summed across partially-filled ones).

Why the count grows without bound: `reconcile` **never reaps** — a surviving kaval's live PTYs are all *adopted* across every server restart (the "terminals survive a kolu update" guarantee), and a terminal is freed only when its child process exits or the user explicitly kills it (<Cite file="packages/server/src/reconcile.ts" label="reconcile.ts" />). Long-lived shells and agents — across many worktrees, over days, ratcheted up by each crash-restart — accumulate, each pinning a 50 K-line mirror. **Not a teardown bug** (teardown is clean): an unbounded, never-reaped population × a large per-terminal retainer.

*Recurrence is real — same signature on 2026-05-27, 06-13 (825 MB), 06-15 (632 MB), 06-16 (809 MB), 06-19 (732 MB), ~every 2–6 days (`coredumpctl list`). Red herrings now ruled out: a ~60-RPC `attach` burst 15 s pre-crash (a single WebSocket disconnect aborting in-flight RPCs); the bounded structures (exit tombstones ≤ 1024, per-subscriber queues ≤ 10 K, exit-waiters) — none leak.*

### Does this hurt kaval's performance? — yes, before it ever crashes

This is the part that bites day-to-day, not just at the moment of death. kaval is the hot path for **all** terminal I/O — every byte in and out of every PTY crosses its event loop. As the heap creeps toward 4 GB, V8's GC runs **more often and for longer**, ending in the "ineffective mark-compacts" the crash banner names: long, stop-the-world pauses that *are* the leak's tail. While GC holds the loop, kaval can't relay output, echo keystrokes, or deliver exit signals — so terminals feel **progressively laggier the longer the server has been up**, worst in the hours before the OOM, then snap back to crisp after the restart-induced fresh heap. So the leak has two costs: a hard crash every few days, and a soft "kolu gets sluggish over time" that a restart silently papers over. (The leak floor that `diagnostics.ts` is tuned for is ~10 MB/min.)

## Why we can't see it yet — the diagnostics gap

kolu already has heap diagnostics: set `KOLU_DIAG_DIR` and the server writes a baseline snapshot, logs subsystem sizes every 5 min, and arms `--heapsnapshot-near-heap-limit=3` so V8 dumps a snapshot *just before* an OOM (<Cite file="packages/server/src/diagnostics.ts" label="diagnostics.ts" />). The home-manager module already exposes it as `services.kolu.diagnostics.dir` (<Cite file="nix/home/module.nix" label="module.nix" />).

**But it only instruments the server (Node 22) — the process that *doesn't* crash.** kaval (Node 24, the one that *does*) is deliberately excluded:

<Callout kind="warn" title="localDriver strips the very flag we need">
`scrubNodeOptions()` removes `--heapsnapshot*`, `--inspect`, `--heap-prof`, `--cpu-prof` from `NODE_OPTIONS` **before** spawning kaval, *"so a kolu launched with diagnostics doesn't make kaval write heap snapshots too."* <Cite file="packages/server/src/ptyHost/localDriver.ts" label="localDriver.ts:98-130" /> That was the right call when the server was the suspected leaker — but it means the prod kaval has **no heap limit and no pre-OOM snapshot**. Every crash so far has thrown away the one artifact that would name the leak.
</Callout>

The leak is now named without it (the in-process repro on a clean box did the job), but the gap still matters: with kaval instrumented, prod would have shown the `terms`-count curve climbing for days, and a near-limit snapshot would confirm the same scrollback grid in the *real* workload — so the fixes below still close it.

## The fix

Two levers, because survivability forbids reaping live shells: cut the **per-terminal retained cost**, and bound the heap with a **loud tripwire**. Shaped by the design philosophy — nothing here silences the crash; limits are baked in, not knobs.

### <Pill variant="ok">1</Pill> Cut the per-terminal mirror cost — the high-leverage fix

The 50 K-line headless mirror *is* the per-terminal retainer (~62 MB full); shrinking it raises the OOM ceiling proportionally (a 5× cut ≈ 54 → ≈ 270+ fully-scrolled terminals). Today `DEFAULT_SCROLLBACK = 50_000` is *one constant* serving **both** the client's visible scrollback **and** the server-side mirror — they're conflated (<Cite file="packages/common/src/config.ts" label="config.ts" />). But the mirror only needs enough to seed a *cold-attaching* client's restore; the live client keeps its own scrollback. **Decouple them** and give the mirror a smaller bound (e.g. 5–10 K), or lower the shared default. The tradeoff is a product call that's yours to make: how many scrollback lines must survive a cold reconnect (a fresh browser with no local buffer)? That number is the mirror size.

### <Pill variant="ok">2</Pill> A deliberate kaval heap limit — a tripwire, not a cap

Give kaval an explicit `--max-old-space-size` chosen to crash **earlier and predictably**. This converts the opaque ~4 GB wall into a bounded, legible one and shrinks the blast-radius window — defence-in-depth behind (1), so an unforeseen accumulator still fails loud and fast rather than after days. **Bake it into kaval's Nix wrapper**, not an env override knob ("being able to override is never a feature").

### <Pill variant="ok">3</Pill> Instrument kaval — confirm in prod, catch regressions

Thread a diagnostics path **through to kaval** (a kaval-side `KOLU_DIAG_DIR`, `--heapsnapshot-near-heap-limit`, `--heapsnapshot-signal=SIGUSR2`) so prod shows the `terms`-count/heap curve and dumps a near-limit snapshot. The repro already named the leak; this confirms it in the *real* workload and guards against a future regression. Concretely: stop scrubbing those flags for kaval (or, cleaner, pass kaval explicit `execArgv` + diag dir via `localKavalDriver` instead of inherited `NODE_OPTIONS`).

### <Pill variant="ok">4</Pill> The home-manager surface — yes, wiring needed there too

Deployment is via the `services.kolu` home-manager module, so the knobs land there. The honest split:

- **The heap *limit* (2) is not a user option** — a baked-in tripwire in kaval's derivation, no `mkOption`, consistent with how the module bakes required values.
- **The diag *dir* (3) reuses the existing option.** `services.kolu.diagnostics.dir` already exists; the work is making that value *reach kaval* — end-to-end: module → server env → `localKavalDriver` forwards a kaval diag dir + execArgv → kaval arms the V8 hooks. The visible module change is small (the plumbing is in `localDriver.ts`); a new `mkOption` is only warranted to toggle kaval diagnostics *independently* of the server's — likely unnecessary.

<Callout kind="note" title="Why this isn't just 'flip the existing option'">
`services.kolu.diagnostics.dir` looks like it should already cover kaval — it doesn't, because of the deliberate `scrubNodeOptions` severing. The option exists; the data path to kaval is cut. That cut is the actual work item, and it's invisible from the module alone.
</Callout>

### <Pill variant="ok">5</Pill> Regression guard

Add a long-running kaval soak test asserting **heap stays proportional and bounded** — open N terminals, fill scrollback, and assert per-terminal heap matches the chosen mirror size (so a future scrollback-size or accumulation regression trips the test, not prod).

<Callout kind="note" title="A separate product decision: should idle terminals ever be reaped?">
The accumulation is *live shells that never exit and are never killed*, kept alive by the survivability guarantee. Reaping (e.g. terminals with no client, idle beyond a long threshold) would bound the count directly — but it trades against "terminals survive a kolu update," so it's a deliberate policy choice, not a code-cleanup. Out of scope for the fix above; flagged for a decision.
</Callout>

## Current state & open questions

**Prod is healthy** — `kolu.service` active since 14:26:09, `NRestarts=1`, a fresh kaval on `/run/user/1000/kaval-7692/pty-host.sock`, no crashes since. But recurrence risk is **HIGH**: with no heap bound and the per-terminal cost unchanged, expect another crash within days. The retained allocation is now **identified** (the 50 K-line per-terminal scrollback mirror × an unbounded live-terminal population). What remains open:

- **The mirror-size decision** (fix 1) — how many scrollback lines must survive a cold reconnect. A product call.
- **Whether to reap idle terminals at all** — the survivability-vs-bounded-count tradeoff above.
- **The exact JS path of the kolu-side `write ECANCELED`** — inferred (a floating promise on the pty-host stdio link), not pinned to a verified line. Low priority: the kolu exit is correct behaviour regardless.

<Callout kind="warn" title="Headline">
justci and mini-ci are not the same program at two maturities — they are **two categories of program, separated by what crosses the ssh wire.** justci ships an opaque **batch artifact** (a git bundle in, scraped `.log` files out) and reconstructs "liveness" out-of-band through a *separate* socket client. mini-ci ships a **live typed reactive surface** in-band over oRPC stdio, so the runner stops being a translator that emits a run and becomes a *service that owns the DAG as state* — one you attach, detach, and reconnect to. That single inversion **replaces the *watch-a-run* half of CI outright** and is generative: a browser dashboard, **an MCP server for agent CLIs**, multi-platform fan-**in**, typed rerun, and self-healing reconnect fall out as *composition* of packages kolu already ships. But it **still owes the *trigger-and-gate* half** that is justci's actual reason to exist — forge-event ingestion, per-commit run identity, commit-status output, and a trust model for a now-remotely-mutable service. "Replaces justci **and** does more" is true on the observation axis and a concrete, costed backlog on the gating axis. This note owns both.
</Callout>

justci's own `dump-yaml` is the confession of category: it proves the pipeline *assembly* — a project's `just` DAG (`[metadata("ci")]` root) into a single `process-compose` document — is a **pure function**, fed a SHA placeholder so it runs offline outside a git checkout. The run *is* the process; there is no serve-only mode. ssh is then a **dumb, one-way, per-node pipe**: a `git bundle` (plus a `nix-store` export of the runner derivation) goes *out* to set the node up; each recipe node re-`ssh`'s to `just --no-deps`; and only an **exit code** comes back. Per-recipe stdout is redirected to `.ci/<sha>/<plat>/<recipe>.log` **files**, and the progress feed carries the on-disk *path*, never the bytes — a later consumer reads them off disk.

"Liveness," then, is something justci reconstructs *out of band*. `status` / `monitor` / `logs` are thin passthroughs to a **separate, baked `process-compose` client** dialed at `.ci/pc.sock`; justci's own guard `die`s before it even reaches the socket when no run is in progress. The introspection is a side-channel that only exists *while a batch lives*.

<Callout kind="accent">
mini-ci moves the **one** load-bearing axis: ssh carries a typed, schema-validated, snapshot-then-delta surface (oRPC stdio). The runner's `nodes` Cell and per-node `nodeLog` Stream are **in-band, over the same surface a client attaches to** — observable whether or not a run is mid-flight, with no second binary and no `pc.sock`. Everything "more" in this note is downstream of that one change.
</Callout>

The honest floor: mini-ci is a *proven substrate*, not a product. A long-lived **runner** owns a task DAG — each node a child process with a per-node log buffer — and serves it as a [`@kolu/surface`](https://kolu.dev/blog/surface-framework/) over stdio. The entire tool is **three primitives** (`src/common/surface.ts`):

| Primitive | Path | What it carries |
| --- | --- | --- |
| **Cell** | `surface.nodes.get({})` | The whole pipeline's node-state — snapshot then deltas. ↔ a list view. |
| **Stream** | `surface.nodeLog.get({ id })` | One node's output: a buffered `snapshot` frame, then `append` deltas (a discriminated union, so a *late* subscriber replays a finished node from the top). ↔ attach. |
| **Procedure** | `surface.node.rerun({ id })` | The *only* mutation: reset the node + its transitive dependents to pending and reschedule. ↔ input. |

It reaches the host **the drishti way** via [`@kolu/surface-nix-host`](../surface-app.html)'s `HostSession`: `nix copy` the prebuilt runner `.drv` to the host, `nix-store --realise` it, then `ssh host mini-ci-runner --stdio`; `localhost` skips the copy. `HostSession` owns the ref-count, reconnect, and a `copying → connecting → connected` state cell — *the same session for local and remote, only the transport differs.*

`mini-ci.test.ts` drives the **real** runner surface through the **real** stdio codec (`createLoopbackPair → serveOverStdio → stdioLink`, byte-identical to the ssh path; the runner spawns real child processes — nothing is mocked), so a green run is genuine evidence. It banks:

- **Cell snapshot-then-delta** — a late subscriber's first frame is the current state.
- **Race-free topo order** — across every captured frame, a node runs only once its dependency is `ok`.
- **Per-node log replay** — a late subscriber to a *finished* node gets its buffered output as the first `snapshot`.
- **Rerun re-runs the closure** — `node.rerun` resets the node + dependents and they settle `ok` again.
- **No false greens** — a failed dependency `skip`s its dependents.

<Callout kind="warn" title="Deliberately not justci — today">
TUI-only (no browser client tree, no `surface-app`); a **singular** `implementSurface` over **one** `HostSession`; a **read-only** `/nix/store` closure, so the default pipeline is **typecheck-only by construction** (`tsc --noEmit` over a real dependency closure); a hand-written `ci.json`; and **client-side** detach/reattach only — runner-side state dies with the runner or its ssh pipe. These are the gaps the rest of this note prices, not hidden costs.
</Callout>

## The half it doesn't replace — trigger, identity, trust

A CI tool is defined at least as much by its *front* half as its execution engine: **what event starts a run, against which immutable commit, and how a verdict reaches a developer who isn't watching.** justci's reason to exist *is* that loop — push → forge event → run pinned to a worktree SHA → commit-status the forge can gate a PR on. mini-ci's verified strengths are all about a developer who is **attached and watching.** The reframe must not overreach into the half it doesn't touch.

- **Trigger.** mini-ci has no webhook / poll / push ingestion and no PR-event story. A live service you *attach* to is orthogonal to the event that *kicks* a run. Liveness answers *how do I watch a run*, not *how does a run begin*.
- **Run identity & history.** justci is per-SHA by design (`.ci/<sha>/…`, an immutable HEAD pin per commit — an *append-only* ledger of runs over commits). mini-ci owns **one** live, mutable DAG; `node.rerun` overwrites the prior result *in place*. There is no model for *which commit this run is* or *where prior runs went* — a missing **data model**, deeper than durability.
- **Trust.** `node.rerun` is remote code execution by construction, over a long-lived, attachable surface. justci's batch model has a bounded attack surface (ssh in, run a sealed batch, ssh out). mini-ci has zero authn/authz and no read-observer-vs-mutator split — fine for a TUI over your own ssh, load-bearing the instant a multi-client dashboard is claimed.
- **Async delivery & concurrent mutation.** justci's deliverable is a status *badge the forge sees with nothing attached*. mini-ci's story is all **pull** — attach and witness. And two clients both calling `node.rerun`, or rerunning a *running* node, has no defined serialization or run ownership. Multi-client is an affordance with a **coordination cost**, not pure upside.

<Callout kind="warn" title="So is it a replacement?">
It **replaces the watch-a-run half outright** and **does genuinely more** there. On the trigger-and-gate half it is a *plan*, not a replacement — and for canonical "run once on a commit, post a verdict, walk away," justci's strict batch (dirty-tree refusal, immutable worktree SHA, fire-and-forget status) may simply be the *right* shape. The live service **enables a different, additional class** of workflow (interactive, agent-driven, reattachable); it is not strictly superior. The novelty is real **against justci specifically** — a small Haskell batch tool — not against the hosted mainstream (GitHub Actions, Buildkite, Dagger) that already streams logs and reattaches. The durable edge is the *substrate*: a typed, reconnecting, composable surface over **your own ssh**, no hosted runner.
</Callout>

## The reckoning — parity to reach, and what only a live service has

### Reaching parity — the honest debt

Capability mini-ci must *grow* to stand in for justci. <Pill variant="warn">harder</Pill> = net-new design + code on a proven substrate; <Pill variant="todo">open</Pill> = unbuilt on either side. "The framework *can* do it" is **not** "mini-ci *does* it."

| justci capability | surface-native path | Status |
| --- | --- | --- |
| Trigger: push → forge event → run on a pinned SHA | No analog. Needs ingestion + commit-pinning at trigger time. The largest unscoped pillar. | <Pill variant="warn">harder</Pill> |
| Per-SHA run identity, history, append-only ledger | A run-identity data model; today one mutable DAG, `rerun` overwrites in place. | <Pill variant="warn">harder</Pill> |
| GitHub commit-statuses `<recipe>@<platform>` + `protect` | A terminal-transition side effect reading the `nodes` Cell + a branch-protection writer. The surface makes state *observable*, not forge integration *free*. | <Pill variant="warn">harder</Pill> |
| Async verdict delivery (a badge seen with nothing attached) | Net-new notify / webhook-out / badge axis; a live dashboard is additive, not a substitute. | <Pill variant="warn">harder</Pill> |
| Bounded trust model | authn/authz + a read-observer vs. mutator split on a remotely-mutable surface. | <Pill variant="warn">harder</Pill> |
| **Real builds** (write-heavy: `vite build`, `nix build`) | A writable scratch workspace (`--workspace` / `MINI_CI_WORKSPACE`) or nix-build-as-task; the read-only `/nix/store` closure is typecheck-only by construction. **The load-bearing execution gap.** Surface contract unchanged — builds are just more nodes. | <Pill variant="warn">harder</Pill> |
| `just`-DAG ingestion + `dump-yaml` / `graph` | A `just → PipelineSpec` translator; mini-ci reads a hand-written `ci.json` today. | <Pill variant="warn">harder</Pill> |
| Strict-by-default hygiene (dirty-tree refuse, HEAD pin, per-sha logs) | Net-new for a gating tool; mini-ci runs whatever cwd it's handed. | <Pill variant="warn">harder</Pill> |
| Durability across observer/process churn | Worse today: runner-side state dies with the pipe. Server-restart survival is the daemon plan's job. | <Pill variant="warn">harder</Pill> |
| Caching / artifact passing between nodes | nix-build-as-task inherits Nix's cache; otherwise net-new. (justci has roadmap gaps here too.) | <Pill variant="todo">open</Pill> |
| Per-recipe `[linux]`/`[macos]` filter | A per-node platform predicate in `PipelineSpec`. justci's one admitted README roadmap item (kolu's `ci/mod.just` notes it: a `[linux]`-only step is still scheduled on macOS and fails today). | <Pill variant="todo">open</Pill> |

And the same three primitives, **to an agent.** Because the runner already serves the surface over a byte pipe (`serveOverStdio`), an **MCP server is just another serve target** — a thin oRPC→MCP adapter (no `@orpc/mcp` package exists; it's ~one file over the official MCP SDK) that maps `node.rerun` to an MCP *tool*, the `nodes` Cell and `nodeLog` Stream to subscribable *resources* — with `get_nodes` / `tail_log` tool mirrors, since agent CLIs surface tools more reliably than resource subscriptions — and adds a `wait_for_settle` verdict tool. Declare it in any agent CLI over stdio, the same transport the TUI already rides:

```jsonc
// .mcp.json (proposed) — Claude Code; Codex, opencode & Gemini CLI take the same shape
{ "mcpServers": { "mini-ci": {
  "type": "stdio", "command": "npx",
  "args": ["-y", "@kolu/mini-ci-mcp", "--host", "srid1"] } } }
```

| mini-ci surface | MCP primitive | |
| --- | --- | --- |
| `nodes` Cell (snapshot-then-delta) | `ci://pipeline` **resource** (subscribe → `updated`) + a `get_nodes()` tool mirror | A Cell *is* read-current-then-get-notified — a subscribable resource's exact twin. |
| `nodeLog` Stream (`{id}`) | `node://{id}/log` **resource template** + a `tail_log({id})` cursor tool | The `{id}`-parameterised Stream maps 1:1 to a URI template; the `MAX_LOG_CHARS` clamp is a natural bounded tail. |
| `node.rerun` Procedure | `rerun_node({id})` **tool** | oRPC procedures → MCP tools 1:1 (Zod → JSON Schema). Where authz must live — this is RCE. |
| *(derived for agents)* | `wait_for_settle()` → verdict, `cancel()` **tools** | Collapse the frame-by-frame watch into one `await` — justci's end-of-run exit code, on the *live* DAG, on demand. |

This is the **agent-driven CI** the live shape makes natural — and the one thing justci reaches for but its model forecloses: <Issue n={22} repo="juspay/justci" label="MCP server for ci, done properly" /> is justci's *own* design note for an agent-controlled MCP mode; it **built it and reverted it** (<PrLink pr={18} repo="juspay/justci" label="reverted in the same PR" />) because, in a batch translator, *launching the server auto-ran every recipe* — `process-compose` has no serve-only / `--no-start` mode, and its MCP surface is "request/response only, no streaming." A runner that owns the DAG as *idle state* has that separation by construction, and `nodeLog`'s snapshot-then-delta *is* the streaming event source #22 can't get. *(Proposed, not built — the MCP face is a designed third serve target on the same router; and `rerun_node` is RCE, so it must land behind the Phase 1 authz boundary.)*

## Graduating to a standalone repo

The delivery vehicle is de-risked by precedent. remote-process-monitor — a full agent/client/common/server browser tree inside kolu — graduated into [srid/drishti](https://github.com/srid/drishti) by adopting `@kolu/surface` + `surface-nix-host` first, then `surface-app` later (<PrLink pr={47} repo="srid/drishti" label="freshness by composition" />). The `@kolu/*` packages are `npins`-pinned by the Nix callers and hydrated by `cp -rL` (not symlinks, for TypeScript's transitive imports); drishti's `KOLU_AGENT_DRV` is the exact analog of mini-ci's `MINI_CI_RUNNER_DRV`. mini-ci's own README already names the path: *"mini-ci could graduate to its own repo the way remote-process-monitor became drishti."*

**License is determined, not chosen.** `surface` and `surface-nix-host` are MIT, but `surface-app` (needed for the PWA dashboard) is **AGPL-3.0-or-later** — already mini-ci's `package.json` license, so adoption forces nothing new.

**It clears the [electricity](../electricity.html) bar.** The ①②③ test — domain-agnostic / hides hard volatility / graduates with a different consumer — is already met by the wire/ssh stack: `electricity.mdx` lists `@kolu/surface`, `surface-nix-host`, and `surface-app` as electricities with **drishti** as the proof-of-③ second consumer. What graduates here is the **product** (mini-ci → its own repo), *riding receptacles that already graduated.* The two-way street: reusable shell concerns (strict hygiene, the trigger model, run identity) extracted from the new repo harden kolu's libraries back.

**Naming** <Pill variant="todo">open</Pill> — srid's *product* line follows a single Sanskrit perception-word (drishti = दृष्टि, "sight"), not his descriptive-English library names (`omnix`, `nixci`). The thesis is *witnessing a live build*, so:

- **sakshi** (साक्षी, "witness / the one who observes") — strongest fit; "sakshi watched the build" reads naturally; parallel in shape to drishti.
- **prahari** (प्रहरी, "sentinel / one who keeps watch") — matches the long-lived-runner / ephemeral-clients model precisely.
- **netra** (नेत्र, "eye") — shortest, a direct sibling to drishti's "sight."

Lead with **sakshi**; the choice is srid's. (A witness-name slightly undersells the eventual *gatekeeper* scope once the trigger half lands — acceptable, since live witnessing is the differentiator.)

**The Atlas index *is* the graph.** Instead of a categorical tree plus a separate
graph page, there is [one entry point](./index.html): a force-directed graph of
every note, with a **Maps of Content** strip below it. The graph answers *"what's
connected to what,"* the hubs answer *"where do I start,"* and together they reach
every note — there is no second index to keep in sync.

> Open the [Atlas](./index.html). Hover a node to trace its neighborhood, click to
> open the note, drag/scroll to move around.

## Kinds are hubs, not a separate skeleton

The old index had a hardcoded skeleton — one section per `kind`. The graph dissolves
that: each of the four kinds (`bug` · `feature` · `analysis` · `reference`) is a
**synthetic hub node**, and every note carries an implicit edge to its kind. Two
things fall out for free:

- **Nothing is unfiled and nothing is an orphan.** Every note is connected — to its
  kind hub at minimum — so the graph is one piece and *every* node is reachable by
  traversal, not just the well-linked ones.
- **The Maps of Content are complete.** A kind hub's cluster is exactly the notes of
  that kind, so the four kind cards together name every note. That's what lets the
  page be browsed — and every node reached — with **no JavaScript at all**.

High-degree *topical* notes (`electricity`, `solid-fileview`, `pty-daemon`, …)
surface as hubs the same way — by degree — so a cross-cutting entry point is
discovered, never declared.

## It reuses the links the Atlas already has

No new frontmatter. The real edges are exactly the two the Atlas already authors —
`parents` and same-directory `./slug.html` prose links — derived by the very same
pass that powers the dead-link gate and the "Referenced by" backlinks. The kind
edges are synthetic, added only in the graph layout. A node's **degree** sets its
size; its **inbound** count ranks it as a topical hub.

## How the page stays a self-contained file

Each Atlas page is a committed, self-contained HTML file that previews in kolu's
Code tab with no dev server, and the build is a pure function of source (the
`ci::atlas-sync` gate rebuilds under a scrambled timezone/locale and fails on any
drift). The graph honors both:

- **Layout is computed at build time** with [`d3-force`](https://github.com/d3/d3-force)
  and **baked** into a static inline `<svg>` as rounded coordinates. Nothing lays
  out at runtime.
- **Determinism is engineered in:** nodes are sorted by id and edges by pair-key
  before the simulation, positions are seeded by that stable index, a fixed number
  of ticks runs synchronously, and every coordinate is rounded. d3-force v3 carries
  no `Math.random`, so the baked geometry is byte-identical on every rebuild.
- **The only client JS is one `is:inline` script** for interaction (zoom/pan,
  hover-highlight, click-to-open) — so no `_astro` bundle is emitted and the page
  stays one file. Nodes are real `<a>` links and the Maps-of-Content cards are plain
  linked lists, so the page **reads and navigates with JavaScript off**.

<Callout kind="accent" title="Why one view instead of two">
A tree and a graph over the same notes are two things to keep in sync, and the tree
only ever showed the skeleton the graph now expresses as structure. Folding them
into one entry point means a single data source — the link graph — drives the
picture *and* the browsable index, so they can never disagree, and "navigate to any
note" is guaranteed by construction.
</Callout>

<Callout kind="good" title="What counts as electricity">
Löwy's receptacle — hide the volatility behind a socket you plug into. Three tests at once: ① **domain-agnostic** — no terminal/canvas/git/xterm leaks into the interface; ② **hides a hard volatility** — transport, reconnect, GPU-context loss, persistence — not a 3-line `===`; ③ **graduates** — ships as `@kolu/*` and a *different* app plugs in. The proof of ③ is real, not aspirational. [`@kolu/surface`](https://kolu.dev/blog/surface-framework/) is **kolu's own** client↔server transport first — and it has since picked up a **second consumer** in [srid/drishti](https://github.com/srid/drishti) (a fleet process-monitor; browser ↔ Bun ↔ ssh-agent) that vendors it from this repo and declares its own per-host schema with none of kolu's domain. The second consumer isn't the *point* of the extraction — it's the *proof* it was real electricity: surface never noticed the domain changed.
</Callout>

<Callout kind="danger" title="How a session mis-identifies electricity — three traps, learned the hard way">
The live example: asked to add back/forward to the Code-tab preview, a session proposed extracting a **history stack**. Wrong three ways, each caught only by re-asking the test above — and the [solid-browser](solid-browser) plan is the corrected end-to-end.

- **Naming an organ, not the concept.** A history stack is a *transformer* — a working part *inside* a larger thing. The electricity was the **browser** (location + history + link-nav + render-host); history is one chamber of it. Test: *is the name self-describing standalone?* "history" → history **of what?**; "browser" → obvious. If you must append "…of X" to explain the name, you've named a part, not the receptacle.
- **A module in `client/src/` is not electricity.** Location *is* structure (Hickey): a generic-looking helper living inside the app is **part of the app**, however agnostic the code reads. Electricity is its **own package**, the dependency arrow pointing *out* — `client → @kolu/x`, never back. (`createSharedRoot` sits in `client/src` marked "publish?" — a leaf *awaiting* graduation, **not** a precedent for calling in-app modules electricity.)
- **Agnostic ≠ electricity — apply test ② honestly.** A clean, reusable, agnostic primitive that hides only a *bounded algorithm* (a stack, a debounce, an `===`) is a **leaf** (`nonempty`/`html-escape`-tier), not a receptacle. "I found a tidy generic module" is the feeling that *precedes* the mistake. Ask what **hard** volatility it hides — transport, GPU-context loss, sandboxing, reconnect — before reaching for `@kolu/`.
</Callout>

> "Decompose based on volatility." Functional decomposition splits a system by what it *does*; volatility-based decomposition isolates each area likely to *change* behind a stable interface — the way a body keeps blood-pressure, pulse, and salinity volatility "encapsulated behind the service called the heart." An *electricity*, here, is one such volatility pulled out into its own package. — Juval Löwy, [Volatility-Based Decomposition](https://www.informit.com/articles/article.aspx?p=2995357&seqNum=2) (excerpt from *Righting Software*).

## Four transports hold the framework up

Strip kolu of its domain and four sockets remain. Each is a *transport* — it moves one kind of thing across the client↔server↔host gap and stays blind to what flows through it. They are why a *different* app ([drishti](https://github.com/srid/drishti)) can stand on kolu's spine at all.

Three carry a *payload* over a live link:

- **State** — **@kolu/surface**. `Cell`/`Collection`/`Stream`/`Event` over oRPC, reconnect, fine-grained reconcile. Doesn't know a terminal tile from a process-monitor row.
- **Process** — **@kolu/pty-host**. One `PtyHost` owns N PTYs and fans VT-derived taps to N consumers; the `createPtyHost` primitive knows nothing of git, PRs, agents, the file tree, or shell-env prep — it spawns a ready `shell`/`args`/`env`. (The package's optional `servePtyHost` serving layer *does* prepare the shell env, via `kolu-pty`'s `prepareShellInit`/`cleanEnv`/`koluIdentityEnv` — that's the kolu-bound half built on top of the agnostic primitive.)
- **Agent identity** — **anyagent**. `AgentProvider<Session, Info>` answers *"which coding agent is in this terminal, and what is it doing?"* with no `claude`/`codex` token anywhere in the contract.

The fourth moves the *server itself*:

- **Host / link** — **@kolu/surface-nix-host**. `HostSession` — `nix copy` a closure to a remote host, realise it, and run a surface `--stdio` link over ssh. It encapsulates a volatility the other three don't touch: *where the server runs* and *whether the host is reachable and provisioned* (connection lifecycle, reconnect, the nix realise). The other transports ride **on top of** it for the remote case; it is the substrate that turns "client↔server" into "client↔ssh↔remote-server" without any payload noticing.

The tell that these are transports and not features: each names a *verb on an opaque payload* — carry state, own a pty, resolve an agent, reach a host — and each already has, or is one schema away from, a consumer that shares none of kolu's domain. surface-nix-host is the most-proven of the four — both in-repo `surface/example/*` apps **and** drishti already stand on it (kolu's own remote backend is still the *future* `RemoteTerminalBackend`, R-2; the kolu server does not yet import `@kolu/surface-nix-host`). The two newest (pty-host, anyagent) were hiding in plain sight: pty-host lived one rung under the terminal subsystem until <PrLink pr={1023} /> pulled it out; anyagent's `AgentProvider` contract was unified in <PrLink pr={603} /> and has been quietly load-bearing ever since. Everything else in the table is either a *UI-adapter* receptacle (xterm, browser, markdown, pwa-install) or a *leaf*.

A note on names — the scope prefix **is** the graduation marker. `@kolu/*` is a **published-shape** package: npm-shaped, the dependency arrow points out, ready for a foreign consumer. `kolu-*` is an **in-tree** workspace package: extracted, but domain-bound or not yet published-shape (`kolu-server`, `kolu-transcript-core`). A few packages drop the scope entirely — `nonempty`, `memorable-names`, and tellingly **`anyagent`**. anyagent shedding even the *kolu* name is the strongest signal it sits at framework level and not app level: it's the receptacle for *coding agents*, a volatility the whole industry shares, not one kolu owns.

<Callout kind="good" title="anyagent — the agent-identity receptacle (the outside-the-box one)">
The non-obvious framework-level electricity, found the way surface was: by asking *what would a different app on this spine still need, with kolu's domain stripped out?* The answer here is **"tell me which coding agent is running, and what it's doing"** — and the volatility behind it is enormous and industry-wide. The set of agents is not fixed and never has been: the package began as `integrations/common`, was renamed in <PrLink pr={544} />, and the contract was unified into one `AgentProvider` in <PrLink pr={603} />; codex and opencode detection landed after (<PrLink pr={658} />, <PrLink pr={677} />). Today **three** agents are fully wired (claude-code / codex / opencode) and **four** more are name-detected (aider / goose / gemini / cursor-agent) — the literal "providers change" volatility Löwy uses to justify a receptacle in the first place.

It passes ① and ② honestly; ③ is still proof-pending. ① **Domain-agnostic** — `AgentTerminalState` carries `foregroundPid`, `cwd`, a lazy `readForegroundBasename()`, and an OSC-633 preexec hint, and *nothing else*; <Cite file="packages/integrations/anyagent/src/agent-provider.ts" lines="27-46" /> has no `claude`/`codex` token. The contract is generic over `<Session, Info extends AgentInfoShape>` <Cite file="packages/integrations/anyagent/src/agent-provider.ts" lines="77-95" /> — integrations *parameterize* it (`AgentProvider<SessionFile, ClaudeCodeInfo>`), they don't widen it. ② **Hides a hard volatility** — session-*identity resolution* differs wildly per agent (claude-code reads a foreground PID → session file; codex/opencode match a process name → a directory-scoped lookup over a SQLite-WAL tail), and the *state-derivation transport* differs too (JSONL `fs.watch` + SDK buffers vs WAL writes). That is genuine transport-grade volatility, not a bounded `===`. ③ **Graduates** — *not yet proven by the note's own bar.* The dependency arrow does point strictly out: `kolu-claude-code` → `anyagent`, never back (`grep` finds zero reverse imports), and three independent integration packages already plug into the unchanged socket — strong *internal* evidence the contract is load-bearing. But ③ asks for a published `@kolu/*` shape and a *different* app plugging in, and anyagent is still `private: true` with only in-tree kolu consumers (`kolu-claude-code` / `kolu-codex` / `kolu-opencode`, plus `kolu-client` / `kolu-server` / `kolu-common`). The three integrations are the same-repo, same-domain workspace — they prove ② (the volatility is real and the socket absorbs it), not ③ (a foreign consumer on a published package). The graduation proof surface gets from drishti, anyagent is one packaging step away from earning.

The tell it sits at framework level and not app level: it sheds even the *kolu* name. `surface` is `@kolu/surface`; the agent receptacle is just `anyagent`, because coding-agent identity is a volatility the whole industry carries, not one kolu owns. The only thing keeping it off the `@kolu/*` published-shape tier is packaging, not soundness.
</Callout>

<Callout kind="good" title="New this pass — two electricities the tracker was missing">

- **@kolu/sandbox-bridge** — the headline. It *resolves* the artifact-sdk re-eval below: "talk to code inside an opaque-origin sandboxed frame" **is** a transport-grade volatility (identity-only trust boundary, N interchangeable clients), and it's already a clean module inside artifact-sdk. - **@kolu/solid-clipboard** — the second `solid-pwa-install`-shaped electricity: cross-engine clipboard fragmentation (secure-context Clipboard API vs `execCommand` fallback vs OSC-52), hidden behind one clean socket, awaiting graduation out of `client/src`.
- **Not** an electricity (recorded so it isn't re-proposed): the **notification/badge** channel (`setAppBadge` + SW `showNotification`) is a real, hard cross-engine volatility but it is *sprayed* across `useTerminalAlerts` + `useActivityAlerts` with no single owner — a Layer-0 *missing seam*, not a receptacle yet. See the considered table.

The other four probes (service-worker → already inside `surface-app`; multi-tab → not a kolu concern; visibility/wake-lock → owned by the forthcoming `solid-xterm`; drag-drop upload → domain-coupled) came back negative — recorded below.
</Callout>

quote extract/re-find/highlight; pure and agnostic, speaks only
`Range`/`Document`/`ShadowRoot`), and (2) the **sandboxed-iframe parent↔frame
bridge** (`client/bridge.ts` + `iframe/` — opaque-origin postMessage with
`event.source` identity as the only trust boundary). The bridge is the *harder*
volatility, and it has quietly accreted clients: comments (selection →
`SelectMsg` → highlights), then **in-frame navigation** (`observeIframeNavigation`),
then **back/forward input** (`observeIframeHistory`, <PrLink pr={1191} />). The
lens-debate in <PrLink pr={1191} /> already reframed the package's *stated*
identity to "the in-iframe sandbox bridge, with comments as one client" — but
that was a docs fix, not a decomposition.

**Resolved:** it is one sound receptacle — a transport-grade volatility with
comments / navigation / input as interchangeable clients — and it should split
from the anchoring core into **@kolu/sandbox-bridge** (table row above). The
deciding evidence: the bridge interface speaks only postMessage + `event.source`
identity + the W3C `Locator`/`SelectMsg`/`ReadyMsg`/`HistoryMsg` wire types (no
comment/artifact token leaks into the transport), `bridge.ts:59` already treats
it as a "network-grade boundary" that tolerates unknown message types, and the
three clients accreted on the *same* `event.source` barrier without depending on
each other — the "set of clients is not fixed" signature. The split is mechanical
(the bridge imports zero anchoring-core functions; the core's `extractQuote` /
`findQuote` / `applyHighlights` stay put). The smell that flagged it — the
back/forward client landing here by *physical necessity* — was real, but the
resolution is "extract the transport," not "leave it bundled." ③ is still
proof-pending (in-tree consumers only; no foreign app yet).
</Callout>

Plus external electricity correctly leaned on instead of hand-rolled: **Corvu** (dialog/drawer/focus-trap/scroll-lock), **@solid-primitives/\*** (resize-observer, media). Origin: a 41-agent Hickey/Lowy audit (16 candidates) → its "ship now" tier landed in <PrLink pr={1089} />; a framework-scale re-test + adversarial skeptic surfaced the missed `solid-xterm`.

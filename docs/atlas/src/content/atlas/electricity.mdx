<Callout kind="good" title="What counts as electricity">
L√∂wy's receptacle ‚Äî hide the volatility behind a socket you plug into. Three tests at once: ‚ëÝ **domain-agnostic** ‚Äî no terminal/canvas/git/xterm leaks into the interface; ‚ë° **encapsulates a real volatility** ‚Äî an axis the system actually *varies* along (transport, reconnect, GPU-context loss, persistence), not a stable 3-line `===`; ‚ë¢ **graduates** ‚Äî ships as `@kolu/*` and a *different* app plugs in. The proof of ‚ë¢ is real, not aspirational. [`@kolu/surface`](https://kolu.dev/blog/surface-framework/) is **kolu's own** client‚Üîserver transport first ‚Äî and it has since picked up a **second consumer** in [srid/drishti](https://github.com/srid/drishti) (a fleet process-monitor; browser ‚Üî Bun ‚Üî ssh-agent) that vendors it from this repo and declares its own per-host schema with none of kolu's domain. The second consumer isn't the *point* of the extraction ‚Äî it's the *proof* it was real electricity: surface never noticed the domain changed.
</Callout>

<Callout kind="danger" title="How a session mis-identifies electricity ‚Äî three traps, learned the hard way">
The live example: asked to add back/forward to the Code-tab preview, a session proposed extracting a **history stack**. Wrong three ways, each caught only by re-asking the test above ‚Äî and the [solid-browser](solid-browser) plan is the corrected end-to-end.

- **Naming an organ, not the concept.** A history stack is a *transformer* ‚Äî a working part *inside* a larger thing. The electricity was the **browser** (location + history + link-nav + render-host); history is one chamber of it. Test: *is the name self-describing standalone?* "history" ‚Üí history **of what?**; "browser" ‚Üí obvious. If you must append "‚Ä¶of X" to explain the name, you've named a part, not the receptacle.
- **A module in `client/src/` is not electricity.** Location *is* structure (Hickey): a generic-looking helper living inside the app is **part of the app**, however agnostic the code reads. Electricity is its **own package**, the dependency arrow pointing *out* ‚Äî `client ‚Üí @kolu/x`, never back. (`createSharedRoot` sits in `client/src` marked "publish?" ‚Äî a leaf *awaiting* graduation, **not** a precedent for calling in-app modules electricity.)
- **Agnostic ‚âÝ electricity ‚Äî apply test ‚ë° honestly.** A clean, reusable, agnostic primitive that hides only a *bounded algorithm* (a stack, a debounce, an `===`) is a **leaf** (`nonempty`/`html-escape`-tier), not a receptacle. "I found a tidy generic module" is the feeling that *precedes* the mistake. Ask what **volatility** it actually encapsulates ‚Äî what axis of *change* (transport, GPU-context loss, sandboxing, reconnect) ‚Äî before reaching for `@kolu/`.
</Callout>

> **"Decompose based on volatility"** ‚Äî L√∂wy's directive: encapsulate each area *likely to change* behind a stable interface, rather than split a system by what it *does*. "Electricity" is *his own* example, and the source of our term: *"the electricity that powers a house ‚Ä¶ is highly volatile: power can be AC or DC; 110 volts or 220 volts ‚Ä¶ produced by solar panels ‚Ä¶ a generator ‚Ä¶ or plain grid connectivity ‚Ä¶ All that volatility is encapsulated behind a receptacle ‚Ä¶ all the user sees is an opaque receptacle."* The body does the same ‚Äî blood-pressure, salinity, and pulse volatility *"encapsulated behind the service called the heart."* An *electricity*, in kolu, is one such volatility lifted into its own `@kolu/*` package; a *receptacle* is the socket a consumer plugs into. ‚Äî Juval L√∂wy, [*Software System Decomposition*](https://www.informit.com/articles/article.aspx?p=2995357&seqNum=2), ch. 2 of *Righting Software*.

<Callout kind="good" title="surface-app's stale-tab gate ‚Äî what graduates, what stays a socket the consumer plugs in">
The restart axis already lived in `createServerLifecycle` (`restartCloseCode` + the `transport:"open"\|"closed"` discriminant), so <PrLink pr={1231} /> only had to graduate the **shared** pieces and leave the **per-socket** ones behind. The split is the whole test ‚ëÝ at work ‚Äî none of the graduated parts knows a terminal from an ssh-agent.

**Graduates** (the second consumer shares the *identical* partysocket + oRPC stack, so the volatility is genuinely common): the `pid`/`4001` constants and a runtime-free `rejectStaleProcess(claimedPid, liveId): boolean`; `serverIdentity()` **returning** its minted `processId` so the gate compares against the SAME id `identity.info` reports (drishti otherwise mints a *second* `randomUUID()` and the gate never matches); and ‚Äî the F1 reversal of the prior "keep it per-consumer" ‚Äî `retireSocket`, whose two reverse-engineered side-effects (`close()` flips partysocket's `_shouldReconnect`; a **throwing** `send` makes oRPC's `ClientPeer` *reject* pending instead of letting `maxEnqueuedMessages:Infinity` grow) bind to internals drishti has byte-for-byte. It takes a structural `{ close; send }`, not a `PartySocket`, so it never drags the concrete type across the boundary.

**Stays the consumer's socket to plug in:** the socket *builder* (drishti's `makeSocket` cold-start deadlines, its `wsUrlFor` query) and the per-runtime `pid` read off the request (kolu's Node `IncomingMessage`, drishti's Bun Fetch-`URL`) ‚Äî surface-app emits the id, never the URL. The client's last-known id stays a per-consumer mutable on purpose: the lifecycle's `serverProcessId()` deliberately reports `undefined` on a stale close (the only id on hand is the *dead* process), but the `pid` echo must keep re-presenting that dead id to get re-rejected ‚Äî two different facts, not one projected twice, so the URL thunk owns its own `lastServerProcessId`.

**Then it graduated too (<PrLink pr={1234} />, [surface-connection](./surface-connection.html)).** Reading drishti's *actual* post-#1231 source ‚Äî not assuming it ‚Äî overturned the "stays the consumer's socket" line above: both consumers hand-roll the *identical* partysocket construction and `pid`-echo threading (drishti's "Bun Fetch-`URL`" runtime is in fact the same `ws` stack as kolu). So the socket builder and the echo *did* graduate, as `createSurfaceSocket` + `createProcessIdEcho` in `@kolu/surface-app/connect`. The "two different facts" insight is **preserved inside the echo** ‚Äî `createProcessIdEcho` still re-presents the dead id, distinct from `serverProcessId()` ‚Äî it just isn't hand-rolled per consumer anymore. What genuinely stays per-consumer narrowed to **lifecycle ownership** (kolu's `rpc.ts` vs drishti's `<SurfaceAppProvider>`) and **socket topology** (one socket vs per-host + admin sharing one echo). Same lesson as the F1 `retireSocket` reversal: the boundary is set by what's *actually* duplicated, found by reading source, not by what looked per-consumer at first.
</Callout>

<Callout kind="good" title="liveWhen ‚Äî predicate-as-data: the fold graduates, the meaning plugs in">
The sharpest proof the receptacle stays domain-blind is the verb it *refuses to learn*. `client.health().live` is the full conjunction ‚Äî transport-live ‚àß every readiness cell's predicate ‚Äî yet `@kolu/surface` core never knows what "connected" *means*. The **fold is the receptacle**: a generic `CellSpec.liveWhen?: (value) => boolean` marks a cell a readiness gate, and core AND-reduces that cell's verdict into `live` (`registry.enrollReadiness`) without ever naming a state literal. The **volatile knowledge rides the cell as data** ‚Äî `surface-nix-host`'s `connectionCell` declares `liveWhen: (v) => v.state === "connected"`, and the ssh vocabulary (the predicate, the four-state enum) stays *there*, beside the schema. It's the runtime sibling of `CellSpec.equals` / `resolveCellVerbs`: core owns the mechanism, the plug owns the meaning ‚Äî the receptacle is domain-agnostic, the volatility plugs in as data. So every surface mirrored through `mirroredSurface` carries its mirror-liveness leg *by construction* ‚Äî no consumer hand-ANDs `connection.state === "connected"`, and no widget can paint a dot green from raw cell state. **Location is structure** in the un-obvious direction too: the `=== "connected"` *can't* migrate into core without dragging ssh's domain across the boundary, so the split isn't a convention you keep ‚Äî it's load-bearing.
</Callout>

<Callout kind="warn" title="Re-evaluating @kolu/artifact-sdk ‚Äî its electricity framing shifted (open; resolve outside this PR)">
"Anchoring" (the original <PrLink pr={922} /> framing) now under-describes the
package. It holds **two** volatilities: (1) the **anchoring core** (`core/` ‚Äî W3C
quote extract/re-find/highlight; pure and agnostic, speaks only
`Range`/`Document`/`ShadowRoot`), and (2) the **sandboxed-iframe parent‚Üîframe
bridge** (`client/bridge.ts` + `iframe/` ‚Äî opaque-origin postMessage with
`event.source` identity as the only trust boundary). The bridge is the *harder*
volatility, and it has quietly accreted clients: comments (selection ‚Üí
`SelectMsg` ‚Üí highlights), then **in-frame navigation** (`observeIframeNavigation`),
then **back/forward input** (`observeIframeHistory`, <PrLink pr={1191} />). The
lens-debate in <PrLink pr={1191} /> already reframed the package's *stated*
identity to "the in-iframe sandbox bridge, with comments as one client" ‚Äî but
that was a docs fix, not a decomposition.

**The open question (work it outside this PR):** is "talk to code inside an
opaque-origin sandboxed frame" one sound receptacle ‚Äî a transport-grade
volatility with comments / navigation / input as interchangeable clients ‚Äî or a
topic-bundle that should split the agnostic **anchoring core** from the **bridge
transport**? Note the smell that prompted this: the back/forward client landed in
this package only by *physical necessity* (the one script inside the sandbox is
this one), which is exactly the "easy to put it here" the three traps above warn
against. Flagged, not decided.
</Callout>

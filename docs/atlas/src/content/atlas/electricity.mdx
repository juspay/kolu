<Callout kind="good" title="What counts as electricity">
Löwy's receptacle — hide the volatility behind a socket you plug into. Three tests at once: ① **domain-agnostic** — no terminal/canvas/git/xterm leaks into the interface; ② **hides a hard volatility** — transport, reconnect, GPU-context loss, persistence — not a 3-line `===`; ③ **graduates** — ships as `@kolu/*` and a *different* app plugs in. The proof of ③ is real, not aspirational. [`@kolu/surface`](https://kolu.dev/blog/surface-framework/) is **kolu's own** client↔server transport first — and it has since picked up a **second consumer** in [srid/drishti](https://github.com/srid/drishti) (a fleet process-monitor; browser ↔ Bun ↔ ssh-agent) that vendors it from this repo and declares its own per-host schema with none of kolu's domain. The second consumer isn't the *point* of the extraction — it's the *proof* it was real electricity: surface never noticed the domain changed.
</Callout>

<Callout kind="danger" title="How a session mis-identifies electricity — three traps, learned the hard way">
The live example: asked to add back/forward to the Code-tab preview, a session proposed extracting a **history stack**. Wrong three ways, each caught only by re-asking the test above — and the [solid-browser](solid-browser) plan is the corrected end-to-end.

- **Naming an organ, not the concept.** A history stack is a *transformer* — a working part *inside* a larger thing. The electricity was the **browser** (location + history + link-nav + render-host); history is one chamber of it. Test: *is the name self-describing standalone?* "history" → history **of what?**; "browser" → obvious. If you must append "…of X" to explain the name, you've named a part, not the receptacle.
- **A module in `client/src/` is not electricity.** Location *is* structure (Hickey): a generic-looking helper living inside the app is **part of the app**, however agnostic the code reads. Electricity is its **own package**, the dependency arrow pointing *out* — `client → @kolu/x`, never back. (`createSharedRoot` sits in `client/src` marked "publish?" — a leaf *awaiting* graduation, **not** a precedent for calling in-app modules electricity.)
- **Agnostic ≠ electricity — apply test ② honestly.** A clean, reusable, agnostic primitive that hides only a *bounded algorithm* (a stack, a debounce, an `===`) is a **leaf** (`nonempty`/`html-escape`-tier), not a receptacle. "I found a tidy generic module" is the feeling that *precedes* the mistake. Ask what **hard** volatility it hides — transport, GPU-context loss, sandboxing, reconnect — before reaching for `@kolu/`.
</Callout>

> "Decompose based on volatility." Functional decomposition splits a system by what it *does*; volatility-based decomposition isolates each area likely to *change* behind a stable interface — the way a body keeps blood-pressure, pulse, and salinity volatility "encapsulated behind the service called the heart." An *electricity*, here, is one such volatility pulled out into its own package. — Juval Löwy, [Volatility-Based Decomposition](https://www.informit.com/articles/article.aspx?p=2995357&seqNum=2) (excerpt from *Righting Software*).

This note does two jobs at once, and they pull in opposite directions. As an **inventory** it tracks what kolu has pulled out from under the app (the table). As a **method** it must *generate* those extractions, not merely record them — because an inventory that only recognises electricity in hindsight is a museum, not a tool. So the real test is **forward** (*"[does it generate, or only recognise?](#the-real-test-does-the-method-generate-or-only-recognise)"*): could it have produced `surface` from *"a client and a server talk over a wire"* before surface existed?

The lens that does both jobs is **three families of volatility** — every electricity below is exactly one of them, and the un-named ones hide in the *seams* between:

- **A · Boundary** — a live value across a place you don't control (network · process · host · origin · *and time*).
- **B · Invariant** — a rule run identically at every call-site, silent when forgotten.
- **C · Appliance** — a typed blob turned into a safe, addressable view.

Read the table as the **inventory**; read the sections after it as the **method**.

## What the table is really a list of — one contract, four boundaries

Step back from the package names and most of the "transport" rows collapse into **one** volatility wearing a different boundary each time:

> A reactive value whose *source* sits on the far side of a boundary you don't control — where the universal contract for surviving the boundary is **snapshot, then deltas, then liveness** (reconnect · re-identify · resume).

| Boundary you can't reach across | What *changes* (the volatility) | Where kolu puts the socket |
| --- | --- | --- |
| **Network** (client ↔ server) | the link drops, reorders, reconnects | `@kolu/surface` — `Cell`/`Collection`/`Stream` over a retrying link |
| **Process** (server ↔ shell) | the PTY dies, restarts, flips build-id | `@kolu/pty-host` — *served over surface*; adds process-lifecycle on top |
| **Host** (local ↔ remote machine) | the box is unreachable until a closure is realised | `@kolu/surface-nix-host` — realises a daemon, then speaks *surface over stdio* |
| **Origin** (parent ↔ opaque-origin iframe) | the frame is sandboxed; `event.source` is the only trust | `artifact-sdk`'s bridge — hand-rolls snapshot/deltas/liveness *again* |

Read down the last column: three of the four are *surface, or surface stretched further*. The fourth — the sandboxed-frame bridge — is the **only** boundary surface's links can't cross (opaque origin = no shared transport, postMessage only), so it re-implements the same snapshot/deltas/liveness contract **by hand**. That hand-rolling isn't a coincidence; it's the *tell*. The real electricity was never "surface the package" — it is the **`Channel` contract itself (snapshot → deltas → liveness), made independent of transport.** Surface owns it for websocket/stdio/in-process links; the bridge is simply where it leaked, because no `postMessageLink` exists yet.

This is the cut the file/module view hides: the same socket is wired four times because four *boundaries* are volatile, not because four *modules* exist. It also sharpens the artifact-sdk re-eval (below): the live question isn't "anchoring vs bridge" — it's *"should the bridge be a **surface link** rather than a private transport?"* — i.e. does the origin boundary deserve a socket of the **same shape** as the network boundary?

## The table misses an entire *kind*: invariant electricities

Every row above is a **boundary** electricity — a socket over a place where bytes cross. But the codebase hides a second family the package-shaped view is blind to, because its members aren't packages — they're *rules enforced by remembering*:

> An **invariant electricity** is a correctness rule run as the same cassette of plumbing at N call-sites, where forgetting it produces a **silent** bug, not a loud one. The volatility isn't a transport — it's that *the set of call-sites keeps growing, and every new one is a latent failure.* (This is `.agency/lowy.md`'s Layer-0 prompt: *the pattern across N call-sites is itself the artifact.*)

| Invariant run everywhere by hand | The silent bug when you forget | Enforce-by-construction receptacle |
| --- | --- | --- |
| **snapshot → *then* deltas** on every streaming handler | client renders deltas onto an empty base; a reconnect shows nothing | `implementSurface` wraps the handler — you declare *read* + *install*, never the ordering. Surface half-owns this; server handlers still hand-roll the snapshot. |
| **owner captured *before* `await`** in any async setup | `onCleanup` lands on a null owner → silent no-op → the subsystem leaks (~900 KB/component; #598/#600/#591) | a `withOwner`-shaped resource: cleanup registers at *attach*, not via a manual `detach()` handle |
| **validate on *read*** from any persisted store | a stale / `NaN` / `"false"` value loads as truthy and corrupts UI state | `persistedPref` (client) · the `state.ts` migration ladder (server) — *the same invariant in two unconnected homes* |
| **silence the *expected* abort, log the rest** | unmount/navigation spews fake errors that bury the real ones | `isExpectedCleanupError` — one predicate every stream consumer must run |

These never surface in a package audit because their *correct* form **deletes code rather than adding a folder**. That is the signature of family B: the receptacle is a *smaller surface*, not a new noun. Where family A asks *"what boundary is volatile?"*, family B asks *"what rule am I trusting myself to remember?"* — and both clear Löwy's bar (a volatility encapsulated so the blast radius of a change stays contained). Note the connection: snapshot→deltas appears in **both** families — surface (A) defines the contract, but its server-side enforcement is still a family-B invariant leaking out. The two families meet exactly where an electricity is half-built.

## A third family: appliance electricities (blob → safe view)

Families A and B miss the *largest* cluster of rows. solid-markdown, solid-pierre, transcript-html, the planned solid-xterm, the anchoring core — none is a boundary, none is an invariant. They are **appliances**: each takes a *typed blob* and returns a *safe, addressable, disposable view*, hiding the volatility of the rendering stack itself.

> An **appliance electricity** absorbs two churning things behind a stable `(blob, host) → view` interface: (1) the *rendering tech* — which parser, which sanitiser after the next CVE, GPU vs canvas, the next xterm major — and (2) the *growing set of content-types*. The domain ("render *this file* in the Code tab") is the appliance that plugs in; the receptacle never learns it.

| Appliance | Blob in | Volatility absorbed |
| --- | --- | --- |
| `@kolu/solid-markdown` | markdown text | marked → DOMPurify → Shiki; the *sanitiser* moves with every XSS CVE |
| `@kolu/solid-pierre` | file / unified diff | Pierre tree/diff internals, theme churn |
| `@kolu/solid-xterm` *(to build)* | PTY byte stream | xterm majors, 8-addon orchestration, WebGL context-loss |
| `@kolu/transcript-core`+`-html` | agent transcript | transcript schema + render |
| anchoring core *(in artifact-sdk)* | a `Range` in a rendered doc | W3C quote re-find as the DOM mutates |

Now the whole table has a shape — every electricity is exactly one of three kinds:

- **A · Boundary** — a live value across a place you don't control. *(surface · pty-host · nix-host · the bridge transport)*
- **B · Invariant** — a rule run the same way at every call-site, silent when forgotten. *(snapshot→deltas · owner-correctness · validate-on-read · expected-abort)*
- **C · Appliance** — a typed blob turned into a safe, addressable view. *(solid-markdown · solid-pierre · solid-xterm · transcript · anchoring)*

And the two **compositions** on the table — `solid-browser` and `solid-fileview` — are revealed for what they are: *not electricities but wiring* that routes a blob (A: link interception) to the right appliance (C) and owns *addressing into* the result (location/history). That is exactly why solid-fileview tested as a leaf in the table above: **a composition of electricities is not itself one.** It also tells you where to look for the *next* one — the seams *between* families, where addressing (anchoring, location) spans A and C, are where the un-named infrastructure hides.

## The seam the families share: addressing (a reference that outlives what it points at)

Walk into that A×C seam and a fourth thing is hiding — not a package, a *capability kolu builds twice and composes by hand*:

> **Addressing** — name a point in content and have the name *survive the content changing under it.* A full address has two levels: **which view** (location) × **where inside it** (anchor). A comment is just `(location, anchor, payload)`.

kolu already owns both halves, in different packages:

- **which view** → solid-browser's location/history (survives *navigation*: the view is replaced)
- **where inside** → artifact-sdk's anchoring core (survives *re-render*: the DOM mutates; re-find by W3C quote + context)

Nothing owns the *pair*. "Jump to this comment after a reload," "deep-link a diff line," "re-anchor a selection across a re-render *and* a navigation" each re-compose the two halves at the call-site — a family-B smell (a rule kept by remembering) riding on top of two family-A/C receptacles.

And here is the step that earns the word *infrastructure*. An anchor is not a string — it is **Hickey's identity over time.** The W3C quote is a stable *identity*; the concrete `Range` it resolves to is a *value* that changes every epoch as the document mutates. Re-finding that Range on each render is *exactly* surface re-delivering a `Cell` value after a delta — the same snapshot-then-deltas contract, except the boundary it crosses is **time**, not the network. Which means family A was never really about *spatial* boundaries: it is **"a live identity whose value is re-resolved across a boundary you don't control"** — and network, process, origin, *and time* are all just values of "which boundary." Addressing is family A with the boundary dial turned to **time**.

<Callout kind="warn" title="Over-merge check (the note's own trap ①)">
Is "addressing" one receptacle, or a topic-bundle of location + anchor smuggled together? Honest answer: **undecided — and that's the finding.** The two halves have genuinely *different* volatilities (navigation vs DOM-mutation) and may well stay two sockets. What is real and demonstrable is that the *pair is used together and owned by no one.* Logged as the strongest un-named candidate at the A×C seam — flagged, **not** declared electricity. (The discipline that catches a forced merge is the same one in the three-traps callout at the top.)
</Callout>

## The real test: does the method *generate*, or only *recognise*?

A tracker recognises electricity after it ships; a method *predicts* it. Surface sits in the "done" column — but surface only *proves* the method if the method would have **produced** it from "a client and a server talk over a wire," before anyone wrote a line. So the honest test of this whole note is **forward**, not backward: run the procedure on kolu *as it stands today* and write down the electricities it predicts that don't exist yet. If they later ship and a second consumer plugs in, the method generated them. If the predicted axis never actually moves, the method hallucinated one (see falsification, below). Three standing bets, in falsifiable form:

| The method predicts… | Derived from | Falsified if… |
| --- | --- | --- |
| **`postMessageLink`** — the iframe bridge becomes a *surface link*, so the origin boundary gets the same snapshot/deltas/liveness socket as the network boundary | family A: four boundaries, one contract; the bridge is the only hand-rolled one | the opaque-origin boundary genuinely needs a *different* contract than the network — it doesn't; both are drop-prone async channels |
| **`createOwnedResource`** — a `withOwner`-shaped primitive that registers cleanup at attach, so `onCleanup` can't land on a null owner | family B: owner-before-await, already seen 3× (#598/#600/#591) | the SolidJS owner model changes so the await-gap closes upstream, or the call-site set stops growing |
| **`@kolu/solid-xterm`** (already tabled) | the xterm lifecycle braid | a second SolidJS app never embeds xterm (③ never arrives) |

### The procedure — run it on any app, before any code

Six steps. Inputs are *interactions*, never features. The falsifiable claim: fed only *"a client and a server talk over a wire,"* step 6 must **emit surface**. If it can't, the method is wrong — and this note is then a catalog of packages wearing a method's clothes.

1. **Frame the interaction, not the feature.** Reject requirements-as-design (Löwy: *"any attempt at designing against the requirements will always guarantee pain, because when the requirements change, so will your design"*). The only legal input here: *a client and a server talk over a wire.*
2. **Sweep the two axes of change** (both *verbatim* Löwy): **① same customer over time** — what changes for *this* user as use evolves? (request/response today → server-pushed live state → deltas → reconnect-with-resume). **② all customers at one instant** — are they all using the wire the same way? (LAN vs flaky mobile vs proxies that kill long-lived sockets → poll / SSE / WS).
3. **Independence check.** Are ① and ② the *same* change in disguise? No: *push semantics over time* ≠ *transport across customers*. Two real axes → a real vault. If they collapse to one, you've found functional decomposition wearing a domain hat — **stop.**
4. **Screen the survivor** (three Löwy filters): *variable, not volatile?* — an if-else/flag can't span poll→SSE→WS, so it survives. *Solution masquerading as requirement?* — "add WebSockets" is the **Cooking**, not the **Feeding**; climb to the parent: *live-state delivery.* *Open-ended & expensive to contain unless encapsulated?* — retrofitting a transport swap across every feature is the loose grenade. Survives.
5. **Name it for the axis, not the tech.** `WebSocketClient` names an *activity* (it fractures on the first transport swap); **`LiveStateTransport` / surface** names the *volatility* (it doesn't). The naming test is the cheap proxy for the sweep test in [§the metaphor, made generative](#the-metaphor-made-generative-axes-in-not-decoration).
6. **The interface is the receptacle.** Verbs that don't change when the volatility behind them does — *snapshot → deltas → liveness*; a signal-shaped handle; no `AsyncIterable` in components. Domain-agnostic, a different app can plug in → **you have specified `@kolu/surface` without having written any kolu.**

Step 2 is where functional decomposition dies: the instant you write *what changes* instead of *what it does*, surface stops being a design choice and becomes **inevitable**. (Honest footnote, [§hindsight](#foresight-names-the-vault-pain-funds-it): the method names *where* the vault is and *what its verbs are* before code exists — but historically it's recurring same-shape **pain** that tells you it's time to pour it. Foresight names the vault; pain funds it.)

The kolu second brain

Capturing not just plans but history and learnings from LLM sessions — without building a second system on top of the one you already have. budding  design · captured 2026-06-01 · reconciles docs/plans with the hk zettelkasten

You don't need a new second brain. You have one — name the roles and govern it by a single axis: mutability.

kolu already owns every organ of a second brain: a zettelkasten (hk on GitHub Issues), an evergreen garden (docs/plans/*.html), a frozen log (docs/proposals), and a dated stream (the blog). The risk isn't a missing system — it's two competing ones that write the same knowledge twice. The fix is one rule for "where does this go," one cheap fix for the gnarliness, and one capture habit.

  • One axis — mutability. Living & cross-linked → hk (Issues, free backlinks). Frozen & rich-enough-to-render → docs/. That's the whole split.
  • Kill the gnarliness cheaply. Stop encoding ancestry in filenames; split the 208 KB monolith; hand-write a Maps-of-Content index. No generator.
  • Capture once. A /retro skill distills a finished session into an hk learning node; only the rich ones graduate to an annotatable docs/plans artifact.
  • Link one direction only. docs → #N. Never build a two-way sync — that's the thing that rots.

1 · The actual problem — two brains, not zero

The instinct "I should design a second brain" hides the real situation: kolu already has two knowledge systems, and they overlap dangerously.

SystemWhat it isAlready claims to hold…
hk (hubkasten)Zettelkasten on GitHub Issues. #N = the only link; backlinks are free via CrossReferencedEvent. "Replacing the issue tracker."roadmaps, bugs, "create a TIL / ADR / decision node" hk/SKILL.md:3
docs/In-repo: plans/*.html (evergreen), proposals/*.md (frozen RFCs), reports.plans, specs, reviews, proposals — and now you want history + learnings here too

The trap the first design walked into. It proposed a brand-new docs/decisions/ + docs/devlog/ and a seven-kind taxonomy. But hk's own skill description already claims TIL / ADR / decision nodes — so ADRs would get two homes. That is exactly "the same knowledge written twice." Both an independent Hickey review (does it complect?) and a Lowy review (are boundaries drawn around what changes together?) rejected it. The seven kinds aren't seven axes of change — they collapse onto two: mutability (living vs frozen) and distilled-vs-raw. "Decision vs proposal" is ceremony of the same concern; "learning vs devlog" is the same capture at two resolutions.

2 · The single axis: mutability

Pick one question to route any artifact: "Will this keep changing, and does it want cheap cross-links — or is it settled and rich?"

Living → hk (GitHub Issues)

High-volatility, frequently re-edited, cheaply cross-linked. The working edge of thought.

  • Roadmaps, open questions, bug lists
  • Settled decisions (a locked node)
  • TILs & tier-2 session learnings before they harden
  • Anything that wants #N backlinks

Frozen & rich → docs/

Hardened, durable, worth rendering & annotating in kolu's own Code tab.

  • plans/*.html — evergreen design & research (the garden)
  • proposals/*.md — frozen RFCs (the formal log)
  • Retros worth re-reading & commenting on

Why mutability and not, say, linkability? Because both surfaces give cheap links, so linkability can't decide. Topic/PARA-actionability can't decide either — it discards the conceptual cross-links the learnings goal needs. Mutability is the one question whose answer is always clear and always picks the right home. This is precisely the Rust pattern (frozen RFC + live tracking issue) and the Oxide pattern (Committed RFD + follow-up Issues).

ADR placement, concretely: a settled technical decision becomes a closed + locked hk node (frozen by the lock, linked by #N) — or, if it needs ceremony, a docs/proposals entry with status: accepted (the template already carries implemented-in: / superseded-by: fields). Do not create docs/decisions/.

4 · Fixing the gnarliness

The complaint is real and has a precise diagnosis. docs/plans/remote-terminals.pty-daemon.tui.html encodes its ancestry into its filename — a hand-rolled folgezettel. Every authority on PKM warns about exactly this failure: a strict path forces one true parent, and tui genuinely relates to both pty-daemon and chrome-bar. Because linking out was expensive, content piled up instead — that's how remote-terminals.html reached 208 KB.

The cheap fix (zero new machinery)

  1. Flat, ancestry-free slugs. remote-terminals.pty-daemon.tui.html → pty-daemon-tui.html. The name is a stable handle, not a path.
  2. Split monoliths into atomic docs. Break the 208 KB file into the children it already wants to be.
  3. Recover structure with a Maps-of-Content index — a hand-written note that links and annotates its children, so a child can appear in multiple maps. See the live one at docs/plans/index.html.
  4. Collapse depth in-page with <details> — the same trick remote-terminals.html already uses 15 times. (This very document uses it in §8.)

Rejected: a just docs::index generator. It would be new machinery to solve a problem the filename scheme itself created — and it adds its own rot (a CI staleness check, an "is the index stale?" gate). Flat slugs + a hand-curated MOC solve it with nothing to maintain but prose. Also rejected: Johnny.Decimal numeric prefixes (its own community found the numbers add friction; keep the shallowness, drop the numbers) and real nested folders (same one-true-parent problem, plus cascading renames).

5 · Capturing history & learnings from LLM sessions

This is the new capability you asked for. The 2025–26 consensus is a three-tier pipeline — and the tiers must live in different homes at different cadences, or the durable layer bloats.

① transcript→ /retro distills→ ② learning node (hk)→ recurs?→ ③ durable rule (.apm/)
TierWhatHome
① EphemeralThe raw agent transcript.Already exported by exportSessionAsHtml.ts (Claude JSONL / Codex JSONL / OpenCode SQLite → self-contained HTML).
② Curated learning"What we found, what worked, what to avoid" — distilled, concept-oriented.hk node by default; a docs/plans/<slug>.retro.html only when rich enough to annotate.
③ Durable ruleAn instruction that changes agent behavior next time..apm/ sources → regenerated into AGENTS.md/.claude by just ai::apm.

The /retro skill

A new APM-sourced skill that runs as a fresh subagent (same-context self-evaluation is overconfident — this mirrors how kolu already isolates /hickey and /lowy). It:

  • Reads the raw transcript (JSONL) — not the rendered HTML. The raw JSONL is what Claude Code deletes after ~30 days, so the distillation must happen before then; the rendered HTML is lossy for distillation. Reuse/extend exportSessionAsHtml.ts rather than re-implementing the parse.
  • Preserves surviving vocabulary verbatim — error strings, file:line refs, param names — while paraphrasing the decisions. The note indexes back to the transcript; it doesn't replace it.
  • Writes one hk learning node, cross-linked by #N to the proposal/plan it touched.
  • Is offered as an optional final step in the /do chain — habit-forming without nagging on trivial diffs.

Promotion to a rule is maintainer judgment, not an automated counter.

The tempting design — "auto-promote a learning once it recurs 2–3 times, tracked by backlink count" — is circular: a backlink only accrues if a human already recognized the recurrence and linked it, which is the very judgment the gate was meant to automate. So either (a) make promotion an explicit call during the per-release ritual ("does this lesson keep biting?"), or (b) have /retro search existing learning nodes at capture time (by the verbatim error string / file:line) and auto-link matches, so the count becomes a real signal. Don't dress manual judgment as automation — nobody maintains a counter that lies. And remember the hard ceiling: always-loaded rules get ~150–200 instructions before models start dropping them (Claude Code's own prompt eats ~50), so the durable tier must stay lean — point to docs, don't inline them.

6 · Why HTML — the dogfooding payoff

This is kolu's unfair advantage and the reason the garden is HTML, not just markdown. kolu's Code tab routes .html/.svg/.pdf to a sandboxed iframe and gives text-anchored comments on the rendered page — the same extractQuote/findQuote anchor model the in-iframe annotator uses. So the second brain is read, navigated, and annotated without leaving kolu, and the comment tray flushes to a Markdown list ready to paste back into an agent prompt.

docs/plans/*.html→ Code tab iframe→ select text → "+ Comment"→ copy tray→ paste into agent

That loop is why "respond in .html" is the right instinct: a plan you can annotate in-app and feed straight back to an LLM closes the loop a markdown file can't. Keep the blog (a public Astro surface) distinct from the garden (in-app artifacts) — same content modes, different boundaries.

7 · How the brain feeds releases

The two systems aren't independent. A changelog and a blog post should be the downstream artifact of captured history — not a separate manual effort. The spine, end to end:

/talk · /do session→ /retro → hk node→ conventional commit + PR (#N)→ git-cliff CHANGELOG→ whatchanged → blog→ evidence demo→ release

Link one direction: per-release CHANGELOG sections and blog posts carry #N links back to the hk nodes and proposals that shipped (free backlinks via CrossReferencedEvent). Traceability flows without a two-way sync invariant to maintain. Surface learnings publicly as a "What we learned" footer on the release post — not a top-of-post section that buries the features. See release-workflow.html for the release mechanics this plugs into.

8 · What we deliberately are not building

The user's #1 ask is the first release. The second-brain reorg is a later project. Both reviewers flagged the original design as a multi-week doc-engineering effort that would delay the release and likely rot. Cut it to the bone:

The deferred / rejected list (and why)
ProposedVerdictWhy
docs/decisions/ (separate ADR dir)cuthk + docs/proposals already cover settled decisions. Two homes = double-writes.
docs/devlog/cutRaw narrative is the transcript (tier ①) + the hk node (tier ②). No third store.
Seven-kind taxonomycutCollapses to one axis (mutability). The kinds blur in practice; deciding "learning or devlog?" is accidental cost.
just docs::index generatorcutMachinery to fix a filename problem; flat slugs + hand MOC need none.
Auto rule-promotion by backlink countmanualThe counter is circular. Promote by judgment at release time.
Two-way hk ↔ docs synccutA cross-store invariant that drifts. Link docs → #N only.

The brain's MVP is small: maturity tags in the house style + one hand-written MOC index + a /retro capture skill + the one-direction link rule. Ship that after the first release, not before.

9 · Migrating the existing 11 files

Incremental, not a big-bang reorg. Order of operations:

  1. Author docs/plans/index.html as the root MOC (done — it groups today's files by kind & maturity).
  2. When you next touch a dotted-name file, rename it to a flat slug and update the MOC link. Don't mass-rename cold.
  3. Split remote-terminals.html into atomic children the next time it's edited; rewrite the parent as a thin map.
  4. Backfill maturity tags opportunistically as you revisit each doc.

The existing #lessons, #r4-lessons, and #r4ui-postmortem anchors inside remote-terminals.html are the proto-version of this system — learnings already captured inline in plan HTML. /retro formalizes that instinct into a repeatable habit.

10 · Open questions

  • Shared CSS. The house style is copy-pasted per file today (these two docs included). A single docs/plans/style.css would stop maturity-tag drift — but adds a load dependency for standalone-rendered artifacts. Decide before the tags spread.
  • Transcript durability. Commit exported transcripts into the repo (diffable, Code-tab-readable, but bloats git) or host out-of-band (like the evidence-assets release)? The 30-day JSONL deletion argues for committing something durable.
  • Public learnings. Should the blog's "What we learned" footer link raw hk Issues (exposing the kasten) or a curated subset? Some postmortem nodes carry internal detail.
  • /retro trigger. Offered automatically after any /do that involved debugging, or fully manual? Auto guarantees capture but risks prompt fatigue on routine sessions.

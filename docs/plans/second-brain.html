The kolu Atlas

The kolu Atlas. Your second brain as an in-repo knowledge garden — capture not just plans but the history and learnings of LLM sessions, without building a second system on top of the one you already have. budding · design · captured 2026-06-01 · revised 2026-06-02 after an adversarial research pass

You already have a second brain. Name the roles — don't build a new system.

  • One store: docs/plans/. Substantial, structured artifacts (plans, designs, reviews, retros, history) live here. GitHub Issues is the lightweight/conversational tier (bugs, tasks); its durable nuggets get extracted into the garden.
  • Markdown for prose; HTML/SVG only for visual artifacts. Render markdown in-app (Code tab, #1093) and on github.com — no per-file CSS, one shared theme.
  • One routing rule, one maturity tag. substantial → file; lightweight → issue. Living-vs-frozen is a seedling/budding/evergreen tag, never a location.
  • Capture by hand first. A manual /atlas retro (fresh subagent) writes one note; automate via /be only after it earns it. Link one direction: docs → #N.

Revised after research 2026-06-02 — a 7-lane adversarial pass (web-researched, fact-checked, Hickey/Lowy/completeness critics) stress-tested the first draft and overturned several decisions. What changed:

  • "HTML all the way" → markdown-first (§4). Token cost, native github rendering, and the fact that the CLIs kolu runs already request markdown all point one way; HTML is a render target, reserved as a source only for visual artifacts.
  • Folding proposals into the garden → keep the numbered log (§6). A decision log's value is its stable number + immutability; don't dissolve it into mutable notes.
  • The docs/plans → docs/garden rename → dropped. Cosmetic, 4+ hardcoded consumers, zero functional gain.
  • The build-failing index gate → softened (§5): dead-links only; "unfiled" becomes a generated fallback, not a red build.
  • Auto-capture on every /be → manual first (§7): the skill isn't built yet; don't automate an empty tier.
  • Fixed two shipped bugs: a contradictory routing rule across these docs, and a false "free #N backlinks" claim (§9).

1 · The problem — two systems, not zero

kolu already has two overlapping stores: GitHub Issues (roadmaps, bugs, quick questions) and docs/ (plans, specs, reviews, proposals). The risk isn't a missing system — it's writing the same thing twice, or not being able to state where a thing goes. The first draft made this worse with new docs/decisions/ + docs/devlog/ dirs and a seven-kind taxonomy; a Hickey (complecting) and a Lowy (volatility) review both rejected it. The kinds collapse: they're all artifacts for one garden, differentiated by a tag, not by new directories.

2 · The one routing rule

Route by a single question, stated once and identically everywhere: is this a substantial, structured artifact — or a lightweight, transient node?

Substantial → docs/plans/ fileLightweight → GitHub Issue
Plans (living trees and settled), designs, research, reviews, retros, history — anything worth reading & annotating in the Code tab.Bugs, tasks, roadmap items, quick questions — the conversational tier.

Living-vs-frozen is NOT the divider — a plan tree like remote-terminals.* lives and evolves in the garden for months. Maturity is a per-note tag (seedling/budding/evergreen), never a routing axis and never a location.

The boundary blurs by design, and that's fine. A concept can have both a garden doc and a tracking issue. Issues aren't a knowledge dead-end — knowledge accretes in threads (it's how Simon Willison runs his whole practice). The rule: when an issue thread becomes the source of truth, extract its summary into a garden note and let the issue stay the conversation. (GitHub Discussions is deliberately not used — not in the checkout, so it fails the "LLM can read it from disk" test.)

History isn't a third place — it's the garden over time. A settled note is just frozen (a maturity tag); git is the history; the changelog is a release artifact, not a garden mode. Two surfaces — public & garden — with Issues alongside as the tracker (§2).

4 · Format — markdown for prose, HTML/SVG for visual artifacts

The first draft said "HTML all the way." The research flipped it. Author prose notes in markdown; reserve HTML/SVG as an authored source only for genuinely visual or interactive artifacts (diagrams, dashboards, this design doc). HTML is otherwise a render target, not the source.

Why markdown wins for proseEvidence
Materially fewer tokens for the same content — the agents pay it on every read.Cloudflare measured ~80% reduction (16,180→3,150 tokens) HTML→markdown on their own page. blog.cloudflare.com/markdown-for-agents
The CLIs kolu exists to run already prefer markdown.Claude Code & OpenCode send Accept: text/markdown — the garden was feeding its own agents the format they ask to avoid.
Renders for free where it matters.github.com renders .md natively (no htmlpreview shim); kolu's Code tab renders .md via #1093.
One content model + one stylesheet.The blog is already markdown; a single @kolu/solid-markdown "paper" theme kills the ~76 KB of per-file CSS duplicated across today's HTML notes.

Two honest caveats (kept, not hidden):

  • HTML genuinely wins for human-read output and visual artifacts (Shihipar/Willison, "the unreasonable effectiveness of HTML"). The split is by regime: markdown for in-repo, agent-ingested, versioned prose; HTML/SVG for visual or one-shot human-facing artifacts.
  • In-app annotation of rendered markdown isn't at parity yet. #1093 renders md, but text-anchored comments work on the source view (v1); HTML still annotates the rendered artifact directly. If a doc is heavily reviewed in the rendered view, HTML is defensible until rendered-md annotation lands.

On Astro (the "make the garden an Astro site, auto-rebuild on edit" idea): not worth it. Markdown already renders in the Code tab with zero build (#1093), so the live-preview payoff is free. Keep Astro for the public blog; don't add a garden build-watch loop. Atomicity (small files) is the orthogonal answer to file size — it was wrongly cited as the answer to per-token noise; the real fix for noise is markdown, and the fix for CSS duplication is the shared theme, not the file count.

5 · Navigation — flat slugs, a curated MOC, a gate that doesn't nag

Dotted names like remote-terminals.pty-daemon.tui.html encode ancestry into the filename — a folgezettel that forces one true parent. Keep the cheap fixes:

  • Flat, ancestry-free slugs (pty-daemon-tui.md) — the name is a handle, not a path. Finish the 3 remaining dotted renames in one commit, not "on next touch" (a half-migrated tree is itself the drift the index advertises but can't enforce).
  • A hand-curated, annotated Maps-of-Content index (index.html) — its ordering, "sibling of", and cross-references are genuinely un-generatable. Keep it as a thin layer.

Split the CI gate (don't train yourself to --no-verify).

  • keep the dead-link check — a real invariant about reality.
  • soften "every file must be hand-linked or the build fails" → an auto-generated "Unfiled" fallback section. A low-value note degrades to appears in Unfiled instead of reddening CI over boilerplate — same orphan-prevention, no authorship tax.
  • Make the gate glob format-agnostic ({md,html}); optionally source each note's one-liner from its own frontmatter so the description isn't a second, drifting copy.

6 · Proposals & decisions — keep the numbered log

The first draft folded proposals into the garden as status: notes. The research rejected the fold: a decision/RFC log's load-bearing value is a stable monotonic identifier (so the title can change without breaking references — Oxide's exact rationale) plus append-only immutability and supersede links. Flat ancestry-free slugs (§5) would kill the numbering that constitutes the log.

So: leave docs/proposals/ numbered, immutable, markdown — exactly as CONTRIBUTING.md ("a short markdown file") and the PR template already require. proposed → accepted → superseded is one lifecycle expressed as a status field — which is why no separate docs/decisions/ dir is needed (the proposal log already fills that role). Retros are not decisions; they stay garden notes (§7).

7 · Capture history & learnings from LLM sessions

Three tiers, each in a different home so the always-loaded layer stays lean:

TierWhat · where
① EphemeralRaw transcript — already exported by exportSessionAsHtml.ts (Claude/Codex JSONL, OpenCode SQLite).
② Curated learning"What we found / what to avoid," distilled — a GitHub Issue, or a <slug>.retro garden note when rich enough to keep.
③ Durable ruleA behaviour-changing instruction — .apm/ sources, regenerated into AGENTS.md. Graduate a must-hold rule to a justci recipe/hook (deterministic) rather than an always-loaded line: the lifecycle is memory → rule → code.
/be session→ /atlas retro (manual, fresh subagent)→ draft→ accept?→ ② note→ recurs?→ ③ rule / hook

Start manual, then automate. The /atlas skill doesn't exist yet and /be currently ends at /evidence. Ship a manual /atlas retro the maintainer invokes when a session taught something; it runs as a fresh subagent (same-context self-eval is overconfident — the reason /hickey//lowy run isolated), preserves verbatim vocabulary (error strings, file:line), and drafts a note that persists only on accept (a staging step — auto-persisting unreviewed agent notes is the documented memory-poisoning failure). Use it ~10 times; only then wire it as a gated final /be stage. Promotion ②→③ is human judgment, never an automated backlink counter. Capture positive patterns ("prefer Y"), not only "avoid X".

8 · How the brain feeds releases

A changelog and blog post are the downstream of captured history, not a separate chore:

/be · /do session→ /atlas retro → note→ conventional commit + PR (#N)→ git-cliff CHANGELOG→ whatchanged → blog→ release

Highest-ROI gate of all — a conventional-commit lint on PRs. Commit hygiene is ~41/100 today, and this whole pipeline (git-cliff CHANGELOG → whatchanged → blog) depends on it. One small check protects the entire release story — worth more than any of the docs machinery. (Start warn-only to avoid the same --no-verify reflex; harden once green.)

9 · Links — one direction, and the #N reality

Keep links one direction (docs → #N, and doc→doc) and no two-way sync — a maintained bidirectional invariant rots. But correct the earlier claim: GitHub does not autolink #N inside committed files, and a file's content creates no backlink on the issue timeline. The real reverse pointer comes from the PR/commit that adds the note (its body mentions #N) — or one deliberate comment on the issue when a note is promoted. There are no "free backlinks" from file content; rely on the PR mention.

10 · Deliberately not building (yet)

cut a separate docs/decisions/ or docs/devlog/ (Issues + the proposal log cover them) · the seven-kind taxonomy (one store, one tag) · a docs::index generator for the curated map (the hand map's value is curation) · two-way Issue↔docs sync · auto rule-promotion by backlink count · the docs/plans→docs/garden rename (cosmetic) · auto-capture on every /be (until manual proves out) · a decay ritual / staging buffer (scale kolu doesn't have yet).

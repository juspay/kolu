The kolu second brain

EPM — Evergreen Project Management. Capture not just plans but the history and learnings of LLM sessions, without building a second system on top of the one you already have. budding · design · 2026-06-01

You already have a second brain. Name the roles and govern it by one axis: mutability.

  • One axis — mutability. Living & cross-linked → GitHub Issues (free #N backlinks). Frozen & rich → docs/ (the garden).
  • Kill the gnarliness cheaply. Flat slugs, atomic notes, a hand-written Maps-of-Content index. No generator.
  • Capture once. A /retro skill distills a finished session into a note; only the rich ones become annotatable garden docs.
  • Link one direction. docs → #N. Never a two-way sync — that's what rots.

1 · The problem — two systems, not zero

kolu already has two overlapping stores: GitHub Issues (roadmaps, bugs, decisions, TILs — #N links with free backlinks) and docs/ (plans, specs, reviews, proposals). The risk isn't a missing system — it's writing the same thing twice. The first design made this worse with new docs/decisions/ + docs/devlog/ dirs and a seven-kind taxonomy; a Hickey (complecting) and a Lowy (volatility) review both rejected it. The kinds collapse onto one real axis: mutability.

2 · The rule: one axis = mutability

Route any artifact by one question: will it keep changing (and want cheap links), or is it settled and rich?

Living → GitHub IssuesFrozen & rich → docs/ (garden)
Roadmaps, open questions, bugs; a settled decision (a locked issue); TILs & session learnings before they harden; anything that wants #N backlinks.Hardened design & research (plans/*); proposals; retros worth re-reading & annotating in the Code tab.

Why mutability and not linkability? Both give cheap links, so linkability can't decide. This is the Rust pattern (frozen RFC + live tracking issue) and Oxide's (Committed RFD + follow-up issues). A settled decision is a closed/locked issue — or, if it needs ceremony, a proposal with status: accepted. Do not add a docs/decisions/ dir.

4 · Fix the gnarliness

remote-terminals.pty-daemon.tui.html encodes its ancestry in the filename — a folgezettel that forces one true parent (yet tui relates to both pty-daemon and chrome-bar), and because linking out was costly, content piled into a 208 KB monolith. The cheap fix, no new machinery:

  • Flat, ancestry-free slugs (pty-daemon-tui.html) — the name is a handle, not a path.
  • Atomic notes — split monoliths into single-concept files. <details> only folds the view; an LLM still reads the whole DOM, so atomicity (not collapsing) is the real fix.
  • A hand-written Maps-of-Content index that links and annotates its children, so one child can sit in many maps. See index.html. No generator, no numeric prefixes.

5 · Capture history & learnings from LLM sessions

Three tiers, each in a different home so the durable layer stays lean:

① transcript→ /retro distills→ ② learning note (issue)→ recurs?→ ③ durable rule (.apm/)
TierWhat · where
① EphemeralRaw transcript — already exported by exportSessionAsHtml.ts (Claude/Codex JSONL, OpenCode SQLite).
② Curated learning"What we found / what to avoid," distilled — a GitHub issue; a <slug>.retro garden doc when rich enough to annotate.
③ Durable ruleA behaviour-changing instruction — .apm/ sources, regenerated into AGENTS.md by just ai::apm.

The /retro skill runs as a fresh subagent (same-context self-eval is overconfident, like /hickey and /lowy): it reads the raw JSONL (deleted after ~30 days), preserves verbatim vocabulary (error strings, file:line) while paraphrasing decisions, writes one learning note linked by #N to the plan it touched, and is offered as an optional final /do step.

Promotion to a rule is judgment, not a counter. "Auto-promote after N backlinks" is circular — a backlink only exists once a human spotted the recurrence. Promote during the release ritual, or have /retro search existing notes by error string and auto-link. Keep the always-loaded rule layer lean (~150–200 instruction ceiling): point to docs, don't inline.

6 · Markdown-first; HTML for visual artifacts

Notes are markdown by default — atomic, LLM-cheap, diffable, rendered natively on GitHub and in the Code tab (#1093). Reserve HTML for layout-heavy visual artifacts (design docs like this one, mockups, diagrams), viewable via htmlpreview.github.io or the Code tab. Text-anchored comments work on any source file, so a note renders and annotates in-app, then the comment tray flushes back into a prompt.

Note kindFormat
Retro · learning · decision · devlogMarkdown
Design doc · spec · review · mockupHTML / SVG

7 · How the brain feeds releases

A changelog and blog post are the downstream of captured history, not a separate chore:

/talk · /do→ /retro → issue→ commit + PR (#N)→ git-cliff CHANGELOG→ whatchanged → blog→ release

CHANGELOG sections and posts carry #N links back to the issues/proposals that shipped (free backlinks) — traceability without a two-way sync. Surface learnings as a "What we learned" footer on the release post, not a feature-burying intro. See release-workflow.html.

8 · Deliberately not building

cut a separate docs/decisions/ or docs/devlog/ (Issues + proposals already cover them — two homes = double writes) · the seven-kind taxonomy (collapses to one axis) · a docs::index generator (flat slugs + a hand MOC need none) · two-way Issue↔docs sync (a drifting invariant — link docs → #N only) · auto rule-promotion by backlink count (circular; promote by judgment).

MVP: maturity tags + one hand-written MOC + a /retro skill + the one-direction link rule. Ship it after the first release.

Release workflow for kolu

Cutting the first public release, then a sustainable frequent cadence with a changelog + blog post per release. budding  research → design · captured 2026-06-01 · grounded in the repo + 8-lane web research, adversarially verified

Ship v0.1.0 as a hand-cut git tag this week; let everything else grow from there.

kolu is unusually close to releasable: the commit-identity rail, a deployed Astro blog, multi-platform CI, and a binary cache already exist. The genuinely missing primitive is a stable per-release anchor — a git tag. Everything users ask for (a changelog, a blog post, demos, a way to pin a known-good build) hangs off that one tag.

  • Version: start at v0.1.0 (0.x SemVer — honestly signals "early, expect breakage"). Graduate to CalVer later if the cadence makes dates the more useful signal. CalVer-now is defensible but not the consensus the hype claims — see § Decision 1.
  • Channels for free: keep nix run github:juspay/kolu as rolling "edge"; github:juspay/kolu/v0.1.0 is "stable" with zero flake changes.
  • Two changelog artifacts, two audiences: a machine CHANGELOG.md (git-cliff) for the audit trail, a curated blog post (the existing whatchanged skill + evidence demos) for humans. Don't conflate them.
  • Cadence: a biweekly rhythm, not a deadline. Small batches, skippable quiet cycles. Automate the mechanical half; keep the narrative human.

2 · Decision 1 — the version scheme

This is the one thing you must settle before the first git tag. The research surfaced a strong "CalVer for apps" narrative; adversarial verification knocked it down to "it's a free choice." Here is the honest landscape.

Correction applied. The claim "CalVer is the modern consensus for apps" is overstated. Verified counter-facts: PEP 2026 (CalVer for Python) was rejected in 2025; Unity abandoned year-based versioning in Unity 6 (2024); the flagship desktop app VS Code ships 1.x SemVer, not 2026.x; GitHub CLI, ripgrep, fd, Neovim, kubectl, Terraform all ship SemVer-format. semver.org never restricts SemVer to libraries — a CLI's flags are a public API. The real axis is "do code-level consumers pin you?", not "app vs library."

OptionFirst tagSignalsVerdict for kolu
0.x SemVerv0.1.0"Early, unstable, expect breakage." Patch = 0.1.1, feature = 0.2.0.Recommended for now. Honest for a project with zero prior releases; the posture Neovim & Zed use. Costs nothing to leave later.
CalVerv2026.6.0"When it shipped." YYYY.MINOR.MICRO; valid SemVer so pnpm/cargo parse it.Defensible later. Erases the pre-1.0 "still early" signal — reads "mature" on day one. Adopt once the CLI/agent surface stabilizes and cadence makes the date the useful fact. Closest precedent: Helix.
SemVer, pinned major as counterv1.0.0 then minor each releaseThe VS Code pattern.Fine, but a 1.0.0 on a first cut over-promises a stable contract kolu doesn't yet make.

Recommendation: tag v0.1.0 now. It is the truthful description of a brand-new public release, keeps every door open, and the changelog — not the digit — is where you communicate user-visible breaks (a persisted-prefs schema change, a renamed keybinding, a CLI-flag change). Re-open this as a one-paragraph decision node the day a date-based number would genuinely serve users better than "how early is this."

If you choose CalVer anyway — the two traps
  • Never zero-pad. 2026.06.01 is unconditionally invalid SemVer (leading zeros) and npm/pnpm silently strip them. Use 2026.6.0.
  • Pick MINOR semantics and write it down. 1-based month (date-legible, gaps in quiet months) or a per-year release counter from 1 (no gaps, less calendar-legible). Don't leave it ambiguous — a script reading the SemVer-valid string can't tell a year rollover from a breaking change.

3 · Two channels for free

The single highest-leverage, lowest-cost insight: the flake-ref third component already accepts a tag. Stable-vs-rolling is pure tagging discipline — zero flake.nix changes.

edge / rolling→nix run github:juspay/kolu·tracks master HEAD
stable / pinned→nix run github:juspay/kolu/v0.1.0·exact tag, reproducible

To make the pinned channel an instant substitute instead of a from-source build, the release job pushes the tagged commit's outputs to the existing cache.nixos.asia/oss Attic cache — for all three platforms (flake.nix:20). README must document both channels; today it documents only rolling (README.md:23-27).

Rejected: a long-lived stable branch with devenv-style promotion (overkill for a solo maintainer — add only when you actually need to backport onto an old release); force-pushing a single rolling tag (Neovim's clobber problem broke source builders).

4 · Stamping the version through the existing rail

The version answers "which release"; the commit answers "exactly which build." Show both: v0.1.0 · 9f2c1ab. The plumbing already exists for the commit — KOLU_VERSION just rides alongside KOLU_COMMIT_HASH.

Source of truth: a checked-in VERSION file, not git describe.

The flake has zero inputs by design (a deliberate ~1.5s/input devshell budget), and github: tarball fetches don't expose revCount — so git describe is impossible in the hermetic sandbox. A VERSION file travels with the tagged source tree and reuses the exact existing chain.

./VERSION→ flake.nix→ default.nix --set KOLU_VERSION→ server/hostname.ts→ router.ts serverInfo→ contract.ts ServerInfo→ IdentityRail.tsx

Mirror the verified commit chain: flake.nix:27 commitHash = self.shortRev or self.dirtyShortRev or "dev" → default.nix:199 --set KOLU_COMMIT_HASH. Add a sibling version = builtins.readFile ./VERSION and a parallel --set KOLU_VERSION.

5 · The changelog: two artifacts, two audiences

Keep a Changelog's core argument is that raw commit logs make bad human changelogs. So produce two things and never conflate them:

Machine CHANGELOG.mdCurated release post
AudienceAnyone diffing "what exactly changed" — the audit trail.Users — "what's new and why I care."
Toolgit-cliff (single Rust binary, in nixpkgs, no Node/registry coupling).whatchanged skill → rewritten as prose in the Astro blog.
SourceAll commits since the prior tag.Merged PRs in the window (judgment: drops refactors/CI/docs).
CadenceRegenerated mechanically on tag.Hand-curated per release.

The commit-hygiene reality bites here. With only ~37% of history conventional, git cliff over all history dumps ~63% of commits into one ungrouped bucket — exactly the raw-dump Keep a Changelog warns against. Two honest moves:

  1. Seed CHANGELOG.md by hand for v0.1.0 — a single curated "Initial public release" entry. Do not ship a git-cliff dump of 600+ noisy commits as your launch changelog.
  2. Enforce conventional commits going forward — a commit-msg hook plus a justci recipe that rejects non-conventional subjects. From the first tag onward, git-cliff produces clean grouped output because the history past that point is clean.

Rejected: semantic-release (no-human-gate, couples version-bump to publish, monorepo support open since 2016); standard-version (deprecated — use commit-and-tag-version if ever needed); changesets/towncrier (their value is published-package semver propagation + per-PR fragment files; kolu publishes no app package and whatchanged already covers the human-intent layer).

6 · Cadence & the per-release ritual

Frequent releases reduce burnout only when each release is small and the mechanics are automated — the toil is per-release manual work, not frequency itself. So: a biweekly rhythm, not a deadline. A feature that isn't ready rides the next train. A quiet fortnight skips silently.

The per-release ritual (steady state — one boring command)

  1. On a green-CI master commit, run whatchanged over the window → draft the post.
  2. Write the one-paragraph lede; skim the auto-grouped features; cut what's noise.
  3. Capture one flagship demo (evidence skill).
  4. Decide any learning→rule promotions (see second-brain.html).
  5. Open the release PR (CHANGELOG + blog) → squash-merge → tag the merge commit → CI does the rest.

Rejected: weekly (Zed/VS Code cadence — sustained by teams); per-merge continuous (would make a blog-post-per-release crushing); fixed-date commitments (the open-source burnout trap).

7 · Release → blog post → demo pipeline

Every stage already exists; the work is wiring them in order. Crucially — kolu is a browser canvas, not a TUI — so its own demos use the evidence skill (headless-Chrome Playwright → .webm → ffmpeg → GitHub-release host), which it already implements end-to-end. Reserve Charm VHS / asciinema only for demoing the CLI agents (claude/codex) running inside kolu's panes.

tag→ whatchanged→ blog draft→ evidence→ MP4/WebM + poster→ website/src/content/blog/*.mdx→ pages.yml→ kolu.dev

Post shape worth copying (Helix / Bun / Ghostty): lead with impact and the "why," embed the demo early, link every feature to its PR, end with a community CTA. Prefer MP4/WebM over GIF for self-hosted docs (10–40× smaller at better quality); reserve GIF for surfaces that can't embed <video> (the README, Slack). OG images come free from the blog's existing astro-og-canvas.

8 · The hard parts the happy path skips

The optimistic design glossed several failure modes. These are the ones that actually hurt.

Hotfix / security / revert — you do need to build from a tag
  • Hotfix: git checkout -b hotfix/v0.1.1 v0.1.0, cherry-pick the fix, tag v0.1.1. This means you build from a non-HEAD ref — so the "no stable branch needed" line is only true until the first hotfix. Branch from the tag on demand; you don't need a permanent stable branch.
  • Security: same path, but skips the biweekly train — define an out-of-band release trigger.
  • Reverting a bad release: tags are immutable, so you roll forward with a new tag, mark the bad GitHub Release as "yanked" in its notes, and overwrite/pull the bad cache artifacts. (HEAD~15 is literally a revert — this is a live scenario, not hypothetical.)
The changelog ↔ tag ordering paradox

The changelog references the tag, but to be inside the tag it must be committed before tagging. Resolution: /release writes CHANGELOG.md + the blog as a PR → squash-merge to master → then tag the merge commit. The tag contains its own changelog and master/tag never diverge. Never have CI commit back after tagging.

The first CI secret, and the cache-substitution check
  • Today the repo has zero CI secrets (pages.yml is secret-free). The release job needs an Attic push token (not the signing key — Attic signs server-side) in GitHub Actions secrets. Document its mint, least-privilege (push-only to oss), and that this is the repo's first secret.
  • After pushing, verify substitution on all three platforms (nix path-info --store https://cache.nixos.asia/oss .#default) before announcing — an incomplete push silently drops users to a multi-minute from-source build.

9 · Automate vs. keep manual — and split /release

Draw the automate/manual line at mechanical & idempotent vs. judgment-bearing. The happy-path design bundled both into one /release skill — that complects "produce the artifact" (deterministic, always identical) with "announce it" (judgment, sometimes skipped). Split them:

Part A — produce no humanPart B — announce human, skippable
On v* tag: build + cache tagged outputs (3 platforms), run git-cliff → CHANGELOG.md, cut the GitHub Release with a compare/vPREV...vNEW link.whatchanged → blog draft → evidence demo → publish. Each runs standalone, so a point release can skip the announce half entirely.

Where Part A lives: ci/mod.just is a push/PR DAG with no tag-trigger concept. A release-on-tag job is a different trigger model — it likely wants its own thin .github/workflows/release.yml (the repo's second workflow, beside pages.yml), not a justci recipe. Confirm before building.

A /release skill is still worth having as the orchestrator for Part B — mirroring how /do chains steps — but only if each half is independently runnable.

10 · The first release — concrete MVP checklist

Resist scope creep. Ship only this for v0.1.0; defer the /release skill, the "N behind stable" rail, and the second-brain reorg until the flow has proven itself once by hand.

  1. Decide the version (§2) → echo '0.1.0' > VERSION with a one-line comment on the scheme.
  2. Wire the rail: read ./VERSION in flake.nix; --set KOLU_VERSION in default.nix; thread through hostname.ts → router.ts → contract.ts → render v0.1.0 · <sha> in IdentityRail.tsx.
  3. Add pkgs.git-cliff + cliff.toml (git cliff --init keepachangelog, with a catch-all parser so stray commits don't error).
  4. Seed CHANGELOG.md by hand — one "Initial public release" section (not a 600-commit dump).
  5. Add the conventional-commit gate (commit-msg hook + justci recipe) so future git-cliff output is clean.
  6. Write the launch post in website/src/content/blog/2026-06-launch.mdx; embed one evidence demo.
  7. Confirm all three platforms green, then git tag -a v0.1.0 + push.
  8. Cut the GitHub Release by hand the first time: gh release create v0.1.0, trim to highlights.
  9. Update README to document both channels.
  10. Verify the tagged build shows v0.1.0 · <sha> and that the pin substitutes from cache on all three platforms.

Per CONTRIBUTING, a user-facing change wants a merged proposal first. The release process is exactly such a change — and notably no proposal has ever merged here (only the template exists). Writing docs/proposals/0001-release-process.md is both the honest first step and the project's chance to exercise its own review loop. Acknowledge the solo-maintainer smell of self-accepting the first one.

12 · Open decisions

  • Version scheme — v0.1.0 recommended; lock it before tagging.
  • Part-A home — thin .github/workflows/release.yml vs. a tag-triggered justci recipe. Confirm whether justci can be cleanly tag-triggered.
  • Preview channel? — two channels (edge/stable) recommended now; add a Zed-style middle "preview" only if real users ask for a buffer.
  • "N commits behind stable" rail — defer past v1. It needs an unauthenticated GitHub compare call (rate-limited 60/hr/IP), has no actionable lane for rolling nix run users (no persistent install), and the home-manager package option has no default. Bolting a network freshness check onto the local build-identity rail complects two volatilities — make it a separate later proposal.
  • Leaf packages — @kolu/log, @kolu/html-escape: if ever published to npm they need independent SemVer, separate from the app's version. Out of scope until then.

ChromeBar identity rail — srv · pty A2 · deliverable #3

UI prototype for the consolidated connection + build/commit readout introduced in R-4 phase A2. Built in its final two-column shape from day one — srv (the server you're connected to) and pty (the pty-host serving your terminals) — even though in A2 they're the same process. That coincidence is the point: it's the live proof A2's identity plumbing works, and it means phase B only has to let the columns diverge, never re-lay-out. Companion to remote-terminals.pty-daemon.html; rendered with the live dark-theme tokens from packages/client/src/index.css and the real ChromeBar.tsx layout.

scope: client rail + server proxy of system.version.identity · absorbs the standalone WebSocket dot · status: A2 not yet built · theme: dark

The design call (revised). The R-4 plan originally deferred the daemon chip to B, reasoning that an always-green "daemon connected" chip would lie about an absent daemon. But in A2 the pty-host isn't absent — it's in-process. So a two-column srv · pty readout where the columns coincide is the literal truth, not a lie. Better still, it's the cleanest acceptance signal A2 could have: the srv column is the server's own commit (from the existing server.info), the pty column is sourced from the new system.version.identity field A2 adds and the server relays. If they don't match in A2, the wiring is broken. And since the WebSocket status dot already is "srv liveness," it folds into the srv column rather than floating beside it as a bare dot.

What's genuinely inert until B: only the divergence — the outdated / dead states — because nothing can diverge from itself. Those branches get wired in A2 (the rail is divergence-capable) but cannot fire until pty is a separate surviving process.

Why both columns in A2

It's the acceptance test

A one-sided readout would show the server's commit and call it done — exercising none of A2's actual wire work. The two-column rail forces the whole path to light up: server boots → fetches ptyHostClient.surface.system.version() → relays .identity onto server.info → client renders the pty column. The columns matching is the green test; a mismatch in A2 means the plumbing B depends on is broken.

The dot was already "srv"

The standalone WebSocket dot represents the client↔server link — i.e. srv liveness. Consolidating it into the rail removes a redundant indicator and gives pty a parallel slot for its liveness (in-process now, daemon-handle in B).

How it's wired (A2)

The rail is thin presentation, but the data takes three hops — because the client speaks kolu-server's surface (kolu-common/contract over the WebSocket), not the pty-host's. So A2 adds identity to system.version, kolu-server reads it once at boot and relays it onto its existing server.info (the same readout that already carries serverProcessId), and the rail renders both columns from one client fact.

// 1. pty-host contract — additive, OPTIONAL field, NO version bump
//    packages/pty-host/src/ptyHostSurface.ts
PtyHostIdentitySchema = z.object({
  staleKey:        z.string(),   // nix hash of the @kolu/pty-host source closure
  navigableCommit: z.string(),   // git commit SHA (nix-baked; no off-nix mode)
})
SystemVersionOutputSchema = z.object({
  contractVersion: z.string(), pid: z.number(), startedAt: z.number(),
  identity: PtyHostIdentitySchema.optional(),  // ← optional so a B-phase
})                                              //   older daemon still validates

// 2. kolu-server RELAYS pty identity onto its own surface, beside its OWN
//    commit. packages/common/src/contract.ts + packages/server/src/router.ts
ServerInfoSchema = z.object({
  // …existing fields (serverProcessId, …)
  commit:  z.string(),                         // srv column (already baked)
  ptyHost: PtyHostIdentitySchema.optional(),   // pty column (relayed)
})
//    server.info handler fills ptyHost once at boot (in-process = always fresh):
//      ptyHost: (await ptyHostClient.surface.system.version({})).identity

// 3. client — one rail component, both columns, reused by B for divergence
//    packages/client/src/ui/IdentityRail.tsx   (rebuilt fresh; #1034's was removed)
const info = await client.server.info();        // the existing one-shot
<IdentityRail
  srv={{ commit: info.commit, live: wsStatus() }}              // dot ← WS
  pty={{ commit: info.ptyHost?.navigableCommit,
         build:  info.ptyHost?.staleKey,
         live:  "inprocess" /* B: derive connected|outdated|dead */ }} />
//    commit link → github.com/juspay/kolu/commit/<FULL sha>, display sha.slice(0,7)

Three constraints baked in. (1) identity is optional on the wire so a B-phase older daemon isn't force-restarted just to add a diagnostic; PTY_HOST_CONTRACT_VERSION stays 2.0. (2) currentBuildId/currentCommitHash re-export as values, not import type — the regression that collapsed the typed client to unknown. (3) The commit href is the full SHA (display slice(0,7)) — matching the recovered 3fd7ea63 renderer, avoiding ambiguous-prefix edge cases.

Decisions (resolved with the maintainer; the load-bearing items came out of the A2 readiness map):

  • Coincidence rendering — DECIDED: always two explicit columns + the ≡ in-process tag. Verbose/explicit is favoured until the feature stabilizes; no collapse-when-equal.
  • Labels — DECIDED: srv / pty.
  • Identity values — DECIDED: nix is first-class, no fallback. KOLU_PTY_HOST_BUILD_ID (staleKey) and KOLU_COMMIT_HASH (commit) are both nix-injected on the koluBin wrapper. Kolu and the pty-host never run outside nix, so there is no dev-derivation, no placeholder dance, and no || "" softening — the values are always present.
  • Closure scope — DECIDED: no severing, no excluded deps. The staleKey is the nix hash of the pty-host package's own source closure (packages/pty-host/src/*.ts minus tests + package.json). That is naturally tight — a provider-DAG change in kolu-common doesn't touch these files, so it can't over-prompt — and complete for the package's own wire+behaviour. The build-time test guards the one real regression: a wire/behaviour dependency landing outside the hashed package (the #1034 mis-scope). It walks index.ts's transitive imports and fails if any reached module is neither in packages/pty-host/src nor on a small allowlist of stable framework/leaf deps (zod, @kolu/surface, the kolu-common leaf, node-pty/@xterm via the pinned package.json). Every dependency is kept; the test merely notices when the closure grows a new external wire edge.

Fresh approach: phased /lowy + /hickey

Everything below this section is the as-built record — the production failure, the hard constraints, the salvage of #1034. This section is the plan now being executed — A1 (#1055) and A2 (#1063) have landed; B — the daemon, the one-way door — is all that remains. It began as the counterfactual: if we discarded that PR and built R-4 afresh knowing all of it, how would we decompose and phase it? It is the output of a fresh /lowy (Lowy / Parnas volatility-based decomposition) and /hickey (Simple Made Easy) pass over the whole feature — every user-facing behaviour, the correctness semantics, and the four production failure modes — taking nothing about the shipped decomposition for granted.

The thesis. Decompose by concept, not by package or process-lifecycle; rank the concepts most-stable → most-volatile (the axes below). But the build order is set by a different fact: the process decouple is the only one-way door — the single step that changes runtime topology (IPC, a surviving process, single-instance, reattach), so the only one that's genuinely risky and hard to bisect. Everything is either before the door or part of it. So the work collapses into a low-risk in-process foundation — A1 (the structural refactor) then A2 (the build/commit identity + the staleness-key hash, isolated for its own review) — whose only job is to shrink the door, and the door itself (B), shipped with its entire safety and honest-state surface.

What's allowed to be a separate phase is decided by hazard, not by module or convenience. The door creates the hazards — a dead daemon (the empty-canvas lie), a wire-incompatible one (a bricked server), a live restart that can lose the session (#1034), an outdated daemon the user can neither see nor act on. Everything that defuses or honestly surfaces those hazards is hazard-coupled to the door and ships with it; nothing in that surface may drift to a follow-up — that drift is the bug #1034 paid for. Only what is neither hazardous nor meaningless-without-the-door can live before it, and that is exactly the in-process foundation. The one move that still matters most: identity/staleness is designed in as A2, before any daemon exists — a typed, build-enforced wire+behaviour closure — so a wire change can't escape the staleness key by construction; currency is then a read-site derivation (staleKey !== currentBuildId()), never a stored boolean. Lowy supplies the volatility ranking; Hickey supplies the within-unit shape ("restart" and "status" are compositions and single-owner projections, not god-functions).

Three units: the foundation (A1 · A2), and the door (B)

No feature flags — each unit ships complete to master and leaves Kolu working. The in-process foundation comes first, as two separately-reviewable PRs: A1 (the structural refactor — extract, contract, serve-in-process, DAG) and A2 (the build/commit identity + the staleness-key hash). Their only job is to make B — the one-way decouple — as small as a topology change can be. B then ships the decouple together with its entire safety and honest-state surface, because shipping it without the canvas / forced-restart / recovery / chip reintroduces the failures the postmortem below catalogues. The boundary rule is asymmetric: hazard-coupled work must stay together (so B is one PR despite its size, and nothing in it may drift to a follow-up), but hazard-free work may be split however helps review — which is exactly why A1/A2 is a clean cut (both are safe and in-process; neither gates the other's safety). Each unit names its LOC estimate (grounded in the analogous code on this branch; excludes one-time relocation), its risk of breaking Kolu, and its user-visible change.

A1 · In-process structural foundation ~720 + ~450 relocated · L risk: low landed #1055

Naming update (#1059, landed): the in-process serving below is now servePtyHost(deps) (transport-agnostic, returns the router) + directLink(router) (the no-wire member of the surface link family — websocketLink / stdioLink / directLink). createInProcessPtyHostClient = directLink(servePtyHost(deps).router); inProcessSurfaceClient was dissolved. Encapsulates. The three structural moves, all in-process and identity-free. Why low-risk and high-leverage. Nothing changes runtime topology, so the full feature suite is the safety net. And by serving the contract in-process, two of the door's scariest surfaces are absorbed here: the contract boundary itself (the consumer speaks ptyHostSurface, not createPtyHost, so it survives the transport swap untouched) and the sync→async spawn race (spawn becomes an async call, so the ready-gate on attach/write is proven before any socket exists). Serialization is the one door-surface A1 does not pre-absorb — the identity link is plain composition with no wire; B's socket adds it.

Review focus. "Nothing changed behaviourally." A big, mostly-mechanical diff (a package move + a transport-shaped refactor) where the question is parity, caught by the full feature suite.

Verify. The full terminal / git / agent feature suites pass unchanged; the in-process contract round-trips (the version handshake is trivially compatible); the DAG compiles with a stub channel set and no PtyHost import, and runs green against an async channel stub.

Cutover (one-time, observed in prod 2026-05-30). Deploying A1 over a running #1034 daemon does not reap it: that daemon survives systemctl --user restart kolu in its own transient cgroup (its whole purpose), but A1 has no daemon client, so it is left orphaned — still holding live PTYs the in-process server can neither see nor reattach (on the prod cutover: 4 running agent sessions + 1 idle shell). So the A1 cutover carries a manual migration step: systemctl --user stop kolu-pty-host.service to reap the orphan + its cgroup. A clean install has no prior daemon, so this is purely a #1034→A1 migration artifact, not part of the steady state.

Why it's all one unit, not a door plus follow-ups. Every bullet after Topology is hazard-coupled to it: the decouple is what creates the dead / incompatible / outdated / restart-loss states, so shipping the door without the canvas, the forced-restart, the recovery mechanism, or the chip reintroduces exactly the postmortem's failures — that drift is the seam #1034 fell through. The honest-state surface is also one concept with one owner: you cannot ship half a chip (dead but not outdated). So it lands whole.

Builds on. All of A, and only A's published boundaries — the contract (now over a socket), the identity (currency derives from staleKey), the in-process-proven DAG (now fed via bridgeStream), and the already-absorbed async-spawn gate. User-visible change. Terminals survive a kolu-server restart / deploy (scrollback + git/agent context intact), and the daemon-health surface comes alive: the srv · pty rail's pty column starts diverging (outdated / dead), plus the update-pending nudge, the honest degraded canvas, and the restart dialog.

Why this is the risk spike — and can't be smaller. The core terminal I/O path now crosses a socket to a separate surviving process; survival, single-instance, reattach, and the async race are new failure surfaces. It's the only step where "type into a terminal" can break, and it can't be reordered because every part of it operates on the daemon. A's job was to shrink it to a transport swap plus survival; it cannot be split further without shipping a known-unsafe intermediate — a door with no honest state or recovery, which is the #1034 regression.

B stays one PR despite its size — hazard, not convenience — built as ordered commits (topology → reattach → recovery → honest-state UI) that land together because no part of its safety surface may ship without the rest. Where Lowy and Hickey disagreed

  • Health cell vs provenance (in B). Lowy would tolerate keeping the identity pair on the status cell (one wider value vs an extra round-trip); Hickey insists on splitting (different concepts, change-rates, consumers). Resolved: split, but provenance is a one-shot buildInfo RPC into a signal — not a streaming cell — so the extra wire surface is a single connect-time call, satisfying Lowy's own "keep the cell minimal" warning too.
  • Recovery as one mechanism (in B). Lowy wants the restart sequence encapsulated once; Hickey warns that forcing the forced-restart (nothing to preserve, PTYs already doomed) and the user-restart (session must survive) through one body risks complecting two failure semantics — easiness masquerading as simplicity. Resolved: one composed mechanism whose step-values parameterise what-to-preserve (captureSession returns an empty context on the forced path) — one ordering owner, no tangled body.
  • Identity shape. Both agree the key-vs-commit value split is correct; the residual tension is encoding + storage. Resolved: PtyHostIdentity is the single emitted value (Lowy's receptacle); currency is derived from its staleKey at the read site (Hickey's derivation); the browser receives only the readout projection, so identity-for-staleness never leaves the server.
  • Live currency's new transition. A read-site derivation introduces a connected→outdated mid-session flip the frozen design never had. Resolved: B scopes currentBuildId as memoised / process-stable and adds the two-snapshot test; a genuinely runtime-changing key is deferred to R-3 with its own UI-transition test, not silently assumed.

The risk profile

Two risk levels: A1 and A2 are low — byte-identical or additive, user-invisible, the full feature suite as their net; B is high — the one-way door. There is no middle. The whole point of A1 and A2 is to be the cheap, confidence-building work you land first so that B, when it comes, is the only variable in play and arrives against an unchanged-behaviour codebase that already speaks the contract. B is an irreducible spike — with no flags, moving PTYs into a surviving process is one atomic step — and it carries its entire safety + honest-state surface, because every part of that surface is hazard-coupled to the door (a dead / incompatible / outdated / restart-loss state the door itself creates). You shrink everything around the door; you do not pretend the door is small, and you do not let its safety surface drift out.

This split is derived from the end-state architecture, not from how the work actually shipped. For reference, the incident-driven path that was taken — extract → parameterise → daemonise → re-cut the UI after a production failure — is the as-built record below; its defining mistake was letting the door's safety surface (honest state + recovery) drift into a follow-up, which is the seam #1034 fell through. Here that surface is, by the hazard rule, part of B; and identity/staleness is a first-class boundary (A2), build-time enforced, so the mis-scoped-key and over-prompting failures are unit tests on day one rather than late retrofits.

Net. Three units, no feature flags, split by the one-way door. The in-process foundation — A1 (extract the primitive · define + serve the contract in-process · decouple the DAG) and A2 (the build/commit identity + the staleness-key hash, carved out for focused review) — is low-risk, user-invisible, and exists only to shrink the door. B — the decouple ships the daemon complete: topology, reattach, currency, the full recovery mechanism (including the #1034 fix), and the entire honest-state UI — because every piece is hazard-coupled to the door and none may drift out. The boundary rule: what's allowed to be a separate phase is decided by hazard, not by module or convenience. The deepest things already proven right — reattach-by-id collapsing two identities into one, the zero-provider survivor, getDaemonHandle refusing a dead client, the metadata write-fence partition — carry over verbatim.

⚠ Production failure (2026-05-30) — the R4c-UI restart that lost a session

The first R4c-UI build (the build-mismatch "update pending" nudge + a Restart local PTY daemon command — #1034) shipped to production (Linux/systemd). The nudge fired correctly on a deploy; clicking restart then destroyed a live 20-terminal session and could not bring the daemon back. macOS was unaffected. Timeline from the user-service journal:

08:26  terminal created, total:20            ← 20 live terminals (Claude Code agents)
08:29  "restarting local PTY-host daemon (user-triggered)"
08:29  "killing all terminals" count:20       ← killAll drains ALL 20 first
08:29  daemon socket closed / old daemon SIGTERM
08:29  (old daemon: 13.5h old, 25G RAM / 40.6G swap — box thrashing)
08:29  "Daemon failed to start: socket did not appear within 30s"  ← RESPAWN TIMED OUT
08:31  new daemon finally binds (~2 min after SIGTERM)

Net effect: killAll ran before the new daemon was confirmed up; the respawn timed out at 30s while the overloaded box took ~2 min to release the old daemon's pid-file + socket. The user was left with zero terminals and a dead daemon handle, and the canvas rendered empty — indistinguishable from "you have no terminals." The 20 PTYs (running agents + scrollback) were gone; restore re-created only fresh empty shells. No second daemon ever ran — single-instance held (pid-file O_EXCL + liveness probe, plus the systemd unit name); the failure was the respawn losing the race against the slow old-daemon exit.

Hard constraints for the redo

  1. Restart must be recoverable, never kill-then-pray. restartLocalPtyHostDaemon drained all 20 terminals before confirming the new daemon could bind; the respawn then failed with no rollback. A daemon restart inherently loses the PTYs (accepted, confirmed cost) — but the failure mode must not be silent loss. Snapshot the session first; after respawn, auto-reattach / offer restore; if respawn fails, surface a loud, recoverable degraded state with the session preserved — never an empty canvas.
  2. Wait for the old daemon to fully exit — the single-instance lock fights the restart. SIGTERM-then-immediately-respawn races the pid-file (O_EXCL + liveness probe) and the --unit=kolu-pty-host name; the new daemon can't acquire until the old one is gone. restartDaemon waited 5s for the socket and polled 30s for the respawn — both far too short. Wait on the actual process exit (pid gone via kill(pid,0)) before spawning, with a generous, load-aware ceiling (prefer event-driven waits over fixed polls).
  3. Timeouts must fit a loaded production box, not an idle dev one. 20 heavy PTYs tearing down + a tsx cold-start under swap pressure >> 30s. macOS passed only because it was fast. Every restart / spawn deadline needs real headroom.
  4. Never lie about state — show "Connecting…/degraded", not an empty canvas. A dead/absent daemon handle (getDaemonHandle() throwing) rendered as silent emptiness. It must surface as an explicit connecting / degraded UI state. The empty canvas is what made the user believe the session was permanently lost.
  5. A persistent PTY status indicator — always visible. The only daemon signal today is the amber nudge, shown only when outdated. There must always be a PTY status chip (distinct from the WebSocket dot) with explicit states — connected · connecting · degraded/dead · update-pending — so the daemon's health is always glanceable, exactly like the WS connection dot. (Being implemented onto #1034 as a first cut.)
  6. Key staleness on pty-host, not the whole binary. currentBuildId() keys on the /nix/store/<hash>-kolu-stamped path = the entire kolu build, so every deploy (even server/client-only, pty-host untouched) fires the nudge — and acting on it costs a terminal-losing restart for zero benefit. Hash the @kolu/pty-host source/closure (or bake a pty-host-specific rev) so the soft nudge only fires when a restart actually gains something. Wire-incompatible contract changes stay on the forced-restart path. (This trade-off was not surfaced at planning — it should have been.)
  7. Back up the session before a destructive restart. The saved session survived here only by luck (6 terminals on disk). Snapshot it to a backup before any kill-all / restart so a botched restart is recoverable. (Done — #1046 shipped Export/Import session commands (sessionTransfer.ts in packages/client/src/ — a ⌘K palette feature): a user-triggered backup/restore hatch for the SavedSession blob. The in-restart snapshot/auto-restore also landed with constraints #1–3. Issue #1040 closed.)
  8. Fix the preferences storm. During the churn the client emitted 200+ rightPanel size/codeTabTreeSize patches (bursts ~100–230/min) — a ResizeObserver-style feedback loop that saturated the server and contended with the autosave the restart relies on. Debounce / guard it (pre-existing, not introduced by #1034). (Filed as #1041.)

Execution postmortem — mistakes made building this PR (don't repeat them)

The Hard constraints above are about the product behaviour to build. This section is a different thing: a postmortem of the build process itself — every mistake the salvage of #1034 made and had to correct mid-flight (production scares, broken builds, duplicate artifacts, git mishaps). If you are re-cutting this PR, treat each as a pre-committed rule. Grouped by kind.

Architecture / scoping

  1. The staleness hash must cover the daemon's entire closure, not just packages/pty-host/. KOLU_PTY_HOST_BUILD_ID hashes only packages/pty-host/**, but the surviving daemon's wire contract (ptyHostSurface.ts + PTY_HOST_CONTRACT_VERSION), its entrypoint (agent/main.ts), and its startup utils (daemonUtils.ts, buildId.ts) all live under packages/server/src/ — outside the hash. So an Axis-2 wire-shape change does not flip outdated; it's caught only if you also remember to bump the contract version. Mis-scoped key = the exact silent-skew bug #6 was meant to kill. Fix = encapsulation, not a wider fileset glob: move the whole daemon closure into @kolu/pty-host so "the survivor" is the package and "hash the package" is correct by construction. A glob re-introduces drift the next time a daemon file lands under a new path. Verification test: "if I edit the wire schema, does the build id change?" — if no, the key is wrong. (Found in prod review; fixed in this PR — the closure now lives in @kolu/pty-host, bar the thin --stdio dispatch shim in index.ts which carries no wire shape and stays outside by design; see the follow-on work.)
  2. Two staleness axes — both must be independently detectable. Axis 1, build drift (pty-host source changed, wire still compatible) → soft "update pending" nudge, user restarts when convenient. Axis 2, wire-incompatible (schema shape changed) → forced restart at connect (output-validation throw → IncompatibleDaemonError), terminals lost. A schema change with no contract-version bump must still be caught — which only works once rule 1 is fixed (the hash covers the contract). Don't rely on humans remembering the bump.
  3. Additive wire fields are optional + no contract bump. A required new field makes a surviving (older) daemon's system.version fail the new server's schema validation → forced restart → terminals lost, just to add a diagnostic. commitHash was added optional for exactly this reason: the live prod daemon kept running; its pty readout simply showed the non-linked fallback until next restart. Reserve breaking (required) changes — and the PTY_HOST_CONTRACT_VERSION bump — for shapes that genuinely can't interoperate.
  4. Identity surfaced to the user must be the navigable one. The build-id readout first shipped the pty-host source hash (b72d6da) — not a git SHA, so it wasn't clickable to a commit. Bake KOLU_COMMIT_HASH (server + daemon) and link that; keep the source hash as the staleness key only. Don't conflate "the staleness key" with "the thing a human clicks".

Process / git discipline

  1. Typecheck every package before commit/push — never after. Two commits (3fd7ea63, b21404ab) shipped a broken server build because Edits had silently failed (file-changed-since-read) and the missing symbols weren't caught until a later sweep. After any edit, re-grep that the change actually landed, then run tsc --noEmit across common + server + client before committing. nix build + tsc tolerate import cycles that full CI (just check's Biome noImportCycles) rejects — so a clean nix build is not proof CI passes.
  2. Never git stash in this worktree. A stash/switch/stash pop dance to check whether a lint error pre-existed popped an unrelated stash into the tree and produced phantom merge conflicts (CanvasMinimap.tsx, Dock.tsx) that had nothing to do with the work. To check history, use git show <ref>:<path> or a throwaway git worktree, not stash.
  3. Resolve all conflict markers before committing a merge. The origin/master merge (#1046) was committed with unresolved <<<<<<< markers still in commands.tsx because one resolving Edit had failed on a string mismatch — which broke nix build in production with "Unexpected token". After resolving, git grep -nE '^(<<<<<<<|=======|>>>>>>>)' must return nothing before the merge commit.
  4. Confirm an Edit actually applied. Several edits silently no-op'd because the file had changed since the last read; the duplicate-line and missing-symbol fallout cost extra commits. Re-read or grep after editing; don't assume success.

Issue / artifact hygiene

  1. Don't double-create GitHub issues on a flaky tool. A buffering shell made gh issue create look like it failed; retries created six issues for two topics (#1040–#1045). Verify with gh issue list before retrying; if dupes happen, close the extras pointing at the kept one. (Cleaned up: kept #1040/#1041, closed #1042–#1045.)
  2. Never defer with a someday-issue. Every gap goes into this plan and is done in this PR after explicit approval — not filed and forgotten. (Exceptions are genuinely-separate, pre-existing work: the preferences storm #1041, which predates this PR.) Issues filed mid-flight as "follow-ups" (#1040 session-backup, #1047 build-id readout) were in fact pulled back into the PR and completed; the issues were closed, not left open.
  3. Don't trust your own tool transcript over ground truth. A glitching shell made a file look corrupted (index.ts appeared to start with a stray git diff line) and made committed work look uncommitted. When the terminal contradicts itself, confirm via an independent path (node -e read, a sub-agent, git show) before acting — do not "fix" a phantom.

The design (forward)

The long-lived survivor is @kolu/pty-host only — node-pty fds + the @xterm/headless mirror + the raw VT taps (cwd · title+fgPid · command-run · exit) + a unix socket. Everything volatile — the provider DAG (git context, agent detection, PR status, the most-edited code in the repo) — runs fresh in kolu-server, consuming the daemon's taps over the socket. That split is the whole point: a kolu-server restart (every deploy) re-runs the providers against the surviving PTYs, so detection is never stale, while the PTYs themselves persist. Only a change to pty-host's own (rarely-changing) wire contract forces terminal loss; a provider change (frequent) restarts the cheap in-server layer with PTYs untouched. This corrects #1031, which daemonized a survivor that held the providers and therefore served stale detection on every deploy.

Surviving a deploy — the cgroup mechanism spike-verified

On Linux/systemd, kolu-server spawns the daemon via systemd-run --user --unit=kolu-pty-host (gated on INVOCATION_ID), landing it in its own transient cgroup — so systemctl --user restart kolu (a deploy) does not kill it. A plain detached / setsid child does not survive on cgroup-v2: the default KillMode=control-group walks cgroup membership, not the session, so the child stays in kolu-server's cgroup and dies with it — the #1031 Linux failure. On macOS the detached spawn already survives launchd; off-systemd (dev) it's a plain detached child. Single-instance is guaranteed two independent ways: the systemd unit name, plus a pid-file (O_EXCL create + liveness probe) the daemon acquires on start.

⚠ Production failure (2026-05-30) — the R4c-UI restart that lost a session

The first R4c-UI build (the build-mismatch "update pending" nudge + a Restart local PTY daemon command — #1034) shipped to production (Linux/systemd). The nudge fired correctly on a deploy; clicking restart then destroyed a live 20-terminal session and could not bring the daemon back. macOS was unaffected. Timeline from the user-service journal:

08:26  terminal created, total:20            ← 20 live terminals (Claude Code agents)
08:29  "restarting local PTY-host daemon (user-triggered)"
08:29  "killing all terminals" count:20       ← killAll drains ALL 20 first
08:29  daemon socket closed / old daemon SIGTERM
08:29  (old daemon: 13.5h old, 25G RAM / 40.6G swap — box thrashing)
08:29  "Daemon failed to start: socket did not appear within 30s"  ← RESPAWN TIMED OUT
08:31  new daemon finally binds (~2 min after SIGTERM)

Net effect: killAll ran before the new daemon was confirmed up; the respawn timed out at 30s while the overloaded box took ~2 min to release the old daemon's pid-file + socket. The user was left with zero terminals and a dead daemon handle, and the canvas rendered empty — indistinguishable from "you have no terminals." The 20 PTYs (running agents + scrollback) were gone; restore re-created only fresh empty shells. No second daemon ever ran — single-instance held (pid-file O_EXCL + liveness probe, plus the systemd unit name); the failure was the respawn losing the race against the slow old-daemon exit.

Hard constraints for the redo

  1. Restart must be recoverable, never kill-then-pray. restartLocalPtyHostDaemon drained all 20 terminals before confirming the new daemon could bind; the respawn then failed with no rollback. A daemon restart inherently loses the PTYs (accepted, confirmed cost) — but the failure mode must not be silent loss. Snapshot the session first; after respawn, auto-reattach / offer restore; if respawn fails, surface a loud, recoverable degraded state with the session preserved — never an empty canvas.
  2. Wait for the old daemon to fully exit — the single-instance lock fights the restart. SIGTERM-then-immediately-respawn races the pid-file (O_EXCL + liveness probe) and the --unit=kolu-pty-host name; the new daemon can't acquire until the old one is gone. restartDaemon waited 5s for the socket and polled 30s for the respawn — both far too short. Wait on the actual process exit (pid gone via kill(pid,0)) before spawning, with a generous, load-aware ceiling (prefer event-driven waits over fixed polls).
  3. Timeouts must fit a loaded production box, not an idle dev one. 20 heavy PTYs tearing down + a tsx cold-start under swap pressure >> 30s. macOS passed only because it was fast. Every restart / spawn deadline needs real headroom.
  4. Never lie about state — show "Connecting…/degraded", not an empty canvas. A dead/absent daemon handle (getDaemonHandle() throwing) rendered as silent emptiness. It must surface as an explicit connecting / degraded UI state. The empty canvas is what made the user believe the session was permanently lost.
  5. A persistent PTY status indicator — always visible. The only daemon signal today is the amber nudge, shown only when outdated. There must always be a PTY status chip (distinct from the WebSocket dot) with explicit states — connected · connecting · degraded/dead · update-pending — so the daemon's health is always glanceable, exactly like the WS connection dot. (Being implemented onto #1034 as a first cut.)
  6. Key staleness on pty-host, not the whole binary. currentBuildId() keys on the /nix/store/<hash>-kolu-stamped path = the entire kolu build, so every deploy (even server/client-only, pty-host untouched) fires the nudge — and acting on it costs a terminal-losing restart for zero benefit. Hash the @kolu/pty-host source/closure (or bake a pty-host-specific rev) so the soft nudge only fires when a restart actually gains something. Wire-incompatible contract changes stay on the forced-restart path. (This trade-off was not surfaced at planning — it should have been.)
  7. Back up the session before a destructive restart. The saved session survived here only by luck (6 terminals on disk). Snapshot it to a backup before any kill-all / restart so a botched restart is recoverable. (Done — #1046 shipped Export/Import session commands (sessionTransfer.ts): a user-triggered backup/restore hatch for the SavedSession blob. The in-restart snapshot/auto-restore also landed with constraints #1–3. Issue #1040 closed.)
  8. Fix the preferences storm. During the churn the client emitted 200+ rightPanel size/codeTabTreeSize patches (bursts ~100–230/min) — a ResizeObserver-style feedback loop that saturated the server and contended with the autosave the restart relies on. Debounce / guard it (pre-existing, not introduced by #1034). (Filed as #1041.)

Execution postmortem — mistakes made building this PR (don't repeat them)

The Hard constraints above are about the product behaviour to build. This section is a different thing: a postmortem of the build process itself — every mistake the salvage of #1034 made and had to correct mid-flight (production scares, broken builds, duplicate artifacts, git mishaps). If you are re-cutting this PR, treat each as a pre-committed rule. Grouped by kind.

Architecture / scoping

  1. The staleness hash must cover the daemon's entire closure, not just packages/pty-host/. KOLU_PTY_HOST_BUILD_ID hashes only packages/pty-host/**, but the surviving daemon's wire contract (ptyHostSurface.ts + PTY_HOST_CONTRACT_VERSION), its entrypoint (agent/main.ts), and its startup utils (daemonUtils.ts, buildId.ts) all live under packages/server/src/ — outside the hash. So an Axis-2 wire-shape change does not flip outdated; it's caught only if you also remember to bump the contract version. Mis-scoped key = the exact silent-skew bug #6 was meant to kill. Fix = encapsulation, not a wider fileset glob: move the whole daemon closure into @kolu/pty-host so "the survivor" is the package and "hash the package" is correct by construction. A glob re-introduces drift the next time a daemon file lands under a new path. Verification test: "if I edit the wire schema, does the build id change?" — if no, the key is wrong. (Found in prod review; fix in this PR.)
  2. Two staleness axes — both must be independently detectable. Axis 1, build drift (pty-host source changed, wire still compatible) → soft "update pending" nudge, user restarts when convenient. Axis 2, wire-incompatible (schema shape changed) → forced restart at connect (output-validation throw → IncompatibleDaemonError), terminals lost. A schema change with no contract-version bump must still be caught — which only works once rule 1 is fixed (the hash covers the contract). Don't rely on humans remembering the bump.
  3. Additive wire fields are optional + no contract bump. A required new field makes a surviving (older) daemon's system.version fail the new server's schema validation → forced restart → terminals lost, just to add a diagnostic. commitHash was added optional for exactly this reason: the live prod daemon kept running; its pty readout simply showed the non-linked fallback until next restart. Reserve breaking (required) changes — and the PTY_HOST_CONTRACT_VERSION bump — for shapes that genuinely can't interoperate.
  4. Identity surfaced to the user must be the navigable one. The build-id readout first shipped the pty-host source hash (b72d6da) — not a git SHA, so it wasn't clickable to a commit. Bake KOLU_COMMIT_HASH (server + daemon) and link that; keep the source hash as the staleness key only. Don't conflate "the staleness key" with "the thing a human clicks".

Process / git discipline

  1. Typecheck every package before commit/push — never after. Two commits (3fd7ea63, b21404ab) shipped a broken server build because Edits had silently failed (file-changed-since-read) and the missing symbols weren't caught until a later sweep. After any edit, re-grep that the change actually landed, then run tsc --noEmit across common + server + client before committing. nix build + tsc tolerate import cycles that full CI (just check's Biome noImportCycles) rejects — so a clean nix build is not proof CI passes.
  2. Never git stash in this worktree. A stash/switch/stash pop dance to check whether a lint error pre-existed popped an unrelated stash into the tree and produced phantom merge conflicts (CanvasMinimap.tsx, Dock.tsx) that had nothing to do with the work. To check history, use git show <ref>:<path> or a throwaway git worktree, not stash.
  3. Resolve all conflict markers before committing a merge. The origin/master merge (#1046) was committed with unresolved <<<<<<< markers still in commands.tsx because one resolving Edit had failed on a string mismatch — which broke nix build in production with "Unexpected token". After resolving, git grep -nE '^(<<<<<<<|=======|>>>>>>>)' must return nothing before the merge commit.
  4. Confirm an Edit actually applied. Several edits silently no-op'd because the file had changed since the last read; the duplicate-line and missing-symbol fallout cost extra commits. Re-read or grep after editing; don't assume success.

Issue / artifact hygiene

  1. Don't double-create GitHub issues on a flaky tool. A buffering shell made gh issue create look like it failed; retries created six issues for two topics (#1040–#1045). Verify with gh issue list before retrying; if dupes happen, close the extras pointing at the kept one. (Cleaned up: kept #1040/#1041, closed #1042–#1045.)
  2. Never defer with a someday-issue. Every gap goes into this plan and is done in this PR after explicit approval — not filed and forgotten. (Exceptions are genuinely-separate, pre-existing work: the preferences storm #1041, which predates this PR.) Issues filed mid-flight as "follow-ups" (#1040 session-backup, #1047 build-id readout) were in fact pulled back into the PR and completed; the issues were closed, not left open.
  3. Don't trust your own tool transcript over ground truth. A glitching shell made a file look corrupted (index.ts appeared to start with a stray git diff line) and made committed work look uncommitted. When the terminal contradicts itself, confirm via an independent path (node -e read, a sub-agent, git show) before acting — do not "fix" a phantom.

The design (forward)

The long-lived survivor is @kolu/pty-host only — node-pty fds + the @xterm/headless mirror + the raw VT taps (cwd · title+fgPid · command-run · exit) + a unix socket. Everything volatile — the provider DAG (git context, agent detection, PR status, the most-edited code in the repo) — runs fresh in kolu-server, consuming the daemon's taps over the socket. That split is the whole point: a kolu-server restart (every deploy) re-runs the providers against the surviving PTYs, so detection is never stale, while the PTYs themselves persist. Only a change to pty-host's own (rarely-changing) wire contract forces terminal loss; a provider change (frequent) restarts the cheap in-server layer with PTYs untouched. This corrects #1031, which daemonized a survivor that held the providers and therefore served stale detection on every deploy.

Surviving a deploy — the cgroup mechanism spike-verified

On Linux/systemd, kolu-server spawns the daemon via systemd-run --user --unit=kolu-pty-host (gated on INVOCATION_ID), landing it in its own transient cgroup — so systemctl --user restart kolu (a deploy) does not kill it. A plain detached / setsid child does not survive on cgroup-v2: the default KillMode=control-group walks cgroup membership, not the session, so the child stays in kolu-server's cgroup and dies with it — the #1031 Linux failure. On macOS the detached spawn already survives launchd; off-systemd (dev) it's a plain detached child. Single-instance is guaranteed two independent ways: the systemd unit name, plus a pid-file (O_EXCL create + liveness probe) the daemon acquires on start.

History, summarized

  • R4a (#1023) + R4b (#1028) shipped clean as pure in-process refactors (byte-identical on screen).
  • Two prototypes seeded the design: #994 (the R-2 remote prototype) proved "providers in the agent", re-provisioned fresh per version; #1010 was a local PTY-only daemon — the thin-survivor model. The redo is their correct combination: #1010's thin survivor (pty-host only) + #994's providers-in-agent, but kept fresh in kolu-server rather than carried by the survivor.
  • R4c first attempt — #1031 — was dropped after a macOS prod failure: it daemonized the R4b in-process agent (pty-host plus the provider DAG), combining the two prototypes wrong, so the survivor served stale detection on every deploy — and the staleness check keyed on the inert pkgVersion="0.1.0" constant, so outdated was always false and a 20-hour-old daemon was reused across deploys. Two fixes fell out: split only pty-host into the survivor, and key staleness on real build identity.
  • The volatility insight (Parnas / Lowy — encapsulate what changes; surfaced by the whole-plan /lowy pass under agency#186): "no legacy agent" ⊗ "tmux feel" conflict only when the long-lived component's code changes, so the survivor must be the stable thin core (pty-host), never the volatile providers.
  • R4c redo — #1034 (this PR) — 2026-05-30 (the postmortem above): build-keyed staleness + the nudge shipped, but the restart was non-atomic and the staleness key was the whole binary (over-prompting). That drove this redo — the constraints are the result.

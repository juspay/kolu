Claude shows “running” forever — a phantom running_background from an orphaned launch marker

Diagnosis & fix plan · implemented in #1109 · introduced by #1015 · the background-task sibling of #1017 · verified against the live reload-html transcript, the on-disk task artifacts, and a 13-agent design/adversarial-verify pass

Shipped (#1109): the recommended fix — the pure deriveState narrowing (promote waiting → running_background only for runId-bearing Workflow runs). The reported Bash/Agent orphan vanishes outright, with no threshold, no journal read, and no heuristic. Covered by flipped + new unit tests and an e2e scenario (background_bash) confirmed red-before / green-after. Not shipped: the watcher-side journal-liveness gate. An earlier draft added liveOutstandingTasks reading <session>/workflows/<runId>.json, but the dynamic-workflow investigation verified that path/shape doesn’t exist on disk — real journals live at <session>/subagents/workflows/<runId>/journal.jsonl as an event log, and workflow-level status is a transcript fact, not a journal fact. Gating on the non-existent snapshot dropped every legitimate live workflow before deriveState, regressing the running indicator, so it was removed. The Workflow-orphan-after-restart residual (below) is therefore not closed here; it belongs with the dynamic-workflow reader repoint, where the journal mtime can be read at its real location.

Root cause: the waiting → running_background promotion is unconditional. A background launch marker with no matching completion is treated as “a task is still running” — forever — and the completion can never arrive once the Claude process that launched it is gone.

deriveState promotes waiting to running_background whenever outstandingBackgroundTasks(lines) is non-empty core.ts:383-386, with no liveness or staleness gate. outstandingBackgroundTasks reconciles launched − completed purely from JSONL text markers core.ts:441-487. The completion <task-notification> is only ever written by the live Claude child when the backgrounded process exits. Claude auto-backgrounded a just ai::apm Bash command after a timeout, then the user restarted Claude — orphaning that child. Its completion can never be written, the launch line is a permanent record in the transcript, so the set stays non-empty and the spinner is permanent. Restarting Claude doesn’t clear it: the re-attached watcher re-reads the same stale JSONL and re-derives the identical verdict.

The report, reproduced

How the promotion is wired

The state derived from the transcript flows, unchanged, all the way to the spinning pip. Nothing on this path consults wall-clock time or process liveness:

tail 256 KB of JSONL→ outstandingBackgroundTasks→ [bi8olsr8z] (never completes)→ deriveState→ waiting → running_background
running_background→ isWorkingState agentState.ts:41-49→ working bucket dockModel.ts:195-208→ working pip pipVariant.ts:23-29→ animate-spin RowPips.tsx:143-144

The promotion exists for a real feature — dynamic-workflow fan-out: when the agent launches a background task and yields its turn (end_turn) while genuinely busy-waiting on it, the bare waiting would wrongly read as idle. The promotion corrects that. The defect is that it has no exit condition other than a completion marker that an orphaned task can never produce.

Provenance — the PR that introduced it, and the issue it already predicted

This was introduced by #1015 — “Detect Claude Code’s running-in-background state for dynamic workflows” c1e8613b · 2026-05-28, the same commit that added running_background, outstandingBackgroundTasks, and all three launch-marker regexes. Two facts from its own history shape the fix:

  • The backgrounded-Bash/Agent promotion was deliberate, not incidental. #1015’s message: “The Bash/Agent coverage was added after dog-fooding caught a session busy-waiting on backgrounded CI still reading as waiting.” So promoting a bare backgrounded Bash to running_background is an intended, field-validated behavior — which is why the “narrow the trigger” alternative below is a genuine product regression, not a free simplification.
  • #1015 predicted this exact bug class and filed it as #1017 — still OPEN. Its follow-ups list: “#1017 — an abandoned session with a stale trailing user/notification entry reads as running (needs an mtime/liveness heuristic).” #1017’s title is “Stale ‘running’ pill: abandoned session… (mtime-staleness heuristic).” Our bug is the background-task manifestation of #1017.

The twist that makes this its own issue. #1017’s “Proposed fix” already sketches the mtime-staleness heuristic — but its open questions carve out one explicit exception:

“Interaction with running_background (the new state from #1015) — a backgrounded task legitimately keeps the session ‘working’ while quiet, so staleness must not override an outstanding-background-task signal.”

That assumption — the background-task signal is trustworthy, so exempt it from staleness — is precisely what our bug breaks. Here the outstanding-background-task signal is itself the stale thing. So the fix isn’t “add staleness around the promotion” (which #1017 forbids); it’s make the background-task signal staleness-aware from the inside — push the decay into outstandingBackgroundTasks so the set it returns is already self-expiring. That reconciles the two: #1017’s outer heuristic can still defer to a fresh outstanding task, because the task stops being “outstanding” once it goes orphaned/talked-past.

There is also a decided precedent and a north star:

  • Precedent — #1019 (“Settle interrupted (Esc) Claude sessions to idle, not a phantom spinner”) closed the sibling bug #1018 with a structural transcript marker (isInterruptMarker → waiting core.ts:244,361-364) rather than a timer. That is exactly the shape of our human-turn guard — prefer a structural signal in the transcript over a wall-clock threshold where one exists.
  • North star — #1011 (“Structured agent-status side-channel via Claude Code hooks”, OPEN) would deliver exact SubagentStart/Stop + Notification signals and moot every transcript heuristic here. Until it lands, the transcript is the only source of truth and these vetoes are the honest best-effort.

Four fix candidates — and why two are traps

Four independent designs were generated and each was handed to an adversarial verifier told to refute it against the real data. Two look obvious and are wrong on this exact transcript:

CandidateIdeaVerdict
Liveness gate
kolu owns the PTY		 Veto the promotion if the Claude process that launched the task is dead, via the session file’s (pid, procStart) against /proc. fails — 22 A restart reuses the same sessionId and JSONL, and the session file re-keys to the new, live foreground pid (verified: pid 2627339, startedAt 22:30:04Z, alive). So “is the session alive?” returns true and the spinner survives. The dead original launcher’s identity is recorded nowhere reachable. Linux-only besides.

poll the .output file		 Treat a task as dead if its tasks/<id>.output is stale / has no /proc fd-holder. fails — 22 The bug command was just ai::apm >/tmp/apm2.log 2>&1 — it redirected output away from .output. The file is 0 bytes, mtime precedes the launch, and no process ever holds it. The probe reads “dead” for live and dead alike, and would prematurely clear the spinner for any redirecting background command. No .pid/.status/.exit artifact exists.
Ordering guard
transcript-only		 Drop a task if a genuine human turn (a real user prompt, not a tool_result) appears after its launch marker. correct, but narrow — 62 Fixes this bug deterministically and is restart-robust (pure over the persistent JSONL). But as the sole fix it breaks a supported case: a genuinely-running backgrounded Bash CI run + an interleaved human prompt would be wrongly de-promoted index.test.ts:397-403. Adopted as a fallback, not the whole fix.
Staleness veto
transcript-relative		 Drop a task whose launch predates the transcript’s newest timestamp by more than STALE_BG_MS. correct & non-breaking — 62 22:29 − 20:52 = 96.7 min > threshold ⇒ dropped. Anchored to the transcript’s own newest line (not Date.now()), so it’s immune to clock skew and to attaching to a paused-but-live old session. breaksLegit = false. Weakness: the threshold is an arbitrary magic number, and a fully-quiet orphan (no further lines at all) never advances newestTs.

The two “obvious” process-liveness fixes are exactly the ones to avoid — they measure an available-but-wrong signal. The lesson: after a restart, Claude is alive yet idle; liveness can’t distinguish a legitimate busy-wait from an orphan. Only the transcript’s own record (later human turns; elapsed transcript time) carries the discriminating fact, and it survives the restart.

Recommended fix — only promote a backed outstanding task

Promote waiting → running_background only when an outstanding task carries a runId — i.e. a genuine Workflow fan-out, whose run journal kolu already watches on disk. A bare backgrounded Bash/Agent (runId: null) has no observable backing signal, so it no longer lights the pip. The reported bug — an orphaned backgrounded Bash — vanishes outright, with no threshold, no timestamp parsing, and no heuristic classifier.

This is the principled line, not merely the small one: kolu should claim “working” only when it can actually see the work. For a Workflow that evidence is its on-disk run journal (the dynamic-workflow plan tracks pointing the reader at its real location, <session>/subagents/workflows/<runId>/journal.jsonl). For a detached Bash command there is nothing to observe but a launch marker that is permanent by construction — which is precisely what spun forever. Removing an over-broad trigger beats bolting two heuristics onto it (Hickey: subtract, don’t add).

The change — one predicate at the promotion site

// packages/integrations/claude-code/src/core.ts — deriveState (383-386)
  let state = stateAndModel.state;
  if (state === "waiting") {
    const bg = outstanding ?? outstandingBackgroundTasks(lines);
-   if (bg.length > 0) state = "running_background";
+   if (bg.some((t) => t.runId !== null)) state = "running_background";  // Workflow runs only — Bash/Agent have no journal
  }

That is the entire behavioral change. outstandingBackgroundTasks stays a faithful “launched − completed” set; only the promotion policy narrows — the correct Lowy seam, because what shifts is “what counts as working,” which is deriveState’s concern, not the marker-reconciliation’s. The BackgroundTask.runId field that already distinguishes the two core.ts:397-400 does all the work; no new state, no new inputs.

What it costs, honestly — two departures from #1015’s behavior:

  • A backgrounded Bash/Agent the agent is genuinely busy-waiting on now reads as waiting — re-opening the exact case #1015 added Bash/Agent coverage for (“dog-fooding caught a session busy-waiting on backgrounded CI still reading as waiting”). Mitigant: a busy-wait that actively polls its output emits tool_use/thinking turns and already reads as working; only a pure end-turn-and-wait loses the pip. The assertion at index.test.ts:397-403 flips to expect waiting.
  • Residual — open, deferred. A Workflow (runId != null) orphaned by a restart can still stick: its terminal <task-notification> never lands, so it stays in the outstanding set. The natural authoritative signal is the run journal’s mtime (a live workflow rewrites it on every fan-out transition), but it lives at <session>/subagents/workflows/<runId>/journal.jsonl — verified by the dynamic-workflow investigation, not the <session>/workflows/<runId>.json snapshot an earlier draft assumed. So the orphaned-Workflow gate is correctly built only once the reader is repointed at the real layout (the dynamic-workflow Phase 1), not on a path that doesn’t exist. Out of scope for this phantom fix.

Net: fully fixes the reported bug (the Bash/Agent orphan) today with the narrowing alone, and leaves the rarer Workflow-orphan-after-restart to be settled with the real journal signal once that reader is corrected.

Alternative — the transcript-only veto (keep Bash/Agent spinning)

If we decide a detached Bash/Agent busy-wait must keep lighting the pip (preserving #1015’s behavior exactly), the fallback is to make the outstanding-set self-expiring instead of narrowing the trigger — a layered, transcript-only veto inside outstandingBackgroundTasks. It’s heavier (a magic threshold + a small classifier) but behavior-preserving for the live case.

// core.ts — outstandingBackgroundTasks (441-487), sketch of the layered veto
  const launched = new Map();      // taskId → { runId, index, atMs }   (was: → runId)
  const completed = new Set();
  let newestMs = null;             // newest timestamp across ALL entries (metadata too)
  let lastHumanTurn = -1;          // index of the newest genuine human prompt

  lines.forEach((raw, i) => {
    let entry; try { entry = JSON.parse(raw); } catch { return; }
    const ms = Date.parse(entry.timestamp ?? "");                 // NaN-safe
    if (!Number.isNaN(ms)) newestMs = Math.max(newestMs ?? ms, ms);

    if (entry.type === "queue-operation") { /* …completed.add(id)… */ return; }
    if (entry.type !== "user") return;
    if (isGenuineHumanTurn(entry)) lastHumanTurn = i;             // real prompt, not machinery
    for (const block of entry.message?.content ?? []) {
      const tr = toolResultBlock(block); if (!tr) continue;
      const taskId = firstMatch(BG_LAUNCH_RES, tr.text); if (!taskId) continue;
      launched.set(taskId, { runId: BG_RUN_ID_RE.exec(tr.text)?.[1] ?? null,
                             index: i, atMs: Number.isNaN(ms) ? null : ms });
    }
  });

  const out = [];
  for (const [taskId, { runId, index, atMs }] of launched) {
    if (completed.has(taskId)) continue;
    if (atMs !== null && newestMs !== null && newestMs - atMs > STALE_BG_MS) continue;  // staleness
    if (lastHumanTurn > index) continue;                                          // human spoke after
    out.push({ taskId, runId });
  }
  return out;

isGenuineHumanTurn reuses the existing isInterruptMarker core.ts:251 and toolResultBlock core.ts:203: a user entry is a real prompt only if it carries no tool_result block, isn’t an interrupt marker, and its text doesn’t start with < (filters injected <command-*> / <task-notification> strings). Both vetoes fail safe: an absent/unparseable timestamp ⇒ not-stale, and the classifier defaults to “machinery” on ambiguity, so older transcripts and every existing timeless fixture keep today’s behavior. Anchoring staleness to the transcript’s own newestMs (not Date.now()) — and counting timestamps on metadata lines like the post-restart pr-link — means a restart’s fresh writes advance the clock past the launch even when the user sends no new prompt, narrowing the quiet-idle gap that staleness-alone would leave.

Why this is the fallback, not the lead: it carries the STALE_BG_MS magic number and a heuristic classifier — accidental complexity the recommended narrowing simply doesn’t have. Take it only if the product answer to “should a detached Bash command light the pip?” is yes.

Test plan — close both coverage gaps

The bug shipped because neither test layer exercises a launch that never completes:

  • Unit gap. Every promotion test supplies a completion marker index.test.ts:389-431; none covers “launch with no completion that should eventually not be running.”
  • E2e gap. The running_background scenario mocks the final state rather than deriving it from JSONL claude-code.feature:72-76 — so a derivation bug is invisible to it by construction.

Unit — recommended fix index.test.ts

No fixture changes, no timestamps — the runId already present on every fixture does the work.

  • Reported bug: deriveState([bashLaunch('bi8olsr8z'), endTurn]) ⇒ waiting — a backgrounded Bash with no completion no longer promotes.
  • Flip the now-wrong assertion: the existing “promotes end_turn to running_background for a backgrounded Bash command” index.test.ts:397-403 becomes ⇒ waiting (rename to “does not promote a backgrounded Bash”).
  • Workflow still promotes: deriveState([bgLaunch('t1','wf_1'), endTurn]) ⇒ running_background (runId != null) — the legitimate fan-out is untouched.
  • Workflow completion still clears: [bgLaunch('t1','wf_1'), bgComplete('t1'), endTurn] ⇒ waiting.

E2e claude_code_steps.ts · claude-code.feature

  • Add “a backgrounded Bash launch does not spin”: buildTranscript claude_code_steps.ts:115-137 writes a Bash launch + end_turn, no completion; assert the dock pip is not the working/spinner variant RowPips.tsx:143-144. Drives the real watcher + deriveState claude_code_steps.ts:243 — this is what closes the e2e mock-vs-derive gap.
  • Keep the existing running_background scenario green, but make its launch a Workflow (carrying a Run ID line) so it proves the legitimate fan-out still spins.

Unit — only if you take the alternative veto

These cover the staleness + human-turn guards instead; they require extending the fixtures with an optional ts and a humanPrompt(text) helper.

  • Staleness — bug regression: deriveState([bashLaunch('bi8olsr8z', T), endTurn(T+31m)]) ⇒ waiting.
  • Staleness — legit preserved: deriveState([bashLaunch(id, T), endTurn(T+5m)]) ⇒ running_background.
  • Human-turn — fallback fires: [bgLaunch('t1','wf_1'), endTurn, humanPrompt('ok'), endTurn] ⇒ waiting (the restart signature, no extra time needed).
  • Human-turn — machinery rejected: an injected <task-notification> user string and an INTERRUPT_TEXT_PREFIX entry must not count — both stay running_background.
  • Fail-safe: the existing timeless tests index.test.ts:389-413 all still pass (absent timestamp ⇒ not-stale; no human turn ⇒ kept).

Open risks & residuals

If we ship the recommended fix (narrow the trigger)

  • Behavior reduction (known, accepted). A detached Bash/Agent the agent genuinely waits on no longer lights the pip — the case #1015 added on purpose. This is the deliberate trade: claim “working” only when kolu can observe the work. If field use shows the pure end-turn-and-wait pattern matters, it’s recoverable by the alternative veto, scoped to runId == null tasks only.
  • Workflow-orphan residual — open, deferred. An earlier draft added liveOutstandingTasks to gate the Workflow promotion on a fresh, non-terminal journal — but it read <session>/workflows/<runId>.json, a snapshot path/shape the dynamic-workflow investigation verified does not exist (real journals are an event log at <session>/subagents/workflows/<runId>/journal.jsonl, with no workflow-level status field). Against a real run the gate dropped every live workflow and regressed the indicator, so it was removed. The journal-mtime idea is still the right authoritative signal — it just has to read the real location, which is the dynamic-workflow reader repoint, deferred out of this fix.
  • Not a data-level fix. The launch marker stays in the transcript forever; only the promotion policy reads it differently. Any future direct consumer of outstandingBackgroundTasks sees the unfiltered set — intended, since narrowing is a promotion decision, not a claim that the task isn’t outstanding.

If we ship the alternative veto instead

  • The threshold is policy. STALE_BG_MS can’t distinguish an orphaned-and-talked-past task from a genuine >threshold run during which the agent services unrelated turns. Pick a value generous enough for real long CI/Workflow runs; keep it a single named constant. The human-turn guard mitigates but doesn’t eliminate this. (The recommended fix has no threshold at all.)
  • Quiet-idle residual. If a task is orphaned and the transcript then produces no further lines and no later human turn, neither disqualifier fires and the spinner persists until the next write — narrowed, but not closed, by counting metadata-line timestamps. A Date.now() anchor would close it but at the cost of clock-skew fragility and the deterministic-fixture test story.
  • Heuristic classifier. The human-vs-machinery rule keys off injected user strings starting with < or INTERRUPT_TEXT_PREFIX. A future Claude Code injected-entry format could weaken it. It only ever drops tasks (fail-safe for this bug), but keep it unit-pinned and in sync with the format.

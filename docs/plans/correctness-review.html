Kolu whole-codebase correctness review

A static correctness audit of the current codebase, focused on logic errors, false safety boundaries, lifecycle leaks, silent failure handling, and restore fidelity. This is not a style review.

Date: 2026-06-01 - Commit reviewed: 1a62eeeb - Method: repo map, targeted static scans, source verification against line anchors

Summary Scope Findings Triage order

Summary

Verdict: I found six correctness issues worth fixing. The high-severity terminal lifecycle leak is fixed by PR #1105, and the OpenCode silent-parse issue by PR #1108. The remaining open set is one high-severity path-authority bug and three medium-severity issues.

Severity Finding Primary surface Status
High Repo-root guard is lexical only; symlinks can escape the repo for reads, previews, and local diffs. kolu-git, iframe preview Open
High Terminal exit subscriptions leak forever for explicitly killed terminals. client terminal lifecycle, server events Fixed #1105
Medium PTY channel aborts leak subscribers when the abort fires between pulls. @kolu/pty-host Open
Medium Session restore drops persisted metadata and active selection for sub-terminals. client session restore Open
Medium OpenCode message JSON parse failure is silently treated as "no messages yet". OpenCode integration watcher Fixed #1108
Medium Terminal kill unregisters local state even when pty-host kill fails; the client then removes the UI anyway. server terminal backend, client terminal CRUD Open

Scope

I reviewed the major correctness-bearing paths across packages/common, packages/server, packages/client, packages/integrations, packages/pty-host, packages/shared, and packages/surface. The search emphasized file authority, streaming and event lifetimes, session persistence, terminal lifecycle, integration parsers, SQLite/watchers, and error fallbacks.

I did not run the product end-to-end for this audit. These findings are source-derived and should be paired with focused tests as each fix lands.

Findings

High Repo-root guard can be bypassed by symlinks

resolveUnder prevents lexical traversal like ../../etc/passwd, but it does not resolve symlinks. A repo path such as leak -> /etc/passwd passes the string check because the path still appears to live under the repo. The later filesystem calls follow the symlink.

packages/integrations/git/src/safe-path.ts:24-36 - path.resolve plus path.relative, no realpath
packages/integrations/git/src/browse.ts:49-68 - Code-tab file reads use the resolved absolute path
packages/server/src/iframePreviewRoute.ts:88-125, 150-171 - preview resolution then stat/readFile
packages/integrations/git/src/review.ts:219-267 - local untracked fallback diffs /dev/null against the resolved absolute path

Risk: the Code tab, binary preview route, and local diff fallback can read or render files outside the repository root. That is both a correctness bug and a security boundary failure for any untrusted workspace.

Fix: split lexical path normalization from filesystem authority checks. For any operation that reads, stats, or diffs an existing file, compare fs.realpath of the root and target before use. Reject targets whose real path is outside the real repo root. Keep the current lexical helper only for paths that may not exist yet.

Tests: add symlink-to-outside fixtures for readFile, preview serving, and local untracked diff. The expected result should be a rejected path, not the external file content.

High Explicit terminal kills leak exit subscriptions

Every terminal gets a per-terminal exit subscription rooted with createRoot, but the disposer is discarded. The server stream yields once only when terminalExit is published. Natural exits publish that event; explicit kills intentionally do not. The result is a root and event stream that wait forever after every explicit kill.

packages/client/src/terminal/useTerminals.ts:56-79 - subscribeExit creates a root and drops the disposer
packages/server/src/surface.ts:249-254 - event source yields only after a bus value
packages/server/src/terminalBackend/local.ts:539-550 - natural exit publishes terminalExit
packages/server/src/terminalBackend/local.ts:553-573 - explicit kill unregisters without publishing terminalExit
packages/client/src/terminal/useTerminalCrud.ts:163-170 - client removes UI after kill and swallows errors

Risk: terminal churn accumulates client roots and server event subscribers for terminals that are already gone. Over time, this can create memory growth and stale event channels.

Fix: make subscribeExit return a disposer and store it by terminal id. Dispose it after a natural exit, after an explicit kill, and whenever the terminal list removes the terminal. An alternative is to derive exit subscriptions from the terminal list with keyed cleanup.

Tests: cover explicit kill lifecycle with a mocked event stream or observable subscriber count: create terminal, subscribe, kill, then assert the per-terminal exit subscription is aborted.

Status - fixed

Implemented in PR #1105, "Fix exit-subscription leak on explicit terminal kills". The fix uses the finding's keyed-cleanup alternative: exit subscriptions are now derived from the live terminal list with mapArray in packages/client/src/terminal/useTerminalExits.ts - the same list-keyed lifecycle useTerminalMetadata already uses. Each terminal id owns a reactive owner that SolidJS disposes the instant the id leaves the list, so explicit kills, killAll, and natural exits all release the subscription with no per-call-site disposer map. The leaky createRoot in useTerminals.ts and the subscribeExit threading through useTerminalCrud / useSessionRestore are gone. Covered by useTerminalExits.test.ts (asserts the subscription is released the moment a terminal leaves the list, including killAll to empty).

Medium PTY channel abort can leak when no pull is pending

Channel.subscribe cleans up immediately when CLOSE is pushed into a pending next(). But an abort that fires between pulls only queues CLOSE. If the consumer never pulls again after cancellation, the subscriber remains in subs and the abort listener remains attached.

packages/pty-host/src/channel.ts:88-138 - abort handler calls sub.push(CLOSE)
packages/pty-host/src/channel.ts:95-99, 107-114 - cleanup happens only on the pending-next close path
packages/pty-host/src/channel.test.ts:75-95 - current abort leak coverage only tests a pending next()

Risk: cancelled attach/tap streams can leave dead subscribers behind if cancellation happens while the consumer is not actively awaiting the next value.

Fix: introduce a single finish() path for abort/close that removes the subscriber and listener immediately, resolves any pending next() as done, and prevents future buffered delivery.

Tests: add cases for "subscribe then abort before first pull" and "publish buffered value then abort before next pull"; both should reduce subscriberCount to zero immediately.

Medium Sub-terminal restore loses saved metadata and active selection

Top-level terminals are restored with their saved metadata, and the old id is mapped to the new id. Sub-terminals are recreated with only cwd. Their saved themeName, canvasLayout, panel state, intent, last activity, and agent command are ignored, and their old id is never mapped. A saved active terminal that points to a sub-terminal therefore cannot be restored.

packages/client/src/terminal/useSessionRestore.ts:247-290 - top-level terminals pass saved metadata and populate oldToNew
packages/client/src/terminal/useSessionRestore.ts:291-294 - sub-terminals pass only cwd
packages/client/src/terminal/useTerminalCrud.ts:151-160 - handleCreateSubTerminal accepts no initial metadata and returns no id
packages/common/src/surface.ts:318-327 - saved sessions persist metadata for every saved terminal

Risk: restoring a workspace with sub-terminals silently changes the saved UI and agent state. If the active terminal was a sub-terminal, the restore path falls back instead of reselecting it.

Fix: change handleCreateSubTerminal to accept InitialTerminalMetadata and return the new id. During restore, pass the same persisted fields used for top-level terminals, seed sub/right-panel state, populate oldToNew, and reselect the mapped active sub-terminal.

Tests: add restore coverage for a sub-terminal with a non-default theme/panel state and for activeTerminalId pointing at the sub-terminal.

Medium OpenCode message parse failures are silently treated as empty state

The OpenCode integration reads OpenCode-owned JSON from SQLite. If the latest message.data is malformed, parseMessageState catches the parse error and returns null. The watcher then logs the same debug message used for a legitimate empty session: "no messages yet".

packages/integrations/opencode/src/core.ts:299-320 - deriveSessionState returns null when parsing returns null
packages/integrations/opencode/src/core.ts:325-331 - JSON parse failure is swallowed
packages/integrations/opencode/src/session-watcher.ts:59-67 - null is logged as "no messages yet"

Risk: upstream schema drift or database corruption hides behind a normal empty-state log. The terminal badge can disappear or stale out without an actionable error.

Fix: return a discriminated parse result or pass logging context into parseMessageState. Treat malformed latest-row JSON as a warn/error condition distinct from "no row". Prefer preserving the last known state or publishing an explicit degraded state.

Tests: assert that malformed message.data logs a parse failure and is not reported as "no messages yet".

Status - fixed

Implemented in PR #1108, "Surface malformed OpenCode message.data instead of empty-state". The fix takes the finding's lighter alternative - pass logging context into parseMessageState rather than thread a discriminated result through the watcher - because the sibling getLatestAssistantContextTokens already logs the same blob's parse failure as an error, so consistency beat a new return type. The two null causes are now logged where each is actually known: parseMessageState logs malformed JSON as error ("opencode message.data parse failed", tagged with the session), and the genuine no-row "no messages yet" debug moved into deriveSessionState at the !row branch. session-watcher.ts no longer flattens both back into one ambiguous line. Covered by unit tests at the parseMessageState and deriveSessionState layers in index.test.ts - malformed data logs the parse error and never the empty-state line, an empty session still logs only the debug, and a valid-but-unmodelled role stays silent.

Medium Kill failures are hidden while local terminal state is removed

The server comment says kill awaits daemon confirmation so a failed kill cannot orphan the PTY. The implementation catches pty-host kill failure, logs it, and unregisters anyway. killAll drains every terminal after a failed killAll as well. The client then swallows kill errors and removes the local UI regardless.

packages/server/src/terminals.ts:97-100 - documented invariant says failed kill cannot orphan the PTY
packages/server/src/terminalBackend/local.ts:553-573 - single kill unregisters after catch
packages/server/src/terminalBackend/local.ts:576-588 - killAll drains after catch
packages/client/src/terminal/useTerminalCrud.ts:163-170 - client catches and removes terminal anyway

Risk: if the pty-host kill RPC or transport fails, Kolu can forget a still-running PTY and remove the user's UI handle to it. The current in-process host makes this less common, but the backend is already written around a daemon/socket confirmation model.

Fix: do not unregister on kill RPC failure. Either propagate the error and leave the terminal visible, mark the terminal as kill-failed while retrying, or unregister only after a natural exit/tombstone confirmation. For killAll, only drain terminals confirmed killed.

Tests: mock ptyHostClient.surface.terminal.kill and killAll to throw; assert registry entries remain and the client does not remove UI on a failed kill.

Triage order

  1. Fix path authority first. It is the only issue that crosses from correctness into a direct workspace boundary violation.
  2. Terminal subscription lifecycle is fixed. PR #1105 landed the list-keyed cleanup and regression coverage.
  3. Fix channel abort cleanup before daemon work expands. It is a small primitive with a large downstream blast radius.
  4. Fix sub-terminal restore and kill-confirmation behavior together with tests. Both are terminal lifecycle fidelity issues and should be covered by restore/CRUD scenarios.
  5. OpenCode parser errors are hardened. PR #1108 landed the malformed-JSON error path and the empty-vs-malformed log split, so future upstream schema drift surfaces instead of failing silently.

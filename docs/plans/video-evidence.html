Video evidence for PRs

Let /do attach video to a PR — not just screenshots — for changes about motion (animations, transitions, multi-step interactions). Shipped as three small pieces across three repos.

Problem

The ## PR evidence flow in .agency/do.md captures stills via the chrome-devtools MCP's take_screenshot, uploads them to a long-lived evidence-assets GitHub release, and embeds them inline. A still can't convey a transition or a multi-step interaction. We want the same agent-driven flow to produce video.

Two independent gaps: capture (the MCP wasn't recording video) and embedding (getting a clip to render in a PR comment).

Capture — the MCP already has it, gated

chrome-devtools-mcp ships screencast_start / screencast_stop (.mp4/.webm) via Puppeteer's page.screencast(), which shells out to ffmpeg. They are hidden behind --experimentalScreencast=true, and the kolu launcher didn't pass it or provide ffmpeg.

chrome-devtools-mcp build/src/tools/screencast.js:18-70 (tools); :53-66 (ffmpeg shell-out + ENOENT error); cli-options screencast flag.

Fix (upstream): the launcher at .apm/skills/nix-chrome-devtools-mcp/bin/serve now runs nix shell …#nodejs …#ffmpeg and passes --experimentalScreencast=true — ffmpeg and the tools resolve through Nix, no host install.
→ juspay/nix-chrome-devtools-mcp#2 (branch screencast). kolu pins it via apm.yml (juspay/nix-chrome-devtools-mcp#screencast) until merged; just ai::apm-update regenerates the vendored serve. Drop the #screencast ref after merge.

Embedding — the real constraint

GitHub auto-mounts a <video> player only for files uploaded through the web composer (a user-attachments URL needing a session cookie + CSRF — gh/PAT can't mint it). A git-committed or release-hosted raw. mp4 renders as a download link, and a hand-authored <video> tag is stripped by the comment sanitizer.

Proof it's the embed, not hosting: srid/ema's README plays an inline clip via a bare user-images.githubusercontent.com URL (a one-time human drag-drop). The repo also commits the same file at docs/static/ema-demo.mp4, yet the README does not use that committed file's raw. URL — because it wouldn't render a player. A dedicated mp4 repo lands on that same non-embedding raw. path.
→ srid/ema README.md:10 (bare user-images URL); git tree has docs/static/ema-demo.mp4.

Architecture — three repos

RepoRoleChange
juspay/nix-chrome-devtools-mcpCapture capabilityLauncher enables screencast + ffmpeg (PR #2).
juspay/video-evidenceShared player hostGitHub Pages evidence.html?repo=<owner/repo>&v=<file>, project-agnostic, org-allowlisted.
juspay/kolu (this PR)Consumerapm.yml pin + regen; .agency/do.md video procedure; this plan.
Key decision (per review): clips live on each project's own evidence-assets release — kolu's on juspay/kolu, exactly where screenshots already go. The video-evidence repo holds only the player, so it is reused across juspay projects with no per-project hosting and no git-blob bloat. One parametrized player (org-allowlisted repo) rather than per-project HTML pages — avoids multiplying artifacts.

Validated end-to-end

Before wiring the launcher, the embed path was proven in a real browser (chrome-devtools): the Pages player loaded a clip from the cross-origin release URL, reached readyState 4, played to completion, and seeked mid-clip — confirming HTTP range requests work on release assets. Test clip generated offline with nix shell nixpkgs#ffmpeg (the same ffmpeg-via-nix the launcher now uses).

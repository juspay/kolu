Detecting AskUserQuestion by scraping the xterm buffer

A spike for #905. The issue's fix is a Claude Code PreToolUse hook + sidecar — editing the user's ~/.claude/settings.json on every machine. This asks the alternative the title poses: light the "Awaiting input" badge by reading the rendered terminal, so the user configures nothing. Reviewed by an Opus /hickey ⇄ /lowy debate + a Löwy electricity-analogy pass — and then re-reviewed by an unbiased lowy run that overturned the separate-package decision: this belongs inside the claude-code integration, not a new package. All folded in; §9 has the record and the meta-lesson. seedling #905

The one-paragraph version, in Löwy's terms. The consumer-facing receptacle already exists: AgentInfo.state === "awaiting_user", which the badge, dock pip, and OS alert already plug into. Behind it sit interchangeable power sources for the one fact "the agent is awaiting the user": the live JSONL classifier (starved during the SDK's buffering window), this screen scrape, and #905's hook. So the work isn't new wiring for consumers — it's one more generator behind a socket that's already standardized, and the volatile part ("what a Claude prompt looks like on screen") is just more Claude-specific awaiting-user knowledge that belongs next to the JSONL classifier in the claude-code package — not in a new shared package. The badge never learns which generator powered it.

1 · Why JSONL polling is blind here

kolu derives every agent's state server-side by tailing the Claude transcript JSONL and classifying the newest message session-watcher.ts:224–274 → core.ts:307 deriveState. The classifier that produces awaiting_user is live, not dead — it already knows these two tools mean "needs the human":

// packages/integrations/claude-code/src/core.ts:231
const AWAITING_USER_TOOLS = new Set(["AskUserQuestion", "ExitPlanMode"]);
// :284 → classifyByAwaiting(awaiting, total) → awaiting_user   (the shared policy, see §5)

The catch, per #905: the Claude Agent SDK buffers the in-flight assistant message (carrying the tool_use block) in memory and flushes it to disk only after the user answers. During the wait the block isn't on disk, so the classifier is input-starved — deriveState reads the prior end_turn entry and reports waiting. schemas.ts:36–43 documents it: "…this case never fires under the current SDK. Fix tracked in #905."

So it's a starved source, not absent machinery. The destination (badge chain) is wired-but-unreached; the source classifier is live-but-starved. Once the scrape ships there are two live signals for one fact (three with the hook), so the plan states a composition rule — see §6. The receptacle is one enum value the whole attention chain keys off schemas.ts:54 · lifecycle.ts:16-17 classifyByAwaiting · AgentIndicator.tsx:26 · agentState.ts:23.

2 · The "xterm contents" already live server-side

kolu's PTY host keeps a headless terminal emulator per PTY, parses every byte through it, and exposes its contents — first-class RPCs ptyHostSurface.ts:214–225, re-surfaced by the server router as terminal.screenText / screenState router.ts:150–158:

// packages/pty-host/src/ptyHost.ts — :4 "…an @xterm/headless screen mirror…"
// :483  PTY data → headless.write(data, () => publish(data))  // buffer fresh before fan-out
getScreenText(id, startLine?, endLine?): string  // PLAIN TEXT, VT-parsed; the region gate (startLine,endLine) already exists — ptyHost.ts:97
  • Plain text out — the emulator already applied SGR/ANSI and resolved the grid; we never parse escape codes ourselves.
  • Always live, every PTY — independent of focus or mount state. TerminalHandle.getScreenText already abstracts local vs remote/SSH (#951) local.ts:231, so detection inherits that coverage for free.

Server-side, decisively. Reading this mirror covers background tiles, mobile (client xterm unmounted), and remote/SSH PTYs — none of which a client-side buffer read can promise. Client-side scraping is off the table; it buys nothing and loses coverage.

Three findings that shape the detector:

  • ExitPlanMode is rock-solid — three exact literals, byte-identical across 82 daily releases; as stable as a tool name. (And it moves on the same clock as the JSONL tool name — see §5.)
  • AskUserQuestion has thinner fixed chrome — its question + option labels are model-supplied, so detection leans on structure (the figures caret/markers + select footer). The figures lib ships an automatic ASCII fallback (❯→>), so glyph drift is a known finite mapping.
  • No box-drawing glyphs anywhere — Claude's prompt boxes use no ╭/┌ corners. The signature must not depend on borders.

Net: fragility is real but bounded and observable. A break is a discrete event (a specific literal changes), caught by a fixture test on the next Claude bump — not the "constant churn" the generic worry implies (zero churn in 82 releases).

4 · What dodging the hook buys — which generators are connected, not how the current is consumed

The hook isn't free: two stanzas in ~/.claude/settings.json (PreToolUse + PostToolUse), per machine and user, plus an unresolved install-UX decision the issue itself flags. The scrape writes zero files and reads a stream kolu already owns. Coverage the hook structurally can't match without N× setup:

The electricity correction (§9). Scrape and hook are interchangeable generators feeding one receptacle, not a "default" and an "upgrade." "Scrape is the zero-install default, hook adds robustness + clean question text" is a true statement about which generators are connected in a given environment — a deployment fact, never an architectural ranking. How the current is consumed (the awaiting_user badge) never varies; which sources are wired in does.

5 · Where it lives: a file in the claude-code integration, not a new package

The first draft proposed a top-level packages/xterm-scrape. An unbiased lowy re-review killed that (§9), and it's right. Kolu already has the volatility-correct decomposition for "is this agent awaiting the user," shipped and working — and a new shared package is not it.

The precedent already in the tree. The cross-domain policy — "awaiting iff every pending tool is awaiting-flavoured" — is one tiny agent-neutral primitive; the per-agent data (which tools count) lives inside each integration:

packages/integrations/anyagent/src/lifecycle.ts:13   classifyByAwaiting(awaiting, total)  // shared policy, names no agent
  ↑ plugged into by all three integrations, each owning its own data:
claude-code/src/core.ts:231   AWAITING_USER_TOOLS = {AskUserQuestion, ExitPlanMode}
codex/src/core.ts:327         AWAITING_USER_TOOLS = {request_user_input}
opencode/src/core.ts:218      AWAITING_USER_TOOLS = {question, plan_exit}

That is the textbook Parnas/Löwy cut, already done. Claude's on-screen prompt signatures are simply more Claude-specific awaiting-user data — so they belong next to AWAITING_USER_TOOLS, in the claude-code package.

Why a new package fails the electricity litmus

  • One domain, not many. Count independent consumers of a detectScreenPrompt: one (the Claude path). Its entire payload — fixtures, literals, a return type that names Claude's tools — is Claude knowledge. A top-level package would be a generic doorknob on a Claude room. classifyByAwaiting (names no agent, three consumers) is what real electricity looks like; this isn't that.
  • Same volatility, two transports. When Anthropic renames ExitPlanMode, the JSONL tool name (core.ts:231) and the on-screen literal move in the same release. They are one change axis (Anthropic's product), seen through two transports — so they belong in one package, with one fixture corpus and one CI-red on drift. Splitting by transport scatters one agent's knowledge across two packages.
  • It's a swappable stopgap. §6 plans to swap the scraper for #905's hook later. You don't mint a shared package for a generator you intend to replace — keep it local to the agent it patches, behind the merge seam, so the swap touches one integration.
  • Purity is a file boundary, not a package boundary. A new claude-code/src/screen.ts with zero node:*/SDK imports + its own screen.test.ts + fixtures gets identical purity, first-class fixtures, and loud-CI-on-drift — without a cross-domain package for single-domain content. (Today's core.ts is fs/SDK-heavy; the file split keeps the recognizer pure.)
  • Promote to anyagent only on real reuse. If a second agent grows a TUI scraper and a genuinely shared mechanic appears, the shared piece is a tiny region-gate / boolean-conjunction runner — never a signatures package. Codex's chrome isn't Claude's; the fixtures and literals don't transfer.
// packages/integrations/claude-code/src/screen.ts — Claude-owned, pure, zero node:*; sibling to core.ts
// per-tool declarative signatures (a stable loop over variable predicates — no per-tool branch):
//   ExitPlanMode    = exact-literal predicate (boolean-certain — moves with AWAITING_USER_TOOLS)
//   AskUserQuestion = structural: caret(❯/>) AND option/marker column AND select-footer literal
// boolean conjunction over a bottom-region gate (NOT a numeric score+threshold); stateless.
detectScreenPrompt(text): { tool: "ExitPlanMode" } | { tool: "AskUserQuestion", question: string | null } | null
// fixtures co-located: claude-code/src/__fixtures__/exit-plan-mode.2.1.159.txt, ask-user-question.2.1.159.txt,
//                      neg-scrollback-proceed.txt, neg-numbered-list.txt, neg-narrow-wrap.txt

Why this keeps fragility tolerable. Every supported Claude version drops a fixture; the test asserts each is detected and each negative rejected. When a bump changes a literal, CI goes red against the new fixture the moment we add it — silent failure (badge quietly stops) becomes a loud, located test failure in the one package that owns Claude.

6 · Wiring it in — at the providers seam, not in the watcher

  • Keep session-watcher.ts JSONL-only. Today it has zero pty-host imports — pure fs.watch + deriveState. It must not grow a pty-host client or a second clock. The recognizer (screen.ts) is a sibling pure file in the same package; the poll + merge happen one level up, at startAgentProvider's createWatcher/setAgentMetadataVia callback providers.ts:454-461, which already holds terminalId and the pty-host client and funnels every emitted ClaudeCodeInfo.
  • No pid→PtyId bridge. TerminalId == PtyId — the same UUID is passed to spawn({id}) and getScreenText({id}) local.ts:361-362, 231-232. There is no namespace to bridge; the screen is reachable directly by the terminalId already in scope, captured as a one-line getScreenText thunk.
  • Leave the shared AgentProvider.createWatcher contract UNCHANGED. Don't thread a screen handle through it anyagent agent-provider.ts:97 — that leaks pty-host into the agent-agnostic contract and forces Codex/OpenCode to stub an unused concept. Resolve the screen capability at providers.ts and hand it only to the Claude path.
  • One fact, stateless merge. At the seam, a pure transform on each emitted ClaudeCodeInfo: awaiting_user fires if any contributor says so; the scrape only promotes waiting/end_turn → awaiting_user (never demotes), filling the gap while the JSONL classifier is starved. The contributors are concordant by construction (same tool set). Write the precedence rule once as a named, commented mergeAwaitingInput; add a "message-flushes-mid-wait" fixture asserting the JSONL and scrape paths agree.
  • Recompute-don't-latch is structural. Because the merge is a stateless transform, "no box detected → input passes through unchanged" — no scrapeOverride variable, no revert step.
  • One lifecycle object owns the clock. The JSONL fs.watch is silent during the wait, so a poller samples getScreenText at ≈0.5–1s — but idle-gate + clock + teardown + merge are one object owned by the seam wrapper: started when state goes idle, stopped the instant it goes non-idle or the watcher tears down (tied to watcher.destroy).
JSONL fs.watch ──▶ deriveState(lines) ──▶ ClaudeCodeInfo (BASE — event-driven, no screen, no clock) │ providers.ts seam (startAgentProvider callback, terminalId in scope) │ state idle? lifecycle obj polls getScreenText(terminalId,-N,-1) ▼ mergeAwaitingInput(info, claude-code/screen.ts → detectScreenPrompt(text)) ← named, stateless │ promote waiting → awaiting_user iff a prompt is on screen ▼ setAgentMetadataVia ──▶ awaiting_user receptacle ──▶ badge

The receptacle is awaiting_user; the generators (JSONL base + Claude's screen promoter, later the #905 hook promoter) plug in behind the named merge rule. A hook becomes a swap inside the claude-code package, not a rewrite — the electricity payoff.

7 · Phase 0 — empirical capture done (static)

Captured from the on-disk binaries (results in §3): the exact ExitPlanMode literals, the figures caret + ASCII fallback, the absence of box glyphs, and byte-stability across 82 releases. One thing static analysis can't settle — confirm with a live capture before Phase 1:

Live confirmation (≈1 hr): run a real Claude session in a kolu PTY, trigger ExitPlanMode (and an AskUserQuestion if offered), and dump terminal.screenText. Verify: (a) the prompt lands in getScreenText; (b) main buffer vs alternate screen (?1049h exists in the binary; buffer.active tracks whichever is active, and alt-screen would remove scrollback false-positives); (c) row position relative to the cursor (sets the region gate's N). The dumps become the first claude-code/src/__fixtures__.

8 · Plan

  1. Phase 0 live capture → first fixtures.
  2. claude-code/src/screen.ts — Claude-owned detectScreenPrompt (boolean conjunction, per-tool signatures, discriminated-union return) + co-located fixtures/negatives. Pure, zero node:*; ship behind a green screen.test.ts first. No new top-level package.
  3. Server wiring at the providers seam — one lifecycle object (idle-gate + poll clock + teardown), the named stateless mergeAwaitingInput, a getScreenText thunk closed over terminalId. session-watcher.ts + the shared createWatcher contract stay untouched. Badge lights, self-clears.
  4. Tooltip enrichment (optional) — surface the parsed question (the union already carries it); the hook does this cleaner, so it can come via the hook path later.
  5. Scope decision (explicit): Codex/OpenCode TUI detection is not in this first PR. Each would get its own screen.ts in its own package if it ever needs one — and only then would a shared mechanic (if any) be promoted to anyagent.

9 · What the reviews changed

This plan went through three review passes, and the third reversed the second.

Pass 1+2 — Opus /hickey ⇄ /lowy debate (2 rounds, consensus) + a Löwy electricity pass. Produced the refinements now in §1–§6: override moved to the providers.ts seam (watcher stays JSONL-only); the pid→PtyId "bridge" deleted (TerminalId == PtyId); "dead code" corrected to live-but-starved with an explicit two-signals-for-one-fact merge; numeric score → boolean conjunction; per-tool signatures + discriminated-union return; recompute-don't-latch made structural; one lifecycle object for clock+gate+teardown; shared createWatcher contract left untouched. The electricity reframe gave the model: awaiting_user = receptacle, sources = interchangeable generators, "scrape=default/hook=opt-in" = a deployment fact.

Pass 3 — an unbiased lowy re-run reversed the package decision. Passes 1–2 gave the packages/xterm-scrape boundary a clean bill — because both lenses accepted the plan's self-description ("agent-agnostic, no claude import") instead of checking it against the codebase. A clean run, told to distrust the framing and read the source, found the decisive fact everyone missed: classifyByAwaiting in anyagent is already the cross-domain electricity, with per-agent tool data living inside each integration. So Claude's screen signatures belong in claude-code/src/screen.ts, not a new top-level package. §5 is the rewritten result.

The meta-lesson (worth carrying into /lens-debate, #1113). The orchestrated 2-round "consensus" laundered a biased premise into confidence: a leading prompt set the frame ("the package is agent-agnostic"), both lenses ratified the self-description, and consensus made it feel settled. A single clean, code-first run overturned it in minutes. Two concrete fixes for the electricity probe:

  • Count independent consumer domains — not "is it decoupled from one named module." One domain + N implementations is a parameterized single-domain utility, not electricity.
  • Search for an existing primitive that already encapsulates the volatility (here: classifyByAwaiting) before minting a new boundary.

11 · Recommendation

Build it — server-side, as claude-code/src/screen.ts (a file, not a package), merged at the providers seam — as the zero-config default.

  1. Right decomposition: Claude's screen signatures join Claude's JSONL classifier in one package (they move on one clock); the only true cross-domain primitive (classifyByAwaiting) stays in anyagent, undiluted. Matches the pattern the codebase already ships for Codex and OpenCode.
  2. Fragility is measured and small: 0 changes across 82 daily releases; any break is a loud, localized fixture failure.
  3. The no-config win is the point: nothing written to the user's home; works on remote/SSH/fresh/shared hosts where the hook needs N× setup or silently misses.
  4. One receptacle, swappable generators: #905's hook drops in later as a swap inside the claude-code package, behind the same named merge — consumers never learn which generator powered the badge.
  5. Next: the Phase-0 live capture (≈1 hr) to lock the buffer behavior and seed the corpus, then drive it with /do.

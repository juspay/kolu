kolu-tui — a CLI terminal client for kolu-server

A terminal-side client of kolu-server's pty-host. kolu-server owns its PTYs in-process today; kolu-tui connects to that same in-process pty-host over a local unix socket (speaking ptyHostSurface) and gives you list / attach / spawn from the terminal — no browser, no daemon. It is the CLI face of kolu and the seed of a tmux/zmx replacement (#671), shipped in phases as a beta. This doc is scoped to kolu-tui-on-the-in-process-server; the parent plan's Phase B (moving the pty-host into a surviving daemon) is a separate plan it later dovetails with — see Later.

Architecture — everything stays in-process in kolu-server

No new process. kolu-server keeps owning the PTYs exactly as it does now; it just exposes the same in-process router over an additional local socket so a second client — the CLI — can reach it. The web path is byte-identical.

┌─────────────┐ ┌──────────────────┐ │ browser │ │ kolu-tui │ │ (the GUI) │ │ packages/pty-tui │ └──────┬──────┘ └─────────┬────────┘ │ ws:// kolu-common/contract │ ptyHostSurface │ rich: PTY + providers + sessions │ raw: PTY + taps ▼ ▼ ┌──────────────────────────────────────────────────────────────────────┐ │ kolu-server — ONE process · all in-process · NO daemon │ │ │ │ servePtyHost(deps).router │ │ (owns node-pty fds + @xterm/headless mirror + the VT taps) │ │ ╱ ╲ │ │ directLink serveOverStdio │ │ (web path — unchanged) (NEW — additive, the SAME │ │ ╱ router on a unix socket) │ │ ▼ ▼ │ │ LocalTerminalBackend ◆ $XDG_RUNTIME_DIR/kolu/ │ │ (providers, sessions…) pty-host.sock ◆ ◀── kolu-tui │ └──────────────────────────────────────────────────────────────────────┘

The split by richness. The browser is the rich client — it speaks kolu-server's public contract and gets the whole world (PTY plus the provider DAG: git context, agent state, PR status, session grouping). kolu-tui is the raw client — it speaks ptyHostSurface directly and gets the bare multiplexer (PTY plus the VT taps), exactly the tmux altitude. Both attach to the one in-process pty-host.

The only server change is one additive socket. servePtyHost(deps).router already exists and is already consumed in-process via directLink. Phase 1 adds serveOverStdio(router) on a unix socket beside it — the same router, a second transport. The in-process web path doesn't change; spawning, killing, and metadata all still flow through LocalTerminalBackend as today. No PTY moves, no process is added, nothing survives a restart that didn't before.

Why a second client at all

Two reasons, both standing on their own without any reference to daemons.

  • It's a feature. A persistent-while-the-server-runs terminal you drive from the shell — the CLI face of kolu, the start of the tmux replacement #671 has always pointed at.
  • It proves the contract is consumer-agnostic. A1 (#1055) asserts ptyHostSurface is a clean seam — kolu-server talks to the pty-host through a contract, not through node-pty internals. The only real proof of that is a second, independent consumer. If writing a terminal client against the surface is clean, the seam is at the right altitude (the "framework needs an end-to-end demo, not just unit tests" lesson — parent plan, lesson #3). If it's awkward, that's a finding worth having early.

Honest scope. Because the pty-host is in-process, restarting kolu-server (a deploy) kills its PTYs — kolu-tui does not make terminals survive a server restart. What it does give is client-side detach/reattach: the CLI can come and go (close the terminal, reopen, re-attach) while kolu-server keeps running and holds the PTY. Surviving a server restart is the separate daemon plan (B), not this.

The surface — ~150 LOC in packages/pty-tui

A small node CLI in its own package, packages/pty-tui, that dials the unix socket through stdioLink (the same link the daemon would later use) and gets a typed ptyHostSurface client. No new wire, no new framing — serveOverStdio is already unit-tested via the loopback fixture.

SubcommandSurface callBehaviour

Forward stdin via write; SIGWINCH → resize({id, cols, rows}). A line-start ~. escape (see below) detaches — the CLI exits, kolu-server keeps the PTY.

The taps (subscribeCwd / subscribeTitle / subscribeForeground / exitPromise) feed the status line — a second, non-browser consumer of the raw VT taps, proving those are transport-clean too. --json makes list machine-readable; --socket <path> points at a non-default server.

Decisions recorded here (to be echoed in code comments)

DecisionChoiceWhy
Homea tiny separate packages/pty-tuiAn independent CLI package, not a bin folded into @kolu/pty-host — keeps the pty-host package focused on the contract + primitive, and keeps the CLI's own deps (raw-tty handling, arg parsing) out of the closure the A2 staleKey hashes.
Render fidelityraw VT passthroughWrite the pty-host's bytes straight to stdout; do not re-render through a second @xterm/headless mirror. Simplest and truest — and it avoids a second rendering path drifting from the server's. This trade-off must be recorded as a design-decision comment at the passthrough site in code.
Detach / escapessh-style line-start ~ escape — never Ctrl+BA passthrough multiplexer must not steal any control char the inner tools need — Ctrl+B and Ctrl+J are reserved by Claude Code (input/prohibitedKeybinds.ts), and editors/shells claim the rest. So the escape is the unambiguous ssh model: a ~ recognised only immediately after a newline you typed. ~. detach · ~k kill · ~~ literal tilde · ~? help. The char is configurable (--escape); a literal ~ mid-line passes through untouched, so every inner-app chord, Ctrl+B included, reaches the program unmodified.

Phasing — three beta increments, all in-process

Phases 1–3 build kolu-tui on the in-process server — each leaves Kolu working, ships on its own, and changes nothing in the web path. They're preceded by a Phase 0 that lives outside kolu entirely: a surface example that rehearses the whole "interactive TUI over oRPC stdio" pattern (detail below).

(the raw multiplexer)		 spawn (+ --print-id), kill, list --json. None. Create and kill PTYs from the CLI — a usable raw multiplexer (beta). The richer tmux-replacement UX (windows, panes, config) grows from here, as its own work.

Phase 1 alone is the smallest honest first ship of kolu-tui — exactly the kolu-tui list starting point: a one-shot RPC, no raw-tty mode, the "hello world" that proves serveOverStdio-over-a-real-socket + stdioLink + the contract round-trip, at near-zero risk.

Phase 0 — a minimal CI-runner TUI (the 3rd surface example)

Before touching kolu at all, prove the pattern in @kolu/surface's examples — the falsifiability test of lesson #3 applied to "interactive TUI over oRPC stdio," the way the worker demo and the remote-process-monitor (which became drishti) validated the earlier patterns. The candidate: a minimal CI runner — justci-flavoured but a self-contained toy, deliberately not the real justci (no Haskell, no GitHub statuses, no multi-platform SSH fan-out). Just a tiny DAG of local shell commands.

It's a clean structural twin of kolu-tui: a long-lived runner owns the DAG + each node's process + its log buffer, and streams to ephemeral TUI clients over stdio. nodes.list() ↔ kolu-tui's list; node.log(id) (snapshot-then-delta) ↔ attach; node.rerun(id) ↔ input. If that's clean to write against the surface primitives, the seam is at the right altitude; if it's awkward, that's a framework finding to fix before kolu-tui inherits it. The example then becomes kolu-tui's copy-paste skeleton (raw-tty setup, the snapshot-paint loop, the ~-escape) and a permanent regression test for the pattern.

The whole user flow — CLI workflows + screens

The interactive loop a user walks, then the same loop scripted as a headless test. The dark screens are what kolu-tui paints. The running example: a PTY 3f9a…c21 in ~/code/kolu that becomes a Claude Code session.

B · Headless test — client-death detach/reattach, no browser

The detach/reattach loop scripted. It asserts that the CLI client can die and a fresh one re-joins the same PTY (the server runs throughout) — fast, no browser.

#!/usr/bin/env bash
set -euo pipefail

# 1 · spawn a PTY with a unique marker in its scrollback (no attach)
id=$(kolu-tui spawn --print-id -- bash -c 'echo "MARK-$$"; exec sleep 1d')
pid_before=$(kolu-tui list --json | jq -r ".[] | select(.id==\"$id\").pid")

# 2 · (no kolu-tui process is attached — simulating client death)

# 3 · a fresh client re-attaches by id and asserts the state is intact
kolu-tui snapshot "$id" | grep -q "MARK-"            # scrollback preserved
pid_after=$(kolu-tui list --json | jq -r ".[] | select(.id==\"$id\").pid")
[ "$pid_before" = "$pid_after" ]                     # same PTY, not a respawn

echo "✓ client came and went; server held the PTY (pid $pid_after unchanged)"

This is the kolu-tui-scoped reattach test — the client can come and go while the server holds the PTY. (Surviving a server restart is a different assertion that belongs to the daemon plan, not here.)

Later — how this dovetails with the daemon plan (out of scope)

Stated only so we don't trip over it: when the parent plan's Phase B eventually moves the pty-host out of kolu-server into a surviving kolu --stdio daemon, kolu-tui's socket target shifts from kolu-server to the daemon — with no contract change, because both serve the same ptyHostSurface. At that point the CLI's terminals gain server-restart survival for free, and the raw-vs-rich client split is unchanged. But that is a separate plan; nothing in kolu-tui's phases above depends on it, and kolu-tui ships and is useful entirely on the in-process server.

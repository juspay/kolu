kolu-tui — the daemon's first client

A de-risking design for Phase B — the one-way door. B moves local PTYs across a socket into a surviving daemon; the irreducible risk is "type into a terminal, across a socket, to a separate process." kolu-tui is a tiny raw-tty client that speaks ptyHostSurface directly — isolating that I/O core from the entire web stack so it can be proven before it's promoted to the GUI. Today it earns its keep by de-risking B — but it is also the seed of a first-class user surface: the CLI face of kolu, a persistent terminal multiplexer that retires tmux/zmx (the original promise of #671). Same code, two lifetimes — so it is built clean, not throwaway.

The thesis — prove the door before you walk the GUI through it

B's danger is concentrated in one place: the core terminal I/O path now crosses a unix socket to a process that must outlive kolu-server. The web path stacks that core under five more layers — browser ↔ WebSocket ↔ kolu-server ↔ socket ↔ daemon, plus ClientRetryPlugin reconnect, the provider DAG, and xterm.js. A bug in "type into a terminal and it survives a restart" could hide in any of them. kolu-tui collapses the stack to tty ↔ stdioLink ↔ daemon — the exact novel surface, nothing else. If attach / write / resize / reattach / survival work in the TUI, any remaining web bug is confined to layers that already work for the in-process server today and are not the one-way-door risk.

It is also the falsification test for A1's central claim. A1 (#1055) asserts that ptyHostSurface is consumer-agnostic — kolu-server talks to it through the exact contract a socket will later carry. The only proof of that is a second, independent consumer. kolu-tui is that consumer. If writing a terminal client against the surface feels awkward, the seam is at the wrong altitude — exactly the "framework needs an end-to-end demo, not just unit tests" lesson (parent plan, lesson #3). A clean TUI is positive evidence the boundary is right.

The de-risking ladder — two novel axes, two rungs, then the GUI

B introduces exactly two genuinely-new, can't-unit-test mechanics — the two that burned #1031 and #1034. Each gets its own cheap rung below the GUI:

RungAxis it provesFidelityStatus
Spike (throwaway) cgroup survival — a systemd-run --user --unit= child (and its own children) outlives systemctl --user restart <parent>; a child left in the parent cgroup is killed. raw net socket, fake in-memory state, node-sleeper "PTY". done — passed

The two rungs together cover both novel axes — survival (spike) and reattach (TUI) — over the real or faithfully-modelled mechanism, before a single browser is involved. The GUI then only adds layers that already work in-process today.

Spike evidence (the rung below — already gathered)

Run on an ephemeral pu box (NixOS, kernel 6.12, cgroup-v2, non-root user with a live systemctl --user manager — faithful to kolu's systemd-run --user model). A survivor spawned via systemd-run --user --unit=spike-survivor owns a child "PTY" + in-memory state behind a unix socket; a control child is left in the parent's cgroup. Then systemctl --user restart the parent (a deploy):

ProcesscgroupAfter restartExpected
survivor (daemon)spike-survivor.service (own)ALIVE, never got SIGTERM✓ survives
PTY grandchildspike-survivor.service (rode survivor)ALIVE✓ survives
detached controlspike-parent.serviceDEAD✓ killed

Reattach (fresh client → surviving socket): counter=3 and survivorPid unchanged, ptyAlive=true — in-memory state and the owned PTY preserved across the restart. The negative control (detached child dying) proves the restart genuinely kills the parent cgroup, so the survival isn't a no-op. kolu-tui climbs from this fake-state spike to the real contract.

What it is — ~150 LOC over the surface kolu-server already uses

A node CLI that dials the daemon's unix socket through stdioLink (the very link kolu-server uses for the same socket) and gets a typed ptyHostSurface client. No new wire, no new framing — serveOverStdio is already unit-tested via the loopback fixture; the TUI just adds a human at the other end of it.

The taps (subscribeCwd / subscribeTitle / subscribeForeground / exitPromise) feed the status line — a second, non-kolu-server consumer of the raw VT taps, proving they're transport-clean too.

Scripting flags: --print-id (spawn prints the minted UUID and does not attach), --json (machine-readable list), --socket <path> (point at a non-default daemon). These collapse the headless gate to a few lines of bash.

The whole user flow — CLI workflows + screens

Three workflows, each a real command sequence; the dark screens are what kolu-tui paints. A is the interactive de-risk loop a developer walks; B is that loop scripted into a headless CI gate; C is the ops diagnostic the tool becomes after B ships. The whole arc — birth → work → detach → reattach → survive a deploy → exit — runs with no browser.

B · Headless CI gate — survival + reattach, no browser

The interactive loop above, scripted. This becomes the fast reattach gate — it asserts the two novel axes in seconds on a pu/CI box, while the web daemon-reattach.feature layers the full-stack path on top.

#!/usr/bin/env bash
set -euo pipefail

# 1 · spawn a PTY with a unique marker in its scrollback (no attach)
id=$(kolu-tui spawn --print-id -- bash -c 'echo "MARK-$$"; exec sleep 1d')
pid_before=$(kolu-tui list --json | jq -r ".[] | select(.id==\"$id\").pid")

# 2 · the deploy: kolu-server dies and respawns; the daemon must outlive it
systemctl --user restart kolu

# 3 · reattach by id and assert
kolu-tui snapshot "$id" | grep -q "MARK-"            # scrollback preserved
pid_after=$(kolu-tui list --json | jq -r ".[] | select(.id==\"$id\").pid")
[ "$pid_before" = "$pid_after" ]                     # reattached, not respawned

echo "✓ survived restart + reattached by id (PTY pid $pid_after unchanged)"

Three assertions, the whole door: the daemon survived a real systemctl --user restart, the scrollback came back, and the PTY pid is unchanged (a join, not a fresh spawn). Green here is the daemon core de-risked before a line of the GUI runs — and it slots straight onto the spike the same way.

What it de-risks — and what it deliberately doesn't

De-risks (the novel, scary core)Does NOT cover (already-working web layers — left to e2e)
  • Socket transport over the real ptyHostSurface (a second consumer).
  • Attach → scrollback snapshot → live deltas (the snapshot-then-delta invariant).
  • Reattach-by-id: daemon outlives its client; fresh client re-joins, no respawn, same pid.
  • Raw input / resize / SIGWINCH round-trips end-to-end.
  • Combined with the spike: the daemon is cgroup-escaped, so the TUI proves reattach across an actual systemctl --user restart of the parent.
  • The provider DAG running fresh in kolu-server (git / agent / PR detection) and bridgeStream into it.
  • The srv · pty rail divergence (outdated / dead) — see the chrome-bar doc.
  • DegradedCanvas, the restart dialog, the update-pending nudge.
  • WebSocket ClientRetryPlugin reconnect and xterm.js rendering.

The right-hand column is display and freshness — lower-risk than the I/O core and exercised by daemon-reattach.feature / daemon-update.feature. The TUI hits the highest-risk part precisely; it is a complement to the e2e suite, not a replacement.

Where it slots in B's build order

After topology + cgroup/supervisor (so there's a real socket and a surviving daemon to attach to), before reattach-into-providers and the honest-state UI — so it gates the daemon core before the heavy server-side and UI work stack on top:

kolu-shared → topology → cgroup/supervisor → [ kolu-tui : gate the core ] →
reattach-by-id (into kolu-server's provider DAG) → recovery → honest-state UI

It also unlocks a headless survival test: on a pu / CI box, kolu-tui attach <id> → systemctl --user restart the daemon's parent → fresh kolu-tui attach <id> → assert the snapshot + pid — proving survival + reattach with no browser, directly easing the "the acceptance signal is the second deploy" friction the B panel calls out.

Scope — de-risk now, product later (the tmux replacement)

This is not a throwaway harness. The persistent daemon B builds is a terminal-multiplexer server; kolu-tui is its native CLI client — a peer of the web GUI on the same ptyHostSurface, and the long-promised tmux/zmx replacement (#671). The two clients split cleanly by richness: the web GUI is the rich client (PTY + the provider DAG — git context, agent state, PR status, fresh in kolu-server), the TUI is the raw client (PTY + the VT taps — the tmux-equivalent), both attaching to the one surviving daemon. So it is built as real software, clean enough to graduate.

What that means for B: the de-risk subset B actually needs is small — list / attach / spawn / snapshot over the socket, ~150 LOC reusing stdioLink + ptyHostSurface. That subset gates the daemon core (and the headless test); B's merge doesn't wait on full multiplexer polish (window/pane management, config, keybindings). The product surface — the tmux-replacing UX — grows on this foundation after B, as its own work, against a contract a real second consumer has already proven. The de-risk use and the product are the same binary at two stages of its life, not two throwaway efforts.

Open questions

  • Home. A bin in @kolu/pty-host (it already owns the contract + the daemon), or a tiny separate packages/pty-tui? Leaning @kolu/pty-host — keeps the second consumer in the package whose closure the staleKey hashes, so the conformance test travels with the contract. (Watch the A2 closure-hash guard: a TUI bin that imports only the contract is fine; if it pulls in unlisted edges the build-time import guard will — correctly — complain.)
  • Render fidelity. Raw VT passthrough (write the daemon's bytes straight to stdout — simplest, truest) vs. re-rendering through an @xterm/headless mirror. Passthrough is enough for a de-risk tool and avoids a second rendering path to keep honest.
  • Canonical headless test? Should the scripted attach → restart → reattach become the canonical headless reattach gate (fast, no browser), with the web daemon-reattach.feature layered on top for the full-stack path? Likely yes.
  • Detach protocol. ^B-prefixed chords collide with tmux/Claude-Code muscle memory — check input/prohibitedKeybinds.ts conventions even for a dev tool, or pick an unambiguous escape.

01  The analogy

Lowy's receptacle: hide the volatility, sell a faceplate

Lowy uses household power to argue that you decompose a system by what changes, not by what it does.

"Power in a house is highly volatile: AC or DC; 110 volts or 220; 50 hertz or 60; produced by solar panels, a generator, or grid connectivity. All that volatility is encapsulated behind a receptacle. When it is time to consume power, all the user sees is an opaque receptacle." Juval Lowy — Righting Software (via informit.com/articles/article.aspx?p=2995357)

The toaster never exposes the wires, measures the frequency with an oscilloscope, or certifies the voltage with a voltmeter. It plugs in. The opposite — every appliance carrying its own voltmeter and deciding what to do with the raw mains — is what Lowy calls functional decomposition, and Hickey calls complecting: one concern braided through everything.

So the question "is mobile electricity?" is precise: does feature code plug into a stable interface that has already resolved "where am I running," or does each consumer measure the voltage (isMobile()) and branch on it itself?

The answer for kolu: two outlets, one good extension cord, and a lot of bare wire.

04  What's wired right

The parts that already are electricity

Credit where due — and these are the templates to copy, not abstract ideals. Each survived a skeptical re-read of the source.

🔋 The two-axis split is a correct un-braiding

useMobile.ts ships two signals for two genuinely independent volatilities and refuses the seductive single god-flag: isMobile = (max-width: 639px) is form-factor; isTouch = (pointer: coarse) is interaction model. A 1366px touch iPad correctly resolves to isMobile()=false / isTouch()=true — so it gets MobileKeyBar and soft-keyboard handling while keeping the desktop layout. Conflating them would have forced a wrong choice for every large-format touch device.

🔌 The top-level layout switch is a true receptacle

App.tsx:546 — match(isMobile()).with(true, MobileTileView).with(false, TerminalCanvas).exhaustive() — routes the whole UI into two compositionally separate subtrees. Leaf components plug into one world and never measure voltage: MobileTileView's swipe handlers carry no isTouch() guard because they simply aren't mounted on the desktop side; the minimap and auto-arrange are absent from mobile by composition, not by per-component if-checks.

05  Where the leads are bare

Five leaks that survived adversarial verification

Each of these was re-read against source. Several map-phase findings were downgraded to "intentional / low" here — what remains is the verified set. The pattern is Lowy's functional decomposition: "mobile" smeared across consumers, each measuring the voltage.

⚡ Dual breakpoint — one threshold, two voltages

The JS media query is max-width:639px while Tailwind sm: utilities fire at min-width:640px (no tailwind.config exists, so sm: is the v4 default 640px). In the 639–640 band the JS appliance thinks it's mobile while the CSS appliance thinks it's desktop. useMobile.ts's own comment claims it "Matches Tailwind's sm: breakpoint (640px)" — contradicting its own query. Worse: ChromeBar's inner hidden sm:flex classes are dead at that boundary, because App.tsx:507 unmounts the whole ChromeBar via isMobile() — a CSS rule orphaned by a JS rule deciding the same thing.

⚡ openInCodeTab forks one intent into two state mutations inline

The producer of a navigation intent measures the voltage itself:

// right-panel/openInCodeTab.ts:76-80 — inside batch()
rp.openCodeAt(req.targetMode);
if (isMobile()) rp.setDrawerOpen(true);        // mobile mechanism
else if (rp.collapsed()) rp.expandPanel();      // desktop mechanism

"Reveal the code view" is one concept; how it becomes visible is the host's job. useRightPanel already owns both mechanisms as separate, documented volatilities (session-local drawerOpen vs persisted collapsed) — it is the natural home for a single rp.reveal() that resolves posture internally. Verification confirmed the branch is real and the store owns both arms; it also noted the branch is currently centralized in one front-door function, which keeps the blast radius to a single call site.

06  Where the analogy honestly breaks

A phone is not a 110-volt desktop
The load-bearing nuance

Electricity's promise is that the same appliance runs on any voltage. That promise does not fully hold for mobile — and pretending it does would be the wrong fix.

Some mobile volatility is not a different voltage of the same signal; it is a categorically different appliance. A soft keyboard is not a 220V keyboard — it is a different input device, with its own focus model, its own dismissal lifecycle, its own contenteditable target. Swipe-to-navigate is not a small-screen pan/zoom; the pan/zoom canvas is genuinely unusable on a phone, so MobileTileView and the dock/right-panel drawers are not accidental complexity — they are a second, correctly-built product surface.

This is why "just move it all to CSS" is wrong. CSS can express the presentational volatility (spacing, hidden labels, max-widths — and AgentIndicator's hidden sm:inline already does this correctly). It cannot express an interaction-model change. So:

Same appliance, different voltage → CSS

Spacing, inline-label hide/show, max-widths, density presets. Belongs in declarative responsive CSS keyed off one breakpoint. Today kolu does almost none of this in CSS (18 JS reads vs 4 classes) — the recoverable ground.

Different appliance → JS, behind a seam

Soft-keyboard input surface, focus suppression, swipe nav, touch-scroll. Must live in JS. Lowy's demand isn't "don't branch" — it's "branch in one named place (an input-strategy seam), not as raw isTouch() poking xterm internals from a setup loop."

So the resolution is clean: there are four concepts wearing one word — viewport-size, touch-modality, layout-posture, and feature-availability. The first two are correctly receptacled. The second two are functionally decomposed. The fix is not to erase the mobile/desktop divergence — it is to add the missing posture and capability receptacles so feature code expresses intent and capability, and one resolver converts that to voltage.

07  The receptacle that should exist

Keep the two-circuit wiring; add three faceplates

Keep the wiring. isMobile and isTouch stay as the two independent axes — but unify the rating: register the breakpoint once (Tailwind v4 @theme { --breakpoint-sm }, or a single shared constant) and derive the JS createMediaQuery from the same number. The 639/640 desync band and the misleading comment both vanish.

Three named seams between raw signals and features
1
useRightPanel.reveal() / ensureVisible() — one verb that internally resolves drawer-open (mobile) vs uncollapse-if-collapsed (desktop). openInCodeTab and every future producer call it and never read isMobile. The store already owns both mechanisms — a ~5-line addition. This is the highest payoff-to-effort fix.
2
A resolved capability object — layout.supportsSpatialCanvas, layout.showsAmbientTips, layout.isCompact — computed once, each from the right axis. useTips, the canvas command section, and the center/switcher guards then ask about capability, not pixels. The difference between every appliance owning a voltmeter and the faceplate resolving voltage once.
3
enableSoftKeyboardInput(term) — owns the "touch needs a contenteditable target" knowledge and the xterm-internal poking, so Terminal.tsx calls one verb. Re-key CodeTab's touch-scroll attachment off this/touch capability rather than viewport.

withKeyboardDismiss is the existence proof these work: a drawer plugs in by passing its setter and never measures the voltage. The other three seams are the same move applied to posture, capability, and input-surface.

The one-line answer: mobile is electricity for the macro layout switch and the keyboard-dismiss wrapper, and bare mains wire everywhere else. The wiring discipline is good — two correctly-separated circuits — but neither terminates in enough receptacles, and the one threshold is rated at two voltages. Add the three faceplates above and mobile becomes electricity for the parts that can be; the rest stays honestly, deliberately, a different appliance.

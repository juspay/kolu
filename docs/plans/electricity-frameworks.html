Correction (2026-06-02) — the sweep missed one, by mis-framing it. The first cut of this document concluded "ZERO frameworks — no second @kolu/surface; the audit holds." That headline was wrong. The sweep pattern-matched "xterm = terminal = kolu domain" and never put the largest dissolvable chunk of the terminal god-procedure on the bar at all: a SolidJS adapter for xterm.js — a domain-agnostic reactive-lifecycle primitive that happens to wear a terminal library's name. The maintainer caught it ("lol i see xterm"). That is the exact inversion of the .agency/lowy.md §6.5 lesson — whose own worked example is, ironically, @kolu/solid-xterm: an agnostic primitive that wears a domain library's name reads as domain, but the §6.5 question — "what library is this?" — gives the right answer. A SolidJS adapter for xterm.js is electricity (reactive-ownership + GPU-lifecycle plumbing), not terminal code.

Corrected verdict: taken to a fresh lowy⇄hickey debate plus an adversarial skeptic, two framework-scale primitives cleared the bar — createSolidXterm (the missed one) and createSharedRoot (already shipped, re-classified from "already-covered" to a genuine domain-agnostic primitive on its own 33-line boundary). @kolu/surface is no longer the only framework-scale receptacle kolu has in it.

The recalibration. An earlier leaf-level dedup sweep found only small dedup wins — a refcounting primitive, three helper merges, one inline. The maintainer pushed back: "those are trivial; I want framework opportunities like surface." This document is the answer. The yardstick here is not "is this duplicated?" but "does this own a hard volatility axis behind a descriptors-not-plumbing programming model — the way @kolu/surface owns live-state transport?" Leaf dedup is explicitly not the goal. The whole-app audit had concluded "the big extraction already happened — surface ate the hardest axis (client↔server state) end-to-end." The corrected headline: at least one new framework was hiding in plain sight. The sweep dismissed a domain-agnostic SolidJS xterm adapter as "terminal domain" and never debated it; on re-examination createSolidXterm clears all four tests and survives the skeptic. A second — createSharedRoot, already shipped — is re-classified as a genuine reusable primitive on its own boundary. The honest negative result still stands for the other probes (the optimistic-write reconciliation framework defineMutation fell short on all four), and that nuance is preserved below.

The surface bar

@kolu/surface is the gold standard because it owns live-state transport and lifecycle behind a descriptors-not-plumbing programming model: consumers declare state shape once with four primitives (Cell, Collection, Stream, Event) via defineSurface, and the framework derives the wire contract, server handlers, and client hooks. It encapsulates the snapshot-then-deltas + reconnect pattern that would otherwise be hand-rolled in N routes, absorbs the AbortController+onCleanup+STREAM_RETRY triplets duplicated at M sites, keeps per-descriptor cost at a 2-file invariant, shipped an escape hatch for wire shapes that don't fit (terminal.attach, custom retry), and proved domain-agnostic by extracting to a publishable package. Every candidate below was held against these four tests at once — plus an adversarial gate.

A · Owns a hard axis end-to-end

Does it own a volatility axis that would force entire rewrites if leaked to consumers? For surface that axis is snapshot+deltas+reconnect transport. For createSolidXterm it is xterm-lifecycle volatility — addon churn, the WebGL single-owner state machine + context-loss recovery, owner-correct async dispose across an await.

B · A new programming model

Does it reduce call-site plumbing from N orthogonal concerns to declaring one shape? surface replaces manual createSubscription/AbortController/onCleanup wiring; createSolidXterm replaces hand-wiring 8 addons + the WebGL focus dance + the async-init owner capture with one createSolidXterm({…}) call. Swapping syntax is ceremony, not a model.

C · Dissolves a pervasive braid

Does extracting dissolve a braid rather than move it across a wall? The braid need not be cross-package: a 972-line god-procedure interleaving six independent volatilities is a braid, and pulling one coherent volatility out of it is dissolution, not relocation.

D · Graduates to a package

Can it extract to a publishable, domain-agnostic @kolu/* package with a working non-Kolu example? The graduation question is the §6.5 question: "what library is this?" — a SolidJS adapter for xterm.js answers cleanly (@kolu/solid-xterm), and the answer is electricity, not terminal code.

The adversarial gate. Clearing the four tests is necessary, not sufficient. Every framework verdict then had to survive a skeptic whose explicit default was to refute it — opening the real call sites, grepping for the claimed recurrence, and asking whether the "hard axis" reduces to UI plumbing the existing receptacles already own. A verdict that the skeptic could deflate to helper-too-small, domain-coupled, or already-covered does not graduate. This is the same adversarial discipline the whole-app audit used ("each candidate handed to an independent skeptic told to refute it by opening the real call sites") — applied here at framework scale. The skeptic's strongest case against createSolidXterm ("the domain braid runs too deep to graduate") is recorded in full on the card below, along with why it was rebutted.

Why the sweep missed it

The first sweep folded the entire terminal subsystem into one "God-procedure decomposition" row and marked it "not a framework — pure intra-module reshaping." That diagnosis collapsed two different things into one: the kolu-specific orchestration in Terminal.tsx (stream attach, paste/drag upload, link activation, keyboard dispatch, diagnostics) — which genuinely is a refactor — and the domain-agnostic xterm-lifecycle core braided through it, which is a framework-scale primitive wearing a terminal library's name.

The lesson: an agnostic primitive that wears a domain library's name reads as domain. "xterm" is in the filename, in the imports, in every comment — so the sweep pattern-matched it to "terminal" and never ran it through the bar. The fix is the .agency/lowy.md §6.5 question: "what library is this?" The answer for a SolidJS wrapper around @xterm/addon-* + WebglAddon + getOwner/runWithOwner is "a SolidJS reactive-lifecycle adapter for xterm.js" — pure electricity, adoptable by any SolidJS app embedding a terminal, not kolu-terminal code. §6.5's own worked example for this trap is, fittingly, @kolu/solid-xterm itself.

Framework-scale candidates that cleared the bar

The probe targeted the hardest remaining axes the audit flagged — the file-watcher zoo, the optimistic-local-vs-server-echo braid, the god-procedures. Two cleared the bar as genuine domain-agnostic frameworks: createSolidXterm (the one the first sweep mis-framed and never debated) and createSharedRoot (already shipped, here re-classified onto its own boundary). The closest failure — the optimistic-mutation reconciliation primitive — is detailed afterward as the one probe that was taken to full adjudication and fell short.

framework createSolidXterm @kolu/solid-xterm · SolidJS adapter for xterm.js

packages/client/src/terminal/Terminal.tsx (972 lines) · debated to consensus · disposition: framework · confidence: strong

The hard axis

The volatility axis is xterm.js lifecycle ownership — six genuinely hard, intertwined concerns: (a) the eight-addon orchestration and disposal, (b) the WebGL single-owner state machine with context-loss recovery, (c) FitAddon resize geometry sync, (d) live theme/font sync, (e) the touch-scroll state machine, and (f) owner-correct async dispose across an await boundary. This axis is documented in .agency/lowy.md as two distinct rows: "Async-initialization cleanup registration" (SolidJS's reactive Owner is null across any await; observed twice as the #598 direct-onCleanup leak inside Terminal.tsx onMount and the #600 @solid-primitives/resize-observer leak — "whole subsystems leak … ~900 KB per leaked component … only heap-snapshot retainer walks catch it") and "Canvas lifecycle + runtime geometry" (the still-alive WebglAddon canvases and whether their GL context is lost — "surfaced during the #591 zombie-context hunt"). The WebGL single-owner discipline is load-bearing: Chrome's ~16-context-per-tab limit (#575), the orphan-canvas leak when onCleanup registers inside a disposed owner (#591, diagnosed via #595's webglTracker), and the texture-atlas corruption fix (#239) all live on this axis. Alternatives — hand-wire addons per call site (the current 972-line state) or pre-bake xterm into a monolithic wrapper (no reuse) — both lose.

What is hand-wired inline today

All of it lives in one onMount body in Terminal.tsx:

  • XTerm constructor + 8 addon loads — new XTerm({…}) at Terminal.tsx:472; FitAddon, WebLinksAddon, SearchAddon, ClipboardAddon, Unicode11Addon, ImageAddon, SerializeAddon all hand-loaded at Terminal.tsx:489-509.
  • The WebGL single-owner dance — loadWebgl()/unloadWebgl() at Terminal.tsx:222-285: new WebglAddon(), the onContextLoss listener, the explicit loseContext() GPU release, and the .xterm-screen canvas:not(.xterm-link-layer) canvas-discovery workaround (Terminal.tsx:241-249) born from the #591/#595 hunt.
  • Owner-correct async init — synchronous getOwner() capture before the await, the disposed bail flag set by a synchronously-registered onCleanup, and the runWithOwner(owner, …) wrap of the post-await body — Terminal.tsx:198, :410-412, :463-471.
  • Resize / theme / font — debouncedFit() at Terminal.tsx:179, the createResizeObserver at Terminal.tsx:750-755, and the theme/font live-sync createEffects at Terminal.tsx:362-398.
  • Touch scroll — the tap-vs-scroll state machine (TAP_THRESHOLD_PX, isTap()) at Terminal.tsx:549-596.

Supporting modules confirm the axis is already partly factored but not yet owned by one primitive: webglTracker.ts (222 lines, the GPU lifecycle ledger, explicitly "Temporary: remove when #591 is root-caused") and terminalRefs.ts (50 lines, per-terminal probe refs behind TerminalProbes).

The domain boundary

Inside the primitive (agnostic core): the XTerm constructor + theme/fontSize options, all addon loading/disposal, the WebGL single-owner + context-loss + canvas-discovery + loseContext(), FitAddon + debouncedFit + resize geometry, theme/font live sync, the touch-scroll state machine, and the async-init owner-capture dance with one synchronized onCleanup.

Stays consumer-injected (kolu-specific, via callbacks): the PTY wiring — kolu wires onData to client.terminal.attach; another app wires a different PTY (Terminal.tsx stream attach + onRetry reset orchestration); the link provider — kolu injects createFileRefLinkProvider for its path:line schema (Terminal.tsx:497) and activateFileRef → openInCodeTab (Terminal.tsx:303); paste/drag upload calling client.terminal.pasteImage/uploadFile; keyboard dispatch; activity-alert and diagnostics hooks (registerDiagnostics at Terminal.tsx:635, registerTerminalRefs at :622); and the e2e bridge container.__xterm = term at Terminal.tsx:619. Terminal.tsx does not become publishable — it is the kolu orchestration layer that wires callbacks to kolu concerns. The primitive is what graduates.

What it dissolves — quantified

The lenses converged on ~240–247 lines of xterm-lifecycle scaffold extractable from Terminal.tsx: ~15 lines of addon loads + ~61 lines of WebGL state machine + ~40 lines of resize geometry + ~23 lines of theme/font effects + ~57 lines of touch scroll + ~50 lines of async-init owner dance + the soft-keyboard bridge. Terminal.tsx shrinks from 972 to roughly ~280–650 lines of pure kolu orchestration (the lenses placed the floor near ~280, the skeptic's more conservative count near ~625 once kolu integration that must stay is honestly accounted). Either way it is dissolution of a real braid, not relocation: the extracted lines are a coherent, independently-reusable volatility, and what remains is the irreducible kolu wiring.

Survived the skeptic? — Yes, but the boundary was sharpened.

Strongest case against (bar D — graduates): "The proposed primitive does not achieve domain-agnostic graduation. kolu-specific integration — stream attachment to client.terminal.attach, scrollLock buffering, soft-keyboard + tap routing, link-provider registration, diagnostics probes — is so braided into initialization and event wiring that the consumer remains the integrator. Another SolidJS app cannot adopt @kolu/solid-xterm without re-implementing the same 90+ lines of assembly logic that stay in Terminal.tsx. The '314 lines deleted' claim is inflated; actual xterm-specific deletion is ~240 lines, with ~90 lines of kolu integration remaining. This is the topic-bundle pattern (the failed @kolu/solid-xterm@0.1 with three exports), not a reusable primitive like @kolu/solid-markdown's single createSolidMarkdown."

Rebuttal (refuted on the interface; conceded on line-count): the charge is valid on the line-count claim — the consensus correspondingly revised the deletion figure down (~240–247, not 314) and the residual Terminal.tsx up. But it is refuted on the primitive interface: the kolu-specific pieces the skeptic lists (stream reconnect, paste/drag upload, link activation) are not xterm-lifecycle concerns — they are consumer-orchestration concerns that must stay in the consumer or be injected via callbacks. That is correct encapsulation, not a leak: the primitive owns xterm volatility; the consumer owns kolu orchestration. The five callbacks (onData, onResize, onLinkActivate, onCustomKeyEvent, plugins) are transport-neutral; LineRef is an xterm buffer reference, not kolu's path:line domain object. The §6.5 test passes precisely because the export is one primitive, not the three-export bundle. The skeptic's residual valid point — that scrollLock/softKeyboardInput are currently kolu-embedded — is a @0.2.0 packaging refinement (separate @kolu/xterm-addons or documented as swappable), not a bar-D failure of the core primitive.

framework createSharedRoot @solid-primitives/shared-root · lazy reactive singleton

packages/client/src/createSharedRoot.ts:23-33 (33 lines) · debated to consensus · disposition: framework (re-classified from "already-covered") · confidence: strong

The hard axis

SolidJS reactive ownership — a lazy module-scoped singleton with app-level cleanup. Three intertwined sub-volatilities: (1) lazy initialization (the factory runs once, on first call, not at module load); (2) single reactive owner — the createRoot owner is the app, retained in the box closure, so disposing the first caller (fast-refresh, test teardown) does not freeze the signal for other consumers; (3) falsy-value boxing — uninitialized state must be distinct from a factory returning null/0/false, or if (!cached) reinit() silently re-runs the factory. Six modules independently reimplemented this before the primitive existed — the box+closure dance is non-obvious and the falsy-value bug is silent.

What is hand-wired inline today

Six active call sites consume the shipped primitive: useTerminalStore, useThemeManager, getNowTicker (staleness), useDockOrder, useIntentEditor, and useModHeld — each previously carried the let box; if (!box) { box = createRoot(...) } null-guard boilerplate, ~5–8 lines apiece.

The single primitive — ONE export, ONE concept

// packages/client/src/createSharedRoot.ts — already shipped.
export function createSharedRoot<T>(factory: () => T): () => T
// wraps createRoot(factory) in a module-scoped box; returns a getter
// that lazily initializes on first call and caches the (possibly falsy) result.

An atomic composition operator: "wrap a factory in a lazy singleton that survives all consumer lifecycles." No .attachCleanup, .getBox, or .reset leaks to callers — not a topic-bundle.

The domain boundary

100% domain-agnostic within its 33-line boundary: the only import is solid-js; the signature <T>(factory: () => T): () => T is shape-agnostic (the factory can return an Accessor, a signal bundle, a store, a memo). The skeptic's counter — that Terminal.tsx (which uses a sibling singleton) is 34% kolu-specific — was the same mis-framing that produced the original miss: it conflates the agnostic primitive with a domain-specific consumer. A consumer being domain-coupled does not make the primitive domain-coupled.

What it dissolves — quantified

~27–48 lines of duplicated lazy-singleton boilerplate across the six call sites (the lenses bracketed the figure; the skeptic's conservative count was ~43). Future singletons inherit the contract for free.

Survived the skeptic? — Yes, by separating primitive from consumer.

Strongest case against (bar D): "The example consumer Terminal.tsx is 34% kolu-specific braided code (332 of 972 lines) — file-ref link provider, mobile tap gestures, upload RPCs, touch scroll, command dispatch, e2e bridges. An xterm-embedding app cannot import that file without the entire kolu domain; therefore the pattern does not graduate." Rebuttal: the skeptic itself conceded the primitive passes bars A, B, and C and "passes D within its own 33-line boundary." The bar-D failure it asserts is against Terminal.tsx-as-publishable-component — which nobody claims. createSharedRoot graduates on its own boundary; Terminal.tsx is evidence of utility, not of agnosticism. (And the genuinely-agnostic terminal primitive that does graduate is createSolidXterm, above — which is exactly the thing the first sweep missed.)

Already proven in-repo with 6+ consumers; the docstring names the problem with real examples; one export, one concept, no configuration surface. The "build once, share across the app, survive first-caller disposal" problem is universal in SolidJS apps.

The probe that fell short — defineMutation

The strongest failure is preserved here unchanged: the optimistic-mutation reconciliation primitive the audit itself under-ranked and re-surfaced as a "missed" candidate, taken to full lowy⇄hickey adjudication and an adversarial skeptic. Consensus: helper-too-small.

grow-surface defineMutation live-state · write side

packages/client/src/canvas/usePendingLayouts.ts:45-111 · debated to consensus · disposition: helper-too-small

The proposed hard axis

The pitch: a fifth surface primitive (or a collection.patch verb) owning per-collection-entry optimistic mutation with echo reconciliation — declarative mutation specs that own the optimistic-apply → RPC-send → server-echo-reconcile lifecycle. The audit's optimisticOverlay named this as "the candidate the backlog under-ranked to invisibility … the mechanical core of the single biggest remaining complecting theme" — the natural neighbor of surface's Cell/Collection, which own server-reconcile but not optimistic-overlay-until-echo. On paper this is the strongest "second surface" story: a real braid (Hickey theme §03: local pending value ⊗ server-saved value ⊗ equality-gated reconcile ⊗ per-key lifecycle), adjacent to the gold-standard framework, parameterizable on an opaque id.

What consumers hand-roll today

Exactly one domain hand-rolls it — canvas tile layouts:

  • usePendingLayouts.ts:69-71 setOne, :72-85 applyMany, :86-106 dropEvicted (the equality-gated GC: "drop entries whose saved layout has caught up to their pending value"), :107-109 clear — a singleton createStore<Record<string, TileLayout>>.
  • Consumed at two sites: TerminalCanvas reads pending in layoutOf() and calls dropEvicted as echoes settle (per the module doc, usePendingLayouts.ts:1-13); useCanvasArrange.ts:56 seeds via applyMany.
  • The write itself is fire-and-forget RPC — useTerminalCrud.ts:50-56 setCanvasLayout is void client.terminal.setCanvasLayout(…).catch(toast); the server echo arrives through the existing metadata subscription that surface already owns.

The primitive that was proposed

A declarative mutation spec colocated with a defineSurface collection — declare the optimistic apply + the reconcile predicate once, let the framework own the pending store and eviction:

// PROPOSED — never built; recorded here to show what was rejected.
// A fifth primitive / collection verb over the surface descriptor model:
collection.patch(key, patch, {
  reconcile(saved, pending): boolean,   // equality gate — when has the echo caught up?
})
// framework would own: per-key pending createStore, optimistic apply,
// dropEvicted-on-echo, clear-on-unmount — the usePendingLayouts body, generically.

Contrast the real surface shape (built, shipped): defineSurface + implementSurface + surfaceClient with Cell/Collection/Stream/Event descriptors — see packages/surface/src/define and its consumption at packages/pty-host/src/ptyHostSurface.ts:155. defineMutation would have been a fifth verb on that model.

What it would dissolve — quantified

One module (usePendingLayouts.ts, 111 lines), two call sites, one domain. The grep is decisive: theme, intent, rename, and right-panel all use plain fire-and-forget RPC with no pending store anywhere (useTerminalCrud.ts:50-56 is the pattern — a void client…catch with the echo arriving over the metadata subscription). The recurrence count is 1. The proposal's own framing conceded "without future patterns, dissolution is zero."

Survived the skeptic? — No. The skeptic prevailed.

Strongest case for: it is adjacent to surface and names a real Hickey braid. The skeptic's rebuttal held on every axis: "Canvas layout mutations are not hard-axis transport problems — they are fine-grained reactive plumbing that fits Surface's existing local-authority Cell model. The claimed hard axis reduces to one-line store mutations + fire-and-forget RPC caught by existing metadata subscriptions. Surface's power is owning snapshot+deltas transport across reconnects; layout mutations never shift that axis." The "dual-store / temporal coupling" issue is a UI-plumbing problem solvable with a memo merging pending+saved — not transport-level hardness. Adding a 5th primitive on the strength of a single domain is speculative, and apply/reconcile semantics differ per-domain (geometry equality here), so the interface would leak domain assumptions the moment a second use case landed. The proposal admitting it was "lower-risk than immediate extraction" was itself the tell: a framework that clears the bar does not need that hedge.

The transport axis (snapshot+deltas+reconnect) is already owned by @kolu/surface; pending is just a UI cache. Parameterizing reconciliation per-domain would leak assumptions and break surface's interface contract. Keep as-is; re-evaluate only if a second mutation domain lands with identical patterns.

hickey says

All four bars fail with codebase evidence. No complecting (the code is simple, isolated, per-domain, well-commented); no fragmentation (pending, setOne, dropEvicted, clear already unified under one module); no concept multiplication (no other code uses pending states, so there is no canonical pattern being reinvented). Graduation

Does not graduate. canvasLayout is a single field, not a collection (useTerminalCrud.ts:50); reconciliation is canvas-specific (geometry equality via layoutsEqual, usePendingLayouts.ts:100). There is no second app — and no second domain in this app — that would adopt it. The honest disposition: leave the module isolated; if a second optimistic-mutation domain (rename, reorder, comment-position) lands with the identical apply+echo shape, re-open the question then. That is a rule-of-two trigger, not a framework today.

Looks big — and one was a framework after all

This is the honest core, now corrected. Across nine probed axes — the file-watcher zoo, the optimistic write-side, server-side publish, intent/command, media capture, the god-procedures, "what should surface grow into?", and two open-ended fresh hunts in client and server — most candidates that looked framework-scale collapse under the bar to one of three diagnoses: helper-too-small (a real but narrow utility), domain-coupled (the interface can't shed the domain, so extracting moves the braid), or already-covered (the hard axis is already owned by surface). The verdict — does kolu have a second surface in it?

Yes — at least @kolu/solid-xterm, which the sweep missed by mis-framing agnostic infra as domain. The honest correction to the original flat "no": held to the surface bar, the largest dissolvable chunk of the terminal god-procedure is a domain-agnostic SolidJS adapter for xterm.js — createSolidXterm, owning the addon/WebGL/resize/theme/touch/async-init volatility behind one primitive — and it clears all four tests and survives the adversarial skeptic. A second, createSharedRoot, is a genuine reusable primitive on its own boundary (already shipped). @kolu/surface is no longer the only framework-scale receptacle kolu has. The first sweep's "ZERO frameworks; the audit holds" was the exact inversion of the §6.5 lesson — it pattern-matched "xterm = terminal = domain" and never put the primitive on the bar. The maintainer caught it.

The nuance the original got right still stands for the rest. The optimistic-write reconciliation framework (defineMutation / the audit's optimisticOverlay) was taken all the way to a lowy⇄hickey debate and an adversarial skeptic and lost on all four tests: it owns a UI cache, not a transport axis; it recurs in exactly one domain; its reconcile semantics are domain-specific; and the write side it claims is already owned by surface's metadata subscription. The file-watcher zoo is a refcounting primitive + fanout micro-utility; geometry is domain-coupled; the server half of surface is surface; commands is a clean engine+bridge split. Those are valuable but not framework-scale. The corrected count is two framework-scale primitives, raising kolu's framework-scale receptacles to three (surface + solid-xterm + shared-root). The right trigger to re-open defineMutation remains concrete: a second optimistic-mutation domain (rename / reorder / comment-position) landing with the identical apply-then-echo shape.

Methodology

the surface bar→ probe the hardest axes→ lowy ⇄ hickey debate→ adversarial skeptic gate→ per-finding consensus

Unlike an earlier leaf-level dedup sweep — which hunted duplicated structure — this pass started from the surface bar as the yardstick and probed the hardest remaining volatility axes the whole-app audit had flagged (the file-watcher zoo, the optimistic-local-vs-server-echo braid, the god-procedures). Each candidate was measured against all four tests at once — owns a hard axis end-to-end, a new programming model, dissolves a pervasive braid, graduates to a domain-agnostic package — using Lowy's volatility-based decomposition and Hickey's structural simplicity. Every framework verdict then faced an adversarial skeptic whose default was to refute it by opening the real call sites and grepping for the claimed recurrence.

The correction is itself a methodology lesson. The first pass did not run createSolidXterm through this pipeline at all — it folded the whole terminal subsystem into a single "god-procedure refactor" row and dismissed it without a debate. The miss was a framing failure, not a verdict failure: an agnostic primitive wearing a domain library's name (xterm) read as domain code. The fix is to apply the .agency/lowy.md §6.5 question — "what library is this?" — to every god-procedure chunk before deciding it is "just a refactor." A SolidJS adapter for xterm.js is electricity, not terminal code; the same lens correctly re-classified createSharedRoot from "already-covered" to a primitive on its own boundary. The adversarial skeptic then did its job on both cleared frameworks — its strongest bar-D attacks (domain-braid-too-deep) were refuted on the primitive interface while its valid points (revised line-counts, submodule packaging) were folded into the plans.

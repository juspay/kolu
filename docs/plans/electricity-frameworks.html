Electricity at framework scale — is there a second @kolu/surface?

The ambitious sweep: held every candidate to the surface bar, not the dedup bar — and reports honestly where a new framework survives. budding  captured 2026-06-02 · sibling of electricity-proposals.html & electricity-review.html

The recalibration. The leaf-level sweep in electricity-proposals.html found only small dedup wins — a refcounting primitive, three helper merges, one inline. The maintainer pushed back: "those are trivial; I want framework opportunities like surface." This document is the answer. The yardstick here is not "is this duplicated?" but "does this own a hard volatility axis behind a descriptors-not-plumbing programming model — the way @kolu/surface owns live-state transport?" Leaf dedup is explicitly not the goal. The whole-app audit already concluded "the big extraction already happened — surface ate the hardest axis (client↔server state) end-to-end." This doc tests that conclusion ambitiously: it probes the hardest remaining axes against the full surface bar, and each framework-scale verdict had to survive an adversarial skeptic whose default was to refute. We report where the conclusion holds and where a genuine new framework remains.

The honest headline: the audit holds. No probe cleared the surface bar. The single most-promising candidate — an optimistic-write reconciliation framework (defineMutation / the audit's optimisticOverlay) — was taken all the way to a lowy⇄hickey debate and an adversarial skeptic, and fell short on all four tests. That negative result is the point of this document: it explains why the obvious-looking "next surface" is not one.

The surface bar

@kolu/surface is the gold standard because it owns live-state transport and lifecycle behind a descriptors-not-plumbing programming model: consumers declare state shape once with four primitives (Cell, Collection, Stream, Event) via defineSurface, and the framework derives the wire contract, server handlers, and client hooks. It encapsulates the snapshot-then-deltas + reconnect pattern that would otherwise be hand-rolled in N routes, absorbs the AbortController+onCleanup+STREAM_RETRY triplets duplicated at M sites, keeps per-descriptor cost at a 2-file invariant, shipped an escape hatch for wire shapes that don't fit (terminal.attach, custom retry), and proved domain-agnostic by extracting to a publishable package. Every candidate below was held against these four tests at once — plus an adversarial gate.

A · Owns a hard axis end-to-end

Does it own a volatility axis that would force entire reactive-surface rewrites if leaked to consumers — the way snapshot+deltas+reconnect would? A pure UI cache or a fire-and-forget RPC owns no transport axis.

B · A new programming model

Does it reduce call-site plumbing from N orthogonal concerns to declaring one shape — replacing manual createSubscription / AbortController / onCleanup wiring? Swapping one method-call syntax for another is ceremony, not a model.

C · Dissolves a pervasive braid

Does the same shape recur across many sites and packages, such that extracting dissolves a braid rather than moving it across a wall? One consumer is a function; a single domain is not a braid.

D · Graduates to a package

Can it extract to a publishable, domain-agnostic @kolu/* package with a working non-Kolu example — encoding invariants as compile-time types and swapping transports without consumer rewrites?

The adversarial gate. Clearing the four tests is necessary, not sufficient. Every framework verdict then had to survive a skeptic whose explicit default was to refute it — opening the real call sites, grepping for the claimed recurrence, and asking whether the "hard axis" reduces to UI plumbing the existing receptacles already own. A verdict that the skeptic could deflate to helper-too-small, domain-coupled, or already-covered does not graduate. This is the same adversarial discipline the whole-app audit used ("each candidate handed to an independent skeptic told to refute it by opening the real call sites") — applied here at framework scale.

Framework-scale candidates

The probe targeted the hardest remaining axes the audit flagged — the file-watcher zoo, the optimistic-local-vs-server-echo braid, the god-procedures. None cleared the bar as a new framework. The closest — the optimistic-mutation reconciliation primitive the audit itself under-ranked and re-surfaced as a "missed" candidate — is detailed below as the one probe taken to full adjudication. It is recorded as grow-surface (a speculative extension to surface, not a new socket), and the consensus was helper-too-small.

The proposed hard axis

The pitch: a fifth surface primitive (or a collection.patch verb) owning per-collection-entry optimistic mutation with echo reconciliation — declarative mutation specs that own the optimistic-apply → RPC-send → server-echo-reconcile lifecycle. The audit's optimisticOverlay named this as "the candidate the backlog under-ranked to invisibility … the mechanical core of the single biggest remaining complecting theme" — the natural neighbor of surface's Cell/Collection, which own server-reconcile but not optimistic-overlay-until-echo. On paper this is the strongest "second surface" story: a real braid (Hickey theme §03: local pending value ⊗ server-saved value ⊗ equality-gated reconcile ⊗ per-key lifecycle), adjacent to the gold-standard framework, parameterizable on an opaque id.

What consumers hand-roll today

Exactly one domain hand-rolls it — canvas tile layouts:

  • usePendingLayouts.ts:69-71 setOne, :72-85 applyMany, :86-106 dropEvicted (the equality-gated GC: "drop entries whose saved layout has caught up to their pending value"), :107-109 clear — a singleton createStore<Record<string, TileLayout>>.
  • Consumed at two sites: TerminalCanvas reads pending in layoutOf() and calls dropEvicted as echoes settle (per the module doc, usePendingLayouts.ts:1-13); useCanvasArrange.ts:56 seeds via applyMany.
  • The write itself is fire-and-forget RPC — useTerminalCrud.ts:50-56 setCanvasLayout is void client.terminal.setCanvasLayout(…).catch(toast); the server echo arrives through the existing metadata subscription that surface already owns.

The primitive that was proposed

A declarative mutation spec colocated with a defineSurface collection — declare the optimistic apply + the reconcile predicate once, let the framework own the pending store and eviction:

// PROPOSED — never built; recorded here to show what was rejected.
// A fifth primitive / collection verb over the surface descriptor model:
collection.patch(key, patch, {
  reconcile(saved, pending): boolean,   // equality gate — when has the echo caught up?
})
// framework would own: per-key pending createStore, optimistic apply,
// dropEvicted-on-echo, clear-on-unmount — the usePendingLayouts body, generically.

Contrast the real surface shape (built, shipped): defineSurface + implementSurface + surfaceClient with Cell/Collection/Stream/Event descriptors — see packages/surface/src/define and its consumption at packages/pty-host/src/ptyHostSurface.ts:155. defineMutation would have been a fifth verb on that model.

What it would dissolve — quantified

One module (usePendingLayouts.ts, 111 lines), two call sites, one domain. The grep is decisive: theme, intent, rename, and right-panel all use plain fire-and-forget RPC with no pending store anywhere (useTerminalCrud.ts:50-56 is the pattern — a void client…catch with the echo arriving over the metadata subscription). The recurrence count is 1. The proposal's own framing conceded "without future patterns, dissolution is zero."

Survived the skeptic? — No. The skeptic prevailed.

Strongest case for: it is adjacent to surface and names a real Hickey braid. The skeptic's rebuttal held on every axis: "Canvas layout mutations are not hard-axis transport problems — they are fine-grained reactive plumbing that fits Surface's existing local-authority Cell model. The claimed hard axis reduces to one-line store mutations + fire-and-forget RPC caught by existing metadata subscriptions. Surface's power is owning snapshot+deltas transport across reconnects; layout mutations never shift that axis." The "dual-store / temporal coupling" issue is a UI-plumbing problem solvable with a memo merging pending+saved — not transport-level hardness. Adding a 5th primitive on the strength of a single domain is speculative, and apply/reconcile semantics differ per-domain (geometry equality here), so the interface would leak domain assumptions the moment a second use case landed. The proposal admitting it was "lower-risk than immediate extraction" was itself the tell: a framework that clears the bar does not need that hedge.

lowy says

Not a hard volatility axis owned by a candidate framework — it is UI plumbing colocated where it belongs. The transport axis (snapshot+deltas+reconnect) is already owned by @kolu/surface; pending is just a UI cache. Parameterizing reconciliation per-domain would leak assumptions and break surface's interface contract. Keep as-is; re-evaluate only if a second mutation domain lands with identical patterns.

hickey says

All four bars fail with codebase evidence. No complecting (the code is simple, isolated, per-domain, well-commented); no fragmentation (pending, setOne, dropEvicted, clear already unified under one module); no concept multiplication (no other code uses pending states, so there is no canonical pattern being reinvented). Replacing colocated function calls with declarative syntax is ceremony, not plumbing reduction. The path forward is: keep it.

Graduation

Does not graduate. canvasLayout is a single field, not a collection (useTerminalCrud.ts:50); reconciliation is canvas-specific (geometry equality via layoutsEqual, usePendingLayouts.ts:100). There is no second app — and no second domain in this app — that would adopt it. The honest disposition: leave the module isolated; if a second optimistic-mutation domain (rename, reorder, comment-position) lands with the identical apply+echo shape, re-open the question then. That is a rule-of-two trigger, not a framework today.

Looks big, isn't the next surface

This is the honest core. Across nine probed axes — the file-watcher zoo, the optimistic write-side, server-side publish, intent/command, media capture, the god-procedures, "what should surface grow into?", and two open-ended fresh hunts in client and server — the candidates that looked framework-scale collapse under the bar to one of three diagnoses: helper-too-small (a real but narrow utility, not a hard axis), domain-coupled (the interface can't shed the domain, so extracting moves the braid rather than dissolving it), or already-covered (the hard axis is already owned by surface or an existing receptacle). The two open-ended hunts returned not-a-real-axis — no unnamed framework volatility surfaced beyond the seeded list.

Unresolved

None. The single probe taken to full adjudication (defineMutation) reached per-finding consensus across both lenses and the adversarial skeptic — all converging on helper-too-small / keep as-is. No framework verdict was left split.

The verdict — does kolu have a second surface in it?

No — the audit largely holds, with zero genuine framework exceptions. Held to the surface bar rather than the dedup bar, the ambitious sweep confirms the whole-app audit's central claim: "the big extraction already happened — surface ate the hardest volatility (client↔server state) end-to-end." The most credible "second surface" — the optimistic-write reconciliation framework adjacent to surface's Cell/Collection — was taken all the way to a lowy⇄hickey debate and an adversarial skeptic, and lost on all four tests: it owns a UI cache, not a transport axis; it recurs in exactly one domain; its reconcile semantics are domain-specific; and the write side it claims is already owned by surface's metadata subscription.

What remains after this probe is what the leaf sweep already found — a refcounting primitive, a fanout micro-utility, a few in-place fixes — plus organization work on the god-procedures. Valuable, but not framework-scale. @kolu/surface remains the only framework-scale receptacle kolu has extracted, and the only one it currently needs. The honest answer to "I want framework opportunities like surface" is: there isn't a second one hiding in the codebase today. The right trigger to re-open the question is concrete — a second optimistic-mutation domain (rename / reorder / comment-position) landing with the identical apply-then-echo shape, at which point defineMutation graduates from speculative to rule-of-two.

Methodology

the surface bar→ probe the hardest axes→ lowy ⇄ hickey debate→ adversarial skeptic gate→ per-finding consensus

Unlike the leaf sweep — which hunted duplicated structure — this pass started from the surface bar as the yardstick and probed the hardest remaining volatility axes the whole-app audit had flagged (the file-watcher zoo, the optimistic-local-vs-server-echo braid, the god-procedures). Each candidate was measured against all four tests at once — owns a hard axis end-to-end, a new programming model, dissolves a pervasive braid, graduates to a domain-agnostic package — using Lowy's volatility-based decomposition and Hickey's structural simplicity. Every framework verdict then faced an adversarial skeptic whose default was to refute it by opening the real call sites and grepping for the claimed recurrence. The skeptic prevailed on the one probe taken to adjudication — which is the load-bearing finding: the candidate that looked most like a second surface is not one.

Grounded against Juval Lowy, Righting Software (the electricity-receptacle metaphor) and Rich Hickey, Simple Made Easy (complecting vs. one-fold simplicity). The surface bar is lifted from the shipped @kolu/surface spec — defineSurface / implementSurface / surfaceClient over Cell/Collection/Stream/Event, with the streaming invariants enforced in .claude/rules/streaming.md.

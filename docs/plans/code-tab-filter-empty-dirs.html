Code-tab filter keeps empty directories — “why is search expanding directories which have no results?”

Diagnosis · branch search-collapsae · confirmed against Kolu source and @pierre/trees@1.0.0-beta.4 internals

Root cause: the filter removes the matching files, but nothing removes the directories those removals leave empty — and Pierre deliberately keeps an emptied directory as a visible “explicit” folder.

Searching docs plans batch-removes every non-matching file. Pierre’s removePath removes only the file node, then promotes the now-empty parent directory to EXPLICIT (its flag for “intentional empty folder, keep it shown”) instead of deleting it. So the whole tree of result-less directories survives. The wrapper comment that claims the opposite — “a file dropped takes its now-empty ancestor directories with it” FileTree.tsx:55-58 — is false for batch/remove; it is only true for resetPaths, which this branch abandoned to preserve hand-expansion.

How the filter is wired

The Code-tab search is host-driven — Pierre’s own search is off, and Kolu projects a path set into the tree:

type in filter→ searchQuery()→ projectFileTreeSearch→ { projectedPaths, expandedAncestors }→ <FileTree> batch + expand
  • projectFileTreeSearch(treePaths(), searchQuery()) CodeTab.tsx:331-333 returns projectedPaths (matching files only) and expandedAncestors fileSearch.ts:44-62. For docs plans that’s the files under docs/plans/ and the ancestors ["docs/", "docs/plans/"].
  • <FileTree> receives paths={…projectedPaths}, search={false}, expandPaths={…expandedAncestors} CodeTab.tsx:557,562-563. search={false} means Pierre’s hide-non-matches machinery is dormant — all filtering is Kolu-side.
  • The wrapper turns the old inventory into the new one as an in-place delta: pathDiffOperations(appliedPaths, paths) emits {type:"remove"} for each dropped file, then tree.batch(ops), then additively .expand()s the ancestors FileTree.tsx:205-227. It uses batch rather than resetPaths specifically so hand-opened folders survive.

Why the screenshot looks the way it does

Why the directories are present at all defect A

pathDiffOperations FileTree.tsx:59-73 diffs only file path lists; directories never appear in projectedPaths, so no remove op ever targets a directory. Every directory that contained only non-matching files is emptied by the file removals and then promoted to a persistent empty folder. Net: the entire pre-search directory skeleton remains, now hollow.

Why they’re shown expanded defect B

Two compounding facts, both about expansion being monotonic:

  • No debounce FileSearchInput.tsx:19 → every intermediate keystroke re-projects. Typing d matches almost every path, so expandedAncestors ≈ every directory, and the wrapper .expand()s them all.
  • As the query narrows to docs plans, the files vanish but the wrapper never collapses FileTree.tsx:198, and Pierre keeps the expanded flag on a directory even after it’s emptied. So the leftover empty dirs stay expanded.

.github/workflows is the lone collapsed row because no intermediate prefix ever expanded it. Even with a single paste of docs plans, the empty directories would still appear — just collapsed rather than expanded.

The trace, end to end

remove non-matching files (batch)→ parents emptied→ promoted to EXPLICIT→ still listed→ + stale expanded flag→ empty expanded rows

Fix direction — /hickey + /lowy verdict

Chosen: prune the emptied directories inside FileTree.tsx, derived from the file set. Reject the collapse pass. Reject resetPaths-on-filter.

Lowy fixes the boundary (the fix belongs in the wrapper; fileSearch.ts and CodeTab.tsx stay inert; do not inject filter-state into the search-agnostic wrapper). Hickey fixes the internals (derive directories from appliedPaths; no parallel appliedDirs state; no collapse pass). They are the intended complement — Lowy gets the seam right, Hickey gets the structure right.

Resolving the disagreement on A — derive, don’t track correctness, not taste

Lowy’s concrete prescription (“diff appliedDirs against new Set(expandPaths)”) has a latent filter-clear bug: on an empty query projectFileTreeSearch returns expandedAncestors: [] by design fileSearch.ts:49-50 while projectedPaths is the full inventory. Using expandPaths as the directory authority would therefore remove every directory the moment the user clears the filter. Deriving the authority from the file set is immune:

// after the file batch + appliedPaths = paths:
const prevDirs = new Set(appliedPaths.flatMap(ancestorDirectoryPaths));
const nextDirs = new Set(paths.flatMap(ancestorDirectoryPaths));   // full set on empty query ⇒ no removals
for (const dir of [...prevDirs].filter(d => !nextDirs.has(d))
                               .sort((a, b) => a.length - b.length)) {  // shallowest first
  const item = tree.getItem(dir);
  if (item) tree.batch([{ type: "remove", path: dir, recursive: true }]); // recursive: emptied dirs still hold empty child dirs
}

Two corrections to the originally-sketched op: (1) recursive: true — an emptied directory still contains its (now-explicit) empty child directories, so a bare remove throws canonical.js:51; (2) shallowest-first so the maximal empty subtree is removed in one op and deeper entries narrow away via the if (item) null-guard. No new tracked state: appliedPaths stays the single source, directories are a pure derivation.

Why defect B folds into defect A no collapse pass

Once empty directories are removed, the “expanded empty rows” symptom is gone — an absent directory has no row to render open. And among the directories that survive, every one is an ancestor of a live match, so it must stay expanded to reveal that match. There is no “surviving directory that should be collapsed” case during an active filter — a collapse pass would be a no-op at best and, where not a no-op, would re-hide matches and fight the user’s own collapse (the #867 “folders collapse while the filter stays active” feature). So Lowy’s symmetry is a false symmetry here: expand-to-reveal is mandatory, collapse-to-hide is never wanted. The expansion “policy” is simply “every surviving dir is open; a user collapse sticks” — the existing design, not a new pass.

Test plan — perfect coverage required

fileSearch.test.ts tests only the pure projection and cannot see this bug — it never drives the Pierre batch where promoteEmptyAncestorsToExplicit lives. Two layers, full coverage:

  • Pure unit — extract directoryRemovalOps(prevFiles, nextFiles) (the derivation above) and table-test it exhaustively:
    • empty/cleared query (nextFiles = full) ⇒ no removals;
    • single deep match ⇒ sibling subtrees removed, match ancestors kept;
    • nested empty subtree ⇒ one recursive op on the shallowest, no throw on the non-empty intermediate;
    • dir with one matching + one non-matching file ⇒ kept (no removal);
    • progressive narrowing (prev = broad matches → next = narrow) ⇒ dirs that lost all matches removed.
  • Integration — mount <FileTree> (jsdom, query the shadow-root rows): filter a populated tree and assert (a) emptied directories are absent, (b) directories still holding a match are present and expanded, and (c) a directory the user collapses during a filter stays collapsed across the next keystroke (the #867 guard — proves the additive expand loop doesn’t re-open hand-collapses).

Decision: Plan B (validated by prototype)

Plan B is the shape we'll build, validated by a stacked prototype: R-1 in #973 (server-spine rewrite around a single Backend interface) and R-2 in #976 (the kolu --stdio agent + RemoteBackend + UI). The prototype runs end-to-end on two real remotes — darwin (sincereintent, aarch64) and linux (vanjaram, x86_64) — but is not merged. Plan A is preserved at the bottom as an archived design alternative — useful for sharpening the contrast but not the path we took.

The remainder of this document is Plan B re-planned as if from scratch, ready for the real implementation — phases ordered by the dependencies and surprises the prototype surfaced, not by the order they happened to land. The "Lessons" section below names the concrete things the prototype taught us; each phase explicitly references the lesson(s) it depends on.

Plan B — Rewrite-freedom (one Backend spine)

One Backend interface owns the per-terminal world. The kolu server holds a backend; that backend produces every per-terminal stream and one-shot op. LocalBackend wraps the in-process providers; RemoteBackend proxies the same interface over oRPC-on-stdio to a kolu --stdio peer running on the remote. The agent is the same kolu binary with a different transport — zero remote-specific business logic.

Prototype: R-1 in #973, R-2 stacked on top in #976. Live on two real remotes; all 24 CI nodes green on R-2 HEAD. Neither PR is merged — the prototype validated the design and informs the real-implementation plan below. Pending: R-3 (PTY ownership migration so terminals survive network blips).

Phases at a glance

Three phases. R-1 is a pure refactor with no user-visible change; R-2 ships the entire remote feature surface in one go; R-3 adds network-blip resilience. R-2's six sub-slices (2a–2f) are an internal build order, not separate ships — none of them is a user-facing feature on its own.

PhaseOne-line gistUser-facing changeStatus
R-1 Server spine collapses around one Backend interface. LocalBackend wraps the in-process providers; meta/*.ts orchestrators dissolve into it. One transport boundary in the system. None. Pure refactor — behavior is byte-identical. prototyped #973
R-2 Same kolu binary runs as kolu --stdio on the remote via ssh nix run --refresh. RemoteBackend proxies the entire Backend surface over oRPC on stdio. Agent has zero remote-specific logic — just another oRPC peer. Everything. Remote terminals work end-to-end: PTY, branch chip, agent badge, GitHub PR, Code tab, file upload, session restore, HostChip on every surface, install flow. prototyped #976
R-3 Stable per-session IDs persist; agent grows attachPty(id) to reattach instead of spawn-fresh; contract version handshake replaces --refresh; idle-TTL GC on abandoned remote sessions. Remote terminals survive network blips — laptop sleep, wifi drop, reconnect with full scrollback restored. planned

Would the split change second time?

Short answer: no. The top-level R-1 / R-2 / R-3 split is unchanged. Phase boundaries were right; what shifted is the content inside each phase, not where the cuts go.

Possible reshuffleVerdict
Combine R-1 and R-2 No. Unreviewable single PR. R-1's "delete legacy meta/* in the same PR; tests are the safety net" bet only holds when it isn't also introducing remote-only code paths.
Split R-1 further (introduce Backend without dissolving meta/*) No. The dissolution is what makes R-2 simple. Splitting just defers the same work and adds a dual-path transitional period.
Split R-2 at slice 2d (bare shell vs. rich surfaces) Maybe. The only real seam the prototype made visible. 2a–2d = working remote PTY, no host chip / no agent badges; 2e–2f = host-aware UI + channel-mirroring on top. The prototype landed both in one branch because the work took a day; a bare-shell-only intermediate ships as half-finished. If review-burden matters more than intermediate-state UX, this is the obvious cut.
Split R-3 into sub-PRs No. R-3's pieces (persist remoteSessionId, attachPty, scrollback replay, idle GC, version handshake) are small enough that 3 PRs is more ceremony than benefit.
Move R-3 (or parts) earlier No. Without R-2 there's no remote feature to make resilient.

The prototype's effect on the plan is internal to each phase, not structural:

  • R-2 grew an internal six-slice build order (2a→2f) that the original Plan B writeup didn't have — it just listed "components" in no particular dependency order. The slice order is now load-bearing for the redo.
  • R-3 is smaller in scope than originally framed. "PTYs live in the backend that hosts them" fell out of R-2 for free — the agent's LocalBackend.spawnPty runs the PTY on the remote. R-3 reduces to: persist remoteSessionId, add attachPty(id), replay scrollback, idle GC, version handshake. No architectural shift.
  • One pattern migrates from R-2 into R-1's interface design. "Register shadow entry sync, do I/O async" is currently retrofitted into RemoteBackend.spawnPty only. Baking it into the Backend.spawnPty contract (every spawn registers an entry sync, then does I/O async) costs LocalBackend nothing — its I/O is instantaneous — but it means RemoteBackend is "just following the contract" rather than "doing a special pattern."

Lessons that shape the redo

The prototype invalidated specific load-bearing assumptions in the original Plan B writeup. These are the surprises that, if we were starting fresh today, would be designed in from the first commit rather than retrofitted after running into the failure mode.

The two volatility axes the design carves out

Lowy's question: what changes for its own reasons, and is each axis isolated to one module? Plan B carves exactly two:

AxisEncapsulated byReason
Where a terminal's state lives (this machine vs. a specific SSH host)Backend interfaceThis is the only thing local-vs-remote changes. Everything downstream — the router, the surface, the orchestrators — talks to the Backend and doesn't know which kind it has.
How RemoteBackend reaches its agent (transport, framing, reconnect)HostSession + StdioRPCLink + stdio-peer framingThe transport changes for its own reasons (ssh today, mTLS-over-TCP tomorrow, websocket-tunnel after that). RemoteBackend doesn't care which — it asks HostSession for a typed client.

Hickey's complement: what concept duplication would have crept in if we'd split differently? The original Plan A "provider-per-domain" cut had N (axis × domain) modules: RemoteGitInfoProvider, RemoteAgentProvider, RemoteFsProvider, etc. — each one a transport adapter wearing a domain costume. Plan B has one transport adapter (RemoteBackend) and zero domain knowledge crosses the transport boundary. The agent runs the unmodified local providers; the boundary just ships their output.

R-1 — Backend abstraction + LocalBackendprototyped #973

User outcome: none. Pure architectural work — the server spine is rewritten around one interface; behavior is byte-identical.

The interface (final shape)

interface Backend {
  readonly id: TerminalLocation;        // { kind: "local" } | { kind: "ssh", host }
  spawnPty(opts: PtySpawnOpts): Promise<TerminalHandle>;
  terminalChannel<K>(terminalId, kind: K, signal?): AsyncIterable<TerminalChannelMap[K]>;
  killTerminal(terminalId): boolean;
  killTerminalEntry(entry): void;       // bulk-kill path, takes entry directly
  uploadFile(terminalId, name, base64Data): Promise<string>;
  readonly fs: BackendFs;               // listAll, readFile, subscribeRepoChange, subscribeFileChange
  readonly git: BackendGit;             // getDiff, getStatus
}

Three invariants the implementation enforces:

  1. Kill convergence. Backend.killTerminal(id) is the sole termination path. TerminalHandle does NOT carry dispose() — handle-as-control-surface and kill-as-lifecycle are two roles that get smuggled through one method if you let them.
  2. Snapshot-then-delta. Every terminalChannel<K> first yield is a full state snapshot, then deltas (the rule in .claude/rules/streaming.md). This is what makes oRPC's ClientRetryPlugin reconnect transparently — and what makes the equivalent on the agent ↔ server hop also reconnect transparently in R-3.
  3. Backend owns its filesystem. BackendFs/BackendGit cover BOTH one-shot ops AND watcher subscriptions. Same axis, one site. (This was the post-implementation Lowy finding; designed in here.)

What R-1 touches

  • packages/common/src/backend.ts — the interface, channel map, fs/git surfaces.
  • packages/server/src/backend/local.ts — LocalBackend. Per-terminal provider startup (claude-code / codex / opencode / github / foreground) moves inside spawnPty; the kolu server's meta/*.ts orchestrators dissolve into private wiring here.
  • packages/server/src/terminals.ts, terminal-registry.ts, router.ts — switch from "import kolu-git directly" to "ask the backend." All per-terminal and per-terminal-host operations resolve through getBackendFor(location).

Risk mitigation

  • No dual path. The legacy meta/* orchestrators get deleted in the same PR that lands LocalBackend. A feature flag would mean two implementations in master with their own drift — the bet is that LocalBackend wraps the same calls the legacy path made, and the test suite is the safety net.
  • Session migration is exhaustively tested. Every persisted-shape revision in git log packages/common/src/surface.ts gets a golden fixture; round-trip + property tests cover migration; e2e loads each historical fixture in a real kolu and verifies the canvas matches.

R-2 — kolu --stdio agent + RemoteBackendprototyped #976

User outcome: the entire remote feature surface lights up at once. Remote terminals work. Branch chip, agent badge, GitHub PR badge, Code tab, file upload — every rich surface — work identically on remote tiles. The only visible difference is the host chip.

For the real implementation, R-2 builds in six self-contained slices. Each slice has a clear "done" criterion you can verify before moving on; if you build them in this order, each one's failure mode is bounded by the slice's surface area. The prototype (#976) landed all six in one branch — splitting them is a review-burden tradeoff (see Would the split change second time?).

2a — Install probe + remote command builder

Done when: installAgent(host) probes nix --version over ssh and surfaces a typed error if nix is missing; remoteAgentCommand(host) returns the argv to spawn.

  • The command: ssh $host nix run --accept-flake-config --refresh <flakeRef> -- --stdio. (Lessons: binary shipping, contract evolution.)
  • Flake ref source: KOLU_AGENT_FLAKE_REF env var; defaults to the active dev branch. R-3 pins to a release tag with an explicit version handshake so --refresh can be dropped.
  • Why not nix copy? See lessons table, row 1. The TL;DR is cross-arch closures can't be realised locally without exactly-right remote-builder config — kolu's user base already has Nix on the remote, so we let the remote build natively.

2b — Stdio framing + narrow agentContract

Done when: a hand-written test wires ServerPeer and ClientPeer through an in-memory pipe and round-trips heartbeat().

  • Framing: base64-encode each message; newlines delimit. Base64 never contains a newline, so framing is unambiguous over the byte stream. One adapter (stdio-peer.ts) used by both ends.
  • Log redirection: log.ts detects process.argv.includes("--stdio") at module load and forces pino output to fd 2 (stderr). MUST be in place before the first import-time log call — otherwise the first server-startup line corrupts the protocol. (Lesson: agent's stdout.)
  • Narrow contract: agentContract is NOT the full appRouter. Exposing terminal.create would let an agent spawn another remote terminal-on-a-terminal (recursion). Exposing surface.* leaks client-facing UX plumbing into a server-internal protocol with a different volatility axis. Agent serves: heartbeat, terminal.{spawn,kill,write,resize,uploadFile}, one channel* procedure per TerminalChannelMap key, and fs.*/git.*.
  • Per-channel procedures, not one routed procedure. terminal.channelData, terminal.channelGit, etc. — one per key. A single terminal.channel(kind: string) erases the per-channel output schema; TypeScript can't catch a kind/schema mismatch. The "one row per kind" boilerplate on both sides is the price of type-safety here.

2c — HostSession with deferred heartbeat

Done when: HostSession.connect() spawns the subprocess and builds the client; markReady() (called externally after the first RPC succeeds) transitions to connected and starts the heartbeat loop; subprocess exit transitions to disconnected; reconnect ladder follows the Zed constants.

  • Constants ported from Zed (/tmp/zed/crates/remote/src/remote_client.rs:149-155): HEARTBEAT_INTERVAL=5s, HEARTBEAT_TIMEOUT=5s, MAX_MISSED=5, MAX_RECONNECT_ATTEMPTS=3.
  • State machine (Zed :157-293): connecting | connected | heartbeat-missed(n) | reconnecting(attempt) | disconnected(reason). projectExternal() collapses to the three-state ConnectionState the UI sees.
  • The deferred-heartbeat invariant (lesson: heartbeat timing): connect() spawns the ssh subprocess and constructs the client BUT does NOT start the heartbeat. The first RPC roundtrip (from RemoteBackend) calls markReady(), which transitions to connected and starts the interval. This survives cold nix run realisation that can take minutes.
  • State listeners fire snapshot-then-delta: onStateChange(cb) calls cb(currentState) synchronously before subscribing to changes. This is what lets RemoteBackend.startStateSubscriber attach BEFORE connect() returns and still see the initial connecting state.

2d — RemoteBackend: instant tile + async tail

Done when: clicking "new terminal on $host" returns a tile in <50ms with a "Connecting…" overlay; the overlay flips to a live shell once nix run realises.

  • Pre-generate the id. spawnPty generates const id = opts.id ?? crypto.randomUUID() BEFORE any RPC. (Lesson: terminal ID authority.)
  • Register the shadow entry synchronously. Build a remoteHandle (PtyHandle-shaped proxy whose write/resize/dispose forward via session.client), construct the TerminalProcess with meta.connectionState = "connecting", call registerTerminal(id, entry). The tile renders.
  • Start the state subscriber synchronously, channel subscribers asynchronously. State subscriber doesn't need session.client (pure in-process listener on HostSession.onStateChange). Channel subscribers DO need a live client; they start in the async tail after markReady(). Starting them before connect() dies on the first iteration. (Lesson: channel subscribers.)
  • The async tail: 
    void (async () => {
  try {
    await this.session.connect();
    await clientOf(this.session).terminal.spawn({ id, cwd, initialMetadata });
    this.session.markReady();             // ← first RPC succeeded; start heartbeat now
    const stopChannels = this.startChannelSubscribers(id, metaMod);
    entry.stopProviders = compose(stopState, stopChannels);
  } catch (err) {
    log.error({ host, id, err }, "RemoteBackend.spawnPty: async tail failed");
    // HostSession's subprocess-exit handler transitions to "disconnected",
    // which the state subscriber mirrors into entry.meta.connectionState,
    // which the DisconnectedOverlay renders. No bespoke failure path needed.
  }
})();
return { id, write: handle.write, resize: handle.resize };
  • DisconnectedOverlay covers connecting AND disconnected. Same component, two states. Connecting shows spinner + "Connecting to $host…" + hint about nix realisation; disconnected shows reason + Reconnect / Close-tile buttons. (Lesson: tile appearance.)

2e — Channel-mirroring for rich surfaces

Done when: a remote tile's agent badge, PR badge, foreground process, cwd, and git info all update from the agent's in-process providers.

  • The agent's LocalBackend already runs the providers — claude-code, codex, opencode, github poll, foreground. They publish to its terminalChannels.
  • RemoteBackend.startChannelSubscribers loops for await on this.terminalChannel(id, "agent") (which RPC-streams from the agent) and applies each update to the kolu-server's entry.meta via updateServerLiveMetadata. Same dance for pr, foreground; cwd and git use updateServerMetadata (fires terminals:dirty so autosave picks it up).
  • The connectionState channel is in-process — it's the kolu server's view of HostSession state, NOT a thing to fetch from the agent. RemoteBackend.terminalChannel(id, "connectionState") wraps HostSession.onStateChange in an AsyncIterable; the agent's channelConnectionState exists for symmetry but RemoteBackend short-circuits before calling it.

2f — Host-aware UI plumbing

Done when: every surface that names a terminal also names its host (when remote); session restore preserves location; sub-terminals inherit location.

  • HostChip on five surfaces: title bar (TerminalMeta.tsx), dock pill (Dock.tsx — QuietRowBody, AwaitingCardBody, WorkingPillBody), workspace switcher card (WorkspaceGrid.tsx), restore card (EmptyState.tsx), saved-terminal list. The chip renders nothing when location.kind === "local" — common case stays clean.
  • SSH host picker: command palette lists hosts from ssh -G $host probes against ~/.ssh/config aliases. No bespoke parser (saves the Zed parser port and stays correct as ssh-config syntax evolves).
  • Session restore preserves location: useSessionRestore.ts's call to handleCreate threads t.location through. e2e test: save a remote session, kill kolu, restart, verify the tile reconnects to the right host. (Lesson: session restore.)
  • Sub-terminal inheritance: getBackendForCreate({ parentId }) reads the parent's meta.location and resolves the backend by that, regardless of what the create input said. Centralizes inheritance to one resolver — createTerminal doesn't pattern-match on parent state.

R-3 — PTY ownership + reattach (network-blip resilience)planned

User outcome: laptop sleep / wifi drop / suspend-resume leaves remote terminals alive with full scrollback. The axis Zed itself doesn't solve (Zed terminals die with the ssh subprocess) — kolu can go further because the agent is already there as the natural home for a tmux-style PTY broker.

  • PTYs live in the backend that hosts them. For RemoteBackend, that means the agent process owns the PTY across ssh drops. R-2 already has this property — the PTY is in the agent's LocalBackend, the kolu server is just a stream consumer.
  • Stable session IDs. Persist remoteSessionId per terminal. On reconnect, call backend.attachPty(remoteSessionId); agent replays a fresh scrollback snapshot (snapshot-then-delta, same shape as the existing data channel's first yield) and resumes streaming.
  • Idle TTL on the agent (e.g. 24h) so abandoned remote sessions don't accumulate.
  • Version handshake replaces --refresh. Agent reports its contract version on first connect; kolu server compares against its compiled-in expectation. Version-skew triggers a single explicit --refresh + reconnect, not blanket --refresh on every spawn.
  • Pre-warm: after installAgent, optionally fire-and-forget ssh $host nix build --no-link <flakeRef> so the first real terminal-spawn doesn't wait on cold realisation. (Today's UX is already acceptable via instant tile + overlay, but pre-warm closes the loop.)

Open refinement: Backend ↔ Surface subsumptionneeds answers

Working through the prototype surfaced a structural question that the current Plan B redo doesn't address: the agent transport in #976 hand-rolls ~300 LOC of oRPC framing, per-channel relay boilerplate, and stdio peer wiring that overlaps conceptually with @kolu/surface's existing snapshot+delta, retry, and pub/sub primitives. Subsuming the agent transport under Surface should shrink Kolu by ~300-400 LOC of glue and expand @kolu/surface by ~150-200 LOC of generic transport adapters — net negative LOC plus a unified mental model for "typed reactive state over an oRPC link."

Status: fully implemented in #978. Q1/Q2/Q4/Q5 answered; Q3 deferred (the phase split shape is now clearer — Phase 1 + Phase 2 both landed in one stacked PR). End-to-end verified on darwin + linux remotes via chrome-devtools-mcp.

Framework gaps in @kolu/surface

Today Surface is browser-coupled at three spots. Closing these gaps is the prerequisite for the subsumption — and the additions are generic: any future consumer of @kolu/surface over a non-WebSocket link benefits.

  1. Transport pluggability. createCellsClient(opts: { websocket }) in packages/surface/src/client.ts hardcodes RPCLink from @orpc/client/websocket. Add a stdio link adapter (~80 LOC) that takes { stdin, stdout } and wires ClientPeer from @orpc/standard-server-peer instead.
  2. Non-Solid client builder. surfaceClient.ts returns Solid Subscriptions built with createMemo — not usable in Node.js (the kolu-server consumes the agent surface but isn't a Solid component). Add a plain-client builder (~50 LOC) returning async iterables for streams/collections and promises for procedures.
  3. implementSurface over stdio. Today the server runtime assumes HTTP/WS. Pair implementSurface's router fragment with ServerPeer reading stdin / writing stdout (~80 LOC).
  4. In-memory CellStore + Channel impls. The agent's cells are ephemeral (no disk persistence) and channels are in-process within the agent (the wire goes through the link). Both already exist as patterns; lift as framework exports (~50 LOC).

Total framework expansion: ~260 LOC, all generic. Total Kolu shrinkage: ~300-400 LOC of hand-rolled glue deleted. Net: smaller, plus a unified mental model.

Open design questions

Q1 ✓ Channel shape on the agent surface. The biggest architectural payoff vs. the smallest semantic change.

  • (a) Full collapse — 1 collection + 1 stream. Eight metadata channels (cwd, title, git, commandRun, agent, pr, foreground, connectionState) become one terminalMetadata collection reusing kolu-common's existing schema. data stays a separate stream (high-throughput PTY bytes). Biggest architectural shrinkage; semantic change on the kolu-server side (one subscription instead of 8 mirroring loops).
  • (b) Preserve 9 streams.
  • (c) Hybrid.
ANSWERED: (a) Full collapse. Chosen. The kolu-common terminalMetadata collection's existing schema is the wire shape; the 8-channel split is a LocalBackend-internal implementation detail. RemoteBackend's channel-mirroring loops collapse to one subscription.

Q2 ✓ Fate of the Backend interface once Surface absorbs the transport.

  • (a) Keep Backend as the routing layer (thinner version of today's interface).
  • (b) Dissolve Backend entirely — kolu-server holds Map<TerminalLocation, SurfaceClient>, no formal interface.
  • (c) Keep Backend as a typed wrapper. Thin facade over a Surface client; each method is a 1-line passthrough. ~80 LOC total.
RECOMMENDED: (c) typed wrapper, given the criterion "Kolu as little code as possible." Reasoning: (b) saves ~30 LOC at the seam but spreads Map<TerminalLocation, SurfaceClient> lookup logic across every call site (each meta consumer writes clientForTerminal(id).rpc.terminal.write(...)), adding ~50-80 LOC in scattered dispatch over time. (c) wins on net LOC across the codebase, not just at the seam, and keeps a named class for future per-backend behavior (caching, retry policy, instrumentation) without paying for them today.

Volatility axes the feature crosses

Four axes, each with its own seam. Each axis lives behind a typed provider interface so the dispatch happens once at terminal creation time, not at every call site.

Glossary.
  • Axis (of change / of volatility). One independent dimension along which the system might evolve later. Lowy's framework (after Parnas 1972) says each module should encapsulate exactly one axis so a change along that axis touches one place.
  • Seam. A spot in the code where you can swap behavior without editing every call site. Term from Michael Feathers, Working Effectively with Legacy Code. Here, each provider interface (AgentProvider, GitInfoProvider, …) is a seam.
  • Receptacle. Lowy's framing for the same idea — a typed slot the consumer holds while the implementation behind it changes. Provider interfaces are receptacles; concrete LocalXxxProvider / RemoteXxxProvider implementations slot in.

Why not a per-host HostContext bundle? An earlier draft proposed bundling { ptyFactory, gitClient, agentClient, fsClient, executor } per host. Hickey + lowy passes flagged it as functional decomposition wearing a discriminated-union costume: it groups by "stuff about a host" rather than by axis of change, and consumers still pattern-match on the union to know what's filled in when. The provider-per-domain shape is the volatility-axis cut — each domain interface changes for its own reasons, and the dispatch is concentrated in terminal-creation only.

Why not thread location: TerminalLocation through every integration's public function? Same review pass: that braids transport dispatch (ssh-or-not) into each integration's domain function (subscribeGitInfo), forcing every caller — even purely local ones — to carry a location argument. The provider pattern keeps integrations domain-pure: the remote variant lives in a separate kolu-remote-client package and never touches kolu-git internals.

}

// Same package — local impl wraps today's code, no behavior change:

// NEW package: packages/kolu-remote-client/src/git.ts
import type { GitInfoProvider } from "kolu-git";
import type { HostSession } from "./hostSession.ts";

export class RemoteGitInfoProvider implements GitInfoProvider {
  constructor(private session: HostSession) {}
  subscribe(cwd, onChange, log) {
    const token = this.session.subscribe("git.info", { cwd }, onChange);
    return {
      setCwd: (next) => token.update({ cwd: next }),
      stop: () => token.close(),
    };
  }
}

The one new top-level abstraction: HostSession

Per-host singleton, owns the ssh-stdio RPC channel and all subscription tokens to that host. Hickey's connection-lifecycle finding: when the ssh connection drops and reconnects, HostSession tears down every live token and re-issues them via the new connection. The integration-level providers never see the churn — their onChange callbacks just keep firing on the new tokens.

interface HostSession {
  // Request/response, awaits result
  call<Req, Res>(method: string, args: Req): Promise<Res>;
  // Streaming subscription. Returns a token whose lifetime HostSession owns.
  // On reconnect, HostSession re-issues all live tokens with the same args
  // and resumes calling the same onEvent — provider code never notices.
  subscribe<Args, Event>(
    method: string,
    args: Args,
    onEvent: (e: Event) => void,
  ): SubscriptionToken<Args>;
  // Lifecycle
  state(): ConnectionState; // Connecting | Connected | Reconnecting | …
}

  • Local xterm-headless emulator stays — for late-join client serialization. The emulator's onData is fed from the RPC subscription instead of a node-pty subprocess.
  • Agent-side GC: idle TTL (e.g. 24h) so abandoned remote sessions don't accumulate.

Why keep terminal bytes and metadata on separate channels? Terminal bytes are high-throughput, latency-sensitive raw streams; metadata is structured, lower-frequency request/response. Multiplexing them onto one channel tangles backpressure semantics — Zed makes the same split for the same reasons. The provider model already gives them separate seams (PtyProvider stream vs. HostSession.call for ops). hickey: separation is the feature.

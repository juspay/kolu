kolu#951 — Remote terminals over SSH (implementation plan)

Four-phase plan to ship remote terminals fresh off master. Distilled from three exploratory prototype PRs (#973, #976, #978) but framed as what to build, not what we built. The prototypes are linked from the implementation reference at the bottom — copy file shapes from there, but make the design choices yourself per this document.

Phases at a glance

Four phases, ordered by what unblocks what. R-1 reshapes the server spine (no user-visible change); R-1.5 lands the generic @kolu/surface primitives the agent transport rides on (no user-visible change); R-2 then drops the entire remote-feature surface on top (the headline user shift); R-3 adds network-blip resilience.

PhaseGistUser-facing change
R-1 Server spine collapses around one Backend interface. LocalBackend wraps the in-process providers; meta/*.ts orchestrators dissolve into it. None. Pure refactor — behavior is byte-identical.
R-1.5 @kolu/surface grows stdio link, loopback link, peer-server, and in-memory channel. Validated by a subprocess-worker demo (search-index worker) in the framework's own example app. None directly. To framework consumers: "@kolu/surface now supports subprocess-worker patterns."
R-2 Same kolu binary runs as kolu --stdio on the remote via ssh nix run. RemoteBackend consumes the agent's typed surface via R-1.5's primitives. Everything. Remote terminals work end-to-end: PTY, branch chip, agent badge, GitHub PR, Code tab, file upload, session restore, host chip on every surface.
R-3 Stable per-session IDs persist; agent grows attachPty(id); explicit contract version handshake replaces the --refresh hack; idle-TTL GC on abandoned remote sessions. Remote terminals survive network blips — laptop sleep, wifi drop, reconnect with full scrollback restored.

Two volatility axes the design carves out

Lowy's question: what changes for its own reasons, and is each axis isolated to one module? The design carves exactly two:

AxisEncapsulated byReason
Where a terminal's state lives (this machine vs. a specific SSH host)Backend interfaceThe only thing local-vs-remote changes. Everything downstream — the router, the surface, the orchestrators — talks to the Backend and doesn't know which kind it has.
How RemoteBackend reaches its agent (transport, framing, reconnect)HostSession + @kolu/surface/links/stdioThe transport changes for its own reasons (ssh today, mTLS-over-TCP tomorrow, websocket-tunnel later). RemoteBackend doesn't care — it asks HostSession for a typed surface client.

Hickey's complement: an earlier design draft had per-domain remote providers (RemoteGitInfoProvider, RemoteAgentProvider, RemoteFsProvider, …) — each a transport adapter wearing a domain costume. The single-Backend cut has zero domain knowledge crossing the transport boundary. The agent runs the unmodified local providers; the boundary just ships their output.

Lessons informing the design

Six lessons from the exploratory prototype that shape the choices below. Each names a specific failure mode someone redoing this would otherwise hit.

#LessonConcrete design choice
1 Hand-rolled transport adapters mask framework work. If the code you're writing in your application would benefit any oRPC consumer (not just yours), it belongs in the framework. R-1.5 lands stdio link, loopback link, peer-server, in-memory channel in @kolu/surface — generic, zero Kolu specifics. R-2 consumes them. Don't write a stdio adapter inside packages/server/.
2 Per-entity reactive state wants collections, not parameterized streams. "What changed about entity X" is what Collection<K, V> is for; snapshot+key-tracking semantics come built-in. Stream is for keyed compute (search results); collection is for keyed state. The agent surface has ONE terminalMetadata collection keyed by terminal id, NOT eight parameterized Stream<{id}, T> entries. Aggregation happens on the agent; RemoteBackend subscribes once.
3 Framework code needs an end-to-end demo, not just unit tests. The demo is a falsifiability test: if writing it against the primitives feels awkward, the primitives are at the wrong altitude. R-1.5 is not "done" until the example app's worker/ sub-demo exercises the stdio link, peer-server, and in-memory channel with snapshot+delta semantics observed via the browser UI.
4 "Shadow entry sync, I/O async" belongs in the Backend.spawnPty contract. Registering a connecting-state entry synchronously, then doing all I/O on a later tick, is what makes the instant-tile UX work. R-1's Backend.spawnPty contract documents the sync-shadow / async-tail pattern. LocalBackend's I/O is instantaneous so it's a no-op there; RemoteBackend follows the same contract.
5 Wire-shape drift is a real bug class — fail fast, no defaults. If kolu-server and the remote agent are built from different sources, every RPC 404s silently (kolu-server speaks surface.X, agent serves something else). R-2's install.ts requires KOLU_AGENT_FLAKE_REF. No default — throws at first remote-spawn with an error message naming the bug class. Operator chooses the ref explicitly (dev branch / release tag).
6 Cold remote realisation can take minutes — design for it. First nix run on a fresh remote builds the closure; without UX accommodation the user stares at a blank tile. R-2 designs in three mitigations from the start: (a) instant tile + DisconnectedOverlay's connecting-state spinner; (b) kolu-server pre-generates the terminal id so it can register before the spawn RPC roundtrips; (c) heartbeat deferred via markReady() — doesn't start until first RPC succeeds, so the 5s × 5 timeout doesn't fire during the build.

R-1 — Backend abstraction + LocalBackend

User outcome: none. Pure architectural work — the server spine is rewritten around one interface; behavior is byte-identical.

What "Backend" means

A Backend is the per-terminal world a terminal lives in. Concretely: what process owns the PTY, what filesystem the Code-tab reads, where the git watcher runs, where the agent-state providers (Claude Code, OpenCode, Codex) observe their session files. Every per-terminal stream and every per-host one-shot op goes through this object.

Two concrete kinds:

  • LocalBackend — this kolu process. The PTY is spawned in-process via node-pty; providers watch local files via @parcel/watcher; git/fs ops shell out locally.
  • RemoteBackend — a specific SSH host. The PTY runs in the kolu --stdio agent on that host; every method proxies via oRPC over the agent's surface (R-2).

What it replaces. Without this interface, the kolu-server's meta/*.ts orchestrators import kolu-git, kolu-anyagent, kolu-github directly and watch local files. Adding remote support without a backend interface means each integration grows a location parameter or a parallel Remote* twin — transport dispatch braided into every domain function. The Backend interface concentrates the local-vs-remote decision at one seam (terminal creation, via getBackendFor(location)); every consumer downstream just talks to backend.X and never asks "which kind?"

What dissolves into it. LocalBackend absorbs the kolu-server's meta/*.ts per-terminal orchestrators (claude-code reconciler, codex reconciler, opencode reconciler, github PR poller, foreground-process detector, git watcher, cwd watcher) as private wiring inside its spawnPty. These files delete as a kolu-server-side layer in R-1 — they survive only as their integration-package source (kolu-claude-code, kolu-git, etc.) that LocalBackend calls.

Why this is the right cut. See the volatility axes: "where a terminal's state lives" is one independent axis. Carving the system at that axis means every other piece of the design (router, surface, restore, UI) treats local and remote as the same shape. The only place that pattern-matches on location.kind is getBackendFor, the resolver itself.

The interface

interface Backend {
  readonly id: TerminalLocation;        // { kind: "local" } | { kind: "ssh", host }
  spawnPty(opts: PtySpawnOpts): Promise<TerminalHandle>;
  terminalChannel<K>(terminalId, kind: K, signal?): AsyncIterable<TerminalChannelMap[K]>;
  killTerminal(terminalId): boolean;
  killTerminalEntry(entry): void;       // bulk-kill path, takes entry directly
  uploadFile(terminalId, name, base64Data): Promise<string>;
  readonly fs: BackendFs;               // listAll, readFile, subscribeRepoChange, subscribeFileChange
  readonly git: BackendGit;             // getDiff, getStatus
}

Invariants

  1. Kill convergence. Backend.killTerminal(id) is the sole termination path. TerminalHandle does NOT carry dispose() — handle-as-control-surface and kill-as-lifecycle are two roles that get smuggled through one method if you let them.
  2. Snapshot-then-delta on every stream. terminalChannel<K> first yield is a full state snapshot; subsequent yields are deltas (the rule in .claude/rules/streaming.md). This is what makes oRPC's ClientRetryPlugin reconnect transparently — and what makes the equivalent on the kolu-server ↔ agent hop also reconnect transparently in R-3.
  3. Backend owns its filesystem. BackendFs/BackendGit cover BOTH one-shot ops AND watcher subscriptions. Same axis ("where the FS lives"), one place to dispatch.
  4. Sync shadow entry, async I/O. spawnPty registers a TerminalProcess entry synchronously (so the tile renders immediately), then performs any I/O on a later tick. LocalBackend's I/O is instantaneous so this is a no-op there; RemoteBackend needs minutes for cold nix run realisation and the contract is what makes the instant-tile UX work.

What R-1 touches

  • packages/common/src/backend.ts — the interface, channel map, fs/git surfaces.
  • packages/server/src/backend/local.ts — LocalBackend. Per-terminal provider startup (claude-code / codex / opencode / github / foreground) moves inside spawnPty; the kolu server's meta/*.ts orchestrators dissolve into private wiring here.
  • packages/server/src/terminals.ts, terminal-registry.ts, router.ts — switch from "import kolu-git directly" to "ask the backend." All per-terminal and per-host operations resolve through getBackendFor(location).

Risk mitigation

  • No dual path. The legacy meta/* orchestrators get deleted in the same PR that lands LocalBackend. A feature flag would mean two implementations in master with their own drift; the bet is that LocalBackend wraps the same calls the legacy path made, and the test suite is the safety net.
  • Session migration exhaustively tested. Every persisted-shape revision in git log packages/common/src/surface.ts gets a golden fixture; round-trip + property tests cover migration; e2e loads each historical fixture in a real kolu and verifies the canvas matches.

R-1.5 — @kolu/surface framework gaps

User outcome: none directly. To framework consumers: "@kolu/surface now supports subprocess-worker patterns" — same typed reactive primitives that power browser UI now work for sandboxed compute workers, IDE-extension language servers, remote SSH workers, etc.

Surface today is browser-coupled at three spots. R-1.5 closes those gaps with generic additions — zero Kolu specifics, useful to any oRPC consumer over a non-WebSocket link.

What lands in @kolu/surface

  1. Stdio link adapter (@kolu/surface/links/stdio) — StdioRPCLink + createStdioCellsClient. Wires an oRPC ClientPeer (from @orpc/standard-server-peer) to a Readable/Writable pair via base64+newline framing. Direction-neutral options (read / write) so client and server use the same shape.
  2. In-process loopback (@kolu/surface/links/loopback) — createLoopbackPair() returns crossed PassThrough streams so a server and client in the same process exercise the same framing as a real subprocess pair. Enables symmetric "local backend wrapped in the same client shape as a remote backend" patterns; also useful for testing.
  3. Peer server (@kolu/surface/peer-server) — serveOverStdio({ router, transport }) pumps any typed oRPC router through ServerPeer. Default transport is process.stdin / process.stdout; loopback consumers pass the server half of a LoopbackPair.
  4. In-memory channel — inMemoryChannel<T>() in @kolu/surface/server next to existing inMemoryStore<T>. Single-process broadcast pub/sub for surfaces served from a Node-only process where publisherChannel's external publisher dep is overkill.

The demo requirement

R-1.5 isn't done until packages/surface/example/ grows a subprocess-worker sub-demo that exercises all four additions with a real user-facing payoff. Reference shape: search-index worker. Child process subscribes to the example app's notes collection over stdio, maintains an inverted index, publishes search results back via its own surface. Parent server's search stream delegates to the worker; note upsert/remove triggers index.update snapshot pushes.

User-facing benefit visible in the example: search stays fast as the notes corpus grows because the index lookup runs in a sandboxed worker. The browser's search UX is unchanged — the architectural shift is invisible at the call site, which is exactly the framework's value.

Why the demo matters for design. If the demo is awkward to write against the new primitives, the primitives are at the wrong altitude. Conversely, when it reads as "of course you'd build it this way," the additions are generic enough that the kolu agent (which has the same structural pattern, just with PTYs instead of an index) can consume the same primitives without further framework changes.

Watch out for

  • implementSurface(...).router returns a fragment, not a top-level router. Re-wrap with implement(contract).router(fragment) before handing to StandardRPCHandler, or every request 404s. Document in peer-server's JSDoc.
  • Stdio's stdout IS the protocol channel in agent processes. Any pino log line corrupts the next frame and the client peer dies with SyntaxError: Unexpected token '«'. Consumers of serveOverStdio need to redirect logs to fd 2. R-2 handles this via log.ts detecting --stdio in process.argv at module load.

R-2 — kolu --stdio agent + RemoteBackend

User outcome: the entire remote feature surface lights up at once. Remote terminals work. Branch chip, agent badge, GitHub PR badge, Code tab, file upload — every rich surface — work identically on remote tiles. The only visible difference is the host chip.

R-2 builds in six self-contained slices. Each slice has a clear "done" criterion you can verify before moving on; in this order, each one's failure mode is bounded by the slice's surface area.

2a — Install probe + remote command builder

Done when: installAgent(host) probes nix --version over ssh and surfaces a typed error if nix is missing; remoteAgentCommand(host) returns the argv to spawn.

  • The command: ssh $host nix run --accept-flake-config --refresh <flakeRef> -- --stdio.
  • Flake ref source: KOLU_AGENT_FLAKE_REF env var, required — no default. The kolu-server's typed client and the remote agent's router must be built from the same source; a baked-in fallback masks wire-shape drift. getAgentFlakeRef throws at first remote-spawn if unset, error message names the bug class.
  • Why not nix copy? Cross-arch closures can't be realised locally without exactly-right remote-builder config (offload + system-features + trusted-user permission). Kolu's user base already has Nix on the remote, so we let the remote build natively via nix run. Zero arch matrix.
  • R-3 retirement: --refresh is a workaround for contract drift between releases. R-3 introduces an explicit version handshake and drops --refresh.

2b — Agent surface declaration

Done when: packages/common/src/agentSurface.ts declares the agent's wire shape via defineSurface({ collections, streams, procedures }). kolu --stdio entry point calls serveOverStdio with that surface's router; round-trips system.heartbeat through an in-memory loopback test.

  • Wire shape:
    • terminalMetadata collection keyed by terminal id, value is an AgentTerminalMetadata subset (cwd, git, agent, pr, foreground, lastAgentCommand, lastActivityAt). Aggregated on the agent side; the kolu-server subscribes once. NOT 8 parameterized streams — lesson #2.
    • terminalData, terminalCommandRun, terminalTitle streams keyed by terminal id — high-throughput PTY bytes + OSC streams stay separate from metadata for backpressure isolation.
    • fsRepoChange, fsFileChange streams.
    • Procedures: system.heartbeat; terminal.{spawn,kill,write,resize,uploadFile}; fs.{listAll,readFile}; git.{getDiff,getStatus}.
  • Narrow contract: agentSurface is NOT the full kolu-server appRouter. Exposing terminal.create would let an agent spawn another remote terminal-on-a-terminal (recursion). The agent serves what the kolu-server's RemoteBackend needs, nothing else.
  • Log redirection: log.ts detects process.argv.includes("--stdio") at module load and forces pino output to fd 2. Must be in place before the first import-time log call.
  • Per-terminal aggregator on the agent: subscribes to the existing per-channel publishers (terminalChannels.{cwd,git,commandRun,agent,pr,foreground}) and republishes the aggregated AgentTerminalMetadata projection through the surface collection's perKeyBus. Starts at terminal.spawn, stops at terminal.kill. title is intentionally excluded — it's an in-process transient consumed by the agent's own providers, not a field the surface ships.

2c — HostSession with deferred heartbeat

Done when: HostSession.connect() spawns the subprocess and builds the surface client; markReady() (called externally after the first RPC succeeds) transitions to connected and starts the heartbeat loop; subprocess exit transitions to disconnected; reconnect ladder follows the Zed constants.

  • Constants ported from Zed (/tmp/zed/crates/remote/src/remote_client.rs:149-155): HEARTBEAT_INTERVAL=5s, HEARTBEAT_TIMEOUT=5s, MAX_MISSED=5, MAX_RECONNECT_ATTEMPTS=3.
  • State machine (Zed :157-293): connecting | connected | heartbeat-missed(n) | reconnecting(attempt) | disconnected(reason). projectExternal() collapses to the three-state ConnectionState the UI sees.
  • The deferred-heartbeat invariant (lesson #6): connect() spawns the ssh subprocess and constructs the client BUT does NOT start the heartbeat. The first RPC roundtrip (from RemoteBackend) calls markReady(), which transitions to connected and starts the interval. This survives cold nix run realisation that can take minutes.
  • State listeners fire snapshot-then-delta: onStateChange(cb) calls cb(currentState) synchronously before subscribing to changes. This is what lets RemoteBackend's connection-state subscriber attach BEFORE connect() returns and still see the initial connecting state.
  • Heartbeat uses surface.system.heartbeat, not a hand-rolled client.heartbeat() call.

2d — RemoteBackend: instant tile + async tail

Done when: clicking "new terminal on $host" returns a tile in <50ms with a "Connecting…" overlay; the overlay flips to a live shell once nix run realises.

  • Pre-generate the id. spawnPty generates const id = opts.id ?? crypto.randomUUID() BEFORE any RPC. The agent honors the same id so kolu-server ↔ agent registries stay in lockstep.
  • Register the shadow entry synchronously. Build a remoteHandle (PtyHandle-shaped proxy whose write/resize/dispose forward via session.surfaceClient), construct the TerminalProcess with meta.connectionState = "connecting", call registerTerminal(id, entry). The tile renders immediately. This is the Backend.spawnPty contract from R-1 (invariant #4) put to work.
  • Start the connection-state subscriber synchronously, metadata subscriber after markReady(). Connection-state subscriber doesn't need session.surfaceClient (pure in-process listener). The metadata subscriber needs a live client; starting it before connect() dies on the first iteration.
  • The async tail: 
    void (async () => {
  try {
    await this.session.connect();
    await surfaceClientOf(this.session).surface.terminal.spawn({ id, cwd, initialMetadata });
    this.session.markReady();             // ← first RPC succeeded; start heartbeat now
    const stopMetadata = this.startMetadataSubscriber(id, metaMod);
    entry.stopProviders = compose(stopState, stopMetadata);
  } catch (err) {
    log.error({ host, id, err }, "RemoteBackend.spawnPty: async tail failed");
    // HostSession's subprocess-exit handler transitions to "disconnected",
    // which the state subscriber mirrors into entry.meta.connectionState,
    // which the DisconnectedOverlay renders. No bespoke failure path needed.
  }
})();
return { id, write: handle.write, resize: handle.resize };
  • DisconnectedOverlay covers connecting AND disconnected. Same component, two states. Connecting shows spinner + "Connecting to $host…" + hint about nix realisation; disconnected shows reason + Reconnect / Close-tile buttons.

2e — Metadata mirroring via one collection subscription

Done when: a remote tile's branch chip, agent badge, PR badge, foreground process, cwd, and git info all update from the agent's in-process providers — through ONE subscription, not many.

  • Single subscription: RemoteBackend.startMetadataSubscriber(id) opens ONE for await loop on surfaceClient.surface.terminalMetadata.get({ key: id }). Each yield is an aggregated AgentTerminalMetadata snapshot. The handler splits across updateServerMetadata (persisted fields: cwd, git, lastAgentCommand, lastActivityAt — fires terminals:dirty for autosave) and updateServerLiveMetadata (transient fields: agent, pr, foreground — no autosave fire).
  • No per-field switch in terminalChannel(kind): when the kolu-server router asks for kind "cwd" / "git" / "agent" / "pr" / "foreground", project from the same collection snapshot: yield snapshot[kind]. The kind names match AgentTerminalMetadata field names exactly — direct indexed access, no switch.
  • connectionState is in-process — it's the kolu-server's view of HostSession state, NOT a thing to fetch from the agent. RemoteBackend.terminalChannel(id, "connectionState") wraps HostSession.onStateChange in an AsyncIterable directly.
  • PTY data and command-run streams stay separate: terminalData, terminalCommandRun, terminalTitle all flow through their own per-id streams. High-throughput, latency-sensitive — keeping them out of the metadata collection avoids backpressure interference.

2f — Host-aware UI plumbing

Done when: every surface that names a terminal also names its host (when remote); session restore preserves location; sub-terminals inherit location.

  • HostChip on five surfaces: title bar (TerminalMeta.tsx), dock pill (Dock.tsx — QuietRowBody, AwaitingCardBody, WorkingPillBody), workspace switcher card (WorkspaceGrid.tsx), restore card (EmptyState.tsx), saved-terminal list. The chip renders nothing when location.kind === "local" — common case stays clean.
  • SSH host picker: command palette lists hosts from ssh -G $host probes against ~/.ssh/config aliases. No bespoke parser — ssh-config syntax evolves and a hand-rolled parser drifts.
  • Session restore preserves location: useSessionRestore.ts's call to handleCreate threads t.location through. e2e test: save a remote session, kill kolu, restart, verify the tile reconnects to the right host.
  • Sub-terminal inheritance: getBackendForCreate({ parentId }) reads the parent's meta.location and resolves the backend by that, regardless of what the create input said. Centralizes inheritance to one resolver — createTerminal doesn't pattern-match on parent state.

R-3 — PTY ownership + reattach (network-blip resilience)

User outcome: laptop sleep / wifi drop / suspend-resume leaves remote terminals alive with full scrollback. This is the axis Zed itself doesn't solve — Zed terminals die with the ssh subprocess. Kolu can go further because the agent is already there as the natural home for a tmux-style PTY broker.

  • PTY ownership. Already mostly satisfied by R-2 — the agent's LocalBackend.spawnPty runs the PTY on the remote machine. The PTY survives ssh drops trivially; R-3 just adds the reattach plumbing on top.
  • Stable session IDs. Persist remoteSessionId per terminal in ServerPersistedTerminalFieldsSchema. On reconnect, the kolu-server calls backend.attachPty(remoteSessionId); the agent yields a fresh scrollback snapshot (snapshot-then-delta, same shape as the existing data channel's first yield) and resumes byte streaming.
  • Version handshake replaces --refresh. Agent reports its contract version on first connect as a system.version procedure or a cell. Kolu-server compares against its compiled-in expectation. Version-skew triggers a single explicit --refresh + reconnect with a clear error, not blanket --refresh on every spawn.
  • Idle TTL on the agent (e.g. 24h) so abandoned remote sessions don't accumulate.
  • Pre-warm: after installAgent, optionally fire-and-forget ssh $host nix build --no-link <flakeRef> so the first real terminal-spawn doesn't wait on cold realisation. (R-2's instant-tile-with-overlay UX is already acceptable, but pre-warm closes the loop.)

Why keep terminal bytes and metadata on separate streams (vs. one fat stream)? Terminal bytes are high-throughput, latency-sensitive raw streams; metadata is structured, lower-frequency snapshots. Multiplexing them onto one channel tangles backpressure semantics. The agent surface already gives them separate slots (terminalData stream vs. terminalMetadata collection); R-3 doesn't change that.

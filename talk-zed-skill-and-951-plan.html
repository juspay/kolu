Two shapes for the work

Two plans, picked according to how much existing code we're willing to rewrite. The additive plan keeps existing kolu architecture in place and adds a provider-per-domain layer to absorb the local-vs-remote dispatch — minimizes blast radius per PR, but reaches the end state in many small slices. The rewrite-freedom plan collapses the server-side architecture around one Backend interface, with the kolu binary doubling as the remote agent — fewer total slices and a simpler end state, but one of the slices is a big-bang rewrite of the server spine.

Plan A — Additive (minimize churn)

Five shipping slices: a Phase 0 prep ladder of pure refactors lands first, then four feature phases (1, 2a, 2b, 3). The design uses the provider-per-domain pattern that already lives in packages/integrations/anyagent/src/agent-provider.ts:77 — kolu's existing AgentProvider interface is the in-repo template, generalized across the other domains (git, fs, github, pty) that need remote support. Phase 0 reshapes the codebase to accept that template with zero behavior change; the feature phases then drop in remote implementations against an already-shaped surface.

Volatility axes the feature crosses

Four axes, each with its own seam. Each axis lives behind a typed provider interface so the dispatch happens once at terminal creation time, not at every call site.

Glossary.
  • Axis (of change / of volatility). One independent dimension along which the system might evolve later. Lowy's framework (after Parnas 1972) says each module should encapsulate exactly one axis so a change along that axis touches one place.
  • Seam. A spot in the code where you can swap behavior without editing every call site. Term from Michael Feathers, Working Effectively with Legacy Code. Here, each provider interface (AgentProvider, GitInfoProvider, …) is a seam.
  • Receptacle. Lowy's framing for the same idea — a typed slot the consumer holds while the implementation behind it changes. Provider interfaces are receptacles; concrete LocalXxxProvider / RemoteXxxProvider implementations slot in.

Why not a per-host HostContext bundle? An earlier draft proposed bundling { ptyFactory, gitClient, agentClient, fsClient, executor } per host. Hickey + lowy passes flagged it as functional decomposition wearing a discriminated-union costume: it groups by "stuff about a host" rather than by axis of change, and consumers still pattern-match on the union to know what's filled in when. The provider-per-domain shape is the volatility-axis cut — each domain interface changes for its own reasons, and the dispatch is concentrated in terminal-creation only.

Why not thread location: TerminalLocation through every integration's public function? Same review pass: that braids transport dispatch (ssh-or-not) into each integration's domain function (subscribeGitInfo), forcing every caller — even purely local ones — to carry a location argument. The provider pattern keeps integrations domain-pure: the remote variant lives in a separate kolu-remote-client package and never touches kolu-git internals.

Shape of a provider, concretely

AgentProvider at packages/integrations/anyagent/src/agent-provider.ts:77-140 is the template — the orchestrator (meta/agent.ts) consumes it and never special-cases concrete implementations. Generalize to other domains:

// packages/integrations/git/src/resolve.ts — extract from existing function
export interface GitInfoProvider {
  subscribe(
    cwd: string,
    onChange: (info: GitInfo | null) => void,
    log: Logger,
  ): { setCwd(next: string): void; stop(): void };
}

// Same package — local impl wraps today's code, no behavior change:
export class LocalGitInfoProvider implements GitInfoProvider { … }

// NEW package: packages/kolu-remote-client/src/git.ts
import type { GitInfoProvider } from "kolu-git";
import type { HostSession } from "./hostSession.ts";

export class RemoteGitInfoProvider implements GitInfoProvider {
  constructor(private session: HostSession) {}
  subscribe(cwd, onChange, log) {
    const token = this.session.subscribe("git.info", { cwd }, onChange);
    return {
      setCwd: (next) => token.update({ cwd: next }),
      stop: () => token.close(),
    };
  }
}

The orchestrator (meta/git.ts) is constructed with a provider; it never sees location:

// terminals.ts — dispatch happens ONCE, here, at terminal creation
const providers = entry.meta.location.kind === "ssh"
  ? remoteProviders(hostSession(entry.meta.location.host))
  : localProviders();
entry.providers = providers;

// meta/git.ts — pure consumer, identical local-or-remote
export function startGitProvider(entry, terminalId) {
  const watcher = entry.providers.git.subscribe(entry.meta.cwd, (info) => {
    terminalChannels.git(terminalId).publish(info);
  }, log);
  …
}

The remote agent on the host imports the same kolu-git, kolu-github, kolu-anyagent packages — but only the local implementations of each provider. The agent's RPC handlers wrap those local providers and route subscription events back over the wire. Zed's "remote runs the same code as local, plus .shared(session)" symmetry, expressed via TypeScript interfaces.

The one new top-level abstraction: HostSession

Per-host singleton, owns the ssh-stdio RPC channel and all subscription tokens to that host. Hickey's connection-lifecycle finding: when the ssh connection drops and reconnects, HostSession tears down every live token and re-issues them via the new connection. The integration-level providers never see the churn — their onChange callbacks just keep firing on the new tokens.

interface HostSession {
  // Request/response, awaits result
  call<Req, Res>(method: string, args: Req): Promise<Res>;
  // Streaming subscription. Returns a token whose lifetime HostSession owns.
  // On reconnect, HostSession re-issues all live tokens with the same args
  // and resumes calling the same onEvent — provider code never notices.
  subscribe<Args, Event>(
    method: string,
    args: Args,
    onEvent: (e: Event) => void,
  ): SubscriptionToken<Args>;
  // Lifecycle
  state(): ConnectionState; // Connecting | Connected | Reconnecting | …
}

  • kolu-github — wrap PR polling in GitHubProvider. Remote variant runs gh pr view on the agent. The gh CLI's auth lives in the remote user's ~/.config/gh — the user is already logged into gh there or they aren't (no kolu-side credential plumbing).

Dispatch site — terminal creation

createTerminal() picks the provider set based on the resolved location and pins them onto entry.providers. No if (remote) branches inside meta/*. Sub-terminals inherit the parent's location and therefore the same provider set.

Phase 3 — PTY ownership migrates to the agent (network-blip resilience)ship 3

User outcome: laptop sleep / wifi drop / suspend-resume leaves remote terminals alive with their scrollback. Same prompt, same in-progress command output. This is the axis Zed itself doesn't solve — Zed terminals die with the ssh subprocess. Kolu can go further because kolu-remote-agent is already there as the natural home for a tmux-style PTY broker.

  • Add remoteSessionId?: string to ServerPersistedTerminalFieldsSchema — the persisted handle the client uses to reattach. lowy: Phase 3 isn't a pure kolu-pty swap; the persisted shape changes too.
  • Replace SshPtyProvider with AgentPtyProvider (still satisfies PtyProvider). Calls HostSession.call("terminal.spawn", { … }) on the agent; agent runs the PTY locally; bytes stream back via a subscription token.
  • Reattach: on reconnect, kolu calls terminal.attach(remoteSessionId); agent yields a fresh scrollback snapshot (matching the snapshot-then-delta streaming invariant in .claude/rules/streaming.md) and resumes byte streaming.
  • Local xterm-headless emulator stays — for late-join client serialization. The emulator's onData is fed from the RPC subscription instead of a node-pty subprocess.
  • Agent-side GC: idle TTL (e.g. 24h) so abandoned remote sessions don't accumulate.

Why keep terminal bytes and metadata on separate channels? Terminal bytes are high-throughput, latency-sensitive raw streams; metadata is structured, lower-frequency request/response. Multiplexing them onto one channel tangles backpressure semantics — Zed makes the same split for the same reasons. The provider model already gives them separate seams (PtyProvider stream vs. HostSession.call for ops). hickey: separation is the feature.

Plan B — Rewrite-freedom (one Backend spine)

If "don't perturb existing code" is dropped as a constraint, the architecture re-cuts substantially. The additive plan's per-domain provider work exists because we can't replace meta/* orchestrators wholesale; lift that constraint and the whole local-vs-remote axis collapses behind a single interface.

The pivot

One Backend interface owns the per-terminal world. A backend produces every per-terminal stream (data, git, agent, github, cwd, …), handles PTY spawn, runs one-shot ops (listAll, readFile, getDiff). The kolu server no longer imports from kolu-git / kolu-anyagent / etc. directly anywhere — it invokes backend.terminalChannel(id, "git") and forwards. There is exactly one transport boundary in the system.

The remote agent is the same binary as the kolu server. kolu serve runs the kolu server with LocalBackend. kolu agent --stdio runs the same code with oRPC's transport set to stdio. The local server's RemoteBackend(connection) proxies every backend op + stream over that oRPC channel. The agent has zero remote-specific logic; it's just another oRPC peer, and oRPC's existing stream wrapper + STREAM_RETRY semantics (see .claude/rules/streaming.md) carry over without modification.

What disappears as per-domain work:

  • No RemoteGitInfoProvider / RemoteAgentProvider / RemoteFsProvider / RemoteGitHubProvider. One RemoteBackend proxies the entire backend protocol.
  • meta/*.ts orchestrators dissolve. LocalBackend internally wires subscribeGitInfo to its git channel — the orchestration is one private function inside the backend, not a server-side layer.
  • The wire-honesty work (host-tagged payloads, terminalKey host prefix, connectionState enum, pid removal) collapses into "design TerminalInfo and channel payloads correctly from the start" — no migrations needed because the schema is rewritten anyway.
  • Don't hand-roll ~/.ssh/config parsing. Use ssh -G $host (prints the resolved config, lets ssh itself enumerate). Saves the Zed parser port and stays correct as ssh-config syntax evolves.

R-1 — Introduce Backend abstractionship R-1

User outcome: none. Pure architectural work, hidden behind a build flag during stabilization. Schema redesigned around backend-tagged terminals from day one — no migration of old persisted sessions if the team is OK with a one-time reset, or a Zod transformer to map old shape to new if not.

The Backend interface

interface Backend {
  readonly id: BackendId;   // "local" | { kind: "ssh"; host: string }

  spawnPty(opts: PtySpawnOpts): Promise<TerminalHandle>;
  attachPty(remoteSessionId: string): Promise<TerminalHandle>; // Phase R-3

  // Per-terminal streaming channels. Snapshot-then-delta invariant.
  terminalChannel<K extends keyof TerminalChannelMap>(
    terminalId: TerminalId, kind: K
  ): AsyncIterable<TerminalChannelMap[K]>;

  // One-shot ops scoped to a terminal's host
  fs: { listAll(path): Promise<Path[]>; readFile(path): Promise<Buffer> };
  git: { getDiff(repoRoot, …): Promise<string>; getStatus(…): Promise<…> };
}

What R-1 touches

  • packages/server/src/terminals.ts, terminal-registry.ts — rewritten around backends. Terminal creation picks a backend by location (only LocalBackend exists in R-1).
  • packages/server/src/meta/{git,agent,github,cwd,process}.ts — collapse into LocalBackend as private wiring. The kolu-server-side layer disappears.
  • packages/server/src/router.ts — terminal route handlers forward backend.terminalChannel(id, kind); no per-domain orchestration on the server side.
  • packages/common/src/contract.ts — terminal + channel schemas redesigned. TerminalLocation is the backend identity, not a parallel field.

Risk mitigation

  • Feature flag: ship Backend alongside the legacy path, controlled by an env var; switch in stages (per-test, per-dev, then default). Old code paths deleted only when the new shape is stable in production.
  • Migration test matrix: golden-replay of saved-session restore against R-1 schema with old session fixtures. Either migrate cleanly or surface "old session — start fresh" with no silent data loss.
  • Per-backend smoke tests: FakeBackend covers every channel + op end-to-end; LocalBackend integration tests pass identically to today's tests.

R-2 — kolu agent --stdio + RemoteBackendship R-2

User outcome: the entire remote feature surface lights up at once. Terminals on remote hosts work. Branch chip, agent badge, GitHub PR badge, Code tab all work for remote tiles immediately — because they're produced by the backend, not by per-domain remote impls. The only thing missing is network-blip survival (that's R-3).

Components

  1. kolu agent --stdio. Same binary. Reads/writes oRPC envelopes on stdin/stdout. Internally constructs LocalBackend and serves its surface as oRPC procedures. Zero remote-specific logic in the agent binary itself — it's just kolu running with a different transport.
  2. Auto-install. Server-side installAgent(host) mirrors Zed's ensure_server_binary (/tmp/zed/crates/remote/src/transport/ssh.rs:785-916): probe arch via ssh host 'uname -sm', check ~/.local/share/kolu/agent/<binary-name> for matching version, three install strategies in order (dev build / remote-side curl / local SFTP upload). The agent binary IS the kolu binary, so distribution is the same as kolu's normal release pipeline — one less moving part.
  3. RemoteBackend(connection). Backend impl whose every method is an oRPC call against the agent. Subscriptions return the agent's streams piped through the local oRPC client. STREAM_RETRY handles reconnect automatically (snapshot-then-delta replays cleanly).
  4. Connection state machine. Heartbeat (Zed constants: 5s interval × 5 misses → reconnect, 3 attempts). State machine ports from /tmp/zed/crates/remote/src/remote_client.rs:157-185. Disconnected overlay UI mirrors /tmp/zed/crates/recent_projects/src/disconnected_overlay.rs.
  5. SSH host picker + UI. Command palette host list via ssh -G $host probe per discovered ~/.ssh/config alias (lighter than porting Zed's parser). Host chip in title bar, dock card, workspace switcher, restore card.

R-3 — PTY ownership + reattach (network-blip resilience)ship R-3

User outcome: laptop sleep / wifi drop / suspend-resume leaves remote terminals alive with full scrollback.

  • PTYs live in the backend that hosts them. For RemoteBackend, that means the agent process owns the PTY; the local server proxies bytes. On ssh drop the PTY keeps running on the remote.
  • Stable session IDs. Persist remoteSessionId per terminal. On reconnect, the local server calls backend.attachPty(remoteSessionId); the agent replays a fresh scrollback snapshot (snapshot-then-delta) and resumes streaming.
  • Idle TTL on the agent (e.g. 24h) so abandoned remote sessions don't accumulate.

The decision is more about team posture than about which design is "correct" — both are structurally sound.

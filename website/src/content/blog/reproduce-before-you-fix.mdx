[Kolu](https://github.com/juspay/kolu) redeploys itself a few times a day. When the client changes, every open tab gets a little card in the corner — *App updated · Reload* — and one click picks up the new build. On two of my machines the card would not die. Click Reload, the page comes back, the card's still there. Click again. Still there.

I'd already taken two shots at it, both guesses, both wrong. The first one shipped: <PR id={1278} /> had **a clean theory** about a stale browser cache and a clever workaround, and it merged two days before the card came back worse than before. A theory, *not a fact*. So I reopened it and typed the least gracious thing I've written to an agent all month:

<Prompt>{"PR 1278 was an utter failure!\n\nWe need ROOT CAUSE with EVIDENCE and a proposal for a correct fix."}</Prompt>

EVIDENCE was the right word, and I'd ignore my own advice for a few more hours. The bug was real, it was ours, and it was hiding in plain sight. But the bug isn't the thing worth writing down. **The thing worth writing down is that we only found it after we stopped guessing and forced the repro to match one hard fact off my actual screen.**

## Start from what the running server says

We keep a file, `docs/cache-bug.md`, that's nothing but headstones for past theories about this exact bug. One rule sits at the top: get ground truth from the running system before you make up a story about it. So before believing anything, `curl` the server.

The server was clean. `GET /` returned `Cache-Control: no-store`, naming the current bundle. A conditional request — `If-Modified-Since`, both the 1970 epoch and a recent date — got a fresh `200`, not a `304`. The HTTPS [Tailscale](https://tailscale.com/) proxy in front behaved the same. Not the server, not the deploy, not the proxy. Every theory about the server was now dead, on facts, before a line of any fix was written. Whatever was wrong lived in the browser.

## A guess dressed up as evidence

So the agent built a repro: swap between two deployed builds, watch the card. Up it came, a click on Reload, the client took the new build. Every time. It converged — and the writeup came back with more confidence than it had earned:

> PR #1278 fixed a bug that doesn't exist. The "App updated" card you keep seeing is the legitimate restart prompt firing on every deploy — and you deploy 3–8 times a day, sometimes minutes apart.

It even had four arguments, each true on its own: a `no-store` response is never stored, so there's nothing to go stale; the server ([Hono](https://hono.dev/)'s `serveStatic`) never sent a `Last-Modified`, so there's no old date to inherit; the browser's sync history showed Reload working; the card fires on a plain restart too, which looks identical. Four for four. **And it never once reproduced what was on my screen.**

That's the trap. A repro that converges *feels* like proof. But it's only proof if it reproduces the actual symptom, and this one didn't. The cleaner the argument got, the further it drifted from the thing I was staring at.

## The one hard fact: a SHA that wouldn't move

I sent back one sentence:

<Prompt>Do you realize I clicked 'Reload' multiple times and the client sha remained 7deb397? Do you understand this is the bug? Did you reproduce it?</Prompt>

One fact, and it kills all four arguments at once. A repro that converges cannot produce a SHA that won't move. If I reload ten times and the client still reads `7deb397`, something is handing the browser the same old bytes ten times running — and a repro that converged was, by construction, never doing that.

So the question stopped being "is there a bug" and became the only one that mattered: how do you *reproduce* that — a client SHA that won't change across reloads? Asked that way, it answers itself. The agent's repro had swapped two builds with *different* bundle filenames, and a different filename forces a fetch — that's the whole point of content hashing — so of course it converged. To get a SHA that won't move you need the opposite: two builds whose bundle has the *same* filename and different bytes.

Once the repro had to match the fact, the cause had nowhere to hide. Same filename, different bytes, served from a year-long cache: that's not a browser bug. That's us shipping two different files under one name.

## The bug, in one loop

Here it is, and it's our own doing end to end. To keep a docs-only commit from busting the [Nix](https://nixos.org/) build cache, we build the client with a *placeholder* commit and `sed` the real SHA in afterward, in a step called `koluStamped`. By the time it runs, [Vite](https://vite.dev/) has already hashed and named the bundle — so the `sed` rewrites the bytes of `/assets/index-D85Q74Rn.js` without changing its name. And that file is served `Cache-Control: public, max-age=31536000, immutable`.

`immutable` is a promise: these bytes never change, cache them for a year, don't even ask. We made the promise and then broke it. Two deploys that differ only outside the client — a doc, a test, the marketing site — ship the same filename with a different SHA inside, and the browser does the rest:

Look for the bug in that loop. It isn't there. **Every box is the browser keeping the promise we broke.** It serves the year-old bundle from disk, the old SHA mismatches the server, the card returns. `cmp` on the two builds running that week said it flat: `7deb397` and `0fab0cc` shipped byte-identical bundles, differing only in the stamp. No amount of cache-busting the shell could touch it — the stale identity rode *inside* the immutable file, and <PR id={1278} /> had been busting the wrong layer.

## The fix: identity rides the no-store shell

Once you can see it, the fix writes itself. A commit SHA is the most volatile thing in the build. It has no business inside a file we swear never changes. Put it on the one resource that's never cached — the `no-store` shell.

The [Vite](https://vite.dev/) plugin injects `window.__SURFACE_APP_COMMIT__` into `index.html` with [`transformIndexHtml`](https://vite.dev/guide/api-plugin#transformindexhtml) instead of baking it into the bundle; `koluStamped` now `sed`s the shell and nothing else; the client reads it with `shellCommit()`. A real code change converges in one reload — new code, new hash, new filename, forced fetch — while a docs-only deploy keeps serving the same cached bundle, because now nothing lies about it. And because a future refactor could quietly undo all this, the build fails loud the moment the placeholder lands back under `dist/assets/`:

```bash
if grep -rl '__KOLU_COMMIT_PLACEHOLDER__' "$out/packages/client/dist/assets" 2>/dev/null; then
  echo "koluStamped: placeholder leaked into a hashed /assets/* file — \
identity must ride the shell, not an immutable bundle (kolu#1319)." >&2
  exit 1
fi
```

A comment would have said the same thing. A comment doesn't fail the build.

## The lesson: reproduce first, then fix

The cache bug is a decent war story, but it isn't the lesson. The lesson is the order of operations. Two fixes shipped before this one, both built on theories that sounded right, and both missed — because neither reproduced the exact thing on my screen. The agent's "there is no bug" was the same mistake pointed the other way: a clean argument standing in for a repro that matched the symptom.

So here's the rule I'd hand an agent, and myself. Don't write a fix until you can make the bug happen on demand, from hard facts, with no guessing in between. Get ground truth from the running system. Find the one fact any real explanation has to satisfy — here, a SHA that wouldn't move — and build the repro that produces it. The cause tends to fall out on its own once the repro is honest. **When the model and the symptom disagree, believe the symptom. It was there first.**

And with an agent, that rule buys more than a correct answer — it buys autonomy. A reproduction isn't just evidence; it's an objective the agent can drive at without me in the loop. Once the bug happens on demand, the work closes into a cycle it can run alone: change something, run the repro, check whether the SHA moved, repeat until it does — and the thing that proves the fix is the same repro that found it. That's exactly how this one ended: once the repro reliably made the SHA stay put on a throwaway box, the agent built the fix and watched it finally move, with me nowhere near it. My one irreplaceable part was the hard fact that anchored the whole thing — *I clicked Reload, and the SHA didn't move.* Hand the agent that, and it can go the rest of the way on its own.

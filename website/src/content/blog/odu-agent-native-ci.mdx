In the [last post](/blog/orpc-over-ssh) I let an app's source of truth move to another machine, with nothing between me and it but ssh. One of the things I built on that transport, almost as an aside, was a CI-runner TUI — [mini-ci](https://github.com/juspay/kolu/tree/master/packages/surface), a recipes×platforms grid you `attach` to over [oRPC](https://orpc.dev/) instead of scraping log files. This post is about what that toy grew into, and the one question that turned it from a nicer dashboard into a different kind of tool.

The question arrived as a three-word insult. I'd spent a day giving the runner — now its own project, [odu](https://github.com/juspay/odu) — an [MCP](https://modelcontextprotocol.io/) server, so a coding agent could drive CI directly. Then I went right back to watching the pipeline by hand and shelling out to a wrapper script to start runs, exactly as before. [Claude Code](https://claude.com/claude-code) was sitting next to a CI server it could talk to and I was still reading tea leaves. Sridhar noticed:

<Prompt>wtf why are not using MCP for ci</Prompt>

That's the whole post, really. The agent that writes the code should run the CI, read the failures, and rerun the one red node — without a human relaying between them. Getting there was less about MCP and more about what shape a CI runner has to be before an agent can hold a conversation with it.

## The runner has to be a service, not a script

The thing odu replaced is [justci](https://github.com/juspay/justci), and justci is a good tool built on a fair assumption: CI is a batch job. You hand it a git bundle, it compiles your [`just`](https://github.com/casey/just) recipe DAG into a [process-compose](https://github.com/F1bonacc1/process-compose) document, runs the whole thing once to a verdict, and leaves you a directory of `.log` files to read. In, out, done. For years that's all I wanted.

An agent wants something else. It wants to *attach to a running system* and ask it questions: what nodes exist, which one is red, show me that log, run this one again. Those aren't batch-job operations. They're the operations you'd run against a live service — and a batch job, by construction, isn't one while it matters and is gone by the time you'd query it.

justci's own tracker is where this gets sharp, because justci tried the exact thing I was about to and backed it out. MCP support landed and was reverted *in the same PR*, and the [issue capturing why](https://github.com/juspay/justci/issues/22) — <PR id={22} repo="juspay/justci" kind="issue" /> — names the defect precisely:

> The agent-introspection model wants **"MCP server up, processes idle"** — the attached agent decides what runs. The reverted shape did the opposite: `process-compose up` started every recipe immediately on MCP launch... there's no separation between "register the project, serve the API" and "drive the pipeline."

That's the wall. process-compose has no serve-only mode; bring it up and it runs everything. So an MCP server bolted onto it can't offer the one state an agent needs to start from — *idle, nothing running, here's the DAG* — because the substrate doesn't have that state. The same issue notes a second miss: process-compose's MCP surface is "request/response only, no streaming," so there's no way to push a node's transition to the agent as it happens.

odu doesn't hit either wall. It's not cleverer — it's the *other shape*. It owns the DAG as idle state by default; a run is something you start against a service that was already up. And it already has the live event source, the same snapshot-then-deltas stream the TUI repaints from. The thing justci's MCP attempt couldn't get for free, odu can't avoid having.

## The verbs an agent actually needs

Once the runner is a service, the MCP face <PR id={3} repo="juspay/odu" /> is a small surface. odu dials its own in-process socket (`.ci/odu.sock`) and exposes a handful of tools and two subscribable resources. But three of the decisions in there are the difference between an agent that can drive CI and one that just has access to it.

### Block until it breaks, not until it finishes

The tool an agent leans on hardest is `wait_for_settle`, and the whole point of it is that it's **fail-fast**: it returns the instant the *first* node goes red, not when the run finishes. If the e2e lane fails in two minutes, the agent shouldn't wait twenty for a nix build it's about to throw away. A batch runner can't offer this — it hands you a verdict when the process exits and not before — but a live `Cell` makes it a one-line change of terminal predicate.

That predicate is also where the most instructive bug of the whole arc lived. The first `wait_for_settle` could report **green when the stream ended before a node reached a terminal state** — a CI tool that lies about success, which is the single worst thing a CI tool can do. The review gauntlet caught it before it shipped. The fix was to compute "done" off the same receptacles the TUI's verdict reads — `summarize(state).done`, `STATUS_META[status].isRed` — instead of re-deriving the rule in the MCP layer, and to hard-code `passed: false` on the timeout and cancel paths so a green can *only* come from a genuinely terminal snapshot. The lesson is older than odu: when two places decide what "success" means, one of them is eventually wrong.

### Snapshot, then notify

The naïve agent API is a subscription. The better one, for a model that only acts when you wake it, turned out to be two plain request/response calls — `get_nodes` for a snapshot and `wait_for_settle` for the blocking finish — because that's exactly the pair process-compose's MCP couldn't give. But notification-aware hosts shouldn't be held to the floor, and Sridhar pushed on that:

<Prompt>Sure. Also I thought MCP had a mechanism to "notify" agents?</Prompt>

So odu also publishes its live state as subscribable resources — `odu://nodes` and `odu://log/{node}` — with `notifications/resources/updated` fired as nodes change. The blocking call is the floor every host can use; the subscription is the ceiling for the ones that can be pushed to. Same underlying `Cell`, served two ways.

### Don't bake in a host

The tempting shortcut is to make `run` take a host. It's wrong, because odu already selects a box from a warm pool under a lease, and an MCP-chosen host would duplicate and fight that:

<Prompt>MCP doesn't have to predetermine a host. Remember we select pu box from a pool.</Prompt>

So the MCP face predetermines nothing — host selection stays the coordinator's job, and the agent just says "run." One less thing for the two paths to disagree about.

That MCP face didn't ship on trust. It went through the project's review gauntlet — a [codex](https://github.com/openai/codex)⇄Claude debate to consensus, a Löwy⇄Hickey structural pass, and a code-police sweep — which turned up six more fixes beyond the false-green: a spawn/exit race in `run`, a **path traversal** in the durable-log fallback, an unbounded `readFileSync` (now tail-capped at 64 KB), request-cancellation threading, subscribe-URI validation, and a leaked test temp dir. The kind of bugs you want a reviewer to find in a tool that runs commands on boxes for you.

## A CI runner you attach to

Somewhere in the middle of this, a naming question turned out to be a design question. The runner had a `monitor` command for watching a live run. Sridhar, having just merged a PR, asked the question that renamed it:

<Prompt>why is it "monitor" instead of say "attach"? Like tmux attach?</Prompt>

It lands because it's true. You don't *monitor* a [tmux](https://github.com/tmux/tmux) session, you attach to it — and the mental model that word carries is exactly odu's: one run, many things attached to it, each one a view of the same live state, none of them owning it. So `monitor` became `attach` <PR id={7} repo="juspay/odu" />, a clean break with no alias.

The rename exposed something worse underneath. `run` and the old `monitor` were painting the same matrix through two separate, hand-written render paths that had quietly drifted apart — a bug a contributor had already filed, <PR id={4} repo="juspay/odu" kind="issue" />, as "inconsistent output, duplicated drifted render paths." The fix wasn't to sync the two paths; it was to delete one. `run` and `attach` now paint from a single renderer <PR id={9} repo="juspay/odu" />, so a live run looks identical whether you started it or walked up to it. *Two faces of one runner had drifted because they were two implementations.* Keep that sentence in mind, because the whole rest of the post is it at a larger scale.

## One surface, three faces

Here is where I was wrong, on the record, and got argued out of it.

By the time odu had a TUI, a CLI, and an MCP face, the obvious refactor was a generic package: take *any* [`@kolu/surface`](https://github.com/juspay/kolu/tree/master/packages/surface) and expose it as an MCP server, so nobody hand-writes the zod-to-JSON-Schema bridge and the subscribe/teardown dance twice. I proposed it, then ran it past the two lenses I trust to kill premature abstractions — [Löwy's volatility test and Hickey's simplicity test](/blog/hickey-lowy) — and they both said no. The "is this generic" bar (I call it the electricity test: does the abstraction carry one thing, unchanged, regardless of what you plug in?) failed. The thing the adapter would have to know — what counts as "settled," as "red" — was odu's `NodeStatus`, not a surface-level concept. And there was no second consumer. One use is not a pattern.

<Prompt>No, don't edit electricity.mdx</Prompt>

I shelved it. That was the right call on the evidence I had, and it was wrong within a week, for a reason worth stating plainly: **odu's own MCP face was the second consumer.** Not a hypothetical one — the hand-rolled `src/mcp/` I'd already written *was* the duplication the package would remove. The test hadn't failed; I'd run it one consumer too early.

What flipped it wasn't stubbornness, it was a directive to stop hedging:

<Prompt>Whatever "bigger framework ask" is necessary must be done in the same PR. Don't shy away from making "big" changes. We want everything perfect in first go.</Prompt>

I'd been quietly planning to migrate odu "about 90%" onto a thin adapter and leave the curation behind as too-odu-specific to generalize. That hedge was the tell. The missing 10% wasn't odu-specific — it was a *framework gap*, and the directive was to close it instead of routing around it.

### A server that's a client

The gap had a name once I stopped avoiding it: I needed a surface that was itself a client of another surface. odu's full internal surface has procedures an agent must never touch and logs that need bounding; what the agent should see is a *curated view* of it. The clean way to express "a curated view of a live surface" is a new primitive, `projectSurface` — it takes a live client of surface A and derives a surface B from it, dropping the dangerous procedures, bounding the logs, deriving the verdict. A server that's a client. It shipped in <PR id={1270} /> alongside the generic adapter `@kolu/surface-mcp`.

Now the curation lives in exactly one place — surface-land — and *every* face reads it. The TUI, the CLI, and the MCP server are no longer three things that each decide what to show; they're three faces of one projected surface. The drift bug from the `attach` rename can't happen here, because there aren't two implementations to drift. There's one.

Then odu migrated back onto its own graduated abstraction <PR id={11} repo="juspay/odu" />, and the diff is the argument. The hand-written `src/mcp/` — the server, the tools, the resources, the zod bridge, the teardown bookkeeping — **collapsed by about 1,500 lines**. What's left is what's genuinely odu's: a projected surface, and two bespoke tools (`run` and `wait_for_settle`) that are call-shaped rather than state-shaped and so ride the adapter's escape hatch instead of being forced through it. The agent-facing surface changed in the move, deliberately — no backward-compat shims, the right shape over the familiar one:

| Before — hand-rolled | After — projected |
| --- | --- |
| tools `get_nodes`, `tail_log` | **resources** `surface://cells/nodes`, `…/collections/logs/{id}` |
| tool `rerun_node` | tool **`node_rerun`** (the projected `node.rerun`) |
| tools `run`, `wait_for_settle` | bespoke tools **`run`, `wait_for_settle`** |
| `run.configure` reachable | **absent by construction** — never on the projected surface |

That last row is the one I like. The adapter is **default-deny**: only what you name in an `expose` allowlist reaches the host. A dangerous procedure isn't hidden, it's *unreachable* — and the migration's smoke test drives a real MCP client and asserts the forbidden names never appear in `tools/list`. Safety by construction, proven rather than promised.

<Tip title="Where the abstraction came from">
This is the same graduation move as the [previous post](/blog/orpc-over-ssh): build the thing inside the app, learn its real shape, then lift it into the framework so the app shrinks. There, Kolu deleted ~400 lines of per-channel relay when the stdio transport moved into `@kolu/surface`. Here, odu deleted ~1,500 when its MCP face became a `projectSurface` projection. **The framework eats the cost so the app stays small** — but only on the second consumer, never the first. I just had to be reminded which one I was looking at.
</Tip>

## What it adds up to

The arc looks like three features — an MCP server <PR id={3} repo="juspay/odu" />, an `attach` rename <PR id={7} repo="juspay/odu" />, a surface adapter <PR id={1270} /> — but it's one idea arriving in stages. A CI runner an agent can drive has to be a live service, not a batch script. A live service grows faces — one you attach to, one you script, one an agent talks to. And the moment you have more than one face, the only way to keep them honest is to give them one source of truth to be faces *of*. odu got there because a hand-rolled integration turned out to be the second consumer that justified pulling the pattern up into the framework — the same shape, deduplicated, until the agent face was no longer a thing I maintained but a projection that fell out of a surface I already had.

The proof it works is mundane, which is the point: odu now runs kolu's own CI, and the agent starts the run, blocks on the first red node, reruns it, and reports green — over the very MCP server this post is about. The thing the agent built became the way the agent works.

It only took a three-word insult to get there.
